# Completed Session Blank Set Persistence

Status: brainstorm / future enhancement
Date: 2026-05-05

## Context

During hosted sync validation, one completed session synced successfully but included a second `exercise_sets` row with blank `weight_value` and blank `reps_value`.

This means sync transported the local state correctly, but the local completed-session save path allowed a placeholder set to persist. Once persisted locally, the sync layer treats it as real domain data and uploads it.

## Problem

Completed sessions should not save or sync blank placeholder sets.

Today the session recorder has UI cleanup behavior for incomplete sets, but the lower-level persistence/sync path still writes every set it receives. That leaves room for blank sets to survive if the UI cleanup path is bypassed, missed, or regressed.

## Proposed Rule

A set is persistable/syncable only if:

- `weight_value` or `reps_value` is non-empty after trimming; and
- any non-empty fields pass validation.

For completed-session submit specifically, blank placeholder sets should be removed before persistence and before sync event generation.

## Product Intent

Blank sets are useful while editing because they give the user a ready-to-type row.

Blank sets are not useful as completed workout history. They should not affect analytics, restored history, or remote sync state.

## Likely Implementation Areas

- `apps/mobile/app/session-recorder.tsx`
  - Review `removeIncompleteSets` and submit cleanup flow.
  - Confirm blank sets are removed for both new completion and completed-session edit.

- `apps/mobile/src/data/session-drafts.ts`
  - Harden `replaceSessionExerciseGraph` or its caller so blank sets are filtered before insert and before `exercise_sets` sync events are emitted.
  - This is the stronger invariant because it protects persistence even if a future UI path forgets cleanup.

## Suggested Shape

Add a shared helper near the session draft persistence boundary, for example:

```ts
const isPersistableCompletedSet = (set: SessionDraftSetInput): boolean =>
  set.weightValue.trim().length > 0 || set.repsValue.trim().length > 0;
```

Then use the filtered set list for:

- local `exercise_sets` inserts;
- `exercise_sets` upsert sync events;
- set reorder indexes.

Keep delete event behavior for existing sets that are removed by the filter, so a previously synced blank set can be cleaned up by a later save.

## Test Ideas

- Saving a completed session with one filled set and one blank placeholder persists only the filled set.
- Sync event emission for that save includes only the filled `exercise_sets` upsert.
- Editing a completed session that already has a blank set removes it and emits an `exercise_sets` delete event.
- A partially filled set with invalid values still triggers validation instead of silently disappearing.

## Open Questions

- Should a partially filled set be allowed in drafts but blocked only when completing?
- Should completed-session edit use the same cleanup prompt as first-time completion, or silently remove blank placeholders?
- Should analytics explicitly ignore legacy blank rows while older synced data still exists?
