# Technical Architecture Decisions (High-Level)

## Purpose

Capture top-level architecture decisions for MVP, with clear `Adopted` vs `Planned` status.

## System map

`Expo React Native UI`
-> `Session domain + autosave orchestration`
-> `Local repository layer (Drizzle + SQLite)`
-> `Auth-gated sync engine + outbox + first-enable bootstrap/merge (M13 baseline)`
-> `Supabase backend (M5 selected; local-first dev/test via Supabase local stack)`

## Decision register

| Autosave policy is explicit and testable; precise timing constants are defined in milestone/task specs | `Adopted` | Preserves high-level architecture stability while allowing milestone-level tuning details. | `docs/specs/milestones/M3-session-domain-local-autosave.md`, `docs/tasks/complete/T-20260220-03-m3-continuous-autosave-and-draft-lifecycle.md` |
| Exercise and mapping metadata semantics are retroactive for history and analytics interpretation | `Adopted` | Aligns with product expectation that edits apply globally and keeps near-term schema/runtime complexity lower before analytics materialization exists. | `docs/specs/00-product.md` |
| Logged session exercises persist canonical `exercise_definition_id` alongside display name/origin metadata | `Adopted` | Prevents name-only drift and gives later tag/analytics features a stable exercise identity for history, grouping, and per-exercise metadata lookup. | `apps/mobile/src/data/schema/session-exercises.ts`, `apps/mobile/src/data/session-drafts.ts`, `docs/tasks/complete/T-20260304-01-m12-session-exercise-definition-link-and-tag-schema.md` |
| Mobile auth bootstrap uses `Supabase Auth` client-safe credentials plus persisted session restore, and missing auth config degrades to logged-out/non-blocking behavior | `Adopted` | Establishes the M11 auth foundation without coupling launch or local tracker flows to backend availability or env configuration. | `apps/mobile/src/auth/`, `docs/specs/milestones/M11-auth-ui-and-profile-management.md` |
| Auth session state is exposed through a shared mobile auth service/provider rather than route-local listeners | `Adopted` | Prevents duplicated subscription logic across screens and gives later auth/profile UI work one canonical session state surface. | `apps/mobile/src/auth/service.ts`, `apps/mobile/src/auth/provider.tsx` |
| M11 profile management uses `app_public.user_profiles` keyed `1:1` to `auth.users(id)` with lazy mobile provisioning on first profile load/save | `Adopted` | Keeps `username` in app-owned storage while leaving email/password in `Supabase Auth`, renders signed-out/signed-in account states in-place from the shared auth snapshot, and confines auth/profile failures to the profile route so local-only tracker flows remain usable. | `supabase/migrations/20260304153000_m11_user_profiles.sql`, `apps/mobile/src/auth/profile.ts`, `apps/mobile/app/profile.tsx` |
| M13 primary sync wire contract is `sync.events.ingest` batched envelopes with required request fields (`device_id`, `batch_id`, `sent_at_ms`) and required event fields (`event_id`, `sequence_in_device`, `occurred_at_ms`, `entity_type`, `entity_id`, `event_type`, `payload`) | `Planned` | Locks one canonical client/backend contract that is explicit enough for deterministic validation, telemetry correlation, and multi-entity projection semantics. | `supabase/session-sync-api-contract.md`, `docs/specs/milestones/M13-simple-backend-sync.md` |
| M13 sync-domain projection tables use composite primary key `(id, owner_user_id)`; every entity is user-scoped from day one, with no shared/global catalog on the backend | `Adopted` | Each user owns their own `id` keyspace, so two users may legitimately hold rows with the same `id` (e.g. seeded `exercise_definitions.id`) without conflict; the wire contract is unchanged but the backend has no cross-owner rejection path by construction. | `supabase/migrations/`, `docs/specs/05-data-model.md`, `docs/tasks/fix-sync/plan.md` |
| M13 ingest semantics are per-event idempotent and batch-ordered (strict request order) with a minimal client response contract (`SUCCESS | FAILURE`, failure `error_index`, `should_retry`, free-text `message`) | `Planned` | Keeps client handling simple while preserving deterministic stop-on-first-failure behavior and retry control. | `supabase/session-sync-api-contract.md`, `docs/specs/milestones/M13-simple-backend-sync.md` |
| Initial mobile sync behavior is auth-gated and non-blocking: one-time bootstrap/merge on enable, then push-dominant periodic sync | `Adopted` | Preserves local-first UX while enabling reliable sync convergence without blocking logged-out/offline usage, including deterministic local-vs-remote merge with convergence flush on enable. | `apps/mobile/src/sync/runtime.ts`, `apps/mobile/src/sync/bootstrap.ts`, `docs/specs/milestones/M13-simple-backend-sync.md` |
| Bootstrap is auth-event-filtered and serialized against the periodic scheduler | `Adopted` | Bootstrap re-runs only on user-change auth events (sign-in/sign-out/user switch), enforces a 30s cooldown between attempts, and the periodic scheduler pauses while bootstrap is in progress to prevent in-flight collisions that previously caused bootstrap loops on `TOKEN_REFRESHED` ticks. | `apps/mobile/src/sync/runtime.ts`, `apps/mobile/src/sync/scheduler.ts`, `apps/mobile/src/sync/engine.ts`, `docs/tasks/fix-sync/plan.md` |
| Bootstrap merge re-runs the local exercise-catalog seeder | `Adopted` | The merge step wipes and reinserts user-owned local rows; for a fresh user with empty remote, calling the seeder at the end of merge ensures the catalog survives, and the per-user composite PK on the backend accepts the per-user seed copies on the next convergence flush. | `apps/mobile/src/sync/bootstrap.ts`, `apps/mobile/src/data/exercise-catalog-seeds.ts`, `docs/tasks/fix-sync/plan.md` |
| Private gym coordinate metadata is synced as part of user-owned `gyms` | `Adopted` | GPS gym support needs backup/restore parity without introducing public places, check-ins, or shared location records; nullable coordinates remain owner-scoped and follow the existing `gyms.upsert`/bootstrap/convergence path. | `apps/mobile/src/data/schema/gyms.ts`, `apps/mobile/src/sync/bootstrap.ts`, `supabase/migrations/20260523120000_m15_gym_coordinates.sql`, `docs/specs/milestones/M15-gps-gym-location-support.md` |
| Mobile GPS access is foreground-only behind an injectable location service and pure matcher | `Adopted` | Keeps native permission/API behavior isolated from recorder and gym-management UI, prevents background location usage in M15, and makes Haversine/radius/accuracy/tie behavior deterministic in unit tests before UI wiring. | `apps/mobile/src/location/foreground-location-service.ts`, `apps/mobile/src/location/gym-location-matcher.ts`, `apps/mobile/app.config.ts`, `docs/specs/milestones/M15-gps-gym-location-support.md` |
| Minimal app diagnostics use Supabase `public.app_logs` plus a mobile logging helper | `Adopted` | Captures production auth/sync failure breadcrumbs without adding an external observability vendor or user-facing dashboard; client access is insert-only and logging is non-blocking. | `supabase/migrations/20260507120000_create_app_logs.sql`, `apps/mobile/src/logging/logEvent.ts`, `docs/specs/milestones/M14-observability-and-diagnostics.md` |
| Pull-side local SQLite FK failures are classified as sync-cycle structural errors | `Adopted` | A pulled child row that violates local FK enforcement is reported as `LOCAL_FK_VIOLATION`, emits a sanitized `sync.pull_local_fk_violation` diagnostic, and preserves the existing page rollback / cursor-not-advanced guarantee; automatic cursor reset or full-repull recovery is deferred. | `apps/mobile/src/sync/cycle.ts`, `apps/mobile/app/__tests__/sync-cycle-convergence.test.ts`, `docs/specs/tech/client-sync-engine.md` |
| Push-side FK closure preflight blocks orphan dirty rows before `sync_push` | `Adopted` | Before sending a push batch the cycle validates FK closure client-side from a declared syncable FK graph (`SYNCABLE_FK_GRAPH`): each dirty child's non-null parent reference must be in the same batch or physically present locally. A local orphan no longer wedges the push — it is quarantined (see next row); the `LOCAL_FK_VIOLATION` code remains the classification stamped on each quarantine record and stays distinct from server `FK_VIOLATION`. | `apps/mobile/src/sync/fk-graph.ts`, `apps/mobile/src/sync/cycle.ts`, `apps/mobile/app/__tests__/sync-cycle-push-preflight.test.ts`, `docs/specs/tech/client-sync-engine.md` |
| Sync quarantine isolates FK-blocked dirty rows so one orphan cannot wedge the backlog | `Adopted` | An orphan dirty row detected by the push preflight is persisted to the local-only `sync_quarantine` table (`(entity_type, entity_id)` key with error code, diagnostic FK context, first/last-seen timestamps, occurrence count), excluded from future push selection, and the valid rows beside it keep pushing — so the cycle converges instead of throwing. Repeated detection is an idempotent upsert (count bump, no duplicates); the state survives app restart; quarantine cascades to children of a quarantined orphan. `getSyncStatus.blockedRowCount` surfaces that blocked rows exist; two sanitized diagnostics (`sync.row_quarantined`, `sync.push_continued_after_quarantine`) are emitted best-effort. A full user-facing repair workflow and automatic destructive graph repair are out of scope. | `apps/mobile/src/sync/quarantine.ts`, `apps/mobile/src/data/schema/sync-quarantine.ts`, `apps/mobile/src/sync/cycle.ts`, `apps/mobile/app/__tests__/sync-cycle-quarantine.test.ts`, `docs/specs/tech/client-sync-engine.md` |
| Scheduler sync success reflects real convergence, not a clean return | `Adopted` | `runSyncCycle` returns a classified `SyncCycleResult` (`converged` / `auth_required` / `retryable_error`; structural errors still throw), and the scheduler advances `lastSuccessAtMs` only on `converged` — `auth_required`, `retryable_error`, and thrown structural errors record no false success, with the retryable code kept visible until a later converged cycle clears it. Scheduler cadence (the four-state machine + long backstop) is unchanged and stays separate from sync-success status; each finished cycle emits one sanitized `sync.cycle_result` diagnostic. | `apps/mobile/src/sync/cycle.ts`, `apps/mobile/src/sync/scheduler.ts`, `apps/mobile/app/__tests__/scheduler-status-accessor.test.ts`, `docs/specs/tech/client-sync-engine.md` |
| MVP backend platform (primary) is `Supabase (Postgres + Auth + RLS)` | `Adopted` | Best fit for M5 scope: secure auth/authz + sync API baseline with strong local-test fidelity and better SQL/data portability than the evaluated fallback (`Cloudflare Workers + D1`). | `docs/tasks/complete/T-20260220-07-m5-backend-stack-decision-and-architecture-update.md`, `docs/specs/milestones/M5-backend-foundation-authz-and-sync-api.md` |
| `Cloudflare Workers + D1` is documented as an M5 contingency fallback, not the primary MVP backend | `Planned (contingency)` | Preserves a zero-spend/lean edge path if needed, but increases auth/authz implementation scope and weakens SQL portability for this milestone. | `docs/tasks/complete/T-20260220-07-m5-backend-stack-decision-and-architecture-update.md` |
| M5 backend tasks must support a local backend runtime with high-fidelity local verification before deployment | `Adopted` | Local runtime/testing was elevated to a hard requirement to minimize post-deployment debugging and keep delivery iteration fast. | `docs/tasks/complete/T-20260220-07-m5-backend-stack-decision-and-architecture-update.md`, `docs/specs/milestones/M5-backend-foundation-authz-and-sync-api.md` |
| Local parallel-agent development uses git worktrees with slot-derived runtime isolation and a harness-independent no-nested-worktree rule | `Adopted` | Multiple agents on one machine need isolated Supabase containers/ports/data, Metro ports, simulator targets, dependency installs, and generated config; refusing nested worktrees prevents parent `node_modules`, TypeScript, and watcher contamination. | `docs/specs/12-worktree-config-and-isolation.md`, `scripts/worktree-setup.sh`, `scripts/worktree-create.sh`, `scripts/worktree-doctor.sh` |
| Social/group features are layered after tracker and sync foundations | `Planned` | Prevents premature complexity while keeping extension path explicit in delivery order. | N/A |
