# Technical Architecture Decisions (High-Level)

## Purpose

Capture top-level architecture decisions for MVP, with clear `Adopted` vs `Planned` status.

## System map

`Expo React Native UI`
-> `Session domain + autosave orchestration`
-> `Local repository layer (Drizzle + SQLite)`
-> `Auth-gated sync v2 cycle (pull→push→pull; per-row dirty bit + LWW) + first-sign-in bootstrap`
-> `Supabase backend (M5 selected; local-first dev/test via Supabase local stack)`

## Decision register

| Decision | Status | Why | Source |
| --- | --- | --- | --- |
| `Expo + React Native + TypeScript` for mobile frontend | `Adopted` | Fastest path to a phone-first app with one codebase and strong AI-assisted delivery. | `apps/mobile/package.json` |
| `Expo Router` for app navigation | `Adopted` | File-based routing keeps structure simple and predictable for AI and humans. | `apps/mobile/package.json`, `apps/mobile/app/` |
| `SQLite (on-device) + Drizzle ORM` for local persistence | `Adopted` | Reliable offline-first data with typed schema and migrations. | `apps/mobile/package.json`, `apps/mobile/src/data/schema/`, `apps/mobile/drizzle/` |
| Continuous local autosave is implemented as a domain/data-layer contract before UI wiring | `Adopted` | Enables isolated verification of persistence behavior before screen integration. | `apps/mobile/src/session-recorder/draft-autosave.ts` |
| React Native lifecycle integration uses adapter contracts | `Adopted` | Maps RN lifecycle events to flush triggers without coupling persistence logic to UI components. | `apps/mobile/src/session-recorder/lifecycle-helpers.ts` |
| Session completion persists `completedAt` and materialized `durationSec` | `Adopted` | Keeps completion reads/sorts deterministic without recomputing duration for completed sessions. | `apps/mobile/src/data/schema/sessions.ts`, `apps/mobile/src/data/session-drafts.ts` |
| Autosave policy is explicit and testable; precise timing constants are defined in milestone/task specs | `Adopted` | Preserves high-level architecture stability while allowing milestone-level tuning details. | — |
| Exercise and mapping metadata semantics are retroactive for history and analytics interpretation | `Adopted` | Aligns with product expectation that edits apply globally and keeps near-term schema/runtime complexity lower before analytics materialization exists. | `docs/specs/00-product.md` |
| Logged session exercises persist canonical `exercise_definition_id` alongside display name/origin metadata | `Adopted` | Prevents name-only drift and gives later tag/analytics features a stable exercise identity for history, grouping, and per-exercise metadata lookup. | `apps/mobile/src/data/schema/session-exercises.ts`, `apps/mobile/src/data/session-drafts.ts` |
| Mobile auth bootstrap uses `Supabase Auth` client-safe credentials plus persisted session restore, and missing auth config degrades to logged-out/non-blocking behavior | `Adopted` | Establishes the M11 auth foundation without coupling launch or local tracker flows to backend availability or env configuration. | `apps/mobile/src/auth/` |
| Auth session state is exposed through a shared mobile auth service/provider rather than route-local listeners | `Adopted` | Prevents duplicated subscription logic across screens and gives later auth/profile UI work one canonical session state surface. | `apps/mobile/src/auth/service.ts`, `apps/mobile/src/auth/provider.tsx` |
| M11 profile management uses `app_public.user_profiles` keyed `1:1` to `auth.users(id)` with lazy mobile provisioning on first profile load/save | `Adopted` | Keeps `username` in app-owned storage while leaving email/password in `Supabase Auth`, renders signed-out/signed-in account states in-place from the shared auth snapshot, and confines auth/profile failures to the profile route so local-only tracker flows remain usable. | `supabase/migrations/20260304153000_m11_user_profiles.sql`, `apps/mobile/src/auth/profile.ts`, `apps/mobile/app/profile.tsx` |
| Sync v2 wire contract is the `sync_push` / `sync_pull` RPC pair with a typed per-row envelope (`{type, id, client_updated_at_ms, fields}`); push sends FK-ordered batches and pull drains per-layer cursors | `Adopted` | One small typed contract replaces the retired v1 `sync.events.ingest` event-log + projection: the server stores typed columns so SQL/analytics work day one and there is no projection layer. | `docs/specs/tech/sync-v2-server-contract.md`, `apps/mobile/src/sync/cycle.ts` |
| Sync v2 server tables use composite primary key `(owner_user_id, id)` with RLS `owner_user_id = auth.uid()`; every entity is user-scoped from day one, with no shared/global catalog on the backend | `Adopted` | Each user owns their own `id` keyspace, so two users may legitimately hold rows with the same `id` (e.g. seeded `exercise_definitions.id`); per-user ownership plus last-write-wins means no cross-owner rejection path exists by construction. | `supabase/migrations/`, `docs/specs/05-data-model.md`, `docs/specs/tech/sync-v2-server-contract.md` |
| Sync v2 push applies per-row last-write-wins on `client_updated_at_ms` (future-clock clamp; `deleted_at` undelete) under a 1..200 batch cap with FK-closure validation; pull is an idempotent per-layer cursor drain | `Adopted` | Keeps the client simple (a per-row dirty bit, no event log/outbox) while convergence stays deterministic and retry-safe. | `docs/specs/tech/sync-v2-server-contract.md`, `apps/mobile/src/sync/cycle.ts` |
| Mobile sync is auth-gated and non-blocking: a periodic foreground cycle converges local SQLite with the server by running PULL→PUSH→PULL until both ends are quiet for one round | `Adopted` | Preserves local-first UX (logged-out/offline usage never blocks) while a retry-safe cycle that reconstructs its state from SQLite each run drives convergence. | `apps/mobile/src/sync/cycle.ts`, `apps/mobile/src/sync/scheduler.ts`, `apps/mobile/src/sync/SyncGate.tsx` |
| The foreground sync scheduler is a four-state machine (offline / long-backstop / short-debounce / running) owning a single timer, driven by network reachability and the foreground edge | `Adopted` | A literal state machine makes every WHEN-to-sync transition a table lookup and guarantees at most one in-flight cycle, replacing the v1 cooldown/loop heuristics. | `apps/mobile/src/sync/scheduler.ts` |
| First-sign-in bootstrap seeds the starter exercise catalog only when the first full pull returns zero server rows, then lets the normal cycle push the seeded (dirty) rows | `Adopted` | Runs once per device per account (gated on `bootstrap_completed_at`); seeding only an empty remote prevents duplicate catalogs while guaranteeing a fresh user keeps the starter exercises. | `apps/mobile/src/sync/bootstrapper.ts`, `apps/mobile/src/data/exercise-catalog-seeds.ts` |
| Private gym coordinate metadata is synced as part of user-owned `gyms` | `Adopted` | GPS gym support needs backup/restore parity without introducing public places, check-ins, or shared location records; nullable coordinates remain owner-scoped and sync as ordinary typed columns through the v2 cycle. | `apps/mobile/src/data/schema/gyms.ts`, `apps/mobile/src/sync/cycle.ts`, `supabase/migrations/20260523120000_m15_gym_coordinates.sql` |
| Mobile GPS access is foreground-only behind an injectable location service and pure matcher | `Adopted` | Keeps native permission/API behavior isolated from recorder and gym-management UI, prevents background location usage in M15, and makes Haversine/radius/accuracy/tie behavior deterministic in unit tests before UI wiring. | `apps/mobile/src/location/foreground-location-service.ts`, `apps/mobile/src/location/gym-location-matcher.ts`, `apps/mobile/app.config.ts` |
| Minimal app diagnostics use Supabase `public.app_logs` plus a mobile logging helper | `Adopted` | Captures production auth/sync failure breadcrumbs without adding an external observability vendor or user-facing dashboard; client access is insert-only and logging is non-blocking. | `supabase/migrations/20260507120000_create_app_logs.sql`, `apps/mobile/src/logging/logEvent.ts`, `docs/specs/milestones/M14-observability-and-diagnostics.md` |
| MVP backend platform (primary) is `Supabase (Postgres + Auth + RLS)` | `Adopted` | Best fit for M5 scope: secure auth/authz + sync API baseline with strong local-test fidelity and better SQL/data portability than the evaluated fallback (`Cloudflare Workers + D1`). | — |
| `Cloudflare Workers + D1` is documented as an M5 contingency fallback, not the primary MVP backend | `Planned (contingency)` | Preserves a zero-spend/lean edge path if needed, but increases auth/authz implementation scope and weakens SQL portability for this milestone. | — |
| M5 backend tasks must support a local backend runtime with high-fidelity local verification before deployment | `Adopted` | Local runtime/testing was elevated to a hard requirement to minimize post-deployment debugging and keep delivery iteration fast. | — |
| Local parallel-agent development uses git worktrees with slot-derived runtime isolation and a harness-independent no-nested-worktree rule | `Adopted` | Multiple agents on one machine need isolated Supabase containers/ports/data, Metro ports, simulator targets, dependency installs, and generated config; refusing nested worktrees prevents parent `node_modules`, TypeScript, and watcher contamination. | `docs/specs/12-worktree-config-and-isolation.md`, `scripts/worktree-setup.sh`, `scripts/worktree-create.sh`, `scripts/worktree-doctor.sh` |
| Social/group features are layered after tracker and sync foundations | `Planned` | Prevents premature complexity while keeping extension path explicit in delivery order. | N/A |
