## 1. Endpoints

Two RPCs. Both exposed as PostgREST RPCs under the `app_public` schema.
Both require an authenticated user; `owner_user_id` is derived from the
JWT and is never sent on the wire (RLS enforces it, per t1 §6). Both are
POSTs.

| RPC | Purpose |
|---|---|
| `sync_push` (`POST /rest/v1/rpc/sync_push`) | Client uploads a batch of dirty rows. Server upserts with LWW inside a single transaction. |
| `sync_pull` (`POST /rest/v1/rpc/sync_pull`) | Client downloads server rows newer than a cursor. Returns up to `limit` rows ordered by `(server_received_at, owner_user_id, type, id)`. |

No third "snapshot" RPC. Snapshot is just `sync_pull` with `cursor = null`.

There is no event log on the wire. There are no sequence numbers, no
device identifiers, no batch ids, no `sent_at` clock. **The data is the
event.**

## 2. Wire types (shared)

### 2.1 `Entity` envelope

Every row crossing the wire — push request, pull response — uses the same
shape. The envelope carries the LWW key; `fields` carries every typed
column on the row (including `deleted_at`).

```json
{
  "type": "exercise_sets",
  "id": "01HXYZ...ULID",
  "client_updated_at_ms": 1733000000123,
  "fields": {
    "session_exercise_id": "01HX...",
    "order_index": 0,
    "weight_value": "100",
    "reps_value": "8",
    "set_type": "rir_2",
    "created_at": 1732999999000,
    "updated_at": 1733000000123,
    "deleted_at": null
  }
}
```

Field rules:

| Field | Type | Notes |
|---|---|---|
| `type` | string enum | One of `gyms`, `sessions`, `session_exercises`, `exercise_sets`, `exercise_definitions`, `exercise_muscle_mappings`, `exercise_tag_definitions`, `session_exercise_tags`. **Plural, snake_case**, matching the t1 table names. |
| `id` | string | Client-assigned. ULID for user-created rows; slug (`seed:bench-press`) for seeds. Stable forever. |
| `client_updated_at_ms` | integer | Epoch ms from the originating client. **The LWW key.** Must be `>= 0`. Produced by the monotonicity guard (§9). |
| `fields` | object | Per-entity typed columns. Keys are wire-canonical snake_case (matching the t1 §2 mapping tables). Values use the JSON types implied by the t1 types: `text` → string, `bigint` → integer, `numeric` → string, `boolean` → boolean, `timestamp_ms`-style → integer epoch ms, nullable → JSON null. The client MUST emit every typed column listed in t1 §2 for that entity, even when the column is nullable (emit JSON null). **Tombstones travel as a normal row with `deleted_at` set to a non-null epoch ms.** |

**No `owner_user_id` on the wire envelope.** It is derived from `auth.uid()`
in the RPC body. Pull responses also omit it (the caller already knows it
is theirs).

**No `extras` blob.** t1's universal `extras jsonb` column is no longer used
by t2; the design now requires server-first deploy for every schema addition
(see §13's "t1 follow-up emitted by t2"). Any new column lands as a typed
`fields` key on the next push after the server migration ships.

**No top-level `deleted` flag.** A tombstone is just a normal row with
`fields.deleted_at != null`; both server upsert and client apply treat
"row is a tombstone" as `deleted_at IS NOT NULL`. (See §13's t1 follow-up
to drop the universal `deleted boolean` column.)

**Server-side schema rollout discipline.** Because t2 carries only typed
`fields`, every new entity column lands on the server first, then on the
client. The window between the two ships is fine: clients written against
the old schema continue to omit the column from `fields`; the server fills
in its default. Clients written against the new schema can write the new
column once the server has it.

### 2.2 Error shape

Both RPCs use the same outer error envelope when they reject the entire
request. Per t1 §1 ground rules ("no server-side validation unless the
server writes"), the failure surface is intentionally narrow.

```json
{
  "error": {
    "code": "FK_VIOLATION",
    "message": "session_exercises 01HX5KAB...: parent sessions(owner_user_id, id)=(<uid>, 01HX5K9V...) not found"
  }
}
```

| `code` | Cause | Client behaviour |
|---|---|---|
| `AUTH_REQUIRED` | No JWT, expired JWT, RLS denies the row. | Trigger token refresh; if it fails, keep dirty bits, surface "Sign in again." |
| `FK_VIOLATION` | Deferrable-FK check failed at COMMIT (t1 §5). Structural bug — see §4.4. | **Non-retriable.** Log to observability; leave dirty bits unchanged; surface non-recoverable error in the sync status panel. No auto-retry. A fix lands by app update. |
| `INTERNAL` | Anything else (transport, 5xx, unmarshalled error). | Retry with backoff per §10. |

**Server validates hierarchy (FKs) only.** No enum, range, length, format,
or business-rule validation. Anything malformed is the client's bug — the
t1 §7 drift gate catches schema-level malformation at PR time.

## 3. `sync_push`

### 3.1 Request

```json
{
  "entities": [
    {
      "type": "exercise_definitions",
      "id": "seed:bench-press",
      "client_updated_at_ms": 1733000000100,
      "fields": {
        "name": "Bench Press",
        "created_at": 1732999900000,
        "updated_at": 1733000000100,
        "deleted_at": null
      }
    },
    {
      "type": "sessions",
      "id": "01HX5K9V...",
      "client_updated_at_ms": 1733000000200,
      "fields": {
        "gym_id": "gym_a",
        "status": "active",
        "started_at": 1733000000000,
        "completed_at": null,
        "duration_sec": null,
        "created_at": 1733000000000,
        "updated_at": 1733000000200,
        "deleted_at": null
      }
    },
    {
      "type": "session_exercises",
      "id": "01HX5KAB...",
      "client_updated_at_ms": 1733000000201,
      "fields": {
        "session_id": "01HX5K9V...",
        "exercise_definition_id": "seed:bench-press",
        "order_index": 0,
        "name": "Bench Press",
        "machine_name": null,
        "created_at": 1733000000201,
        "updated_at": 1733000000201,
        "deleted_at": null
      }
    }
  ]
}
```

| Field | Validation (structural only) |
|---|---|
| `entities` | Array, length `1..200`. v2 batch cap is **200** rows. Larger batches are split client-side (§4). |

### 3.2 Ordering inside the batch

- Clients may send entities in any order. The full hierarchy must be present — either in the batch itself or already on the server. (This is what the rest of the doc calls "closure.")
- Server implements this by deferring constraints: a single transaction with `SET CONSTRAINTS ALL DEFERRED`; all FKs are checked at COMMIT. If any FK is unsatisfied, the entire batch rolls back with `FK_VIOLATION` (§2.2, §4.3).

### 3.3 What goes in `fields`

For each typed server column on this entity (per t1 §2 mapping table), put
its current local value under `fields.<wire_key>`. The wire_key is the
snake_case Postgres column name from the mapping table. Nullable columns
get a JSON null when the local value is nullish.

Every other client column is either (a) a local-only sync column (§12) —
must not appear on the wire — or (b) drift, caught at PR time by the t1
§7 checker.

### 3.4 Building the batch

Per t1 §11 #6, the client MUST NOT push a child row whose parent is
neither in the same batch nor already on the server. The client
satisfies this by walking the eight entity tables in a fixed
**topological order** when collecting dirty rows.

#### 3.4.1 Topological order of the eight tables

The eight tables form a DAG under their FK relations. The fixed order
the client iterates is:

| Layer | Tables (any intra-layer order) | FKs to earlier layers |
|---|---|---|
| 0 | `gyms`, `exercise_definitions`, `exercise_tag_definitions` | none |
| 1 | `sessions`, `exercise_muscle_mappings` | `sessions → gyms`; `exercise_muscle_mappings → exercise_definitions` |
| 2 | `session_exercises` | `→ sessions`, `→ exercise_definitions` |
| 3 | `exercise_sets`, `session_exercise_tags` | `exercise_sets → session_exercises`; `session_exercise_tags → session_exercises`, `→ exercise_tag_definitions` |

No table has a within-layer FK and no table is self-referential, so any
order inside a layer is safe.

This list is hardcoded client-side. **The t1 §7 drift checker is
extended to assert that this hardcoded order is a valid topological
sort of t1's actual FK graph** — adding a new entity table or FK in a
future migration fails CI until the order is updated. Recorded as a
t1 follow-up in §13.3.

#### 3.4.2 Algorithm `selectPushBatch(batchCap = 200) → Entity[]`

```
batch = []
for table in TOPO_ORDER:
  if len(batch) >= batchCap: break
  rows = SELECT * FROM <table>
         WHERE local_dirty = 1
         ORDER BY local_updated_at_ms ASC
         LIMIT (batchCap - len(batch))
  for row in rows:
    batch.append(serialize(row))
return batch
```

The cycle (t4 §11) calls this repeatedly. Each call returns at most
`batchCap` rows from the topologically-earliest tables that still have
dirty rows; the cycle pushes the batch, the server commits it, and the
next call picks up where this one left off. When the call returns an
empty array, the push leg of the cycle is done.

Intra-table ordering by `local_updated_at_ms ASC` is a quality-of-life
choice — it makes the dirty queue drain FIFO so the oldest-dirty
row never starves — but the FK guarantee does not depend on it.

#### 3.4.3 Why this gives the no-orphan-child guarantee

Consider any dirty row `R` from table `T(R)` being pushed in some batch
`B`. Its FK references point to rows in tables that, by construction,
appear in earlier layers of `TOPO_ORDER`. For each such referenced row
`P`:

1. **`P` is dirty and in `B`** — its layer is strictly earlier than
   `T(R)`, so the algorithm visited `P`'s table first and selected it
   (modulo batch-cap, see case 3). The deferred FK in the push
   transaction (`SET CONSTRAINTS ALL DEFERRED`, t1 §5.1) resolves at
   COMMIT.
2. **`P` is clean** — `local_dirty = 0` means the only events that
   could have produced that state are a successful push ack or an
   inbound pull apply (§8.2). Either way, the server has `P`. The
   deferred FK resolves at COMMIT against the server-resident row.
3. **`P` is dirty but didn't fit in `B`** — the algorithm processes
   layer 0 before layer 1 before layer 2 before layer 3. If `P`'s
   table is in an earlier layer than `T(R)`'s, the algorithm consumes
   the entire dirty queue of `P`'s table before reaching `T(R)` at
   all. So `P` either fits in `B` (case 1) or `T(R)`'s rows are never
   reached in this call (algorithm broke out at the batch-cap on the
   earlier table). In that branch, `R` is not in `B` either — it will
   appear in a later batch in the same cycle, by which time `P` has
   already been pushed and committed (case 2 from `R`'s next-call
   perspective).

There is no scenario in which `R` is in `B` with a dirty `P` from an
earlier layer that is missing.

#### 3.4.4 What this design deliberately rejects

- **No FIFO-across-tables.** A naïve `SELECT … WHERE local_dirty = 1
  ORDER BY local_updated_at_ms ASC` across all eight tables breaks the
  guarantee: a parent edited *after* a child's creation lands later
  in FIFO than its child, so the batch cap can cut a child without
  its still-unpushed parent. Topological iteration sidesteps this.
- **No transitive closure walk per dirty row.** The prior design
  (with a `server_known` bit) walked the FK graph upward per row to
  collect ancestors. Topological iteration achieves the same FK
  closure with less code and no auxiliary column. Both are correct;
  topological iteration is the chosen one.

### 3.5 Success response

```json
{
  "ok": true,
  "server_received_at": "2026-05-23T10:14:32.871Z"
}
```

A single ack body. No per-row outcomes. The whole batch committed; every
row in the request has been processed (either UPSERT-applied because LWW
accepted the incoming, or no-op because LWW rejected it — either way the
server's state is now at-or-newer than what the client sent for every
row, which is all the dirty bit needs to know).

The client clears `local_dirty` for every row in the in-memory batch
snapshot, subject to the §8.3 concurrent-edit check on each row.

`server_received_at` is the server's wall-clock `now()` at the moment of
the upsert (single `now()` per transaction, stable across all rows in
the batch). It is the cursor axis for pull (§5.3); the client uses it
only for observability on push.

### 3.6 Idempotency — t4 §11 #4

Push is idempotent under retry because LWW resolves duplicates server-side:
a re-push of the same row with the same `client_updated_at_ms` finds
`incoming == stored`, the LWW predicate `incoming > stored` is false, no
UPDATE happens, the server returns `{ ok: true }` and the client clears
the dirty bit. If the client edited the row between push attempts, the
retry carries a newer `client_updated_at_ms` and is a normal write, not
a retry. Honors t4 §11 #4.

### 3.7 Dedup within a single batch

If the same `(type, id)` appears twice in one request, the server's
`ON CONFLICT` clause will conflict with itself inside the same statement
(Postgres rejects this with `cannot affect row a second time`). The
local table's primary key `(type, id)` ensures the dirty stream
(§3.4) cannot produce two rows with the same key, so the batch is
naturally deduped.

If a dedup bug slips through (somehow two dirty rows with the same id —
a bug in repo code), the server's response will be `INTERNAL` from the
ON CONFLICT failure, and the client retries after backoff.

### 3.8 Tombstones in push

A delete push is a normal row push with `fields.deleted_at` set to a
non-null epoch ms and `client_updated_at_ms = nowMonotonic()`. The full
last-known row travels in `fields` so the row's non-PK columns reflect
the last-known state at delete time, making undelete-by-LWW lossless
(t1 §1.1.4).

For join tables (`exercise_muscle_mappings`, `session_exercise_tags`),
t1 currently lacks a per-entity `deleted_at` column. The t1 follow-up
in §13 adds one (`deleted_at bigint nullable`) to both join tables so
the wire shape is uniform: every entity carries `deleted_at` in its
typed columns; a tombstone is `deleted_at IS NOT NULL`.

**Example tombstone push payload:**

```json
{
  "type": "exercise_sets",
  "id": "01HX5KCD...",
  "client_updated_at_ms": 1733000060000,
  "fields": {
    "session_exercise_id": "01HX5KAB...",
    "order_index": 0,
    "weight_value": "100",
    "reps_value": "8",
    "set_type": "rir_2",
    "created_at": 1732999999000,
    "updated_at": 1733000060000,
    "deleted_at": 1733000060000
  }
}
```

Same envelope shape as a live write; `deleted_at != null` is the only
tombstone marker.

## 4. Aggregate grouping, batching, and batch failure

### 4.1 Aggregate grouping rule

**A child row's parents must be batched with it if the parents are not
already on the server.** This is t1 §11 #6. The rule is satisfied
naturally by the dirty-bit discipline (§3.4): dirty parents flow with
dirty children through the FIFO dirty stream because the repo write
paths flip dirty on both; clean parents are already on the server.

The grouping is per-batch, not per-session. A single batch may include
fragments from multiple sessions, multiple aggregates, multiple unrelated
entities — what it MAY NOT include is a child whose parent is missing
from both this batch and the server.

### 4.2 Batch failure: whole-batch reject by transaction boundary

A single push is one Postgres transaction. Either every row in the batch
commits, or none of them do. There is no partial-accept code path.

The success body (§3.5) is `{ ok: true }`; there is no per-row failure
line. The only structural failure is `FK_VIOLATION` at COMMIT, treated
as non-retriable per §4.3.

### 4.3 `FK_VIOLATION` is a structural bug — log and give up

`FK_VIOLATION` indicates the client batched a child without its parent
AND the parent was not already on the server. Given §3.4's dirty-bit
discipline, this should be impossible. When it happens anyway it is a
structural bug in the repo write paths (a parent dirtying path that
missed a child, or vice versa).

On `FK_VIOLATION`:

1. The client logs the error (`code`, `message`, the offending row id)
   to observability.
2. Dirty bits stay unchanged (nothing was acked, nothing was
   re-classified).
3. The client surfaces a non-recoverable error in the sync status panel
   ("Sync error — please contact support").
4. The cycle does NOT auto-retry. The next scheduler tick MAY trigger
   another cycle, but if the underlying dirty state is the same the
   same `FK_VIOLATION` will recur; the structural bug must be fixed in
   code. A fix lands by app update.

There is no closure-walk re-run, no second-chance retry. This was a v1
behaviour that contributed to the M13 wedge; v2 does not repeat it.

### 4.4 Batch size

`batchCap = 200`. Justification:

- v1 used 100; the bottleneck there was the projection function.
- In v2 the work per row is one UPSERT and one LWW comparison.
- Postgres comfortably handles 200-row UPSERTs; round-trip transport time
  dominates over per-row CPU.
- t4 §6.1 sets the foreground cycle target at ≤ 5 seconds and the
  background cycle at ≤ 20 seconds; at 200 rows / batch and typical net
  latencies, three batches comfortably fit inside foreground.
- Larger batches risk hitting the Supabase RPC request-size cap (default
  1 MiB after base64). 200 typical rows ≈ a few hundred KB; safe.

If the dirty stream exceeds 200 rows, the cycle issues multiple push
batches in sequence (§7.3). Each batch is independently committed.

## 5. `sync_pull`

### 5.1 Request

```json
{
  "cursor": {
    "server_received_at": "2026-05-23T10:14:32.871Z",
    "owner_user_id": "00000000-0000-0000-0000-000000000aaa",
    "type": "sessions",
    "id": "01HX5K9V..."
  },
  "limit": 500
}
```

| Field | Type | Notes |
|---|---|---|
| `cursor` | object or `null` | Opaque to the client (round-trip the value the server emitted). `null` for snapshot / initial pull. |
| `cursor.server_received_at` | string (ISO 8601 with ms) | The `server_received_at` of the last row returned by the previous response. |
| `cursor.owner_user_id` | string (UUID) | Same-millisecond tiebreak axis. Echoed back to the server. |
| `cursor.type` | string enum | Same-millisecond tiebreak axis. |
| `cursor.id` | string | Same-millisecond tiebreak axis. |
| `limit` | integer | `1..1000`. Default 500. The cycle (t4) chooses based on background-vs-foreground budget. |

### 5.2 Response

```json
{
  "entities": [
    {
      "type": "sessions",
      "id": "01HX5K9V...",
      "client_updated_at_ms": 1733000000200,
      "fields": { "...": "..." }
    },
    {
      "type": "session_exercises",
      "id": "01HX5KAB...",
      "client_updated_at_ms": 1733000000201,
      "fields": { "...": "..." }
    }
  ],
  "next_cursor": {
    "server_received_at": "2026-05-23T10:14:33.000Z",
    "owner_user_id": "00000000-0000-0000-0000-000000000aaa",
    "type": "session_exercises",
    "id": "01HX5KAB..."
  },
  "has_more": true
}
```

| Field | Notes |
|---|---|
| `entities` | Up to `limit` Entity envelopes, ordered by `(server_received_at ASC, owner_user_id ASC, type ASC, id ASC)`. |
| `next_cursor` | The cursor to use on the next pull. Always present (echoes input on an empty page). |
| `has_more` | `true` iff the server has at least one more row strictly greater than `next_cursor`. Cycle uses this to decide whether to issue another pull (t4 §11 #1). |

On an empty page (`entities.length === 0`), `has_more === false` and
`next_cursor` echoes the input `cursor` unchanged (the client persists it
unchanged — the cycle exits the pull loop).

### 5.3 Pagination — monotonicity and tiebreak

The server orders rows by:

```sql
ORDER BY server_received_at ASC,
         owner_user_id     ASC,
         type              ASC,
         id                ASC
LIMIT :limit
```

with the WHERE clause:

```sql
WHERE owner_user_id = auth.uid()
  AND (
    :cursor IS NULL
    OR (server_received_at, owner_user_id, type, id)
        > (:cursor.server_received_at, :cursor.owner_user_id,
           :cursor.type, :cursor.id)
  )
```

**Why `server_received_at` is monotonic.** It is set by the server's `now()`
on every successful UPSERT (`INSERT ... DEFAULT now()` for new rows; a
trigger touches it on UPDATE, per t1 §1 ground rules — the touch is
"structural, not validation"). Supabase Postgres is single-primary, so
`now()` is monotonic across all writes from a single client's view —
there's no replica clock skew to reason about.

**Same-millisecond rows.** Two rows in the same Postgres transaction share
a `now()` value (transactions hold a stable `now()`). The
`(owner_user_id, type, id)` lexicographic tiebreak gives every row a
unique sort key. The row-value comparison `>` ensures the next pull
strictly advances past the last-emitted row — it never repeats and never
skips.

**Why the cursor carries `owner_user_id`.** Within a single user's pull,
all rows share `owner_user_id` so it's strictly redundant. The field is
preserved for forward-compat with multi-owner pulls (purpose #5 from
`plan.md`). The cost is one UUID per cursor — negligible.

**No "high watermark" race.** v1 had a window where two concurrent writes
could share `server_received_at` and the cursor advance could skip one.
The row-value tiebreak closes this.

### 5.4 Tombstones in pull responses

A tombstone in a pull response is a normal Entity with `fields.deleted_at`
set to a non-null epoch ms. The client applies it the same way it applies
any other row (§5.6); local readers filter `deleted_at IS NOT NULL` to
hide deleted rows from the UI.

### 5.5 Pull and concurrent local edits

If a local mutation lands while a pull page is being applied:

- The local write flips `local_dirty = 1` and bumps `local_updated_at_ms`
  via the monotonicity guard.
- The pull apply runs LWW locally per row: if
  `incoming.client_updated_at_ms > local.local_updated_at_ms`, apply
  (clear `local_dirty`, update timestamp from incoming); otherwise no-op.

The pull-page transaction (§5.6) holds the connection for the whole page,
so the local write either lands fully before the page or fully after; in
both orderings LWW resolves correctly.

### 5.6 Applying a pull page — transaction shape, FK handling

The client schema has cross-entity FKs (t1 §5.2). Naive per-row INSERT/
UPDATE inside a page can violate FKs when a child row arrives before
its parent. The protocol does NOT require the server to emit rows in
parent-first order; the client handles it locally.

**Each pull page applies inside a single SQLite transaction with deferred
foreign keys:**

```sql
BEGIN;
PRAGMA defer_foreign_keys = ON;   -- scoped to this transaction
-- INSERT/UPDATE one row at a time per the LWW rule below
COMMIT;                            -- FK constraints checked here
```

`PRAGMA defer_foreign_keys = ON` is set immediately after `BEGIN` on the
same connection; SQLite checks deferred FKs at COMMIT instead of at each
statement. This lets the client apply rows in arrival order without
worrying about ordering.

**Per-row LWW rule (inside the transaction):**

1. SELECT the row by `(type, id)`.
2. If the row does not exist: INSERT every column from `fields`; set
   `local_dirty = 0`, `local_updated_at_ms = incoming.client_updated_at_ms`.
3. If the row exists and `incoming.client_updated_at_ms >
   local.local_updated_at_ms`: UPDATE every column from `fields`; set
   `local_dirty = 0`, `local_updated_at_ms = incoming.client_updated_at_ms`.
4. If the row exists and `incoming.client_updated_at_ms <=
   local.local_updated_at_ms`: no-op. Do not clear `local_dirty` — the
   local write is newer.

**If COMMIT fails on FK violation:** the page is rolled back; the cursor
does NOT advance; the cycle terminates with a logged error (same posture
as `FK_VIOLATION` on push per §4.3 — non-retriable, structural bug, fix
lands by app update). In practice, a client-side FK violation at COMMIT
means the server emitted a row whose parent is neither in the same page
nor present locally — extraordinarily unlikely under normal operation
(t1 §5 deferred-FK push semantics make orphan rows on the server
impossible) and treated as such.

**Idempotency.** Step 4's no-op means re-applying a page is safe. An
interrupted apply (process killed mid-page) rolls back the transaction;
the cursor stays at the pre-page value; the next cycle re-fetches the
same page; the second apply commits cleanly.

> Note for t4: "batched independent commits" in t4 §11 #3 means independent
> *across pages*, not within a page. Each pull page is one transaction;
> the cursor advances between pages.

### 5.7 Cursor lifecycle

The cursor advances **only** when a pull page is fully applied
(committed). Push has no effect on the cursor.

| Cycle outcome | Cursor after cycle |
|---|---|
| No pull (cycle pushed only). | Unchanged. |
| Pull issued, page applied, COMMIT succeeded. | Advanced to the `next_cursor` from that page. |
| Pull issued, multiple pages applied. | Advanced incrementally — after each page's COMMIT, the cursor is persisted before the next page is fetched. |
| Pull issued, page apply rolled back (FK violation at COMMIT, transport drop mid-apply, process killed). | Unchanged. Next cycle re-fetches the same page. |
| Pull issued, empty response (`entities = []`, `has_more = false`). | Unchanged (the response echoes the input cursor). |

The cursor is stored in `sync_runtime_state.pull_cursor` (singleton per
device, see §12).

## 6. First sign-in

On the first successful sign-in (no `bootstrap_completed_at` set in
`sync_runtime_state`), the runtime simply starts the scheduler. The first
cycle pulls everything the server has (cursor = null) and pushes whatever
rows are locally dirty. LWW reconciles any overlap row-by-row. No modal.
No user choice. The `bootstrap_completed_at` field is set after the first
cycle completes successfully.

**Out of scope: cross-account local data.** If a device has local rows
left over from a previous account (e.g. the user signs in as account B on
a device that previously held account A's data), those rows will push as
account B's data on the next cycle — RLS will set `owner_user_id =
auth.uid()` server-side. Preventing this is the responsibility of the
auth/runtime layer, not the sync protocol: it must wipe local entity
tables on sign-out (or on detecting a new `auth.uid()` mid-session)
before the sync runtime starts. Flagged as a follow-up task to land
against the auth surface; t2 does not solve it here.

## 7. The cycle interface (consumed by t4)

t4 §11 #1 requires that "a cycle continues issuing PULLs until the server
has no new rows beyond the cursor, and PUSHes until no local rows are
dirty." t2 owns the wire shape that makes this work; t4 owns the
scheduling. This section spells out the interface between them.

### 7.1 What a cycle does

The cycle is the routine the t4 scheduler invokes when it transitions
from `Scheduled(t)` to `Running`. Its job is to converge local and remote
state, with no in-memory state surviving the call:

1. **Pull phase** — repeatedly call `sync_pull` with the persisted cursor
   until `has_more === false`. Apply each page per §5.6, advancing the
   cursor after each page (t4 §11 #5).
2. **Push phase** — repeatedly call `selectPushBatch` (§3.4) and
   `sync_push` until `selectPushBatch` returns an empty array (dirty
   stream exhausted).
3. **Re-pull phase** — call `sync_pull` once more with the (now-advanced)
   cursor. If `has_more === false` AND `entities.length === 0`, the cycle
   has converged. If new entities arrive (a concurrent device pushed
   while we were pushing), apply them and continue draining.
4. **Repeat until** both ends are quiet for one full round (no rows
   pulled, no rows pushed). The standard convergence pattern.

> t4 explicitly does NOT prescribe the PULL/PUSH ordering. This doc
> recommends **PULL → PUSH → PULL**: pull first so a remote tombstone or
> remote edit lands before we re-push potentially-stale data; push
> second; pull a third time to drain anything that arrived during the
> push. t4 §11 says "convergence is what matters." Builders may swap
> the order if there's a reason; the requirements §7.2 must hold.

### 7.2 What a cycle MUST guarantee (interface for t4)

- **Drain to empty.** When the cycle returns, either (a) both ends are
  quiet (no rows dirty, no remote rows newer than cursor), or (b) the
  cycle aborted on a budget or error and made forward progress in at
  least one batch (t4 §11 #1, §11 #3).
- **Per-batch commits.** Each `sync_push` is one transaction; each
  `sync_pull` page apply is committed before the cursor advances. An
  interruption mid-cycle leaves the database consistent at the last
  committed batch (t4 §11 #3).
- **Idempotency.** Re-running the cycle from the same starting state
  produces the same final state (t4 §11 #4, §11 #5).
- **Surviving state.** Between cycles, only `local_dirty`,
  `local_updated_at_ms`, the pull cursor, and the bootstrap flag
  survive — all of which are in SQLite (t4 §11 #6).
- **Cycle duration target.** Background cycles abort cleanly within 20
  seconds (t4 §11 #2). Each batch's HTTP request times out at 15
  seconds; the cycle checks elapsed time after each batch and breaks
  out if the next batch would push past the budget.

### 7.3 Multi-batch push

When the dirty stream exceeds 200 rows, the push phase issues multiple
batches in sequence. The acknowledgement of batch N must be applied to
local state (dirty bits cleared on the rows that committed) before batch
N+1 is built; otherwise batch N+1 would re-send the same rows.

### 7.4 Why the cursor is single-valued (not per-table)

One cursor for the whole user, not one per table. Reasons:

1. The cursor is `(server_received_at, owner_user_id, type, id)` — the
   `type` is part of the cursor, so a single cursor traverses all tables
   in interleaved order.
2. Per-table cursors would mean the cycle issues 8 parallel pulls; the
   client would need to merge by `server_received_at` to apply in the
   right order. Single cursor pre-merges server-side.
3. The pull RPC returns up to `limit` rows across all tables; the
   server's ORDER BY interleaves them naturally.

## 8. Dirty-bit lifecycle

### 8.1 The dirty bit

Per §12 (canonical column inventory), every entity table has two
local-only columns:

- `local_dirty integer not null default 0` — `1` iff the row needs
  pushing.
- `local_updated_at_ms integer not null default 0` — the row's
  client-monotonic timestamp (§9). Same value the wire envelope's
  `client_updated_at_ms` carries on push.

Neither column ever appears on the wire.

A row with `local_dirty = 0` is in sync with the server: the only ways
to flip dirty → 0 are a successful push ack or a pull apply, both of
which mean the server holds the row. So `local_dirty = 0` ⇒ server has
the row.

### 8.2 Setting and clearing — full table

| Event | `local_dirty` | `local_updated_at_ms` |
|---|---|---|
| Repo write (create / update / softDelete / cascade) | `1` | `nowMonotonic()` |
| Pull apply (incoming wins, §5.6 step 2 or 3) | `0` | `incoming.client_updated_at_ms` |
| Pull apply (incoming loses, §5.6 step 4) | unchanged | unchanged |
| Push ack (no concurrent edit — `current == sent_at_ms`) | `0` | unchanged |
| Push ack (concurrent edit detected — `current != sent_at_ms`) | unchanged (stays `1`) | unchanged |
| Push error (`FK_VIOLATION`, transport, etc.) | unchanged | unchanged |

All repo write paths run inside a single Drizzle transaction that
updates the row AND sets the dirty bit. No "set dirty bit later" path
exists. This is the **single biggest contract repos must obey** — t1's
agent-reminder spec (t1 §8) calls it out.

### 8.3 Push-in-flight race

The race: while a row's push is in flight, the user edits the row again.
A naive ack-handler would clear the dirty bit, losing the new edit.

**Mechanism.** At request time the cycle captures
`Map<(type, id), sent_at_ms>` for every row in the batch, where
`sent_at_ms` is the value of `local_updated_at_ms` at the moment the
row was serialised. The map lives in cycle-local memory.

On receipt of the success ack, the cycle, in one Drizzle transaction
per row:

1. SELECT the row's current `local_updated_at_ms`.
2. If `current == sent_at_ms`: clear `local_dirty = 0`.
3. If `current != sent_at_ms` (in practice always `current > sent_at_ms`
   because the monotonicity guard never goes backwards): leave
   `local_dirty = 1`. Next cycle re-pushes the newer value.

If the suspension drops the in-memory map before the ack handler runs
(extraordinarily rare — both happen in microseconds inside the same
cycle), the rows stay dirty; the next cycle re-pushes and the server's
LWW handles the duplicate idempotently (§3.6).

## 9. Clock-monotonicity guard

### 9.1 The formula

```
local_updated_at_ms = max(Date.now(), last_emitted_ms + 1)
```

Every repo write call that bumps `local_updated_at_ms` calls a single
helper:

```ts
function nowMonotonic(): number {
  const wall = Date.now();
  const previous = cached_last_emitted ?? readPersisted();
  const next = Math.max(wall, previous + 1);
  cached_last_emitted = next;
  persistAsync(next);
  return next;
}
```

### 9.2 Location

A single helper function in `apps/mobile/src/data/clock.ts` (file name
chosen to be conspicuous — `clock` is the entire purpose of this
module). All repos import from this one file. The session-recorder and
the seed loader also use this helper. No other monotonic-time helper
exists.

### 9.3 Persistence across restarts

The device clock can move backwards between launches (NTP correction,
user-set time, time-zone change interpreted by the OS). To prevent the
monotonic counter from regressing, `last_emitted_ms` is persisted to
SQLite.

Storage: **a new column `last_emitted_ms integer not null default 0` on
the `sync_runtime_state` singleton table** (already used by the
bootstrap and the v1 scheduler runtime; see
`apps/mobile/src/data/schema/sync-runtime-state.ts`). One row, one column,
trivial to read/write.

`persistAsync` is fire-and-forget. If the process dies before the write
flushes, the next launch reads a slightly-stale value — the worst case
is one wasted millisecond of monotonic headroom (the next write bumps
past it on the first call).

**The counter is device-global, not per-user.** On sign-out / sign-in,
the counter does **not** reset. Two users on the same device share the
counter (which is what we want — monotonic timestamps across the device's
clock are the invariant). Any future local-wipe-on-account-switch path
(see §6) must leave `last_emitted_ms` alone.

### 9.4 In-memory cache

The helper caches `last_emitted_ms` in a module-scoped variable so tight
loops (50 row writes in the same transaction) don't hit SQLite 50 times.
The cache is invalidated on each `persistAsync` flush; the read path
prefers the cache over the persisted value when both are present.

On cold start the cache is empty, so the first call reads from SQLite.

### 9.5 Why client-side, not server-side

LWW resolution happens server-side, but the timestamps it compares are
client-generated. The monotonicity guard prevents two timestamps from
the **same device** being equal — without it, two same-millisecond
writes could tie and LWW would pick arbitrarily.

The server does not need to know about the guard. It just sees the
incoming `client_updated_at_ms` and compares it to its stored value.

Cross-device clock skew remains unguarded — this is the brainstorm's §9
"Clock skew on LWW: trust client clocks for v2" position, confirmed here.

## 10. Retry and backoff

### 10.1 Push retry schedule

| Parameter | Value |
|---|---|
| Initial delay | 2 seconds |
| Multiplier | 2× |
| Cap | 2 minutes |
| Jitter | ±25% (uniform) |
| Reset triggers | (a) a successful push (any `{ ok: true }` response); (b) user toggles sync off then on; (c) user taps "Sync now"; (d) sign-out (state cleared) |

After a failed push, the next retry happens at:

```
now + (2s × 2^(failures-1)) × (1 ± 0.25)   clamped to ≤ 2 min
```

`consecutive_failures`, `next_attempt_at`, and `last_error_message` are
persisted in `sync_runtime_state` (extending the existing singleton with
three new columns — `consecutive_failures int default 0`,
`next_attempt_at integer`, `last_error_message text`).

### 10.2 Pull retry schedule

Same schedule as push.

### 10.3 Offline mid-push

If the device goes offline while a push request is in flight:

- HTTP layer reports a transport failure.
- Dirty bits remain set (nothing was acked).
- `consecutive_failures` increments; `next_attempt_at` advances per
  §10.1.
- When NetInfo reports `isInternetReachable = true` (t4 §3 — online
  edge), the t4 scheduler fires `requestSync('online')`. This **clears
  `next_attempt_at`** (treats the online edge as a reset trigger for the
  immediate retry) and starts a cycle.

Per t4 §11 #4, the retry is idempotent: re-pushing the same rows
produces the same final state on the server, even if the server actually
did receive the original request and the failure was on the response
path.

### 10.4 Non-retryable errors

Per §2.2, the only failure surface that is non-retryable is
`FK_VIOLATION` (structural bug — see §4.3). The client does not auto-
retry; the dirty bits stay unchanged and the user sees a non-recoverable
status.

### 10.5 Backoff reset on cycle boundaries

If a cycle completes successfully (drained both ends), the backoff state
is reset to `(0, null, null)`. The next cycle starts on the t4
scheduler's normal cadence (foreground edge, edit, safety tick).

## 11. Tombstones — end-to-end recap

This consolidates tombstone behaviour scattered above. Builders should
read this section to confirm there are no surprises.

| Event | Behaviour |
|---|---|
| User soft-deletes a row locally | Repo sets `deleted_at = Date.now()`, `local_dirty = 1`, `local_updated_at_ms = nowMonotonic()`. Row stays in SQLite. |
| Local readers query rows | All queries filter `deleted_at IS NULL`. |
| Client pushes tombstone | Normal Entity envelope with `fields.deleted_at` set to a non-null epoch ms. The full last-known row travels in `fields`. |
| Server applies tombstone | Standard LWW upsert. `deleted_at` lands as a typed column under `fields`. Non-PK columns are written from `fields` verbatim — they reflect the row's last-known state, **not** cleared. Tombstone stays in `app_public.<entity>` indefinitely (no v2 hard purge). |
| Server returns tombstone in pull | Standard Entity envelope with `fields.deleted_at != null`. |
| Client applies inbound tombstone | LWW comparison per §5.6. If incoming wins: write every field (including `fields.deleted_at`); set `local_updated_at_ms = incoming.client_updated_at_ms`; set `local_dirty = 0`. If incoming loses: no-op. |
| Undelete (resurrect) | Repo writes `deleted_at = null` and dirties the row. Next push sends the full row with `fields.deleted_at = null`. Server LWW upsert restores the row. |

**The v1 "tombstone for missing row" idempotency case** — deleting a row
the server doesn't have — works by construction in v2: the server just
stores the tombstone as a new row.

### 11.1 Tombstone retention

The server keeps tombstones forever in v2. No GC job, no purge schedule.
Two reasons:

1. Group/sharing (purpose #5) will eventually want audit; tombstones
   are part of that.
2. Without retention, a peer device that's been offline for longer than
   the GC window would miss the deletion and resurrect the row on its
   next push. Retention is correctness, not just observability.

Tombstones are small (PK + flags + last-known row), and the user base
will not generate enough deletes to overflow Supabase's free tier in any
realistic window.

## 12. Local-only sync-bookkeeping columns

Per t1 §11 #7, t2 owns the canonical inventory. This section pins it down

### 12.1 The two columns

**Every entity client table** (all eight: `gyms`, `sessions`,
`session_exercises`, `exercise_sets`, `exercise_definitions`,
`exercise_muscle_mappings`, `exercise_tag_definitions`,
`session_exercise_tags`) gains these two columns in the t2 build wave's
local Drizzle migration:

| Column | Type | Default | Semantic |
|---|---|---|---|
| `local_dirty` | `integer` (SQLite-style boolean: 0/1) | `0` | `1` iff the row needs pushing. Set by every repo write path. Cleared by the push ack handler (§8). |
| `local_updated_at_ms` | `integer` (epoch ms) | `0` | The client-monotonic timestamp used as the LWW key. Sent on the wire as `client_updated_at_ms` (§2.1). Bumped by `nowMonotonic()` on every dirtying write (§9). |

Both columns are local-only; the push serialiser (§3.3) omits them; the
pull apply (§5.6) writes `local_dirty=0`,

`local_updated_at_ms`) and camelCase in Drizzle TypeScript (`localDirty`,
`localUpdatedAtMs`). Same as existing columns.

Stored as a new singleton column on the existing `sync_runtime_state`
table (one row per device):

| Column | Type | Default | Semantic |
|---|---|---|---|
| `pull_cursor` | `text` | `null` | The opaque cursor blob (JSON-stringified, `{server_received_at, owner_user_id, type, id}`). NULL means "snapshot from the beginning." Advanced after each pull page is fully applied (§5.6). |

`pull_cursor` is **not** a per-entity column; it is a singleton because
the cursor traverses all entities together (§7.4).

### 12.3 The monotonic-clock state

Also a column on `sync_runtime_state`:

| Column | Type | Default | Semantic |
|---|---|---|---|
| `last_emitted_ms` | `integer` | `0` | The largest `local_updated_at_ms` this device has emitted. Persisted by `nowMonotonic()`'s fire-and-forget write. See §9.3. |

### 12.4 The retry-state columns

Per §10.1, three more singleton columns:

| Column | Type | Default | Semantic |
|---|---|---|---|
| `consecutive_failures` | `integer` | `0` | Count of consecutive push (or pull) failures since the last successful cycle. Used to compute the next backoff. |
| `next_attempt_at` | `integer` (epoch ms) | `null` | Earliest wall-clock time the next attempt may run. NULL = no backoff in effect. |
| `last_error_message` | `text` | `null` | Human-readable last error, surfaced via the sync status panel (t4 §8). |

### 12.5 Drift-checker registration

Per t1 §3.3 and §11 #7, all eight entity tables' two local-only columns
must be listed in `apps/mobile/src/data/schema/sync-extras.json` under
`exemptions.local_only_columns`. The registration is global (not
per-entity), because the same two columns apply to all eight tables:

```jsonc
{
  "exemptions": {
    "local_only_columns": [
      "local_dirty",
      "local_updated_at_ms"
    ],
    "untyped_text_references": [
      // ... per t1 §3.3 ...
    ]
  }
}
```

The `sync_runtime_state` table is **not** in the drift checker's scope —
it has no server counterpart by design; it's a client-only singleton (the
drift checker scans only the eight user-owned entity tables, per t1 §7).
The singleton-state columns (`pull_cursor`, `last_emitted_ms`,
`consecutive_failures`, `next_attempt_at`, `last_error_message`,
`bootstrap_completed_at`) live on that table and don't need any
registration.

### 12.6 Push serialiser exclusion

The push serialiser (§3.3) MUST NOT emit `local_dirty` or
`local_updated_at_ms` in `fields`. t1 §11 #7 explicitly assigns this
responsibility to t2:

> "t2 is also responsible for ensuring the push serialiser does not emit
> these columns on the wire."

The build wave implements this by routing the serialiser through a
single function (`entityToWire`) per entity, which closes over the
typed server-column list from the t1 §2 mapping —
`local_only_columns` are not in that list, so they're not emitted.

| # | Constraint | Honoured in |
|---|---|---|
| 6 | Only dirty bits + pull cursor survive suspension. | **§8.3; §12.** Concretely, the persisted state is: per-entity `local_dirty`, `local_updated_at_ms`, plus singleton `pull_cursor`, `last_emitted_ms`, `consecutive_failures`, `next_attempt_at`, `last_error_message`. Everything else (cycle progress, in-flight batch metadata, the `Map<(type,id), sent_at_ms>` push-state) is in-memory and reconstructed on the next cycle from the dirty stream and the cursor. |

### 13.3 t1 follow-up emitted by t2

This subsection records changes to t1's design that t2 now requires.
Each item is a follow-up task to land against t1 (or against t1's
artifacts: the server schema, the drift checker, the `sync-extras.json`
registry).

**Server-first deploy is now mandatory for every schema addition, no
exceptions.** The forward-compat hatch that `extras` provided is gone.
Any new entity column lands on the server first, then on the client.

1. **Drop `extras jsonb` from the universal server column set.** t1 §11
   #8 / #9 are dead; the column is no longer written to and no longer
   read by anything. Migration to drop it can be timed independently of
   the t2 ship (existing rows can keep the column with no readers).
2. **Drop the `extras_only` allowlist machinery.** t1 §3.3 /
   `sync-extras.json`'s `extras_only_columns` per-entity entries become
   dead code. The drift checker's `extras_only` handling (t1 §4 and the
   §7 algorithm step that walks the allowlist) should be removed.
   `exemptions.local_only_columns` remains, with two entries:
   `local_dirty`, `local_updated_at_ms`.
3. **Drop `deleted boolean` from the universal server column set.** The
   `deleted_at` column (nullable bigint epoch-ms, currently per-entity
   per t1 §4) becomes the sole tombstone marker. Local readers continue
   to filter `WHERE deleted_at IS NULL`. Server upsert / pull paths
   treat `deleted_at IS NOT NULL` as "tombstone."
4. **Add `deleted_at` to the join tables** (`exercise_muscle_mappings`,
   `session_exercise_tags`). t1 §4.2 lists `deleted_at` only on the six
   non-join entities; t2's wire shape requires every synced row to carry
   `deleted_at` because that column is the only tombstone marker. The
   follow-up adds `deleted_at bigint nullable` to both join tables on
   both the server and the client.
5. **Extend the drift checker to assert the topological FK order** used
   by t2 §3.4.1. The push-batch algorithm depends on a hardcoded
   per-table order (Layer 0..3) being a valid topological sort of the
   live FK graph. Adding a new entity table or FK in a future migration
   must fail CI until the order is updated. Concretely: the checker
   reads the FK edges from the live introspected schema (per t1 §7.2),
   reads the hardcoded `TOPO_ORDER` from a known client-side location
   (e.g., `apps/mobile/src/sync/topo-order.ts`), and asserts every FK
   `T_child → T_parent` has `index(T_parent) < index(T_child)` in the
   list.

### 13.4 No deferred or impossible t2 constraints

The active t1 §11 constraints (1–7, 10–12) and all 6 t4 §11 constraints
are honoured by this doc. Constraints 8 and 9 are dead by t2's choice
(see §13.3). If the build wave finds any of the remaining constraints
untrue when implementing, the gap is a t2 design bug to flag back to
this doc.

## 14. Sequence diagrams

### 14.1 Device-reinstall login

```
Device A — fresh install, user signs in to existing account.

  ┌─ Auth: sign in, JWT obtained, userId = U
  │
  ├─ Runtime starts the scheduler (bootstrap_completed_at is NULL —
  │  but the only effect is "we'll set it after cycle 1 succeeds").
  │
  ├─ Cycle 1 fires immediately (foreground/online edge per t4).
  │    - Pull phase: cursor = null.
  │        sync_pull { cursor: null, limit: 500 }
  │          ◀── { entities: [...500...], next_cursor: C1,
  │                has_more: true }
  │        Apply 500 rows (§5.6). Persist C1.
  │        sync_pull { cursor: C1, limit: 500 }
  │          ◀── { entities: [...500...], next_cursor: C2,
  │                has_more: true }
  │        Apply, persist C2.
  │        sync_pull { cursor: C2, limit: 500 }
  │          ◀── { entities: [...37...], next_cursor: C3,
  │                has_more: false }
  │        Apply, persist C3.
  │    - Push phase: dirty count = 0 → no-op.
  │    - Re-pull phase: cursor = C3 → empty, has_more = false.
  │      Cycle converged.
  │
  ├─ Runtime sets bootstrap_completed_at = now().
  │
  └─ Subsequent cycles run on t4's normal cadence.

End state: Device A has the same data as the server. Pull cursor at C3.
Dirty count: 0.
```

### 14.2 Two devices both online, editing the same row

```
Device A and Device B both signed in as user U. Both have completed
bootstrap. t4 scheduler on both at SAFETY_INTERVAL = 60s.

Initial state — row R on both devices:
  { local_updated_at_ms: 1000, name: "Bench Press",
    local_dirty: 0 }

T = 100 (wall): User on Device A renames R → "My Bench A"
  Repo write:
    nowMonotonic() = max(100, 1000+1) = 1001  ← guard prevents tie
    Row R = { local_updated_at_ms: 1001, name: "My Bench A",
              local_dirty: 1 }
  requestSync('mutation') → scheduler: Scheduled(MIN(t, now+500ms))
  500ms passes; no further edits.

T = 600 (wall): Device A cycle fires.
  selectPushBatch → [R @ 1001]
  POST sync_push { entities: [R @ 1001] }
  Server:
    UPSERT R; existing stored has client_updated_at_ms = 1000.
    Predicate 1001 > 1000 → APPLY.
    Server row R = { client_updated_at_ms: 1001, name: "My Bench A",
                     server_received_at: T_server_1 }
    Response: { ok: true, server_received_at: T_server_1 }
  Device A:
    Row R.local_updated_at_ms still 1001 (no concurrent edit during
    flight). Clear local_dirty=0.

T = 150 (wall on Device B, before Device A's push lands at the server
         — but B's clock may run slightly behind):
  User on Device B renames R → "My Bench B"
  B's nowMonotonic() = max(150, B's last_emitted+1) ≈ 1150 (assuming
  B's local clock is at 1150 — different than A's because no shared
  monotonic counter across devices).
  Row R on B = { local_updated_at_ms: 1150, name: "My Bench B",
                 local_dirty: 1 }

T = 700 (wall): Device B cycle fires.
  selectPushBatch → [R @ 1150]
  POST sync_push { entities: [R @ 1150] }
  Server (now sees R with stored client_updated_at_ms = 1001 from A):
    Predicate 1150 > 1001 → APPLY.
    Server row R = { client_updated_at_ms: 1150, name: "My Bench B",
                     server_received_at: T_server_2 }
    Response: { ok: true, server_received_at: T_server_2 }
  Device B: clear local_dirty=0.

T = 750 (wall): Device A's next scheduler tick (foreground stayed up).
  selectPushBatch → empty (no dirty rows).
  sync_pull { cursor: C_A, limit: 500 }
    ◀── { entities: [R @ 1150 with name="My Bench B"],
          next_cursor: C_A', has_more: false }
  Apply R: incoming 1150 > local 1001 → UPDATE.
  Row R on A = { local_updated_at_ms: 1150, name: "My Bench B",
                 local_dirty: 0 }
  Cursor advances to C_A'.

Final: both devices converged on "My Bench B". B's edit, at wall=150,
won because B's local_updated_at_ms (1150) was greater than A's
(1001). LWW resolved correctly. A's earlier rename was overwritten —
accepted behaviour for purpose #2 ("two-device support — last-write-
wins is acceptable", per plan.md).
```

**Alternative ordering in 14.2.** What if Device A's push (T=600) is
slow and Device B's push (T=700) hits the server first?

- B's push at 1150 lands first → server row at 1150, name="My Bench B".
- A's push at 1001 arrives next → predicate 1001 > 1150 is false → no
  UPDATE; server unchanged. Response is still `{ ok: true }`; the dirty
  bit's job is done (the server is at-or-newer than what A sent).
- A's next pull delivers B's value as in the original sequence.

Same convergence; LWW is determined by `client_updated_at_ms`, not by
request arrival order.

## 15. What this design does not cover

Closed in companion design docs:

- **t1** — the typed server schema, RLS, deferrable-FK declarations, LWW
  semantics, the drift checker.
- **t3** — the seed loader's decision tree, seed-version drift across
  reinstall, user-edited seeds.
- **t4** — iOS lifecycle triggers, the two-state scheduler, BGTaskScheduler
  integration, the "Sync now" button, the sync status surface.

Out of scope here as deliberate boundaries:

- The PostgREST RPC source code for `sync_push` and `sync_pull`. The build
  wave writes these.
- The client cycle implementation in `apps/mobile/src/sync/` (the v1
  `engine.ts` / `outbox.ts` deletion and the v2 replacement). Build wave.
- The local Drizzle migration adding `local_dirty`, `local_updated_at_ms`
  to the eight entity tables, and `pull_cursor`, `last_emitted_ms`,
  `consecutive_failures`, `next_attempt_at`, `last_error_message` to
  `sync_runtime_state`. Build wave.
- The Maestro / e2e tests verifying the cycle. Build wave.
- The session-recorder integration with `requestSync('mutation')`. Build
  wave (and t4 already specifies the trigger surface).
- Tombstone hard-purge policy (we keep them forever in v2 per §11.1).
- Group / sharing semantics (purpose #5 from `plan.md`; future work).
- Web client and MCP read paths against the typed server schema (purposes
  #3 / #4; they consume t1's schema directly, no t2 protocol involvement).
