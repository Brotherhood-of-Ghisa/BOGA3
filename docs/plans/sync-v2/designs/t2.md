## 0. What this doc covers

1. The two RPCs — `sync.push` and `sync.pull` — request/response JSON shape,

2. Push batching: topological table ordering, deferrable-FK commit, the

3. Pull: per-layer cursors symmetric to the push topological batch

6. Dirty-bit lifecycle (§7), clock-monotonicity guard (§8), local-only
   column inventory (§9).
7. Constraint reconciliation against t1 §11 and t4 §11 (§10).

## 1. Endpoints

Two RPCs. Both exposed as PostgREST RPCs under the `app_public` schema.
Both require an authenticated user; `owner_user_id` is derived from the
JWT and is never sent on the wire (RLS enforces it, per t1 §6). Both are
POSTs.

| RPC | Purpose |
|---|---|
| `sync_push` (`POST /rest/v1/rpc/sync_push`) | Client uploads a batch of dirty rows. Server upserts with LWW inside a single transaction. |
| `sync_pull` (`POST /rest/v1/rpc/sync_pull`) | Client downloads server rows newer than a cursor. Returns up to `limit` rows ordered by `(server_received_at, owner_user_id, type, id)`. |

No third "snapshot" RPC. Snapshot is just `sync_pull` with `cursor = null`.

There is no event log on the wire. There are no sequence numbers, no
device identifiers, no batch ids, no `sent_at` clock. **The data is the
event.**

**Bigint serialization.** PostgREST returns Postgres `bigint` values as
JSON strings by default. Both RPCs MUST be exposed with
`Prefer: return=representation` and the database role's PostgREST
config set to `db-aggregates-enabled=true` and the
`request.headers.accept` handling pinned so that `bigint` returns as a
JSON integer. All `client_updated_at_ms`, `local_updated_at_ms`-derived
values, and any `bigint` typed column on the wire is sent and received
as a JSON integer, never a JSON string. (Values fit safely in
JavaScript Number: epoch-ms < 2^53.) The build wave is responsible for
the PostgREST role configuration; the protocol spec assumes integers
on the wire.

## 2. Wire types (shared)

### 2.1 `Entity` envelope

Every row crossing the wire — push request, pull response — uses the same
shape. The envelope carries the LWW key; `fields` carries every typed
column on the row (including `deleted_at`).

```json
{
  "type": "exercise_sets",
  "id": "01HXYZ...ULID",
  "client_updated_at_ms": 1733000000123,
  "fields": {
    "session_exercise_id": "01HX...",
    "order_index": 0,
    "weight_value": "100",
    "reps_value": "8",
    "set_type": "rir_2",
    "created_at": 1732999999000,
    "updated_at": 1733000000123,
    "deleted_at": null
  }
}
```

Field rules:

| Field | Type | Notes |
|---|---|---|
| `type` | string enum | One of `gyms`, `sessions`, `session_exercises`, `exercise_sets`, `exercise_definitions`, `exercise_muscle_mappings`, `exercise_tag_definitions`, `session_exercise_tags`. **Plural, snake_case**, matching the t1 table names. |
| `id` | string | Client-assigned. ULID for user-created rows; slug (`seed:bench-press`) for seeds. Stable forever. |
| `client_updated_at_ms` | integer | Epoch ms from the originating client. **The LWW key.** Must be `>= 0`. Produced by the monotonicity guard (§8). |
| `fields` | object | Per-entity typed columns. Keys are wire-canonical snake_case (matching the t1 §2 mapping tables). Values use the JSON types implied by the t1 types: `text` → string, `bigint` → JSON integer (see §1 for the PostgREST serialization pin), `numeric` → string, `boolean` → boolean, `timestamp_ms`-style → integer epoch ms, nullable → JSON null. The client MUST emit every typed column listed in t1 §2 for that entity, even when the column is nullable (emit JSON null). `deleted_at` is one of those columns under LWW like any other (per t1 §1.1). |

**No `owner_user_id` on the wire envelope.** It is derived from `auth.uid()`
in the RPC body. Pull responses also omit it (the caller already knows it
is theirs).

**No `extras` blob, no top-level `deleted` flag.** Wire shape is
`fields` only; deletion is `fields.deleted_at != null`. Local readers
filter `deleted_at IS NULL`.

**Server-side schema rollout discipline.** Every new entity column lands
on the server first, then on the client. Clients written against
the old schema continue to omit the column from `fields`; the server fills
in its default. Clients written against the new schema can write the new
column once the server has it.

### 2.2 Error shape

Both RPCs use the same outer error envelope when they reject the entire
request. Per t1 §1 ground rules ("no server-side validation unless the
server writes"), the failure surface is intentionally narrow.

```json
{
  "error": {
    "code": "FK_VIOLATION",
    "message": "session_exercises 01HX5KAB...: parent sessions(owner_user_id, id)=(<uid>, 01HX5K9V...) not found"
  }
}
```

| `code` | Cause | Client behaviour |
|---|---|---|
| `AUTH_REQUIRED` | No JWT, expired JWT, RLS denies the row. | Trigger token refresh; if it fails, keep dirty bits, surface "Sign in again." |
| `FK_VIOLATION` | Deferrable-FK check failed at COMMIT (t1 §5). Structural bug — see §3.2 (push), §4.5 (pull-apply). | **Non-retriable.** Log to observability; leave dirty bits unchanged; surface non-recoverable error in the sync status panel. No auto-retry. A fix lands by app update. |
| `INTERNAL` | Anything else (transport, 5xx, unmarshalled error). | The cycle returns with an error; dirty bits stay set; the next scheduler tick (t4 safety interval, foreground edge, online edge, or next mutation) starts a fresh cycle that re-pushes the same rows. No exponential backoff, no `next_attempt_at` bookkeeping. |

**Server validates hierarchy (FKs) only.** No enum, range, length, format,
or business-rule validation. Anything malformed is the client's bug — the
t1 §7 drift gate catches schema-level malformation at PR time.

## 3. `sync_push`

### 3.1 Request

> as-built RPC has signature `app_public.sync_push(entities jsonb
> default '[]'::jsonb)` — named parameter. Granted `execute` to
> `authenticated` AND `anon` (the function's own AUTH_REQUIRED guard
> per §2.2 is the first statement, so anonymous callers never reach
> state-mutating code; the `anon` grant lets PostgREST route them into
> the function body in the first place). Clients POST the JSON below
> as-is — PostgREST's named-parameter dispatch handles the mapping.

```json
{
  "entities": [
    {
      "type": "exercise_definitions",
      "id": "seed:bench-press",
      "client_updated_at_ms": 1733000000100,
      "fields": {

      "id": "01HX5K9V...",
      "client_updated_at_ms": 1733000000200,
      "fields": {
        "gym_id": "gym_a",
        "status": "active",
        "started_at": 1733000000000,
        "completed_at": null,
        "duration_sec": null,
        "created_at": 1733000000000,
        "updated_at": 1733000000200,
        "deleted_at": null
      }
    },
    {
      "type": "session_exercises",
      "id": "01HX5KAB...",

        "session_id": "01HX5K9V...",

        "created_at": 1733000000201,
        "updated_at": 1733000000201,
        "deleted_at": null
      }
    }
  ]
}
```

| Field | Validation (structural only) |
|---|---|
| `entities` | Array, length `1..200`. v2 batch cap is **200** rows. Larger batches are split client-side (§4). |

### 3.2 Ordering inside the batch

- Clients may send entities in any order. The full hierarchy must be present — either in the batch itself or already on the server. (This is what the rest of the doc calls "closure.")
- Server implements this by deferring constraints: a single transaction with `SET CONSTRAINTS ALL DEFERRED`; all FKs are checked at COMMIT. If any FK is unsatisfied, the entire batch rolls back with `FK_VIOLATION` (§2.2, §4.3).

### 3.3 What goes in `fields`

For each typed server column on this entity (per t1 §2 mapping table), put
its current local value under `fields.<wire_key>`. The wire_key is the
snake_case Postgres column name from the mapping table. Nullable columns
get a JSON null when the local value is nullish.

Every other client column is either (a) a local-only sync column (§9) —
must not appear on the wire — or (b) drift, caught at PR time by the t1
§6 checker.

### 3.4 Building the batch

Per t1 §11 #6, the client MUST NOT push a child row whose parent is
neither in the same batch nor already on the server. The client
satisfies this by walking the eight entity tables in a fixed
**topological order** when collecting dirty rows.

#### 3.4.1 Topological order of the eight tables

The eight tables form a DAG under their FK relations. The fixed order
the client iterates is:

| Layer | Tables (any intra-layer order) | FKs to earlier layers |
|---|---|---|
| 0 | `gyms`, `exercise_definitions`, `exercise_tag_definitions` | none |
| 1 | `sessions`, `exercise_muscle_mappings` | `sessions → gyms`; `exercise_muscle_mappings → exercise_definitions` |
| 2 | `session_exercises` | `→ sessions`, `→ exercise_definitions` |
| 3 | `exercise_sets`, `session_exercise_tags` | `exercise_sets → session_exercises`; `session_exercise_tags → session_exercises`, `→ exercise_tag_definitions` |

No table has a within-layer FK and no table is self-referential, so any
order inside a layer is safe.

This list is hardcoded client-side. **The t1 §7 drift checker asserts
that this hardcoded order is a valid topological sort of t1's actual FK
graph** (see t1 §7.7) — adding a new entity table or FK in a future
migration fails CI until the order is updated.

#### 3.4.2 Algorithm `selectPushBatch(batchCap = 200) → Entity[]`

```
batch = []
for table in TOPO_ORDER:
  if len(batch) >= batchCap: break
  rows = SELECT * FROM <table>
         WHERE local_dirty = 1
         ORDER BY local_updated_at_ms ASC
         LIMIT (batchCap - len(batch))
  for row in rows:
    batch.append(serialize(row))
return batch
```

The cycle (t4 §11) calls this repeatedly. Each call returns at most
`batchCap` rows from the topologically-earliest tables that still have
dirty rows; the cycle pushes the batch, the server commits it, and the
next call picks up where this one left off. When the call returns an
empty array, the push leg of the cycle is done.

Intra-table ordering by `local_updated_at_ms ASC` is a quality-of-life
choice — it makes the dirty queue drain FIFO so the oldest-dirty
row never starves — but the FK guarantee does not depend on it.

#### 3.4.3 Why this gives the no-orphan-child guarantee

The algorithm visits layers in topological order. For any dirty row
`R` in some batch, every FK target either (a) is in the same batch
from an earlier layer (deferred FK resolves at COMMIT), (b) is
`local_dirty = 0` and therefore already on the server (§7.2's
lifecycle — dirty clears only via push ack or pull apply), or (c) is
dirty but in a layer the algorithm hasn't reached yet, in which case
`R` is not in this batch either (the batch-cap broke out earlier in
the topological walk). Cross-account residue is the one way (b) can
fail; see §5's auth-precondition note.

### 3.5 Success response

```json
{
  "ok": true,
  "server_received_at": "2026-05-23T10:14:32.871Z"
}
```

A single ack body. No per-row outcomes. The whole batch committed; every
row in the request has been processed (either UPSERT-applied because LWW
accepted the incoming, or no-op because LWW rejected it — either way the
server's state is now at-or-newer than what the client sent for every
row, which is all the dirty bit needs to know).

The client clears `local_dirty` for every row in the in-memory batch
snapshot, subject to the §7.3 concurrent-edit check on each row.

`server_received_at` is the server's wall-clock `now()` at the moment of
the upsert (single `now()` per transaction, stable across all rows in
the batch). It is the cursor axis for pull (§4.3); the client uses it
only for observability on push.

### 3.6 Idempotency — t4 §11 #4

Re-push under retry is a server-side no-op: LWW finds `incoming ==
stored`, no UPDATE happens, the ack still clears the dirty bit.

### 3.7 Batch size

`batchCap = 200`. Postgres handles 200-row UPSERTs well, transport
round-trip dominates per-row CPU, and the payload stays safely under the
Supabase RPC 1 MiB request-size cap (a typical row is a few hundred
bytes). Three batches at this size comfortably fit inside t4's
foreground-cycle target of 5 seconds. If the dirty stream exceeds 200
rows, the cycle issues multiple push batches in sequence (§6.3); each
batch commits independently.

## 4. `sync_pull`

Pull is symmetric to push in two ways: 200-row batches, and topological
table ordering. The push side iterates tables in topological order so a
dirty child's dirty parent is in the same or an earlier batch (§3.4);
the pull side does the same on the server, drained layer-by-layer, so a
child page never lands before its parents are on the client. Without
that, a parent edited *after* a child was created appears later in
`server_received_at` order and a batch-cap cut between the two would
leave the client with a child whose parent isn't yet local — the same
FK pathology §3.4 fixes for push.

### 4.1 Request

> **Build-wave note (sync-v2-server PR #73, merged 2026-05-26).** The
> as-built RPC has signature `app_public.sync_pull(jsonb)` — UNNAMED
> parameter. PostgREST's single-jsonb fallback routes the raw POST
> body to the unnamed param; the function then reads
> `payload->>'layer'`, `payload->'cursor'`, `payload->>'limit'`.
> Granted `execute` to `authenticated` AND `anon` (same rationale as
> §3.1's note: `auth.uid() IS NULL` guard fires first; `anon` grant
> required to route into the function body so AUTH_REQUIRED surfaces
> instead of PostgREST 42501). Clients POST the JSON below as-is.

```json
{
  "layer": 0,
  "cursor": null,
  "limit": 200
}
```

Or with a non-empty cursor:

```json
{
  "layer": 2,
  "cursor": {
    "server_received_at": "2026-05-23T10:14:32.871Z",
    "owner_user_id": "00000000-0000-0000-0000-000000000aaa",
    "type": "session_exercises",
    "id": "01HX5KAB..."
  },
  "limit": 200
}
```

| Field | Type | Notes |
|---|---|---|
| `layer` | integer | `0..3`. The topological layer being drained. Mapping is the same as §3.4.1 push-side: 0 = `gyms, exercise_definitions, exercise_tag_definitions`; 1 = `sessions, exercise_muscle_mappings`; 2 = `session_exercises`; 3 = `exercise_sets, session_exercise_tags`. |
| `cursor` | object or `null` | Opaque to the client (round-trip the value the server emitted). `null` for snapshot / initial pull of this layer. **Each layer has its own cursor** — see §4.4 and §9.2. |
| `cursor.server_received_at` | string (ISO 8601 with ms) | The `server_received_at` of the last row returned by the previous response. |
| `cursor.owner_user_id` | string (UUID) | Same-millisecond tiebreak axis. Echoed back. |
| `cursor.type` | string enum | Same-millisecond tiebreak axis. Always one of the types in `layer`. |
| `cursor.id` | string | Same-millisecond tiebreak axis. |
| `limit` | integer | `1..200`. Default 200. Symmetric to push `batchCap` (§3.7). The cycle (t4) may pick a smaller value if it knows it's tight on budget. |

### 4.2 Response

```json
{
  "entities": [
    {
      "type": "session_exercises",
      "id": "01HX5KAB...",
      "client_updated_at_ms": 1733000000201,
      "fields": { "...": "..." }
    }
  ],
  "next_cursor": {
    "server_received_at": "2026-05-23T10:14:33.000Z",
    "owner_user_id": "00000000-0000-0000-0000-000000000aaa",
    "type": "session_exercises",
    "id": "01HX5KAB..."
  },
  "has_more": true
}
```

| Field | Notes |
|---|---|
| `entities` | Up to `limit` Entity envelopes, **all from tables in the requested `layer`**, ordered by `(server_received_at ASC, owner_user_id ASC, type ASC, id ASC)`. |
| `next_cursor` | The cursor to use on the next pull **of this same layer**. Always present (echoes input on an empty page). |
| `has_more` | `true` iff this layer has at least one more row strictly greater than `next_cursor`. `false` means the layer is drained. |

On an empty page (`entities.length === 0`), `has_more === false` and
`next_cursor` echoes the input `cursor` unchanged. The cycle advances
to the next layer.

### 4.3 Pagination — monotonicity and tiebreak

The server orders rows in the requested layer by:

```sql
ORDER BY server_received_at ASC,
         owner_user_id     ASC,
         type              ASC,
         id                ASC
LIMIT :limit
```

with the WHERE clause:

```sql
WHERE owner_user_id = auth.uid()
  AND type IN (<tables in :layer>)
  AND (
    :cursor IS NULL
    OR (server_received_at, owner_user_id, type, id)
        > (:cursor.server_received_at, :cursor.owner_user_id,
           :cursor.type, :cursor.id)
  )
```

The only delta from a single-cursor design is the `type IN (...)`
filter scoping rows to the current layer.

**Known limitation: concurrent-commit cursor race.**
`server_received_at` is set by the touch trigger at INSERT/UPDATE time,
which is *before* COMMIT. `now()` returns transaction-start time, so
two concurrent push transactions for the same owner (two devices
pushing within milliseconds) can commit in inverted order vs their
`server_received_at` values: A starts at T=100 (gets `server_received_at
= 100`), B starts at T=101 (gets `= 101`); B commits at T=102, A at
T=103. A pull landing between the two commits sees only B at 101,
advances the cursor past 101, and the next pull's `> 101` predicate
skips A's row at 100.

Bounded by the realistic two-device window (purpose-#2 low-priority).
**Self-healing**: any subsequent edit of the skipped row produces a
fresh `server_received_at` that advances past the cursor. A row stays
invisible only if never touched again. Acknowledged v2 limitation;
future hardening could switch the cursor axis to
`pg_xact_commit_timestamp(xmin)` (requires `track_commit_timestamp =
on`) or clamp cursor advance via `pg_stat_activity`.

**Same-millisecond rows.** Two rows in the same Postgres transaction
share a `now()` value. The `(owner_user_id, type, id)` lexicographic
tiebreak gives every row a unique sort key. The row-value `>` ensures
the next pull strictly advances past the last-emitted row — never
repeats, never skips. `type` is compared as a string; the server casts
it to a fixed enum at the RPC boundary so values are canonical
(plural snake_case as listed in §2.1) and case/whitespace drift is
impossible.

**Why the cursor carries `owner_user_id`.** Forward-compat with
multi-owner pulls (purpose #5 from `plan.md`). One UUID per cursor —
negligible.

### 4.4 Layer-by-layer drain (the cycle's pull leg)

> **Build-wave correction (sync-v2-server, merged 2026-05-26).** The
> layer→type mapping referenced throughout this section (and the
> example below) inherits t1 §7.7's original mapping which placed
> `exercise_tag_definitions` in Layer 0. That is structurally
> impossible against the t1 §5.2 FK graph; see t1 §7.7's build-wave
> correction note for the rationale. The corrected partition is
> Layer 0: `gyms`, `exercise_definitions`; Layer 1: `sessions`,
> `exercise_muscle_mappings`, `exercise_tag_definitions`; Layer 2:
> `session_exercises`; Layer 3: `exercise_sets`,
> `session_exercise_tags`. The as-built `sync_pull` SQL `case` mapping
> and `apps/mobile/src/sync/topo-order.ts` both encode this corrected
> partition; downstream client code MUST match.

The client cycle drains layers in topological order. By the time a
layer-K page applies, every parent table (in layers 0..K-1) is already
fully on the client from earlier iterations of the layer loop, so every
FK reference in the page resolves immediately. **No `PRAGMA
defer_foreign_keys` is needed** (§4.5).

```
for layer in [0, 1, 2, 3]:
  while True:
    resp = sync_pull({ layer, cursor: cursors[layer], limit: 200 })
    applyPage(resp.entities)             // §4.5
    cursors[layer] = resp.next_cursor    // persist after each COMMIT
    if not resp.has_more: break
```

**Cycle-aborts mid-drain.** t4's BG budget may cut the drain. The
per-layer cursors persist after every page COMMIT, so the next cycle
resumes from `cursors[K]` for the layer it was working on, then proceeds
through subsequent layers. The DB is FK-consistent at every commit
boundary because no layer-K page commits before layers 0..K-1 are
complete (and layer K rows reference only those layers).

> Open follow-up: an idle cycle issues 4 round-trips (one empty pull
> per layer). At typical Supabase latencies that's ~200 ms of overhead
> per safety-interval tick — well inside t4's foreground budget but
> worth collapsing into a single "any layer dirty?" probe before ship.
> Tracked as a t2/t4 build-wave follow-up.

### 4.5 Applying a pull page

Each pull page applies inside a single SQLite transaction. **No deferred
foreign keys are needed** because (a) layers 0..K-1 are already fully
on the client when applying a layer-K page, and (b) within a single
layer, no FK references point to other rows in the same layer (per
§3.4.1, the layer split has no intra-layer FKs).

```sql
BEGIN;
-- INSERT/UPDATE one row at a time per the LWW rule below
COMMIT;
```

**Per-row LWW rule (inside the transaction):**

1. SELECT the row by `(type, id)`.
2. If the row does not exist: INSERT every column from `fields`; set
   `local_dirty = 0`, `local_updated_at_ms = incoming.client_updated_at_ms`.
3. If the row exists and `incoming.client_updated_at_ms >
   local.local_updated_at_ms`: UPDATE every column from `fields`; set
   `local_dirty = 0`, `local_updated_at_ms = incoming.client_updated_at_ms`.
4. If the row exists and `incoming.client_updated_at_ms <=
   local.local_updated_at_ms`: no-op. Do not clear `local_dirty` — the
   local write is newer.

**If a per-row INSERT fails on FK violation:** the page is rolled back;
`cursors[layer]` does NOT advance; the cycle terminates with a logged
error (same posture as `FK_VIOLATION` on push — non-retriable,
structural bug, fix lands by app update). Under the layered drain this
should be impossible in practice: every FK in a layer-K row points
into layers 0..K-1, which are fully present.

### 4.6 Pull and concurrent local edits

If a local mutation lands while a pull page is being applied:

- The local write flips `local_dirty = 1` and bumps `local_updated_at_ms`
  via the monotonicity guard.
- The pull apply runs LWW locally per row (§4.5 steps 2–4): if
  `incoming.client_updated_at_ms > local.local_updated_at_ms`, apply
  (clear `local_dirty`, update timestamp from incoming); otherwise no-op.

The pull-page transaction holds the SQLite connection for the whole
page, so the local write either lands fully before the page or fully
after; in both orderings LWW resolves correctly.

### 4.7 Cursor lifecycle

There are **four cursors**, one per layer. Each advances independently
of the others. Push has no effect on any of them.

| Cycle outcome | Cursors after cycle |
|---|---|
| No pull (cycle pushed only). | Unchanged. |
| Pull issued, layer-K page applied, COMMIT succeeded. | `cursors[K]` advanced to the `next_cursor` from that page; other cursors unchanged. |
| Pull issued, multiple pages within a layer applied. | `cursors[K]` advanced incrementally — persisted after each page's COMMIT. |
| Pull issued, page apply rolled back (FK violation, transport drop, process killed). | All cursors unchanged. Next cycle re-fetches the same page in the same layer. |
| Pull issued, empty response (`entities = []`, `has_more = false`). | `cursors[K]` unchanged (echoes input). Cycle moves to layer K+1. |
| All four layers drained within the cycle. | All four cursors advanced to wherever the respective layer's last page landed; the cycle's pull leg returns. |

The four cursors are stored in `sync_runtime_state.pull_cursor` as a
JSON object keyed by layer index (`"0"`, `"1"`, `"2"`, `"3"`); see
§9.2.

## 5. First sign-in

On the first successful sign-in (no `bootstrap_completed_at` set in
`sync_runtime_state`), the runtime simply starts the scheduler. The
first cycle pulls everything the server has (each layer's cursor =
null) and pushes whatever rows are locally dirty. LWW reconciles
overlap row-by-row. No modal. No user choice.

**Precondition the auth layer must satisfy** — t2 assumes the local
entity tables hold rows owned by the currently-signed-in user, not a
prior user. If a device has residue from a previous account, those
rows would push to the new account on the next cycle (RLS sets
`owner_user_id = auth.uid()`) and §3.4.3's "clean ⇒ server has the
row" invariant would not hold. Wiping local entity tables on
sign-out (or on a `auth.uid()` change) is the auth-layer's
responsibility. Flagged as a t2-emitted follow-up against the auth
surface; t2 does not solve it here.

`bootstrap_completed_at` is set the first time **all four layers
drain to `has_more = false`** in the same cycle. A cycle that aborts
on BG budget (t4 §11 #2) before reaching that state leaves the flag
null and the next cycle picks up from per-layer cursors. The flag is
purely a "cold-start UX surface" — the protocol does not branch on
it.

## 6. The cycle interface (consumed by t4)

t4 §11 #1 requires that "a cycle continues issuing PULLs until the server
has no new rows beyond the cursor, and PUSHes until no local rows are
dirty." t2 owns the wire shape that makes this work; t4 owns the
scheduling. This section spells out the interface between them.

### 6.1 What a cycle does

The cycle is the routine the t4 scheduler invokes when it transitions
from `Scheduled(t)` to `Running`. Its job is to converge local and remote
state, with no in-memory state surviving the call:

1. **Pull phase** — drain all four topological layers in order (§4.4).
   For each layer 0..3, repeatedly call `sync_pull` with the per-layer
   cursor until `has_more === false`. Apply each page per §4.5,
   advancing `cursors[layer]` after each page (t4 §11 #5).
2. **Push phase** — repeatedly call `selectPushBatch` (§3.4) and
   `sync_push` until `selectPushBatch` returns an empty array (dirty
   stream exhausted).
3. **Re-pull phase** — drain all four layers again. If every layer
   returns `has_more === false` AND `entities.length === 0` on its first
   probe, the cycle has converged. If new entities arrive in any layer
   (a concurrent device pushed while we were pushing), apply them and
   continue draining.
4. **Repeat until** both ends are quiet for one full round (no rows
   pulled in any layer, no rows pushed). The standard convergence
   pattern.

> t4 explicitly does NOT prescribe the PULL/PUSH ordering. This doc
> recommends **PULL → PUSH → PULL**: pull first so any remote edit
> lands before we re-push potentially-stale data; push second; pull a
> third time to drain anything that arrived during the push. t4 §11
> says "convergence is what matters." Builders may swap the order if
> there's a reason; the requirements §6.2 must hold.

### 6.2 What a cycle MUST guarantee (interface for t4)

- **Drain to empty.** When the cycle returns, either (a) both ends are
  quiet (no rows dirty, no remote rows newer than cursor), or (b) the
  cycle aborted on a budget or error and made forward progress in at
  least one batch (t4 §11 #1, t4 §11 #3).
- **Per-batch commits.** Each `sync_push` is one transaction; each
  `sync_pull` page apply is committed before the cursor advances. An
  interruption mid-cycle leaves the database consistent at the last
  committed batch (t4 §11 #3).
- **Idempotency.** Re-running the cycle from the same starting state
  produces the same final state (t4 §11 #4, t4 §11 #5).
- **Surviving state.** Between cycles, only `local_dirty`,
  `local_updated_at_ms`, the four per-layer pull cursors, and the
  bootstrap flag survive — all of which are in SQLite (t4 §11 #6).
- **Cycle duration target.** Background cycles abort cleanly within 20
  seconds (t4 §11 #2). Each batch's HTTP request times out at 15
  seconds; the cycle checks elapsed time after each batch and breaks
  out if the next batch would push past the budget.

### 6.3 Multi-batch push

When the dirty stream exceeds 200 rows, the push phase issues multiple
batches in sequence. The acknowledgement of batch N must be applied to
local state (dirty bits cleared on the rows that committed) before batch
N+1 is built; otherwise batch N+1 would re-send the same rows.

### 6.4 Why four cursors, not one or eight

Per-layer cursors give the FK-closure guarantee on apply (§4.4): each
layer-K page applies after layers 0..K-1 are fully on the client. A
single cursor across all eight tables can't do this (same FK pathology
as §3.4's push case); per-table cursors give nothing extra because the
topological layers (§3.4.1) have no intra-layer FKs.

## 7. Dirty-bit lifecycle

### 7.1 The dirty bit

Per §9 (canonical column inventory), every entity table has two
local-only columns:

- `local_dirty integer not null default 0` — `1` iff the row needs
  pushing.
- `local_updated_at_ms integer not null default 0` — the row's
  client-monotonic timestamp (§8). Same value the wire envelope's
  `client_updated_at_ms` carries on push.

Neither column ever appears on the wire.

A row with `local_dirty = 0` is in sync with the server: the only ways
to flip dirty → 0 are a successful push ack or a pull apply, both of
which mean the server holds the row. So `local_dirty = 0` ⇒ server has
the row.

### 7.2 Setting and clearing — full table

| Event | `local_dirty` | `local_updated_at_ms` |
|---|---|---|
| Repo write (create / update / softDelete / cascade) | `1` | `nowMonotonic()` |
| Pull apply (incoming wins, §4.5 step 2 or 3) | `0` | `incoming.client_updated_at_ms` |
| Pull apply (incoming loses, §4.5 step 4) | unchanged | unchanged |
| Push ack (no concurrent edit — `current == sent_at_ms`) | `0` | unchanged |
| Push ack (concurrent edit detected — `current != sent_at_ms`) | unchanged (stays `1`) | unchanged |
| Push error (`FK_VIOLATION`, transport, etc.) | unchanged | unchanged |

All repo write paths run inside a single Drizzle transaction that
updates the row AND sets the dirty bit. No "set dirty bit later" path
exists. This is the **single biggest contract repos must obey** — t1's
agent-reminder spec (t1 §8) calls it out.

### 7.3 Push-in-flight race

The race: while a row's push is in flight, the user edits the row again.
A naive ack-handler would clear the dirty bit, losing the new edit.

**Mechanism.** At request time the cycle captures
`Map<(type, id), sent_at_ms>` for every row in the batch, where
`sent_at_ms` is the value of `local_updated_at_ms` at the moment the
row was serialised. The map lives in cycle-local memory.

On receipt of the success ack, the cycle, in one Drizzle transaction
per row:

1. SELECT the row's current `local_updated_at_ms`.
2. If `current == sent_at_ms`: clear `local_dirty = 0`.
3. If `current != sent_at_ms` (in practice always `current > sent_at_ms`
   because the monotonicity guard never goes backwards): leave
   `local_dirty = 1`. Next cycle re-pushes the newer value.

If the suspension drops the in-memory map before the ack handler runs
(extraordinarily rare — both happen in microseconds inside the same
cycle), the rows stay dirty; the next cycle re-pushes and the server's
LWW handles the duplicate idempotently (§3.6).

## 8. Clock-monotonicity guard

### 8.1 The formula

```
local_updated_at_ms = max(Date.now(), last_emitted_ms + 1)
```

Every repo write call that bumps `local_updated_at_ms` calls a single
helper:

```ts
function nowMonotonic(): number {
  const wall = Date.now();
  const previous = cached_last_emitted ?? readPersisted();
  const next = Math.max(wall, previous + 1);
  cached_last_emitted = next;
  persistAsync(next);
  return next;
}
```

### 8.2 Location

A single helper function in `apps/mobile/src/data/clock.ts` (file name
chosen to be conspicuous — `clock` is the entire purpose of this
module). All repos import from this one file. The session-recorder and
the seed loader also use this helper. No other monotonic-time helper
exists.

### 8.3 Persistence across restarts

`last_emitted_ms` is stored as an integer column on the
`sync_runtime_state` singleton row.

**Persistence MUST be synchronous within the same SQLite transaction
as the row write that produced the value.** Concretely: every repo
write that bumps `local_updated_at_ms` on an entity row writes the new
value to `sync_runtime_state.last_emitted_ms` in the same
`BEGIN; … COMMIT;`. Fire-and-forget persistence is not safe — a
crash between row commit and counter persist would leave the counter
behind, so the next launch would emit a timestamp ≤ already-persisted
entity timestamps, and subsequent local edits would be silently
LWW-rejected by the server (since `incoming ≤ stored`).

The counter is device-global, not per-user. Any future
local-wipe-on-account-switch path (see §5) must leave `last_emitted_ms`
alone.

### 8.4 In-memory cache

A module-scoped variable mirrors the persisted value so tight loops
don't re-read SQLite per write. The cache is updated inside the same
transaction that persists the value (§8.3). Cold start reads from
SQLite.

## 9. Local-only sync-bookkeeping columns

Per t1 §4.1, t2 owns the canonical inventory of local-only sync columns
and registers them with the drift checker. This section pins it down.

### 9.1 The two columns

**Every entity client table** (all eight: `gyms`, `sessions`,
`session_exercises`, `exercise_sets`, `exercise_definitions`,
`exercise_muscle_mappings`, `exercise_tag_definitions`,
`session_exercise_tags`) gains these two columns in the t2 build wave's
local Drizzle migration:

| Column | Type | Default | Semantic |
|---|---|---|---|
| `local_dirty` | `integer` (SQLite-style boolean: 0/1) | `0` | `1` iff the row needs pushing. Set by every repo write path. Cleared by the push ack handler (§7). |
| `local_updated_at_ms` | `integer` (epoch ms) | `0` | The client-monotonic timestamp used as the LWW key. Sent on the wire as `client_updated_at_ms` (§2.1). Bumped by `nowMonotonic()` on every dirtying write (§8). |

Both columns are local-only; the push serialiser (§3.3) omits them; the
pull apply (§4.5) writes `local_dirty=0`,
`local_updated_at_ms=incoming.client_updated_at_ms`.

**Naming convention.** snake_case in SQLite (`local_dirty`,
`local_updated_at_ms`) and camelCase in Drizzle TypeScript (`localDirty`,
`localUpdatedAtMs`). Same as existing columns.

### 9.2 The pull cursors

Stored as a single JSON column on the existing `sync_runtime_state`
table (one row per device), holding **one cursor per topological layer**
(§4.4):

| Column | Type | Default | Semantic |
|---|---|---|---|
| `pull_cursor` | `jsonb` (or text holding JSON) | `{}` | Map keyed by layer index (`"0"`, `"1"`, `"2"`, `"3"`). Each entry is either absent / `null` (snapshot — start from the beginning of that layer) or the opaque cursor blob `{server_received_at, owner_user_id, type, id}` returned by the previous `sync_pull` of that layer. Each entry advances independently after that layer's page COMMIT (§4.7). |

Example value mid-drain (layers 0–1 fully drained; layer 2 partial;
layer 3 not yet started):

```json
{
  "0": { "server_received_at": "2026-05-23T10:00:00.000Z",
         "owner_user_id": "00000000-...", "type": "gyms", "id": "gym_a" },
  "1": { "server_received_at": "2026-05-23T10:05:11.123Z",
         "owner_user_id": "00000000-...", "type": "sessions",
         "id": "01HX..." },
  "2": { "server_received_at": "2026-05-23T10:14:32.871Z",
         "owner_user_id": "00000000-...", "type": "session_exercises",
         "id": "01HX5KAB..." }
}
```

`pull_cursor` is **not** per-entity. It's per-layer, per-device, stored
as one JSON value on the singleton `sync_runtime_state` row.

### 9.3 The monotonic-clock state

Also a column on `sync_runtime_state`:

| Column | Type | Default | Semantic |
|---|---|---|---|
| `last_emitted_ms` | `integer` | `0` | The largest `local_updated_at_ms` this device has emitted. Persisted by `nowMonotonic()`'s fire-and-forget write. See §8.3. |

### 9.4 Drift-checker registration

Per t1 §4.1, all eight entity tables' two local-only columns
must be listed in `apps/mobile/src/data/schema/sync-extras.json` under
`exemptions.local_only_columns`. The registration is global (not
per-entity), because the same two columns apply to all eight tables:

```jsonc
{
  "exemptions": {
    "local_only_columns": [
      "local_dirty",
      "local_updated_at_ms"
    ],
    "untyped_text_references": [
      // ... per t1 §4.1 ...
    ]
  }
}
```

The `sync_runtime_state` table is **not** in the drift checker's scope —
it has no server counterpart by design; it's a client-only singleton (the
drift checker scans only the eight user-owned entity tables, per t1 §7).
The singleton-state columns (`pull_cursor`, `last_emitted_ms`,
`bootstrap_completed_at`) live on that table and don't need any
registration.

### 9.5 Push serialiser exclusion

The push serialiser (§3.3) MUST NOT emit `local_dirty` or
`local_updated_at_ms` in `fields`. t1 §4.1 explicitly assigns this
responsibility to t2:

> "t2 is also responsible for ensuring the push serialiser does not emit
> these columns on the wire."

The build wave implements this by routing the serialiser through a
single function (`entityToWire`) per entity, which closes over the
typed server-column list from the t1 §2 mapping —
`local_only_columns` are not in that list, so they're not emitted.

## 10. Constraint reconciliation against t1 §11 and t4 §11

This subsection walks the two upstream §9 lists in order and points to
which section of this doc honours each constraint.

### 10.1 t1 §11 constraints

**t1 §11.1 Uniqueness invariants (client-enforced).**

| # | Constraint | Honoured in |
|---|---|---|
| 1 | Active `session_exercises` no duplicate `(session_id, order_index)`. | Repos serialise reorder writes through a single Drizzle transaction so all affected siblings dirty together and ship in the same push batch (§3.4 dirty-stream order). |
| 2 | Active `exercise_sets` no duplicate `(session_exercise_id, order_index)`. | Same as #1. |
| 3 | `exercise_muscle_mappings` no duplicate `(exercise_definition_id, muscle_group_id)`. | Client repo enforces on upsert (existing repo behaviour pre-v2). t2's protocol doesn't change this. |
| 4 | `exercise_tag_definitions` no duplicate active `(exercise_definition_id, normalized_name)`. | Same as #3. |
| 5 | `session_exercise_tags` no duplicate `(session_exercise_id, exercise_tag_definition_id)`. | Same as #3. |

t2's protocol enforces these by **not relaxing them** — the client's
existing uniqueness logic (in repos) is what the server trusts. The server
has no UNIQUE index for any of them; a duplicate landing on the server is
a client bug. The protocol simply doesn't reject it.

**t1 §11.2 Foreign-key closure on push batches.**

| # | Constraint | Honoured in |
|---|---|---|
| 6 | No orphan-child pushes. | **§3.4: the batch builder walks the eight tables in topological order (Layer 0..3 per §3.4.1). A dirty child's dirty parent is either in the same batch (deferred FKs at COMMIT) or in an earlier batch already committed to the server. Clean parents (`local_dirty = 0`) are already server-resident. No closure walk needed.** |

**t1 §11.3 LWW semantics that t2 implements.**

| # | Constraint | Honoured in |
|---|---|---|
| 10 | Row-level LWW on `client_updated_at_ms`. | §3.5 / §4.5 (server-side UPSERT and client-side apply both check the row-level predicate). |

### 10.2 t4 §11 constraints

**t4 §11.1 Cycle shape.**

| # | Constraint | Honoured in |
|---|---|---|
| 1 | Cycle drains until empty. | **§6.1 (pull → push → re-pull, layer-by-layer on the pull side); §6.2 ("Drain to empty").** The wire shape that makes drain possible is `has_more` + opaque per-layer `cursor` (§4.2). |
| 2 | Cycle duration target (20s background, ≤5s foreground soft). | §6.2 ("Cycle duration target" sub-bullet); §4.4 (batch cap of 200 chosen with this budget in mind). |

**t4 §11.2 Per-batch atomicity.**

| # | Constraint | Honoured in |
|---|---|---|
| 3 | Batched independent commits. | **§3.2 (each `sync_push` is one transaction); §4.5 (each pull page applies in one transaction; per-layer cursor advances after each commit); §6.2 ("Per-batch commits").** Independence is *across* pages/batches; within a page, all rows commit together. On any failure the cycle simply returns — the next scheduler tick starts a fresh cycle that re-pushes the same dirty rows (no backoff bookkeeping). |

**t4 §11.3 Idempotency under interruption.**

| # | Constraint | Honoured in |
|---|---|---|
| 4 | Push is idempotent. | **§3.6 (a re-pushed row with the same `client_updated_at_ms` is a server-side no-op under LWW; the success ack clears the dirty bit either way).** |
| 5 | Pull is idempotent by cursor. | **§4.5 / §4.6 (cursor advances only after page COMMIT; an interrupted apply rolls back and the next cycle re-fetches the same page).** |

**t4 §11.4 Surviving state.**

| # | Constraint | Honoured in |
|---|---|---|
| 6 | Only dirty bits + pull cursor survive suspension. | **§7.3; §9.** Concretely, the persisted state is: per-entity `local_dirty`, `local_updated_at_ms`, plus singleton `pull_cursor`, `last_emitted_ms`, `bootstrap_completed_at`. Everything else (cycle progress, in-flight batch metadata, the `Map<(type,id), sent_at_ms>` push-state) is in-memory and reconstructed on the next cycle from the dirty stream and the cursor. |
