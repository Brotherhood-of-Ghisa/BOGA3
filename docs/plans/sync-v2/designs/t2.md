# t2 — Sync v2 protocol detail design

> **Status:** design doc. A future builder implements this verbatim. No code, no
> migrations, no UI mocks beyond modal copy strings.
>
> **Companion design:** [t1.md](t1.md) (server schema) defines the per-entity
> column shape and the two new local columns (`local_dirty`, `local_updated_at_ms`)
> that this protocol uses. t2 references those names; t1 defines them.

## 0. Scope and key positions

This doc covers everything on the wire and in the client/server state machines for
v2. It does NOT cover seed lifecycle (see t3), iOS trigger reliability (see t4),
or the typed server schema itself (see t1).

The brainstorm took six positions. I confirm or override each one upfront so the
rest of the doc is unambiguous.

| Position from brainstorm | Verdict | Reasoning |
|---|---|---|
| Per-row push granularity (not per-aggregate) | **Confirm** | Aggregates encourage server-side referential reasoning we explicitly do not want. Deferrable FKs handle ordering inside a batch. |
| LWW on client clocks, with monotonicity guard | **Confirm** | Purpose #2 (two-device) is explicitly low-priority. Clock-skew lost writes are acceptable. The monotonicity guard prevents a much worse failure: a user typing into the same row faster than the device clock ticks, where two edits would tie and the wrong one could win. |
| Conflict UI modal only when **both** sides have data | **Confirm** | One-sided cases need no human input. Modal only on the genuinely ambiguous case. |
| "Has local edits" tracked by a single dirty bit per row | **Confirm** | But "has local data" (the question the initial-sync state machine asks) is a different question — see §6. |
| Migration off v1: clean cut, no carry-over | **Confirm** | The brainstorm already settled this; v2 ingest never reads v1 data. |
| `pull` cursor is server-time, not client-time | **Confirm and refine** | Cursor is `server_received_at` (timestamptz) with a `(owner_user_id, entity_type, id)` tiebreak. See §5. |

New position taken in this doc (not pre-committed by the brainstorm):

| Decision | Choice |
|---|---|
| Push batch failure mode (whole-batch reject vs partial accept) | **Whole-batch reject.** See §3. |
| Aggregate grouping (detached children allowed?) | **Detached children allowed.** Server has no FK enforcement that would reject a child whose parent isn't in the same push. Deferrable FKs cover the in-same-push case; for cross-push cases the child is buffered locally as orphaned-but-acked. See §4. |
| Per-row outcome reporting on success | **Echo `(type, id, server_updated_at_ms, applied)` for every row in the request, including LWW no-ops.** See §3.3. |
| Initial-sync "Replace local with cloud" — is the user's local data lost? | **Yes, permanently and explicitly.** Modal copy says so. The brainstorm referenced "Replace local with cloud" without specifying preservation; we choose hard wipe because anything else creates a fourth path (preserve-but-not-merge) that nobody understands. |

## 1. Endpoints

Two RPCs. Both exposed under the `app_public` schema as PostgREST RPCs. Both
require an authenticated user; `owner_user_id` is derived from the JWT, never
sent on the wire. Both are POSTs.

| RPC | Purpose |
|---|---|
| `sync.push` (`POST /rest/v1/rpc/sync_push`) | Client uploads a batch of dirty rows. Server upserts with LWW. |
| `sync.pull` (`POST /rest/v1/rpc/sync_pull`) | Client downloads server changes since a cursor. Returns up to `limit` rows ordered by `server_received_at`. |

No third "snapshot" RPC. Snapshot is `sync.pull` with `since = null`.

There is no event log on the wire. There are no sequence numbers, no device
identifiers, no batch ids. The data IS the event.

## 2. Wire types (shared)

### 2.1 `Entity` envelope

Every row crossing the wire (in either direction, push or pull) uses the same
shape:

```json
{
  "type": "exercise_set",
  "id": "01HXYZ...ULID",
  "local_updated_at_ms": 1733000000123,
  "deleted": false,
  "payload": {
    "session_exercise_id": "01HX...",
    "order_index": 0,
    "weight_value": "100",
    "reps_value": "8",
    "set_type": "rir_2",
    "created_at_ms": 1732999999000,
    "extras": {}
  }
}
```

Field rules:

| Field | Type | Notes |
|---|---|---|
| `type` | string enum | One of `gym`, `session`, `session_exercise`, `exercise_set`, `exercise_definition`, `exercise_muscle_mapping`, `exercise_tag_definition`, `session_exercise_tag`. Singular form. |
| `id` | string | Client-assigned. ULID for user-created rows, slug for seeds (`seed:bench-press`). Stable forever. |
| `local_updated_at_ms` | integer | Epoch ms from the originating client. **Authoritative for LWW.** Must be `>= 0`. See §7 for monotonicity. |
| `deleted` | boolean | `true` is a tombstone. When `true`, `payload` MAY be `{}` or MAY include the last-known fields (optional, server stores whatever is sent). |
| `payload` | object | The row body. Schema is per-entity, defined in t1.md. `payload.id` MAY be omitted (the envelope-level `id` is authoritative); if present, it MUST match. `payload.extras` (object) is forward-compat; unknown keys live there. |

The wire envelope deliberately mirrors the local row shape (no separate "event"
abstraction). When the client decides to push, it scans for rows with
`local_dirty = true` and converts each row directly to an `Entity` envelope.

### 2.2 Error shape

Both RPCs use the same error envelope when they reject the entire request:

```json
{
  "error": {
    "code": "VALIDATION_FAILED",
    "message": "exercise_sets[3].order_index: expected non-negative integer",
    "row_index": 3
  }
}
```

Error codes (free-text `message` is for humans; never parse it):

| `code` | Meaning | Client behavior |
|---|---|---|
| `AUTH_REQUIRED` | No JWT or expired | Trigger token refresh; if it fails, drop transport but keep dirty bits. |
| `VALIDATION_FAILED` | Request shape invalid (bad enum, missing required, type mismatch) | **Non-retryable.** Log and surface. See §3.4 for why and what to do. |
| `RATE_LIMITED` | Server is shedding load | Retry with backoff (see §8). |
| `INTERNAL` | Anything else | Retry with backoff. |

`row_index` is present on `VALIDATION_FAILED` whenever the failure is
attributable to a specific entry in the request `entities` array; otherwise
omitted.

## 3. `sync.push`

### 3.1 Request

```json
{
  "entities": [
    {
      "type": "exercise_definition",
      "id": "seed:bench-press",
      "local_updated_at_ms": 1733000000100,
      "deleted": false,
      "payload": {
        "name": "Bench Press",
        "created_at_ms": 1732999900000,
        "extras": {}
      }
    },
    {
      "type": "session",
      "id": "01HX5K9V...",
      "local_updated_at_ms": 1733000000200,
      "deleted": false,
      "payload": {
        "gym_id": "gym_a",
        "status": "active",
        "started_at_ms": 1733000000000,
        "completed_at_ms": null,
        "duration_sec": null,
        "deleted_at_ms": null,
        "created_at_ms": 1733000000000,
        "extras": {}
      }
    },
    {
      "type": "session_exercise",
      "id": "01HX5KAB...",
      "local_updated_at_ms": 1733000000201,
      "deleted": false,
      "payload": {
        "session_id": "01HX5K9V...",
        "exercise_definition_id": "seed:bench-press",
        "order_index": 0,
        "name": "Bench Press",
        "machine_name": null,
        "created_at_ms": 1733000000000,
        "extras": {}
      }
    }
  ]
}
```

| Field | Validation |
|---|---|
| `entities` | Array, length `1..200`. v2 batch size limit is 200 (v1 was 100; we relax because no per-event work happens — it's bulk upsert). |

No `device_id`. No `batch_id`. No `sent_at_ms`. The request is idempotent by
content (see §3.6).

### 3.2 Ordering inside the batch

**Push entities go in arbitrary order from the client's perspective.** The
client SHOULD emit them in the order they were dirtied (FIFO from the dirty
scan), but the server MUST NOT depend on this.

The server makes the batch commit atomically using a single transaction with
`SET CONSTRAINTS ALL DEFERRED` at the top. Per t1.md, the seven FKs that
matter (`sessions.gym_id`, `session_exercises.session_id`,
`session_exercises.exercise_definition_id`, `exercise_sets.session_exercise_id`,
`exercise_muscle_mappings.exercise_definition_id`,
`exercise_tag_definitions.exercise_definition_id`,
`session_exercise_tags.session_exercise_id`,
`session_exercise_tags.exercise_tag_definition_id`) are declared
`DEFERRABLE INITIALLY DEFERRED`. So a single push transaction can contain a
session + its child session_exercise + its child set in any order; FK checks
run at commit and pass as long as all parents are present (either previously
committed or in this batch).

If a child references a parent that exists neither in the batch nor in the
database, the deferred FK fails at commit → the **whole batch rejects**.

### 3.3 Success response

```json
{
  "results": [
    {
      "type": "exercise_definition",
      "id": "seed:bench-press",
      "outcome": "applied",
      "server_updated_at_ms": 1733000000100
    },
    {
      "type": "session",
      "id": "01HX5K9V...",
      "outcome": "applied",
      "server_updated_at_ms": 1733000000200
    },
    {
      "type": "session_exercise",
      "id": "01HX5KAB...",
      "outcome": "lww_skipped",
      "server_updated_at_ms": 1733000000999
    }
  ]
}
```

`results` length == request `entities` length, in the same order.

| `outcome` | Meaning |
|---|---|
| `applied` | Server stored the row. `server_updated_at_ms` echoes the request's `local_updated_at_ms`. |
| `lww_skipped` | Server already had a row with `local_updated_at_ms >=` the request's value. No write happened. `server_updated_at_ms` is the existing stored value. |

**Both outcomes mean the client clears `local_dirty` for that row.** A
`lww_skipped` is not a failure from the client's perspective — the server's
state already reflects something newer or equal, so the dirty bit has served
its purpose. The client also adopts the server's `server_updated_at_ms` into
its own `local_updated_at_ms` if the server's value is newer (so the next push
won't re-send).

### 3.4 Failure response (whole-batch reject)

If any row in the batch fails server-side validation (bad shape, FK at commit,
RLS-impossible value, etc.), the entire batch transaction rolls back and the
server returns the error envelope (§2.2). No rows are stored.

```json
{
  "error": {
    "code": "VALIDATION_FAILED",
    "message": "session_exercise.01HX5KAB...: session_id references unknown session 01HX5K9V...",
    "row_index": 2
  }
}
```

**Why whole-batch reject and not partial accept:**

1. Partial-accept needs per-row error indices (`error_index` in v1, plus
   `should_retry`) and a prefix-commit contract. v1 had this; it interacts badly
   with the dirty-bit model — the client has to clear N bits where N is some
   prefix, and the rest stay dirty. Pretty quickly you reproduce the v1 outbox
   contract.
2. With deferrable FKs, a corrupt child row genuinely poisons the batch
   regardless — partial accept would have to drop the bad child AND its
   transitively-dependent rows.
3. The only common cause of per-row failure in practice is FK-to-missing-parent,
   which the client can detect and re-batch on retry (see §4).

**Client behavior on `VALIDATION_FAILED`:**

- The dirty bits stay set (nothing was acked).
- Log the error (`row_index` + `message`) for observability. Surface to user
  via the sync status panel ("Action required").
- Do **not** auto-retry. v1 retried these forever. The next manual sync
  (or app-restart-bumped attempt) will retry; in the meantime the user can
  inspect.
- Exception: if the failure is FK-related and the parent exists locally but
  wasn't in the batch (e.g., parent wasn't dirty), the client SHOULD add the
  parent to the next push attempt. The mechanism: after a VALIDATION_FAILED
  with a `row_index` that points at a child row, mark that child's parent
  dirty as well (regardless of whether its content actually changed) so the
  next push includes it. This is the "detached child rescue" path; see §4.

### 3.5 Ordering and uniqueness of bits in the transaction

The server processes the batch in a single transaction, in arbitrary order
(implementation MAY use `INSERT ... ON CONFLICT (owner_user_id, id) DO UPDATE
... WHERE incoming.local_updated_at_ms > stored.local_updated_at_ms` and rely
on `RETURNING` to populate the results array). LWW is enforced by the
predicate in the `DO UPDATE WHERE` clause; a stale row produces an `INSERT`
no-op (LWW lost) and is reported as `lww_skipped`.

If the same `(type, id)` appears twice in a single batch request, the server
MUST reject with `code: VALIDATION_FAILED` and `message` indicating the
duplicate. (Client must dedup before push; this is a programming error.)

### 3.6 Idempotency

Re-sending the identical request produces the identical response. There is no
event log to dedup against; the LWW predicate makes replay safe. If a request
times out, the client retries the entire request unmodified after backoff.

### 3.7 Tombstones

A push entry with `deleted: true` upserts a tombstone row. The tombstone has the
same `local_updated_at_ms` LWW semantics as any other row — if the stored row
is newer than the incoming tombstone, the tombstone is `lww_skipped` (i.e., the
delete is lost; this is the correct LWW behavior). `payload` MAY be `{}` or MAY
echo the last-known fields; per t1.md the server stores whatever is sent
verbatim. Tombstones are permanent (no hard purge in v2; future maintenance
job).

## 4. Aggregate grouping rule

**Detached children are acceptable.** A push that contains a set whose session
is not in this push (and not yet on the server) is valid only if the server
already has the parent session. Otherwise the deferred FK fails at commit and
the batch rejects (§3.4).

This is the **detached-child rescue path**:

1. User adds a set to a session that hasn't yet been pushed.
2. Client dirty-scans; both rows are dirty.
3. First push attempt includes both (because both are dirty). FK passes.

The pathological case requires a bug elsewhere: a child gets dirtied without
its parent ever having been pushed. The mechanism in §3.4 handles it — on FK
failure, mark the parent dirty and retry.

**The client SHOULD NOT pre-validate FKs.** It is a "send what's dirty, let
the server reject if it's broken" model. The cost of a stricter client
algorithm (build a dependency graph, walk it) is high and the failure mode
is rare and self-healing.

## 5. `sync.pull`

### 5.1 Request

```json
{
  "since": {
    "server_received_at_iso": "2026-05-23T10:14:32.871Z",
    "owner_user_id": "00000000-0000-0000-0000-000000000aaa",
    "type": "session",
    "id": "01HX5K9V..."
  },
  "limit": 500
}
```

| Field | Type | Notes |
|---|---|---|
| `since` | object or `null` | Opaque cursor produced by a previous pull response. `null` for snapshot/initial sync. |
| `since.server_received_at_iso` | string | The server's timestamp on the **last row received** in the previous response. |
| `since.owner_user_id` | string | UUID. Always equals `auth.uid()` server-side, but echoed for tiebreak comparison. |
| `since.type` | string enum | Last row's type. |
| `since.id` | string | Last row's id. |
| `limit` | integer | `1..1000`. Default 500. |

The cursor is opaque to the client — it stores what the server hands back and
returns it on the next call. The shape above is for designer reference; the
client treats the whole object as a blob it round-trips.

### 5.2 Response

```json
{
  "entities": [
    {
      "type": "session",
      "id": "01HX5K9V...",
      "local_updated_at_ms": 1733000000200,
      "deleted": false,
      "payload": { "...": "..." }
    },
    {
      "type": "session_exercise",
      "id": "01HX5KAB...",
      "local_updated_at_ms": 1733000000201,
      "deleted": false,
      "payload": { "...": "..." }
    }
  ],
  "next_cursor": {
    "server_received_at_iso": "2026-05-23T10:14:33.000Z",
    "owner_user_id": "00000000-0000-0000-0000-000000000aaa",
    "type": "session_exercise",
    "id": "01HX5KAB..."
  },
  "has_more": true
}
```

`entities` is the page. When `has_more` is `true`, the client SHOULD immediately
issue another `sync.pull` with `since = next_cursor` until `has_more` is
`false`. Pull is fast enough that we don't need to interleave anything.

When the response has `entities.length === 0`, `has_more` is `false` and
`next_cursor` echoes the input `since` (the client persists it unchanged).

### 5.3 Pagination, monotonicity, and tiebreak

The server orders rows by:

```
ORDER BY server_received_at ASC,
         owner_user_id ASC,
         type ASC,
         id ASC
LIMIT :limit
```

with the WHERE clause:

```sql
WHERE owner_user_id = auth.uid()
  AND (
    :since IS NULL
    OR (server_received_at, owner_user_id, type, id)
       > (:since.server_received_at, :since.owner_user_id, :since.type, :since.id)
  )
```

**Why `server_received_at` is monotonic:** `server_received_at` is set by the
server (`default now()` on push, or `now()` in any background re-projection
job) at the moment the row is written. Two rows written in the same transaction
share a `now()` value (timestamps are stable inside a transaction in
Postgres) — that's the same-millisecond case below.

**Same-millisecond rows:** rows with identical `server_received_at` are
disambiguated by `(owner_user_id, type, id)` lexicographic order. Since the
cursor includes all four components and the WHERE clause uses the
row-value-comparison `>` (strict), we always advance past the last seen row,
never repeat it, never miss one.

**Why the cursor includes `owner_user_id`:** within a single user's pull, all
rows share `owner_user_id` so it's strictly redundant. But it future-proofs
group/multi-owner pulls (purpose #5) where an admin pulls across several
owners — and the cost of carrying a UUID through the cursor is nil.

**Clock skew across server replicas:** Supabase Postgres is single-primary,
so `now()` is monotonic across all writes from a single client's perspective.
We do not need a "high watermark" sentinel — a row with a `server_received_at`
earlier than the cursor's high water mark cannot appear because writes are
serialized.

### 5.4 Tombstones in pull responses

A tombstone row is just an `Entity` with `deleted: true`. The client applies
it by:

1. Setting the local row's tombstone column (per-entity, e.g., `deleted_at`
   on `exercise_definitions` per t1.md) to a wall-clock value derived from
   `local_updated_at_ms`.
2. Setting `local_updated_at_ms` to the server's value.
3. Leaving `local_dirty = false` (this is an inbound apply, not a local edit).

The local repo queries MUST treat `deleted_at IS NOT NULL` as
"row does not exist for product purposes" — same convention as v1.

### 5.5 Pull and concurrent local edits

If the client is mid-pull and a local mutation arrives for a row that's in the
pull page, the rule is:

- **The local edit wins by setting `local_dirty = true` and bumping
  `local_updated_at_ms`.**
- The pull apply step uses LWW locally too: compare incoming
  `local_updated_at_ms` to the row's current `local_updated_at_ms`. If
  incoming is `<=` current, the pull apply is a local no-op. The dirty bit
  stays.

This is symmetric to server-side LWW. The next push attempt will re-emit the
locally-newer value to the server.

## 6. Initial-sync state machine

This is the decision tree run **after authenticated sign-in and before the
scheduler is started**. It runs exactly once per (user, device) pair under
normal operation — once `bootstrap_completed_at` is set, it never re-runs for
that user on that device.

### 6.1 "Has local data" definition

A device "has local data" iff **any of the eight entity tables has at least
one row whose `local_dirty = true`**.

Reasoning, in priority order:

1. We're asking: *is there any local content that, if we naively snapshot the
   server, would be lost?* The answer is "yes" iff something is locally newer
   than the server.
2. `local_dirty = true` precisely captures "this row has not been acked by the
   server." Whether it was user-created, user-edited, or a seed write that
   never got pushed (the most common case for "remote-only" branches — see
   below), the same flag covers all three.
3. Alternative definitions and why they're worse:
   - `count(*) > 0`: a fresh install with seeds loaded but never pushed would
     trip "has local data" and prompt the modal even though there's nothing to
     preserve (the seeds will re-seed identically on the other device too).
   - "user-created rows exist": requires distinguishing user vs seed, which
     v2 explicitly refuses (t3.md). Dead end.
   - A separate boolean flag set by the first non-seed write: works but
     duplicates information already in `local_dirty`.

Edge case: seeds. Seeds are loaded on first install (per t3.md). Loading a seed
sets `local_dirty = true` because the seed needs to push to the server (seeds
are per-user data). So a "fresh install that has only loaded seeds" technically
*has local data* by this definition.

This is fine because:

- If the user is fresh too (remote empty), branch A ("empty/empty") absorbs
  it.
- If remote has data, the seed dirty bits are mostly harmless duplicates —
  the server's LWW will accept or reject them in the next push regardless of
  which initial-sync branch we took. The modal still asks the right question
  ("you have local content — keep it or replace?") and the answer "keep
  local and merge" will let LWW resolve it correctly.

### 6.2 "Has remote data" definition

The bootstrap pulls one page (`limit: 1`, `since: null`). If
`entities.length === 1` OR `has_more === true`, the user has remote data.

This costs one tiny RPC. We do not need a separate "is remote empty" check.

### 6.3 The four branches

```
                     remote_empty            remote_has_data
                  ┌───────────────────────┬──────────────────────┐
local_empty       │  A. empty/empty       │  C. remote-only      │
                  │  → done                │  → auto snapshot     │
                  ├───────────────────────┼──────────────────────┤
local_has_data    │  B. local-only        │  D. both-have-data   │
                  │  → auto push           │  → MODAL             │
                  └───────────────────────┴──────────────────────┘
```

Note: "local_empty" here means `local_dirty = true` count == 0. "local_has_data"
means `local_dirty = true` count > 0.

#### Branch A — empty/empty

**Trigger:** zero dirty rows locally, zero rows in remote pull(limit=1).

**Action:**
1. Run the seed loader (t3.md decides whether anything actually writes). If
   the seed loader writes anything, those writes will be `local_dirty = true`
   and the next scheduled push will sync them. No modal.
2. Set `bootstrap_completed_at = now()`.
3. Start the scheduler.

**Walk:** User installs the app, signs in. Server has nothing. Seeds load
locally. Bootstrap completes. Scheduler starts, pushes the seeds in the next
30s window, server stores them.

**Modal:** none.

#### Branch B — local-only

**Trigger:** at least one row with `local_dirty = true`, zero rows in remote
pull(limit=1).

**Action:**
1. Run the seed loader (per t3.md).
2. Push **all** dirty rows in batches of up to 200 until the dirty count is
   0. (Bootstrap convergence loop, similar to v1's `flushSyncOutboxUntilSettled`
   but driven by the dirty bit, not an outbox.)
3. Set `bootstrap_completed_at = now()`.
4. Start the scheduler.

**Walk:** User opens an app that's been used offline for a week. Sessions and
sets are sitting locally with `local_dirty = true`. They sign in for the first
time. The server is empty for them. Everything pushes in 2–3 batches. Bootstrap
completes.

**Modal:** none. (The user's data didn't conflict with anything; pushing is the
obvious, non-destructive action.)

#### Branch C — remote-only

**Trigger:** zero dirty rows locally, at least one row in remote
pull(limit=1).

**Action:**
1. (Do not run the seed loader yet — defer to step 4. Reason: the server may
   already have seed rows for this user from a prior install, and we don't
   want to double-write them.)
2. Pull all pages until `has_more = false`, applying each `Entity` to local
   tables. Persist `next_cursor` after each page.
3. After full pull: run the seed loader. t3.md handles the "seed already
   present locally because we just pulled it" case as a no-op.
4. Set `bootstrap_completed_at = now()`.
5. Start the scheduler.

**Walk:** User installed the app on device 2 (or reinstalled on device 1).
Server has everything. Local is bare. Snapshot pull drains all rows in 5–10
pages of 500. Seed loader sees seeds already present, does nothing. Bootstrap
completes.

**Modal:** none. (No local content to preserve; the action is unambiguous.)

#### Branch D — both-have-data

**Trigger:** at least one row with `local_dirty = true` AND at least one row
in remote pull(limit=1).

**Action:**
1. **Show the modal.** Block on user choice. The app is unusable until they
   choose (the modal is presented above the home screen with no dismiss
   gesture; the only way out is the two buttons). The choice persists across
   app restarts — if the user kills the app mid-decision, the modal re-appears
   on next launch. Implementation note for the future builder: the
   `bootstrap_completed_at` column staying null is sufficient state to
   re-trigger on next launch.
2. On choice **"Keep local"**:
   a. Pull all pages. For each incoming `Entity`, apply with LWW (compare
      incoming `local_updated_at_ms` to local `local_updated_at_ms`; the
      newer wins; ties resolved by adopting incoming).
   b. After full pull, push all dirty rows until dirty count is 0.
   c. Run the seed loader (defensive no-op if seeds present).
   d. Set `bootstrap_completed_at = now()`. Start scheduler.

   Net effect: union of local and remote, with LWW deciding per-row collisions.

3. On choice **"Replace local"**:
   a. Wipe all eight local entity tables (`DELETE FROM <table>` for each).
   b. Reset `local_dirty = false` everywhere (vacuously true after wipe).
   c. Reset the pull cursor to `null`.
   d. Pull all pages and apply (no LWW conflict possible; local is empty).
   e. Run the seed loader (handles "seed already pulled" case).
   f. Set `bootstrap_completed_at = now()`. Start scheduler.

   Net effect: local is reset to the server's view. The user's
   previously-dirty rows are **gone**.

**Modal copy (exact strings — future builder copies verbatim):**

```
Title:    Cloud data found
Body:     This device has unsynced changes, and there is also workout data
          in the cloud from another device or a previous install.

          What would you like to do?

Button 1: Keep this device's changes
          (We will merge them with the cloud data. Where two changes
          conflict, the most recent one wins.)

Button 2: Use cloud data instead
          (This device's unsynced changes will be discarded. This cannot
          be undone.)
```

**Walk D-1 (Keep local):** User has been editing on device A offline. Earlier
they synced on device B. They sign into device A while online. Modal appears.
They tap "Keep this device's changes." Bootstrap pulls device B's data
(applying with LWW), then pushes device A's pending edits. Device A now has
the union; device B will see device A's updates on its next pull.

**Walk D-2 (Replace local):** User accidentally created garbage data on device
A while signed out. They sign in. Modal appears. They tap "Use cloud data
instead." Local wipes, server snapshot replays. Garbage is gone.

### 6.4 Bootstrap failure handling

If any RPC inside the bootstrap fails:

- Transport/timeout/network: retry with backoff (per §8), up to a hard
  cap of 50 attempts (matches v1 `BOOTSTRAP_CONVERGENCE_MAX_ATTEMPTS`). On
  cap, surface "Initial sync did not complete" in profile status; user must
  toggle sync off and on (or force-sync) to retry.
- `VALIDATION_FAILED`: the bootstrap stops, `bootstrap_completed_at` stays
  null, error message goes to profile status. Manual retry only.

The bootstrap MUST be transactional at the wall-clock level: until
`bootstrap_completed_at` is set, the scheduler does not run (the scheduler
checks it before each tick). This prevents the v1 "scheduler races
bootstrap" bug from re-emerging.

## 7. Dirty-bit lifecycle and clock-monotonicity guard

### 7.1 The dirty bit

Per t1.md, every local entity table has:

- `local_dirty boolean not null default false`
- `local_updated_at_ms integer not null` (epoch ms; mode `timestamp_ms` in Drizzle)

**Set:** every write path in every repo MUST set `local_dirty = true` and bump
`local_updated_at_ms` (using the monotonicity guard, §7.3). Specifically:

| Repo write path | Action |
|---|---|
| `create*` (new row) | `local_dirty = true`, `local_updated_at_ms = now_monotonic()` |
| `update*` | `local_dirty = true`, `local_updated_at_ms = now_monotonic()` |
| `soft-delete*` (set `deleted_at`) | `local_dirty = true`, `local_updated_at_ms = now_monotonic()` |
| Repo internal mutation via foreign-key cascade (e.g., reorder siblings) | same as above for each affected row |
| Seed loader, on first install | `local_dirty = true`, `local_updated_at_ms = now_monotonic()` |
| Pull apply (inbound) | `local_dirty = false`, `local_updated_at_ms = entity.local_updated_at_ms` |
| Push ack (outbound success) | `local_dirty = false` |
| Push LWW-skipped (server's value was newer) | `local_dirty = false`, `local_updated_at_ms = max(local, server_updated_at_ms)` |

All repo write paths run inside a single Drizzle transaction that updates the
row AND sets the dirty bit. No "set dirty bit later" path exists. This is the
single biggest contract repos must obey; t1.md should call it out in the
agent-reminder section.

### 7.2 Push-in-flight mutations

The race we must handle:

```
T0: Row R is dirty, push batch picks it up (snapshot of its current value).
T1: Push request leaves the device.
T2: User edits R again locally. Row R is now dirty again with a newer
    local_updated_at_ms. (The repo doesn't know a push is in flight; it just
    does its job.)
T3: Server commits the T0 value.
T4: Server response arrives. Client sees R was "applied".
```

If the client naively clears `local_dirty = false` on R, the T2 edit is lost.

**Mechanism (must-have):** the push routine snapshots `(id, local_updated_at_ms)`
for every row it sends. On response, it clears `local_dirty` **only if the
row's current `local_updated_at_ms` is still equal to the snapshotted value.**
If the row has been edited since (newer `local_updated_at_ms`), the dirty bit
stays set; the next push will pick up the newer value.

Wire-level shape:

- The push REQUEST doesn't need to carry the snapshotted value as a separate
  field — it's already in `local_updated_at_ms`.
- The push RESPONSE already carries the same value back in `server_updated_at_ms`
  (it's what the server stored, which is what the client sent).
- So the client compares `result.server_updated_at_ms` to the row's current
  `local_updated_at_ms`. Equal → clear dirty. Different → leave dirty.

(For `lww_skipped` results, the server's `server_updated_at_ms` is the
stored value which is `>=` what we sent. The client compares to what it
SENT, not what's stored. The push routine MUST remember the per-row sent
value for this comparison. The simplest implementation: capture the array
of sent values at request time and pair them with the response results
array.)

### 7.3 Clock-monotonicity guard

```
local_updated_at_ms = max(Date.now(), last_emitted_ms + 1)
```

**Where it lives:** a single helper function in the data layer (e.g.,
`apps/mobile/src/data/clock.ts`), called by every repo write path. The
helper takes no arguments and returns the next monotonic timestamp.

**Persistence:** `last_emitted_ms` must survive app restarts (the device
clock could go backwards between launches — NTP correction, manual time
change, etc.). Persist it as a single row in `sync_runtime_state` (the
existing singleton table per t1.md, adding one column `last_emitted_ms
integer not null default 0`).

The helper:

```
let cached_last_emitted: number | null = null;
function nowMonotonic(): number {
  const wall = Date.now();
  const previous = cached_last_emitted ?? readPersisted();
  const next = Math.max(wall, previous + 1);
  cached_last_emitted = next;
  persistAsync(next);  // fire-and-forget; on next launch we read from disk
  return next;
}
```

Notes:

- The in-memory cache means tight loops (e.g., 50 row writes in the same
  transaction) don't hit storage 50 times.
- `persistAsync` is fire-and-forget: if the process dies before the write
  hits disk, the next launch reads a slightly-stale value, but that's
  fine — the worst case is one wasted millisecond of monotonic headroom.
- On user-switch (sign out / sign in), the helper does NOT reset. The
  guard is device-global, not per-user. Two users on the same device share
  the monotonic counter, which is exactly correct.

**Why this lives on the client, not the server:** LWW resolution happens
server-side, but the timestamps it compares are client-generated. The
guard is purely about preventing two timestamps from this device from being
equal (which would let LWW pick arbitrarily). The server doesn't need to
know.

## 8. Retry and backoff

### 8.1 Push retry schedule

Reuse the v1 constants where they're sensible (they were never the problem):

| Parameter | Value |
|---|---|
| Initial delay | 2 seconds |
| Multiplier | 2x |
| Cap | 2 minutes |
| Jitter | ±25% (uniform) |
| Reset | on a successful push (any `success` response) OR on user-toggle of sync OR on user-initiated "Sync now" |

After a failed push, the next retry happens at
`now + 2s * 2^(failures-1) * (1 ± 0.25)` clamped to 2 minutes max.

### 8.2 Pull retry schedule

Same schedule. Pull failures during the bootstrap (Branch C/D first pull) count
against the bootstrap's 50-attempt cap (§6.4). Pull failures during steady
state (post-bootstrap, scheduled pulls) reset the bootstrap attempt count and
follow the regular schedule.

### 8.3 Offline mid-push

If the device goes offline while a push request is in flight:

- The HTTP layer reports a transport failure. The dirty bits remain set
  (nothing was acked).
- The engine schedules the next retry per §8.1.
- When the network comes back (network-online event from the scheduler, per
  t4.md), the next retry happens immediately (the engine pulls the
  next-attempt-at into the past on online-event, triggering an immediate
  flush).

### 8.4 Non-retryable failures

`VALIDATION_FAILED` is non-retryable automatically. The dirty bits stay set,
but no retry is scheduled. The next time the user (or app) explicitly
triggers a sync (toggle off/on, "Sync now" button, app launch), the engine
attempts again.

### 8.5 Backoff state reset

Backoff state lives in the runtime singleton (per t1.md's
`sync_runtime_state`):

- `consecutive_failures int` — number of consecutive failures
- `next_attempt_at timestamptz` — when the next push may run
- `last_error_message text` — surfaced in profile-status

Reset to (0, null, null) on:
- A successful push
- Sync toggled off then on
- User taps "Sync now"
- User signs out (state cleared as part of bootstrap reset for the new user)

## 9. Tombstones — end-to-end recap

This section consolidates tombstone behavior, scattered above, into one
checklist for the builder.

| Event | Behavior |
|---|---|
| User soft-deletes a row locally | Repo sets `deleted_at = now()`, `local_dirty = true`, `local_updated_at_ms = nowMonotonic()`. Row stays in table. |
| Local readers query rows | All queries filter `deleted_at IS NULL`. |
| Client pushes tombstone | `Entity { deleted: true, payload: { ...last known fields, deleted_at_ms }, local_updated_at_ms }`. |
| Server applies tombstone | Standard LWW upsert. The row's `deleted` boolean column becomes `true`. Tombstone stays in `app_public.<entity>` (no purge). |
| Server returns tombstone in pull | Standard `Entity { deleted: true, ... }`. |
| Client applies inbound tombstone | LWW comparison; if incoming wins, set local `deleted_at` to a wall-clock time derived from the entity's `local_updated_at_ms`, update `local_updated_at_ms`, leave `local_dirty = false`. |
| Undelete (resurrection) | Soft-delete is just `deleted_at IS NOT NULL`. To resurrect, repo sets `deleted_at = null`, dirties the row. Next push sends `deleted: false`. Server LWW applies (if the resurrect timestamp is newer than the tombstone — which it will be, given monotonicity). |

**The v1 "tombstone for missing row" idempotency case** (delete a row that the
server doesn't have): in v2, the server just stores the tombstone as a new row.
Idempotent by construction; no special-case logic.

  │         ◀── { entities: [...500...], next_cursor: C1, has_more: true }
  │       Apply all 500 to local (dirty stays false)
  │     - sync.pull { since: C1, limit: 500 }
  │         ◀── { entities: [...500...], next_cursor: C2, has_more: true }

### 10.2 Two devices both online editing the same row

```
Device A and Device B both signed in as user U.
Both have completed bootstrap. Both syncing every 60s.

Initial state: row R = { local_updated_at_ms: 1000, name: "Bench Press" }
On both devices (synced).

T = 100: User on Device A renames R → "My Bench A"
  A: nowMonotonic() = 1100 (wall is 1100)
  A: local row R = { local_updated_at_ms: 1100, name: "My Bench A",
                     local_dirty: true }

T = 150: User on Device B renames R → "My Bench B"
  B: nowMonotonic() = 1150 (wall is 1150)
  B: local row R = { local_updated_at_ms: 1150, name: "My Bench B",
                     local_dirty: true }

T = 200: Device A's scheduler tick fires push
  A → server: push { entities: [R @ 1100] }
  Server: stored R has local_updated_at_ms = 1000; incoming 1100 > 1000 → APPLY
  Server: row R = { local_updated_at_ms: 1100, name: "My Bench A" }
        ◀── { results: [{ id: R, outcome: "applied",
                          server_updated_at_ms: 1100 }] }
  A: row R's current local_updated_at_ms is still 1100 (no further edits)
     → clear dirty. R = { local_updated_at_ms: 1100, name: "My Bench A",
                          local_dirty: false }

T = 220: Device B's scheduler tick fires push
  B → server: push { entities: [R @ 1150] }
  Server: stored 1100; incoming 1150 > 1100 → APPLY
  Server: row R = { local_updated_at_ms: 1150, name: "My Bench B" }
        ◀── { results: [{ id: R, outcome: "applied",
                          server_updated_at_ms: 1150 }] }
  B: dirty cleared.

T = 260: Device A's next scheduler tick fires pull
  A → server: pull { since: C_A, limit: 500 }
  Server: returns R as updated since C_A
        ◀── { entities: [R @ 1150 name="My Bench B"], next_cursor: C_A',
              has_more: false }
  A: incoming 1150 > local 1100 → APPLY (R isn't dirty on A; nothing to
                                          conflict)
  A: row R = { local_updated_at_ms: 1150, name: "My Bench B",
               local_dirty: false }

Final: both devices converged on "My Bench B". B's edit, at T=150, won
because it was later by client clock (monotonicity guard prevents ties).
A's earlier edit was overwritten — accepted LWW behavior for purpose #2.
```

**Edge case in this scenario:** what if A's push and B's push arrive at the
server in the opposite order (T=200 push from A but it's slower; T=220 push
from B arrives first)? Answer: LWW is determined by `local_updated_at_ms`, not
arrival order. B's push at 1150 lands first → stored 1150. A's push at 1100
arrives next → incoming 1100 < stored 1150 → `lww_skipped`. A clears its dirty
bit (the server already has a value at or newer than what A sent). Next pull
on A delivers B's value. Same convergence.
