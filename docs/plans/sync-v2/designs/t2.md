> Status: design doc only. A future builder implements this verbatim. No code,
> no migrations, no UI mocks beyond modal copy strings.
>
> **Restart note.** This is the second version of t2. The first draft (PR #56)
> was written before t1 and t4 landed and made several positions that t1
> reversed:
>
> - Whole-batch reject vs partial accept: the old draft chose whole-batch
>   reject because partial accept "reproduces the v1 outbox contract." This
>   doc keeps whole-batch reject but for different reasons (t1 §5 deferrable
>   FKs + t1 §11 no-orphan-child constraint + t4 §11 idempotency).

>   without its parent. **Reversed.** t1 §11 constraint #6 forbids it.

> - "Extras blob" — the old draft was vague on merge vs replace. t1 §11.4
>   pins it: client sends the **full** extras blob; server **replaces**.
> - Sync cycle had no formal shape. **t4 §11 owns it now.** This doc lays
>   the wire surface (pull cursor + `has_more`, per-row push outcomes)
>   that lets the t4-shaped cycle "drain to empty."
>

## 1. Endpoints

Two RPCs. Both exposed as PostgREST RPCs under the `app_public` schema.
Both require an authenticated user; `owner_user_id` is derived from the
JWT and is never sent on the wire (RLS enforces it, per t1 §6). Both are
POSTs.

| RPC | Purpose |
|---|---|
| `sync_push` (`POST /rest/v1/rpc/sync_push`) | Client uploads a batch of dirty rows. Server upserts with LWW inside a single transaction. |
| `sync_pull` (`POST /rest/v1/rpc/sync_pull`) | Client downloads server rows newer than a cursor. Returns up to `limit` rows ordered by `(server_received_at, owner_user_id, type, id)`. |

No third "snapshot" RPC. Snapshot is just `sync_pull` with `cursor = null`.

There is no event log on the wire. There are no sequence numbers, no
device identifiers, no batch ids, no `sent_at` clock. **The data is the
event.**

## 2. Wire types (shared)

### 2.1 `Entity` envelope

Every row crossing the wire — push request, push response (echoes), pull
response — uses the same shape. The envelope carries the LWW key and the
wire-level `deleted` flag; `fields` carries typed-promoted columns;
`extras` carries unknown / forward-compat keys.

```json
{
  "type": "exercise_sets",
  "id": "01HXYZ...ULID",
  "client_updated_at_ms": 1733000000123,
  "deleted": false,
  "fields": {
    "session_exercise_id": "01HX...",
    "order_index": 0,
    "weight_value": "100",
    "reps_value": "8",
    "set_type": "rir_2",
    "created_at": 1732999999000,
    "updated_at": 1733000000123,
    "deleted_at": null
  },
  "extras": {
    "rpe": 8.5
  }
}
```

Field rules:

| Field | Type | Notes |
|---|---|---|
| `type` | string enum | One of `gyms`, `sessions`, `session_exercises`, `exercise_sets`, `exercise_definitions`, `exercise_muscle_mappings`, `exercise_tag_definitions`, `session_exercise_tags`. **Plural, snake_case**, matching the t1 table names. |
| `id` | string | Client-assigned. ULID for user-created rows; slug (`seed:bench-press`) for seeds. Stable forever. |
| `client_updated_at_ms` | integer | Epoch ms from the originating client. **The LWW key.** Must be `>= 0`. Produced by the monotonicity guard (§9). |
| `deleted` | boolean | `true` = tombstone. Per t1 §1.1.3, **`fields` and `extras` still carry the row's last-known values when `deleted = true`** — tombstones preserve the full row so undelete-by-LWW is lossless. |
| `fields` | object | Per-entity typed columns. Keys are wire-canonical snake_case (matching the t1 §2 mapping tables). Values use the JSON types implied by the t1 types: `text` → string, `bigint` → integer, `numeric` → string, `boolean` → boolean, `timestamp_ms`-style → integer epoch ms, nullable → JSON null. The client MUST emit every typed column listed in t1 §2 for that entity, even when the column is nullable (emit JSON null). |
| `extras` | object | Forward-compat hatch. **Full blob, replace semantics** (t1 §11 #8). Every key MUST be registered in `apps/mobile/src/data/schema/sync-extras.json → <entity>.extras_only_columns` (t1 §3.3); the drift checker enforces this at PR time. The server does not enforce it. |

**No `owner_user_id` on the wire envelope.** It is derived from `auth.uid()`
in the RPC body. Pull responses also omit it (the caller already knows it
is theirs).

**No `payload` key.** The old PR-#56 draft used `payload` as a single bag.
We split into `fields` (typed) and `extras` (untyped) so the drift gate has
something to lint against: any key in `fields` must have a matching server
column (or an `extras_only_columns` entry that says "this lives under
extras now"), any key in `extras` must be an `extras_only_columns` entry.

### 2.2 Error shape

Both RPCs use the same outer error envelope when they reject the entire
request. Per t1 §1 ground rules ("no server-side validation unless the
server writes"), the failure surface is intentionally narrow.

```json
{
  "error": {
    "code": "FK_VIOLATION",
    "message": "session_exercises 01HX5KAB...: parent sessions(owner_user_id, id)=(<uid>, 01HX5K9V...) not found"
  }
}
```

| `code` | Cause | Client behaviour |
|---|---|---|
| `AUTH_REQUIRED` | No JWT, expired JWT, RLS denies the row. | Trigger token refresh; if it fails, keep dirty bits, surface "Sign in again." |
| `FK_VIOLATION` | Deferrable-FK check failed at COMMIT (t1 §5). This is **the only data-shape failure** that v2 surfaces, and it indicates a t2 batching bug per t1 §11 #6. | **Defense-in-depth path:** see §4.4. Dirty bits stay set. Log the constraint and the offending row to observability; the next cycle will re-batch with parents included. Not auto-retryable on a tight loop. |
| `RATE_LIMITED` | Server is shedding load. | Retry with backoff per §10. |
| `INTERNAL` | Anything else (transport, 5xx, unmarshalled error). | Retry with backoff per §10. |

**There is no `VALIDATION_FAILED`.** v1 had one; the previous PR-#56 draft
preserved it; t1's no-server-validation rule eliminates it. The server does
not look at enum values, numeric ranges, string lengths, or required-vs-null
fields beyond the structural ones (PK columns, `client_updated_at_ms`,
`deleted`, `extras`). Anything malformed is the client's bug to find;
nothing the server emits will tell the client about it because the server
won't notice.

Examples of "malformed" that the server cheerfully accepts:

- `fields.order_index = -3` — stored.
- `fields.status = "fluffy"` for a `sessions` row — stored.
- `fields.name = ""` — stored.
- `extras = { "totally_made_up_key": [1, 2, 3] }` — stored if it slipped
  the drift gate (it can't, but if it did, the server would not catch it).

The drift gate (t1 §7) catches drift at PR review time. The push RPC
catches PK / FK / RLS violations at runtime. Nothing else is checked.

## 3. `sync_push`

### 3.1 Request

```json
{
  "entities": [
    {
      "type": "exercise_definitions",
      "id": "seed:bench-press",
      "client_updated_at_ms": 1733000000100,
      "deleted": false,
      "fields": {
        "name": "Bench Press",
        "created_at": 1732999900000,
        "updated_at": 1733000000100,
        "deleted_at": null
      },
      "extras": {}
    },
    {
      "type": "sessions",
      "id": "01HX5K9V...",
      "client_updated_at_ms": 1733000000200,
      "deleted": false,
      "fields": {
        "gym_id": "gym_a",
        "status": "active",
        "started_at": 1733000000000,
        "completed_at": null,
        "duration_sec": null,
        "created_at": 1733000000000,
        "updated_at": 1733000000200,
        "deleted_at": null
      },
      "extras": {}
    },
    {
      "type": "session_exercises",
      "id": "01HX5KAB...",
      "client_updated_at_ms": 1733000000201,
      "deleted": false,
      "fields": {
        "session_id": "01HX5K9V...",
        "exercise_definition_id": "seed:bench-press",
        "order_index": 0,
        "name": "Bench Press",
        "machine_name": null,
        "created_at": 1733000000201,
        "updated_at": 1733000000201,
        "deleted_at": null
      },
      "extras": {}
    }
  ]
}
```

| Field | Validation (structural only) |
|---|---|
| `entities` | Array, length `1..200`. v2 batch cap is **200** rows. Larger batches are split client-side (§4). |

No `device_id`, no `batch_id`, no `sent_at_ms`. The request is idempotent
by content (§3.6).

### 3.2 Ordering inside the batch

**Push entities may be in any order from the client's perspective.** The
client MAY emit them in dirty-scan order (FIFO from the dirty cursor); the
server MUST NOT depend on order.

The server processes the batch in a **single transaction with all FKs
deferred** (t1 §5.1):

```sql
BEGIN;
SET CONSTRAINTS ALL DEFERRED;
-- one UPSERT per entity (could be a single SQL statement with VALUES if
-- we want; the implementation choice is the build wave's, not this doc's).
COMMIT;
```

At COMMIT, all deferrable FKs are checked. If every FK resolves (either to
a row inside this batch or to one already on the server from a prior
commit), the transaction commits and the server responds with per-row
results (§3.5).

If any FK is unsatisfied at COMMIT, the entire batch rolls back and the
server returns `FK_VIOLATION` (§2.2, §4.4).

**This is the only failure mode where the server reports anything other
than transport/auth/rate-limit errors.** The deferrable-FK mechanism is
specifically designed so that "child row in the same batch, parent row in
the same batch, any order" succeeds — t1 §5.3.

### 3.3 `fields` vs `extras` — split rule

Per t1 §3 and §11 #8 / #9, the split is:

1. For each typed-promoted server column on this entity (per t1 §2 mapping
   table), put its current local value under `fields.<wire_key>`. The
   wire_key is the snake_case Postgres column name from the mapping table.
   Nullable columns get a JSON null when the local value is nullish.
2. For each `extras_only_columns` entry registered for this entity in
   `sync-extras.json` whose local value is non-null, put it under
   `extras.<wire_key>`.
3. Every other client column is either (a) a local-only sync column
   (§12) — must not appear on the wire — or (b) drift, caught at PR time
   by the t1 §7 checker.

**The full `extras` blob is sent on every push.** No partial / merge
semantics; the server's UPSERT writes `extras = <incoming object>` verbatim
(t1 §11 #8). To delete an extras key, the client sends a payload that
omits the key.

### 3.4 Building the batch — the no-orphan-child rule

Per t1 §11 #6, the client MUST NOT push a child row whose parent is
neither in the same batch nor already on the server. The build wave
implements the rule with this algorithm; the design is fixed here so the
implementation is unambiguous:

**Algorithm `selectPushBatch(dirtyCursor, batchCap = 200) → Entity[]`:**

1. Read all rows with `local_dirty = true` across the eight entity tables,
   ordered by `local_updated_at_ms` ASC (FIFO oldest-dirty first). Call
   this the **dirty stream**.
2. Open an empty `batch` (array of Entity envelopes) and an empty
   `included_ids` set (`Set<(type, id)>`).
3. For each row `R` in the dirty stream until either the stream is
   exhausted or `batch.length >= batchCap`:
   1. Compute `R`'s **dirty ancestor closure**: every row reachable from `R`
      by following FK pointers up the parent-of relation, where the
      ancestor is itself either (a) `local_dirty = true` and not yet in
      `included_ids`, or (b) **does not exist on the server yet** — which
      the client knows via a separate `server_known` bit per row, set when
      a push response acks the row (§3.6, §8). The closure stops at
      ancestors that are clean AND server-known.
   2. If `batch.length + closure.size > batchCap`: stop including new
      ancestors. Drop `R` from this batch (it will be included in a later
      batch when its closure fits). Continue to the next `R`.
   3. Otherwise, add the closure to the batch in topological-irrelevant
      order (any order — FK deferral handles it) and add each
      `(type, id)` to `included_ids`.
4. Return `batch`.

**Closure precedent:** the FK relations are exactly the seven cross-entity
FKs in t1 §5.2. Walking the closure is cheap; the deepest path is
`exercise_sets → session_exercises → sessions → gyms` (length 4) or
`exercise_sets → session_exercises → exercise_definitions →
exercise_muscle_mappings → muscle_groups`, but `muscle_groups` is
client-only taxonomy (t1 §2.6.1), so the FK chain that crosses the wire
has depth ≤ 4.

**The `server_known` bit.** A new local-only column on every entity table
(§12.2 codifies it: `server_known integer not null default 0`). Set to `1`
on any successful push ack (`applied` or `lww_no_op` — see §3.5) for that
row, and on any inbound pull apply. The closure walk treats `server_known
= 0` as "the server does not yet have this row", which combined with
"clean (`local_dirty = 0`)" — an unlikely-but-possible state for a row
that was pulled but then locally untouched — narrows to its real meaning:
include this ancestor if it might be needed for FK closure.

> Why not always include the parent unconditionally? Because the steady-state
> case is many dirty children of clean, long-ago-pushed parents. Re-pushing
> the parent every time would inflate batches and produce a lot of
> `lww_no_op` responses for no gain.

### 3.5 Success response

```json
{
  "results": [
    {
      "type": "exercise_definitions",
      "id": "seed:bench-press",
      "outcome": "applied",
      "server_received_at": "2026-05-23T10:14:32.871Z"
    },
    {
      "type": "sessions",
      "id": "01HX5K9V...",
      "outcome": "applied",
      "server_received_at": "2026-05-23T10:14:32.871Z"
    },
    {
      "type": "session_exercises",
      "id": "01HX5KAB...",
      "outcome": "lww_no_op",
      "server_received_at": "2026-05-23T10:14:32.871Z"
    }
  ]
}
```

`results` length equals request `entities` length, in the same order.

| `outcome` | Meaning | Client action |
|---|---|---|
| `applied` | Server stored the row (UPSERT INSERT or UPDATE that the LWW predicate accepted). | Clear `local_dirty` if no concurrent edit (§8.2); set `server_known = 1`. |
| `lww_no_op` | Server already had a row with `client_updated_at_ms >= incoming.client_updated_at_ms`. No write happened (UPSERT branch where the LWW predicate rejected the update). | Clear `local_dirty` if no concurrent edit; set `server_known = 1`. The stored value is already at-or-newer than what the client sent, so the dirty bit has served its purpose. The next pull will deliver the newer stored value if the client wants it. |

**Both outcomes ack the dirty bit.** `lww_no_op` is not a failure from the
client's perspective; the server's state is acceptable.

The response field `server_received_at` is the server's wall-clock value
of `now()` at the moment of the upsert (single `now()` per transaction,
stable across all rows in the batch — Postgres guarantees this). It is
the cursor axis for pull (§5.3); the client does not have to use it for
anything but observability on push. It is **not** the LWW key (which
remains `client_updated_at_ms`).

### 3.6 Idempotency — t4 §11 #4

A retry of a previously-acked row returns `applied` (or `lww_no_op` —
whichever the server arrived at the first time), **not** an error. The
client clears the dirty bit either way. This is what makes "the same
cycle resumed after iOS suspension" safe: the same dirty rows re-pushed
twice converge to the same final state.

The server-side mechanism that makes this so:

- `INSERT ... ON CONFLICT (owner_user_id, id) DO UPDATE SET ... WHERE
  incoming.client_updated_at_ms > stored.client_updated_at_ms`.
- The first push: row didn't exist → INSERT, `applied`.
- The retried push: row exists with `stored.client_updated_at_ms ==
  incoming.client_updated_at_ms` → predicate `incoming > stored` is false,
  no UPDATE. The RPC reports `lww_no_op`.

> Subtle case. If a retry happens after the client has already locally
> bumped `client_updated_at_ms` (because the user edited the row again
> between push attempts), the second push carries the new value. Server
> sees `incoming > stored` → UPDATE, `applied`. The retry is no longer a
> retry of the original payload; it's a new write. Either way the dirty
> bit gets cleared (assuming `local_updated_at_ms` matches the snapshotted
> sent value per §8.2 — if it doesn't match, the next cycle picks it up).

**Acceptance criterion (t4 §11 #4 verbatim):** "Re-pushing the same row
tuple `(owner_user_id, entity_type, id, client_updated_at_ms, payload)`
twice yields the same final server state and the same per-row outcome on
the response. A retry of a previously-acked row returns an 'applied'
outcome." Reading "applied" strictly: in this design, the second response
will be `lww_no_op` (`incoming == stored`), which is **functionally
identical to `applied`** from the client's perspective — the dirty bit is
cleared, the server state is correct. The choice of which outcome label
the server emits on the strict-equality case is a server-implementation
detail; for clarity we emit `lww_no_op` because that matches the SQL
predicate ("the WHERE clause did not match, no UPDATE happened"). The
contract t4 cares about — "the client clears the dirty bit on retry" —
holds in both `applied` and `lww_no_op` cases.

### 3.7 Dedup within a single batch

If the same `(type, id)` appears twice in one request, the server's
`ON CONFLICT` clause will conflict with itself inside the same statement
(Postgres rejects this with `cannot affect row a second time`). The
implementation MUST dedup client-side before sending; the algorithm in
§3.4 walks the dirty stream once and skips rows whose `(type, id)` is
already in `included_ids`, so this is naturally satisfied.

If a dedup bug slips through (somehow two dirty rows with the same id —
a bug in repo code), the server's response will be `INTERNAL` from the
ON CONFLICT failure, and the client retries after backoff. The bug will
be diagnosed from logs; this doc doesn't try to recover from a corrupted
local state.

### 3.8 Tombstones in push

Per t1 §1.1.3, a delete push sets `deleted = true` AND `client_updated_at_ms
= nowMonotonic()` AND carries the full last-known row in `fields` and
`extras`. The server's upsert writes everything verbatim — the row's
non-PK columns reflect the last-known state at delete time. This makes
undelete-by-LWW lossless (t1 §1.1.4).

For entities with a domain `deleted_at` column (`gyms`, `sessions`,
`session_exercises`, `exercise_sets`, `exercise_definitions`,
`exercise_tag_definitions` — see t1 §4.2), the client also sets
`fields.deleted_at = Date.now()` on the delete.

For join tables (`exercise_muscle_mappings`, `session_exercise_tags`),
there is no `deleted_at`; only the wire-envelope `deleted` carries the
state (t1 §1.1.4).

**Example tombstone push payload:**

```json
{
  "type": "exercise_sets",
  "id": "01HX5KCD...",
  "client_updated_at_ms": 1733000060000,
  "deleted": true,
  "fields": {
    "session_exercise_id": "01HX5KAB...",
    "order_index": 0,
    "weight_value": "100",
    "reps_value": "8",
    "set_type": "rir_2",
    "created_at": 1732999999000,
    "updated_at": 1733000060000,
    "deleted_at": 1733000060000
  },
  "extras": {
    "rpe": 8.5
  }
}
```

Same envelope shape as a live write; `deleted: true` is the only flag.
**The previous PR-#56 draft permitted `"payload": {}` on tombstones; that
is dropped.** Tombstones carry the full row, no exceptions.

## 4. Aggregate grouping, batching, and batch failure

### 4.1 Aggregate grouping rule

**A child row's parents must be batched with it if the parents are not
already on the server.** This is t1 §11 #6 (the "no orphan-child pushes"
constraint), originating from t1 round 2's review comment that the prior
"detached children allowed" position was a bug. The algorithm in §3.4 is
how this doc codifies it.

The grouping is per-batch, not per-session. A single batch may include
fragments from multiple sessions, multiple aggregates, multiple unrelated
entities — what it MAY NOT include is a child whose parent is missing
from both this batch and the server.

### 4.2 Batch failure: whole-batch reject by transaction boundary

A single push is one Postgres transaction. Either every row in the batch
commits, or none of them do. There is no partial-accept code path.

**Why whole-batch reject is the right choice (revisited from PR #56):**

1. **t1 §5.1 transactional semantics + §11 #6 no-orphan rule combine to
   eliminate partial-accept's value.** If the client builds batches by the
   §3.4 algorithm, the only way an FK can fail at COMMIT is a t2 batching
   bug. Partial-accept is a feature that helps when "most rows are good,
   one is bad"; in v2 the design makes all rows good or signals the bug
   loudly.
2. **Partial-accept reintroduces v1 outbox semantics.** The client would
   need to clear N dirty bits where N is a prefix of the batch, with a
   shape that interacts badly with §8.2's "is the row still at the same
   `local_updated_at_ms`" check.
3. **t4 §11 #4 idempotency requires "same final state on retry."** If a
   partial accept dropped rows 5–8 and the client retried, the server
   would now see rows 5–8 freshly — partial-accept's reasoning becomes
   tangled with retry idempotency. Whole-batch reject keeps both clean:
   on FK_VIOLATION, nothing happened, and the retry attempts the same
   batch again (after the client re-runs the §3.4 closure walk, which by
   now includes the missing parent).

### 4.3 Per-row outcome reporting

Per §3.5, the response carries one result per request entity, in order.
Both `applied` and `lww_no_op` ack the dirty bit. There is no per-row
**failure** outcome on a successful response — failure is whole-batch
(§4.2).

This is the client's signal to clear dirty bits. **A response without an
explicit per-row outcome line is treated as if the row was not in the
batch** (it stays dirty). The contract is: response is a 1:1 echo of the
request in order, with one outcome per row.

### 4.4 Defense-in-depth: what to do on `FK_VIOLATION`

`FK_VIOLATION` is a t2 batching bug per §3.4 — it should be impossible
in well-behaved code. When it happens anyway (e.g., the closure walk had
a bug, or the local DB ended up with a child whose parent never existed
locally — a repo invariant violation):

1. The client logs the error (`code`, `message`, the offending row id) to
   observability.
2. Dirty bits stay set (nothing was acked).
3. **The next cycle's `selectPushBatch` MUST re-run the §3.4 closure walk
   from a clean state.** If the parent exists locally but wasn't included
   (because its `server_known` bit was wrongly `1`, say), the second pass
   should catch it.
4. If the second cycle also produces `FK_VIOLATION` on the same row, the
   client surfaces a non-recoverable error in the sync status panel
   ("Sync error — please contact support") and stops auto-retrying. The
   dirty bits stay; a future fix lands by app update.

This is not "auto-retry forever on FK violations." That was a v1
behaviour and contributed to the M13 wedge. We retry **once** on a
suspected closure-walk bug and then escalate.

### 4.5 Batch size

`batchCap = 200`. Justification:

- v1 used 100; the bottleneck there was the projection function.
- In v2 the work per row is one UPSERT and one LWW comparison.
- Postgres comfortably handles 200-row UPSERTs; round-trip transport time
  dominates over per-row CPU.
- t4 §6.1 sets the foreground cycle target at ≤ 5 seconds and the
  background cycle at ≤ 20 seconds; at 200 rows / batch and typical net
  latencies, three batches comfortably fit inside foreground.
- Larger batches risk hitting the Supabase RPC request-size cap (default
  1 MiB after base64). 200 typical rows ≈ a few hundred KB; safe.

If the dirty stream exceeds 200 rows, the cycle issues multiple push
batches in sequence (§7.3). Each batch is independently committed.

## 5. `sync_pull`

### 5.1 Request

```json
{
  "cursor": {
    "server_received_at": "2026-05-23T10:14:32.871Z",
    "owner_user_id": "00000000-0000-0000-0000-000000000aaa",
    "type": "sessions",
    "id": "01HX5K9V..."
  },
  "limit": 500
}
```

| Field | Type | Notes |
|---|---|---|
| `cursor` | object or `null` | Opaque to the client (round-trip the value the server emitted). `null` for snapshot / initial pull. |
| `cursor.server_received_at` | string (ISO 8601 with ms) | The `server_received_at` of the last row returned by the previous response. |
| `cursor.owner_user_id` | string (UUID) | Same-millisecond tiebreak axis. Echoed back to the server. |
| `cursor.type` | string enum | Same-millisecond tiebreak axis. |
| `cursor.id` | string | Same-millisecond tiebreak axis. |
| `limit` | integer | `1..1000`. Default 500. The cycle (t4) chooses based on background-vs-foreground budget. |

### 5.2 Response

```json
{
  "entities": [
    {
      "type": "sessions",
      "id": "01HX5K9V...",
      "client_updated_at_ms": 1733000000200,
      "deleted": false,
      "fields": { "...": "..." },
      "extras": {}
    },
    {
      "type": "session_exercises",
      "id": "01HX5KAB...",
      "client_updated_at_ms": 1733000000201,
      "deleted": false,
      "fields": { "...": "..." },
      "extras": {}
    }
  ],
  "next_cursor": {
    "server_received_at": "2026-05-23T10:14:33.000Z",
    "owner_user_id": "00000000-0000-0000-0000-000000000aaa",
    "type": "session_exercises",
    "id": "01HX5KAB..."
  },
  "has_more": true
}
```

| Field | Notes |
|---|---|
| `entities` | Up to `limit` Entity envelopes, ordered by `(server_received_at ASC, owner_user_id ASC, type ASC, id ASC)`. |
| `next_cursor` | The cursor to use on the next pull. Always present (echoes input on an empty page). |
| `has_more` | `true` iff the server has at least one more row strictly greater than `next_cursor`. Cycle uses this to decide whether to issue another pull (t4 §11 #1). |

On an empty page (`entities.length === 0`), `has_more === false` and
`next_cursor` echoes the input `cursor` unchanged (the client persists it
unchanged — the cycle exits the pull loop).

### 5.3 Pagination — monotonicity and tiebreak

The server orders rows by:

```sql
ORDER BY server_received_at ASC,
         owner_user_id     ASC,
         type              ASC,
         id                ASC
LIMIT :limit
```

with the WHERE clause:

```sql
WHERE owner_user_id = auth.uid()
  AND (
    :cursor IS NULL
    OR (server_received_at, owner_user_id, type, id)
        > (:cursor.server_received_at, :cursor.owner_user_id,
           :cursor.type, :cursor.id)
  )
```

**Why `server_received_at` is monotonic.** It is set by the server's `now()`
on every successful UPSERT (`INSERT ... DEFAULT now()` for new rows; a
trigger touches it on UPDATE, per t1 §1 ground rules — the touch is
"structural, not validation"). Supabase Postgres is single-primary, so
`now()` is monotonic across all writes from a single client's view —
there's no replica clock skew to reason about.

**Same-millisecond rows.** Two rows in the same Postgres transaction share
a `now()` value (transactions hold a stable `now()`). The
`(owner_user_id, type, id)` lexicographic tiebreak gives every row a
unique sort key. The row-value comparison `>` ensures the next pull
strictly advances past the last-emitted row — it never repeats and never
skips.

**Why the cursor carries `owner_user_id`.** Within a single user's pull,
all rows share `owner_user_id` so it's strictly redundant. The field is
preserved for forward-compat with multi-owner pulls (purpose #5 from
`plan.md`). The cost is one UUID per cursor — negligible.

**No "high watermark" race.** v1 had a window where two concurrent writes
could share `server_received_at` and the cursor advance could skip one.
The row-value tiebreak closes this.

### 5.4 Tombstones in pull responses

A tombstone in a pull response is just an Entity with `deleted: true` and
the full row in `fields`/`extras` (t1 §1.1.3). The client applies it by:

1. Comparing incoming `client_updated_at_ms` to the local row's
   `local_updated_at_ms` (§8.4).
2. If incoming wins: write every field including the wire-envelope
   `deleted = true`, set the local row's `deleted_at` from
   `fields.deleted_at` (entities with that column), set
   `local_updated_at_ms = fields.updated_at` (or `client_updated_at_ms`,
   whichever t3 / repo convention chooses — see §11), set
   `local_dirty = false`, set `server_known = 1`.
3. If incoming loses (local has a newer write): no-op. The dirty bit stays
   set; the next push will overwrite the server.

Local readers MUST treat `deleted_at IS NOT NULL` as "row does not exist
for product purposes." This is the same convention as v1.

### 5.5 Pull and concurrent local edits

If a local mutation arrives for a row that's mid-apply from a pull:

- **The local write flips `local_dirty = true` and bumps
  `local_updated_at_ms` via the monotonicity guard.** This happens inside
  the repo's write transaction.
- **The pull apply runs LWW locally:** for each incoming Entity, compare
  `incoming.client_updated_at_ms` to the local row's current
  `local_updated_at_ms`. If incoming > local, apply (and clear `local_dirty`
  to `false`, since the inbound now reflects the state of record — unless
  the local row was just dirtied between the two reads, in which case the
  LWW comparison rejects the apply). If incoming <= local, no-op.

In practice the cycle pulls inside a single read-then-write transaction
per row (or per page) so the race window is microseconds; if a write
sneaks in, LWW resolves it.

### 5.6 Inbound apply rules — per-row

For each Entity in the response, the client:

1. SELECT the row by `(type, id)`.
2. If the row does not exist: INSERT, set `server_known = 1`,
   `local_dirty = 0`, `local_updated_at_ms = incoming.client_updated_at_ms`.
3. If the row exists and `incoming.client_updated_at_ms >
   local.local_updated_at_ms`: UPDATE every column from
   `fields` + `extras` + `deleted` + `deleted_at` (where present); set
   `server_known = 1`, `local_dirty = 0`,
   `local_updated_at_ms = incoming.client_updated_at_ms`.
4. If the row exists and `incoming.client_updated_at_ms <=
   local.local_updated_at_ms`: no-op (other than ensuring
   `server_known = 1`). Do not clear `local_dirty` — the local write is
   newer.

The cursor is advanced **after** the entire page is applied (t4 §11 #5).
An interrupted apply (process killed mid-page) leaves the cursor at its
pre-page value; the next cycle re-fetches the same page; idempotency at
the row level (step 4's no-op) makes the second apply safe.

## 6. Initial-sync state machine

This is the decision tree the bootstrap runs immediately after sign-in and
before the scheduler is started. Bootstrap runs **at most once per
`(user_id, device)` pair** under normal operation: the
`sync_runtime_state.bootstrap_completed_at` column staying NULL is the
"bootstrap not done" signal, and any RPC failure that occurs during
bootstrap leaves it NULL so the next launch re-runs.

### 6.1 "Has local data" definition

> A device "has local data" iff **at least one row in any of the eight
> entity tables has `local_dirty = true`.**

Reasoning, in priority order:

1. The question the modal asks is: *is there local content that, if we
   naively snapshot the server, would be lost?* The answer is "yes" iff
   something is locally newer than the server.
2. `local_dirty = true` precisely captures "this row has not been acked
   by the server." Whether it was user-created, user-edited, or a seed
   write that never got pushed, the same flag covers all three.
3. Rejected alternatives:
   - `count(*) > 0` per table: a fresh install with seeds loaded but never
     pushed would trip "has local data" and prompt the modal even when
     there's nothing to preserve.
   - "user-created rows exist" (distinguishing user vs seed): v2
     explicitly does not distinguish (t3.md owns that decision).
   - A separate boolean profile flag set on first non-seed write:
     duplicates `local_dirty`.

**Edge case — seeds.** The seed loader (t3) sets `local_dirty = true` on
fresh seed writes because seeds need to push as ordinary per-user data.
A fresh install that has only loaded seeds technically "has local data"
by this definition. This is fine: if remote is also empty, Branch A
absorbs it; if remote has data, the seed dirty bits are mostly harmless
duplicates because LWW will reconcile them.

### 6.2 "Has remote data" definition

The bootstrap issues `sync_pull { cursor: null, limit: 1 }`. If
`entities.length === 1` OR `has_more === true`, the user has remote data.

One tiny RPC; no separate "is remote empty" probe.

### 6.3 The four branches

```
                     remote_empty            remote_has_data
                  ┌───────────────────────┬──────────────────────┐
local_empty       │  A. empty/empty       │  C. remote-only      │
                  │  → seed + done        │  → auto snapshot     │
                  ├───────────────────────┼──────────────────────┤
local_has_data    │  B. local-only        │  D. both-have-data   │
                  │  → auto push          │  → MODAL             │
                  └───────────────────────┴──────────────────────┘
```

"local_empty" = no row has `local_dirty = true`. "local_has_data" = at
least one row does. "remote_empty" = §6.2 probe returns no rows.
"remote_has_data" = §6.2 probe returns at least one row.

#### 6.3.1 Branch A — empty/empty

**Trigger:** zero dirty rows locally, zero rows in the probe pull.

**Action:**

1. Run the seed loader (t3 decides what gets written). Writes flip
   `local_dirty = true` on the seed rows.
2. Set `bootstrap_completed_at = now()`.
3. Start the scheduler. Next cycle pushes the seeds.

**Walk:** User installs the app, signs in for the first time. Server has
nothing. Seeds load. Bootstrap completes. Within the next debounced cycle
(per t4), the seeds push to the server.

**Modal:** none.

#### 6.3.2 Branch B — local-only

**Trigger:** at least one row with `local_dirty = true`, zero rows in the
probe pull.

**Action:**

1. Run the seed loader (t3).
2. **Drain the dirty stream to zero** by issuing push batches per §3.4
   until either dirty count == 0 or push fails. (Convergence loop;
   conceptually identical to t4's `oneCycle.push` phase but invoked
   eagerly from bootstrap before the scheduler starts.)
3. Set `bootstrap_completed_at = now()`.
4. Start the scheduler.

**Walk:** User has been editing offline for a week. Server is empty.
They sign in. Everything pushes in 2–3 batches. Done.

**Modal:** none. (Pushing is unambiguous; nothing is at risk.)

#### 6.3.3 Branch C — remote-only

**Trigger:** zero dirty rows locally, at least one row in the probe pull.

**Action:**

1. **Defer the seed loader to step 3.** The server may already have the
   user's seed rows from a prior install; pulling them avoids double-write.
2. **Drain the pull cursor** by issuing `sync_pull` calls until
   `has_more === false`, applying each page per §5.6 and advancing the
   cursor after each apply.
3. Run the seed loader. t3 sees seeds already present (because we just
   pulled them) and writes nothing.
4. Set `bootstrap_completed_at = now()`.
5. Start the scheduler.

**Walk:** User reinstalled the app or signed in on device 2. Server has
everything. Local is bare. Snapshot drains in (e.g.) 5 pages of 500 rows.
Seed loader is a no-op. Done.

**Modal:** none.

#### 6.3.4 Branch D — both-have-data

**Trigger:** at least one dirty row locally AND the probe pull returns
data.

**Action:**

1. **Show the modal.** Block the app on user choice; the modal is
   presented above the home screen with no dismiss gesture; only the two
   buttons proceed. If the user kills the app, the modal re-appears next
   launch (the `bootstrap_completed_at` column remains NULL).
2. On choice **"Keep this device's changes"**:
   a. Drain the pull cursor (apply each page with LWW per §5.6, which
      handles per-row conflicts the same way a steady-state pull does).
   b. Drain the dirty stream (push batches per §3.4 until dirty count ==
      0). The server's LWW will skip rows where the pulled value was
      newer; everything else is preserved.
   c. Run the seed loader (defensive no-op if seeds already present).
   d. Set `bootstrap_completed_at = now()`. Start the scheduler.

   Net effect: union of local and remote; per-row collisions resolve by
   LWW.

3. On choice **"Use cloud data instead"**:
   a. Wipe all eight local entity tables (`DELETE FROM <table>` per table).
   b. Reset `local_dirty = 0` everywhere (vacuous after wipe).
   c. Reset `last_emitted_ms` is **not** reset — it is device-global and
      must survive user-switch (§9.3).
   d. Reset the pull cursor to NULL.
   e. Drain the pull cursor (apply each page; no LWW conflict possible).
   f. Run the seed loader (defensive no-op if seeds came via pull).
   g. Set `bootstrap_completed_at = now()`. Start the scheduler.

   Net effect: local is replaced by the server's view. The user's
   previously-dirty rows are **gone**.

**Modal copy — exact strings (future builder copies verbatim):**

```
Title:    Cloud data found
Body:     This device has unsynced changes, and there is also workout
          data in the cloud from another device or a previous install.

          What would you like to do?

Button 1: Keep this device's changes
          (We will merge them with the cloud data. Where two changes
          conflict, the most recent one wins.)

Button 2: Use cloud data instead
          (This device's unsynced changes will be discarded. This
          cannot be undone.)
```

**Walk D-1 (Keep local):** User has been editing on device A offline.
Earlier they synced on device B. They sign into device A online. Modal.
Tap "Keep this device's changes." Pulls device B's state with LWW
(reconciling per-row), then pushes device A's dirty rows. Both devices
converge.

**Walk D-2 (Replace local):** User accidentally created garbage data on
device A while signed out. They sign in. Modal. Tap "Use cloud data
instead." Local wipes; server snapshot replays. Garbage is gone.

### 6.4 Bootstrap failure handling

If any RPC inside bootstrap fails:

- **Transport / 5xx / timeout / `INTERNAL` / `RATE_LIMITED`:** retry with
  backoff per §10. Bootstrap's local attempt counter is bounded at **50
  attempts** (matches v1's `BOOTSTRAP_CONVERGENCE_MAX_ATTEMPTS`). On cap,
  bootstrap surfaces "Initial sync did not complete — open Settings and
  tap 'Sync now' to retry" in the sync status panel; the user (or
  app-restart) must explicitly retry.
- **`AUTH_REQUIRED`:** trigger token refresh; if it fails, bootstrap
  pauses, status surface shows "Please sign in again." Bootstrap resumes
  when a fresh JWT lands.
- **`FK_VIOLATION`:** this would indicate the bootstrap pushed a batch
  with an orphan child — i.e., a t2 bug. Bootstrap stops, logs the
  violation, surfaces a non-recoverable error. A fix lands by app update.

**Bootstrap is transactional at the wall-clock level:** until
`bootstrap_completed_at` is set, the scheduler does not run (it checks
that field on every tick — see t4 §1 and §2). This prevents v1's
"scheduler races bootstrap" failure mode.

## 7. The cycle interface (consumed by t4)

t4 §11 #1 requires that "a cycle continues issuing PULLs until the server
has no new rows beyond the cursor, and PUSHes until no local rows are
dirty." t2 owns the wire shape that makes this work; t4 owns the
scheduling. This section spells out the interface between them.

### 7.1 What a cycle does

The cycle is the routine the t4 scheduler invokes when it transitions
from `Scheduled(t)` to `Running`. Its job is to converge local and remote
state, with no in-memory state surviving the call:

1. **Pull phase** — repeatedly call `sync_pull` with the persisted cursor
   until `has_more === false`. Apply each page per §5.6, advancing the
   cursor after each page (t4 §11 #5).
2. **Push phase** — repeatedly call `selectPushBatch` (§3.4) and
   `sync_push` until `selectPushBatch` returns an empty array (dirty
   stream exhausted).
3. **Re-pull phase** — call `sync_pull` once more with the (now-advanced)
   cursor. If `has_more === false` AND `entities.length === 0`, the cycle
   has converged. If new entities arrive (a concurrent device pushed
   while we were pushing), apply them and continue draining.
4. **Repeat until** both ends are quiet for one full round (no rows
   pulled, no rows pushed). The standard convergence pattern.

> t4 explicitly does NOT prescribe the PULL/PUSH ordering. This doc
> recommends **PULL → PUSH → PULL**: pull first so a remote tombstone or
> remote edit lands before we re-push potentially-stale data; push
> second; pull a third time to drain anything that arrived during the
> push. t4 §11 says "convergence is what matters." Builders may swap
> the order if there's a reason; the requirements §7.2 must hold.

### 7.2 What a cycle MUST guarantee (interface for t4)

- **Drain to empty.** When the cycle returns, either (a) both ends are
  quiet (no rows dirty, no remote rows newer than cursor), or (b) the
  cycle aborted on a budget or error and made forward progress in at
  least one batch (t4 §11 #1, §11 #3).
- **Per-batch commits.** Each `sync_push` is one transaction; each
  `sync_pull` page apply is committed before the cursor advances. An
  interruption mid-cycle leaves the database consistent at the last
  committed batch (t4 §11 #3).
- **Idempotency.** Re-running the cycle from the same starting state
  produces the same final state (t4 §11 #4, §11 #5).
- **Surviving state.** Between cycles, only `local_dirty`,
  `server_known`, `local_updated_at_ms`, the pull cursor, and the
  bootstrap flag survive — all of which are in SQLite (t4 §11 #6).
- **Cycle duration target.** Background cycles abort cleanly within 20
  seconds (t4 §11 #2). Each batch's HTTP request times out at 15
  seconds; the cycle checks elapsed time after each batch and breaks
  out if the next batch would push past the budget.

### 7.3 Multi-batch push

When the dirty stream exceeds 200 rows, the push phase issues multiple
batches in sequence. Each batch's `selectPushBatch` is recomputed from
fresh state (dirty stream + `server_known` bits) — this lets earlier
batches' ack'd rows become "server-known" and stop being re-included as
parents in later batches.

The acknowledgement of batch N must be applied to local state before
batch N+1 is built. Otherwise batch N+1 would re-send the same rows.

### 7.4 Why the cursor is single-valued (not per-table)

One cursor for the whole user, not one per table. Reasons:

1. The cursor is `(server_received_at, owner_user_id, type, id)` — the
   `type` is part of the cursor, so a single cursor traverses all tables
   in interleaved order.
2. Per-table cursors would mean the cycle issues 8 parallel pulls; the
   client would need to merge by `server_received_at` to apply in the
   right order. Single cursor pre-merges server-side.
3. The pull RPC returns up to `limit` rows across all tables; the
   server's ORDER BY interleaves them naturally.

## 8. Dirty-bit lifecycle

### 8.1 The dirty bit and the "server-known" bit

Per §12 (canonical column inventory), every entity table has:

- `local_dirty integer not null default 0` — `1` iff the row needs pushing.
- `server_known integer not null default 0` — `1` iff the server has acked
  this row at least once. Used by §3.4 to decide whether to include the row
  as an ancestor in a closure walk.
- `local_updated_at_ms integer not null default 0` — the row's
  client-monotonic timestamp (§9). Same value the wire envelope's
  `client_updated_at_ms` carries on push.

The three columns are local-only; they never appear on the wire.

### 8.2 Setting and clearing — full table

| Event | `local_dirty` | `local_updated_at_ms` | `server_known` |
|---|---|---|---|
| Repo `create<Entity>(...)` | `1` | `nowMonotonic()` | `0` |
| Repo `update<Entity>(...)` | `1` | `nowMonotonic()` | unchanged |
| Repo `softDelete<Entity>(...)` | `1` | `nowMonotonic()` | unchanged |
| Repo cascade-update (sibling reorder, etc.) | `1` | `nowMonotonic()` | unchanged |
| Seed loader insert on fresh install | `1` | `nowMonotonic()` | `0` |
| Pull apply (incoming wins, §5.6 step 2 or 3) | `0` | `incoming.client_updated_at_ms` | `1` |
| Pull apply (incoming loses, §5.6 step 4) | unchanged | unchanged | `1` |
| Push ack with `outcome = applied` (no concurrent edit) | `0` | unchanged | `1` |
| Push ack with `outcome = applied` (concurrent edit detected, §8.3) | `1` | unchanged | `1` |
| Push ack with `outcome = lww_no_op` (no concurrent edit) | `0` | bumped to `max(local, server_value)` if we got a newer one (§8.4) | `1` |
| Push batch error (`FK_VIOLATION`, transport, etc.) | unchanged | unchanged | unchanged |
| Bootstrap Branch D "Replace local" wipe | n/a — row deleted | n/a | n/a |

All repo write paths run inside a single Drizzle transaction that updates
the row AND sets the dirty bit. No "set dirty bit later" path exists.
This is the **single biggest contract repos must obey** — t1's agent-
reminder spec (t1 §8) calls it out.

### 8.3 Push-in-flight race

The race we must handle:

```
T0: Row R is dirty (local_dirty=1, local_updated_at_ms=1000). Cycle
    picks it up in batch B. Snapshot of (R.id, 1000) captured in
    push-state.
T1: HTTP request leaves the device with R @ 1000.
T2: User edits R again locally.
    Repo write: local_dirty=1 (still), local_updated_at_ms=1100.
T3: Server commits R @ 1000.
T4: Response: { id: R, outcome: applied, server_received_at: ... }.
T5: Client applies the ack.
```

If the client naively sets `local_dirty = 0` on receipt of the ack, the
T2 edit is lost. The fix:

**Mechanism — `sent_at_ms` snapshot in push-state:** at request time,
the cycle captures `Map<id, sent_at_ms>` for every row in the batch
where `sent_at_ms` is the value of `local_updated_at_ms` at the moment
the row was serialised. This map lives in cycle-local memory only (it
does not need to survive suspension; if suspension drops the map, the
next cycle picks up the rows from the dirty stream again — t4 §11 #6
allows this).

**On receipt of the push response, per row:**

1. SELECT the row's current `local_updated_at_ms`.
2. If `current == sent_at_ms`: clear `local_dirty = 0`,
   set `server_known = 1`.
3. If `current > sent_at_ms`: leave `local_dirty = 1`, set
   `server_known = 1`. The dirty bit stays; next cycle will re-push the
   newer value.
4. If `current < sent_at_ms`: this should be impossible (the
   monotonicity guard prevents the local clock going backwards). If it
   happens (e.g., a future bug clears the cache and the persisted
   `last_emitted_ms` is stale), the safe behaviour is the same as case
   3: leave the dirty bit set.

The comparison happens **inside a Drizzle transaction** to prevent another
race between the SELECT and the UPDATE. The repo `update*` paths use the
same transaction discipline.

### 8.4 `lww_no_op` clock bump

When the response is `lww_no_op`, the server's stored value has
`client_updated_at_ms >= incoming.client_updated_at_ms`. The client knows
the server is at-or-newer than what it sent, but does not (from the push
response alone) know the server's exact `client_updated_at_ms`.

**The push response does not echo the server's `client_updated_at_ms`**
because the next pull will bring it. Specifically: after `lww_no_op`, the
client clears `local_dirty` (the dirty bit's job is done — the server is
at-or-newer than the local row) and waits for the next pull cycle to
deliver the actually-newer remote value. The client's
`local_updated_at_ms` stays at `sent_at_ms`; the LWW comparison in §5.6
will correctly accept the inbound update because `incoming.client_updated_at_ms
> local.local_updated_at_ms`.

**Consequence.** Between `lww_no_op` and the next pull, the local row is
stale relative to the server. This is fine: the user just edited the row
themselves moments ago, and a peer device wrote it even more recently —
they will see the peer's value within one cycle. There is no correctness
loss; there is a brief consistency lag, which is the price of clean
LWW + no per-column merge.

### 8.5 Survival of `local_dirty` across suspension

Per t4 §11 #6, `local_dirty`, `local_updated_at_ms`, and the pull cursor
are the only surfaces that survive iOS suspension/termination. They all
live in SQLite. The cycle holds no other state that needs persistence.

If iOS terminates mid-push (network in flight, no ack yet), the dirty
bits remain set; the next cycle re-pushes those rows; the server's LWW
makes the retry idempotent (§3.6).

## 9. Clock-monotonicity guard

### 9.1 The formula

```
local_updated_at_ms = max(Date.now(), last_emitted_ms + 1)
```

Every repo write call that bumps `local_updated_at_ms` calls a single
helper:

```ts
function nowMonotonic(): number {
  const wall = Date.now();
  const previous = cached_last_emitted ?? readPersisted();
  const next = Math.max(wall, previous + 1);
  cached_last_emitted = next;
  persistAsync(next);
  return next;
}
```

### 9.2 Location

A single helper function in `apps/mobile/src/data/clock.ts` (file name
chosen to be conspicuous — `clock` is the entire purpose of this
module). All repos import from this one file. The session-recorder and
the seed loader also use this helper. No other monotonic-time helper
exists.

### 9.3 Persistence across restarts

The device clock can move backwards between launches (NTP correction,
user-set time, time-zone change interpreted by the OS). To prevent the
monotonic counter from regressing, `last_emitted_ms` is persisted to
SQLite.

Storage: **a new column `last_emitted_ms integer not null default 0` on
the `sync_runtime_state` singleton table** (already used by the
bootstrap and the v1 scheduler runtime; see
`apps/mobile/src/data/schema/sync-runtime-state.ts`). One row, one column,
trivial to read/write.

`persistAsync` is fire-and-forget. If the process dies before the write
flushes, the next launch reads a slightly-stale value — the worst case
is one wasted millisecond of monotonic headroom (the next write bumps
past it on the first call).

**The counter is device-global, not per-user.** On sign-out / sign-in,
the counter does **not** reset. Two users on the same device share the
counter (which is what we want — monotonic timestamps across the device's
clock are the invariant). This is one of the reasons Branch D "Replace
local" does not reset `last_emitted_ms` (§6.3.4).

### 9.4 In-memory cache

The helper caches `last_emitted_ms` in a module-scoped variable so tight
loops (50 row writes in the same transaction) don't hit SQLite 50 times.
The cache is invalidated on each `persistAsync` flush; the read path
prefers the cache over the persisted value when both are present.

On cold start the cache is empty, so the first call reads from SQLite.

### 9.5 Why client-side, not server-side

LWW resolution happens server-side, but the timestamps it compares are
client-generated. The monotonicity guard prevents two timestamps from
the **same device** being equal — without it, two same-millisecond
writes could tie and LWW would pick arbitrarily.

The server does not need to know about the guard. It just sees the
incoming `client_updated_at_ms` and compares it to its stored value.

Cross-device clock skew remains unguarded — this is the brainstorm's §9
"Clock skew on LWW: trust client clocks for v2" position, confirmed here.

## 10. Retry and backoff

### 10.1 Push retry schedule

| Parameter | Value |
|---|---|
| Initial delay | 2 seconds |
| Multiplier | 2× |
| Cap | 2 minutes |
| Jitter | ±25% (uniform) |
| Reset triggers | (a) a successful push (`applied` or `lww_no_op` for any row in the response — i.e., any non-error response); (b) user toggles sync off then on; (c) user taps "Sync now"; (d) sign-out (state cleared as part of bootstrap reset) |

After a failed push, the next retry happens at:

```
now + (2s × 2^(failures-1)) × (1 ± 0.25)   clamped to ≤ 2 min
```

`consecutive_failures`, `next_attempt_at`, and `last_error_message` are
persisted in `sync_runtime_state` (extending the existing singleton with
three new columns — `consecutive_failures int default 0`,
`next_attempt_at integer`, `last_error_message text`).

### 10.2 Pull retry schedule

Same schedule as push. Pull failures during bootstrap (Branches C/D)
count against bootstrap's 50-attempt cap (§6.4). Pull failures during
steady state (post-bootstrap, scheduled cycles) follow §10.1.

### 10.3 Offline mid-push

If the device goes offline while a push request is in flight:

- HTTP layer reports a transport failure.
- Dirty bits remain set (nothing was acked).
- `consecutive_failures` increments; `next_attempt_at` advances per
  §10.1.
- When NetInfo reports `isInternetReachable = true` (t4 §3 — online
  edge), the t4 scheduler fires `requestSync('online')`. This **clears
  `next_attempt_at`** (treats the online edge as a reset trigger for the
  immediate retry) and starts a cycle.

Per t4 §11 #4, the retry is idempotent: re-pushing the same rows
produces the same final state on the server, even if the server actually
did receive the original request and the failure was on the response
path.

### 10.4 Non-retryable errors

Per §2.2, the only failure surface that is non-retryable on a tight loop
is `FK_VIOLATION` (which indicates a t2 batching bug — see §4.4). The
client retries once on the next cycle; if it fails again, the dirty bits
stay set and the user sees a non-recoverable status. There is no
`VALIDATION_FAILED` in v2.

### 10.5 Backoff reset on cycle boundaries

If a cycle completes successfully (drained both ends), the backoff state
is reset to `(0, null, null)`. The next cycle starts on the t4
scheduler's normal cadence (foreground edge, edit, safety tick).

## 11. Tombstones — end-to-end recap

This consolidates tombstone behaviour scattered above. Builders should
read this section to confirm there are no surprises.

| Event | Behaviour |
|---|---|
| User soft-deletes a row locally | Repo sets `deleted_at = Date.now()`, `local_dirty = 1`, `local_updated_at_ms = nowMonotonic()`. Row stays in SQLite. |
| Local readers query rows | All queries filter `deleted_at IS NULL`. |
| Client pushes tombstone | Entity `{ deleted: true, fields: { ...full row including deleted_at }, extras: { ...full blob } }`. The full last-known row travels (t1 §1.1.3). |
| Server applies tombstone | Standard LWW upsert. The row's `deleted` column becomes `true`. Non-PK columns are written from `fields`/`extras` verbatim — they reflect the row's last-known state, **not** cleared. Tombstone stays in `app_public.<entity>` indefinitely (no v2 hard purge — t1 §11 "Out of t1's scope but t2 must own"). |
| Server returns tombstone in pull | Standard Entity `{ deleted: true, fields: {...}, extras: {...} }`. |
| Client applies inbound tombstone | LWW comparison per §5.6. If incoming wins: write every field, set local `deleted_at` from `fields.deleted_at` (or current time if the field is absent — defensive; should be present), set `local_updated_at_ms = incoming.client_updated_at_ms`, set `local_dirty = 0`, set `server_known = 1`. If incoming loses: no-op. |
| Undelete (resurrect) | Repo writes `deleted_at = null` and dirties the row. Next push sends `deleted: false` with the full row in `fields`. Server LWW upsert restores the row (t1 §1.1.2 scenario B; t1 §1.1.3 ensures this is lossless because the prior tombstone preserved every field). |

**The v1 "tombstone for missing row" idempotency case** — deleting a row
the server doesn't have — works by construction in v2: the server just
stores the tombstone as a new row.

### 11.1 Tombstone retention

The server keeps tombstones forever in v2. No GC job, no purge schedule.
Two reasons:

1. Group/sharing (purpose #5) will eventually want audit; tombstones
   are part of that.
2. Without retention, a peer device that's been offline for longer than
   the GC window would miss the deletion and resurrect the row on its
   next push. Retention is correctness, not just observability.

Tombstones are small (PK + flags + last-known row), and the user base
will not generate enough deletes to overflow Supabase's free tier in any
realistic window.

## 12. Local-only sync-bookkeeping columns

Per t1 §11 #7, t2 owns the canonical inventory. This section pins it down
and registers each column with the drift checker.

### 12.1 The three columns

**Every entity client table** (all eight: `gyms`, `sessions`,
`session_exercises`, `exercise_sets`, `exercise_definitions`,
`exercise_muscle_mappings`, `exercise_tag_definitions`,
`session_exercise_tags`) gains these three columns in the t2 build wave's
local Drizzle migration:

| Column | Type | Default | Semantic |
|---|---|---|---|
| `local_dirty` | `integer` (SQLite-style boolean: 0/1) | `0` | `1` iff the row needs pushing. Set by every repo write path. Cleared by the push ack handler (§8). |
| `local_updated_at_ms` | `integer` (epoch ms) | `0` | The client-monotonic timestamp used as the LWW key. Sent on the wire as `client_updated_at_ms` (§2.1). Bumped by `nowMonotonic()` on every dirtying write (§9). |
| `server_known` | `integer` (SQLite-style boolean: 0/1) | `0` | `1` iff the server has acked this row at least once. Used by the §3.4 closure walk to decide whether to re-include the row as an ancestor. Set by push ack (`applied` or `lww_no_op`) and by pull apply. |

The three columns are local-only; the push serialiser (§3.3 split rule)
omits them; the pull apply (§5.6) writes `local_dirty=0`,
`local_updated_at_ms=incoming.client_updated_at_ms`, `server_known=1`.

**Naming convention.** All three columns use snake_case in SQLite (`local_dirty`,
`local_updated_at_ms`, `server_known`) and camelCase in Drizzle TypeScript
(`localDirty`, `localUpdatedAtMs`, `serverKnown`). Same as existing columns.

### 12.2 The pull cursor

Stored as a new singleton column on the existing `sync_runtime_state`
table (one row per device):

| Column | Type | Default | Semantic |
|---|---|---|---|
| `pull_cursor` | `text` | `null` | The opaque cursor blob (JSON-stringified, `{server_received_at, owner_user_id, type, id}`). NULL means "snapshot from the beginning." Advanced after each pull page is fully applied (§5.6). |

`pull_cursor` is **not** a per-entity column; it is a singleton because
the cursor traverses all entities together (§7.4).

### 12.3 The monotonic-clock state

Also a column on `sync_runtime_state`:

| Column | Type | Default | Semantic |
|---|---|---|---|
| `last_emitted_ms` | `integer` | `0` | The largest `local_updated_at_ms` this device has emitted. Persisted by `nowMonotonic()`'s fire-and-forget write. See §9.3. |

### 12.4 The retry-state columns

Per §10.1, three more singleton columns:

| Column | Type | Default | Semantic |
|---|---|---|---|
| `consecutive_failures` | `integer` | `0` | Count of consecutive push (or pull) failures since the last successful cycle. Used to compute the next backoff. |
| `next_attempt_at` | `integer` (epoch ms) | `null` | Earliest wall-clock time the next attempt may run. NULL = no backoff in effect. |
| `last_error_message` | `text` | `null` | Human-readable last error, surfaced via the sync status panel (t4 §8). |

### 12.5 Drift-checker registration

Per t1 §3.3 and §11 #7, all eight entity tables' three local-only columns
must be listed in `apps/mobile/src/data/schema/sync-extras.json` under
`exemptions.local_only_columns`. The registration is global (not
per-entity), because the same three columns apply to all eight tables:

```jsonc
{
  // ... per-entity extras_only_columns entries ...
  "exemptions": {
    "local_only_columns": [
      "local_dirty",
      "local_updated_at_ms",
      "server_known"
    ],
    "untyped_text_references": [
      // ... per t1 §3.3 ...
    ]
  }
}
```

The `sync_runtime_state` table is **not** in the drift checker's scope —
it has no server counterpart by design; it's a client-only singleton (the
drift checker scans only the eight user-owned entity tables, per t1 §7).
The four singleton-state columns (`pull_cursor`, `last_emitted_ms`,
`consecutive_failures`, `next_attempt_at`, `last_error_message`) are local
to that table and don't need any registration.

> Why three local-only columns per entity and not one combined? Each one
> has a distinct lifecycle and gets read by a distinct piece of the
> protocol. Compressing them into a single bitfield or JSON blob would
> require parsing on every read, and Drizzle would lose its type
> information. The space cost is 3 × 8 bytes per row in the worst case
> (SQLite stores small integers compactly); on a 100k-row local DB this
> is ~2 MB total. Negligible.

### 12.6 Push serialiser exclusion

The push serialiser (§3.3 split rule) MUST NOT emit `local_dirty`,
`local_updated_at_ms`, or `server_known` in either `fields` or `extras`.
t1 §11 #7 explicitly assigns this responsibility to t2:

> "t2 is also responsible for ensuring the push serialiser does not emit
> these columns on the wire."

The build wave implements this by routing the serialiser through a
single function (`entityToWire`) per entity, which closes over the
typed-promoted column list from the t1 §2 mapping and the
`extras_only_columns` registry — `local_only_columns` are by definition
in neither, and so they're not emitted.

### 14.2 Two devices both online, editing the same row

```
Device A and Device B both signed in as user U. Both have completed
bootstrap. t4 scheduler on both at SAFETY_INTERVAL = 60s.

Initial state — row R on both devices:
  { local_updated_at_ms: 1000, name: "Bench Press",
    local_dirty: 0, server_known: 1 }

T = 100 (wall): User on Device A renames R → "My Bench A"
  Repo write:

  Device A:
    Row R.local_updated_at_ms still 1001 (no concurrent edit during
    flight).
    Clear local_dirty=0. server_known=1 (already was).

T = 150 (wall on Device B, before Device A's push lands at the server
         — but B's clock may run slightly behind):
  User on Device B renames R → "My Bench B"
  B's nowMonotonic() = max(150, B's last_emitted+1) ≈ 1150 (assuming
  B's local clock is at 1150 — different than A's because no shared
  monotonic counter across devices).
  Row R on B = { local_updated_at_ms: 1150, name: "My Bench B",
                 local_dirty: 1, server_known: 1 }

T = 700 (wall): Device B cycle fires.
  selectPushBatch → [R @ 1150]
  POST sync_push { entities: [R @ 1150] }
  Server (now sees R with stored client_updated_at_ms = 1001 from A):
    Predicate 1150 > 1001 → APPLY.
    Server row R = { client_updated_at_ms: 1150, name: "My Bench B",
                     server_received_at: T_server_2 }
    Response: { results: [{ id: R, outcome: "applied",
                            server_received_at: T_server_2 }] }
  Device B: clear local_dirty=0.

T = 750 (wall): Device A's next scheduler tick (foreground stayed up).
  selectPushBatch → empty (no dirty rows).
  sync_pull { cursor: C_A, limit: 500 }
    ◀── { entities: [R @ 1150 with name="My Bench B"],
          next_cursor: C_A', has_more: false }

Final: both devices converged on "My Bench B". B's edit, at wall=150,
won because B's local_updated_at_ms (1150) was greater than A's

```

**Alternative ordering in 14.2.** What if Device A's push (T=600) is
slow and Device B's push (T=700) hits the server first? Answer:
LWW is determined by `client_updated_at_ms`, not arrival order.

- B's push at 1150 lands first → server row at 1150, name="My Bench B".
- A's push at 1001 arrives next → predicate 1001 > 1150 is false →
  `lww_no_op`. Server unchanged.
- A's ack on `lww_no_op` clears local_dirty (the dirty bit's job is
  done — the server is at-or-newer than what A sent).
- A's next pull delivers B's value as in the original sequence.

Same convergence, different sequence of `applied`/`lww_no_op` labels on
the responses. The client doesn't care which path it took.

## 15. What this design does not cover

Closed in companion design docs:

- **t1** — the typed server schema, RLS, deferrable-FK declarations, LWW
  semantics, the `extras` convention, the drift checker.
- **t3** — the seed loader's decision tree, seed-version drift across
  reinstall, user-edited seeds.
- **t4** — iOS lifecycle triggers, the two-state scheduler, BGTaskScheduler
  integration, the "Sync now" button, the sync status surface.

Out of scope here as deliberate boundaries:

- The PostgREST RPC source code for `sync_push` and `sync_pull`. The build
  wave writes these.
- The client cycle implementation in `apps/mobile/src/sync/` (the v1
  `engine.ts` / `outbox.ts` deletion and the v2 replacement). Build wave.
- The local Drizzle migration adding `local_dirty`, `local_updated_at_ms`,
  `server_known` to the eight entity tables, and `pull_cursor`,
  `last_emitted_ms`, `consecutive_failures`, `next_attempt_at`,
  `last_error_message` to `sync_runtime_state`. Build wave.
- The Maestro / e2e tests verifying the cycle. Build wave.
- The session-recorder integration with `requestSync('mutation')`. Build
  wave (and t4 already specifies the trigger surface).
- Tombstone hard-purge policy (we keep them forever in v2 per §11.1).
- Group / sharing semantics (purpose #5 from `plan.md`; future work).
- Web client and MCP read paths against the typed server schema (purposes
  #3 / #4; they consume t1's schema directly, no t2 protocol involvement).
