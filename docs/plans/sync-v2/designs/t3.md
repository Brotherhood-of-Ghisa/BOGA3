## 1. The bootstrapper

### 1.1 What it does

A bootstrapper routine runs once per device, on the first successful
sign-in to a given account. Concretely:

```
bootstrapper(): // called on sign-in iff sync_runtime_state.bootstrap_completed_at IS NULL
  rowsPulled := runFirstFullPull()        // drains all 4 layers per t2 §4.4
  if rowsPulled == 0:
    runSeeder()                            // §3.4
  sync_runtime_state.bootstrap_completed_at := now()
```

- `runFirstFullPull()` drains all four topological layers (t2 §4.4) and
  returns the total number of entity rows applied to the local DB. The
  local entity tables are empty before this call (it runs on a fresh
  device-account pairing — see §1.2).
- `runSeeder()` inserts every bundle row through the normal repo path.
  Rows become `local_dirty = 1` with `local_updated_at_ms = nowMonotonic()`
  and are pushed by the next cycle (t2 §3.4).
- `bootstrap_completed_at` is set last, so a crash mid-bootstrapper
  leaves the flag NULL and the next sign-in re-attempts cleanly.

After this single pass, `bootstrap_completed_at` is non-NULL and the
bootstrapper never runs again on this device for this account.

### 1.2 Why local is empty before the first pull

The bootstrapper fires only when `bootstrap_completed_at IS NULL`, which
is true exactly in two cases:

- **First install of the app on this device** with this account signed
  in. The 8 syncable entity tables were created empty by the v2
  migration; nothing has been written to them yet.
- **Sign-out / account switch wipe.** The auth-layer wipe clears the 8
  entity tables and resets `bootstrap_completed_at` to NULL (see §6.2).
  After the wipe, the tables are empty again from the perspective of
  the newly-signed-in account.

In both cases, the entity tables hold zero rows when the bootstrapper
starts. `rowsPulled` therefore equals the post-pull row count.

### 1.3 Pre-sign-in / offline — dev affordance

A user who has not signed in, or who is signed in but offline, sees an
empty exercise catalog. The bootstrapper cannot complete without a
successful first pull, so the seeder never runs. **Accepted for shipped
builds.** The product purposes in `plan.md` are all account-linked; the
offline-pre-sign-in catalog is not a supported journey.

For local development this is a productivity wall — every reinstall
forces a real sign-in + network round-trip before the app is usable.
**The t3 build wave ships a dev affordance that bypasses this:**

- A pre-baked dev profile fixture (a fully-populated local SQLite snapshot
  plus a mock signed-in session) is loaded behind `isDevMode()` (per the
  no-`__DEV__` rule). The fixture's local rows match a designated dev
  remote database state so sync remains a no-op on the first cycle.
- The fixture is loaded on `apps/mobile`'s dev-build first launch when
  no real session exists, leaves `bootstrap_completed_at` set as part
  of the fixture data, and is wiped by `resetLocalDataAndReseed`.
- Production (TestFlight, App Store) builds never see the fixture —
  `isDevMode()` gates it off so the `com.phano.boga3.dev` TestFlight
  build still exercises the real sign-in flow.

The fixture is a build-wave concern; the design contract here is just:

1. Dev mode MAY pre-populate the local DB and the auth session.
2. The bootstrapper MUST observe `bootstrap_completed_at` honestly — if
   the fixture set it, the bootstrapper does not run, and the seeder is
   not invoked.
3. The fixture's data is the authoritative source of seeds in dev mode;
   the bundled seed list is not consulted on top of it.

### 1.4 What the seeder writes

The seeder uses the **same repo paths** the rest of the app uses for
inserts. Every row it produces is created via the existing repo
functions that wrap `nowMonotonic()` (t2 §8) and set `local_dirty = 1`.
The repos do not know — and do not need to know — that the caller is
the seeder.

Consequence: seed rows are dirty after the seeder completes; the next
cycle picks them up via `selectPushBatch()` (t2 §3.4.2) and pushes them
under the topological order. The server stores them with
`server_received_at = now()` and the row's own `client_updated_at_ms`.

There is no sync-invisible write path. There is no sentinel timestamp.
There is no second seeder pass.

## 2. Identity

### 2.1 Slug IDs

Seed rows have stable, deterministic IDs of the form
`seed_<snake_case_slug>`:

- `exercise_definitions.id`: `seed_bench_press`, `seed_barbell_back_squat`, etc.
- `exercise_muscle_mappings.id`: `seed_map_bench_press__chest`, where the
  `__` separates the exercise slug from the muscle group slug. The pair is
  also indexed for lookup performance (t1 §2.6); the slug PK is the
  authoritative identity.
- `muscle_groups.id`: bare slugs (`chest`, `back_lats`) — these are
  client-only taxonomy IDs (t1 §2.6.1) and don't need a prefix.

### 2.2 Why slugs, not UUIDs

If two devices on the same account both reach the bootstrapper with
`bootstrap_completed_at = NULL` (two fresh installs signing in back-to-
back), both will see `rowsPulled = 0` on their first pull and both will
seed.

- With slugs: both push `(owner, seed_bench_press)`. The server upserts
  by composite PK; LWW resolves on `client_updated_at_ms`. Whichever
  push commits last wins; the content (name="Bench Press", canonical
  defaults) is identical on both sides, so the LWW winner has the same
  visible content as the loser. Convergence is automatic.
- With UUIDs: both push `(owner, <random_a>)` and `(owner, <random_b>)`.
  The server now has two distinct rows for "Bench Press" because the PK
  `(owner, id)` makes them distinct. The user sees "Bench Press" twice.

Slugs make the seed loader a **pure upsert** with no race window.

### 2.3 The `sys_` → `seed_` rename

The current seed bundle uses `sys_*` (e.g. `sys_barbell_bench_press`).
v2 renames to `seed_*` because "system" implies an ownership tier we
explicitly deleted from the v2 server schema (t1 §1: every row is
per-user from day one). The prefix should reflect *lineage*, not
*ownership*.

The rename is a one-time bundle-file change in the t3 build wave. Since
the v2 migration drops every v1 server table (t1 §1, "hard cut from
v1") and the hosted DB is wiped post-merge, there are **no existing
`sys_*` rows on the server** to migrate. On the client side, the v2
client-side migration that lands alongside the schema wipe also drops
and re-creates the entity tables. Every device that upgrades to v2
enters with empty entity tables and runs the bootstrapper from scratch.

## 3. The decision

### 3.1 The check

```
rowsPulled := runFirstFullPull()  // total rows applied across all 8 entity tables
if rowsPulled == 0:
  seed
else:
  no-op
```

That is the entire gate. The bootstrapper runs once, after the first
full pull has drained every layer to `has_more = false` (t2 §4.4). The
local entity tables are empty pre-pull (§1.2), so `rowsPulled` is the
exact count of rows the server delivered for this user.

`rowsPulled` **counts tombstoned rows** (rows with `deleted_at IS NOT
NULL`). A user who deleted everything on their other device still has
the tombstones pulled to this device; `rowsPulled > 0` → seeder no-op.
That is the deletion-survives-reinstall guarantee (plan.md outcome #1).

### 3.2 The decision table

| `rowsPulled` | Action |
|---|---|
| `0` | Seed every row in the bundle via the normal repo path. |
| `> 0` | No-op. The server is authoritative. |

There is no per-seed branch, no row-level inspection, no per-table
counting. One number, one comparison.

### 3.3 Why no per-row lineage tracking

The simpler model needs no per-row lineage because the seeder makes no
per-row decision and never visits this device again:

- "Did the user edit this seed?" — irrelevant; the seeder never re-runs.
- "Is this row deleted?" — irrelevant; same reason.
- "Did the bundle add a new field?" — irrelevant; bundle updates do not
  reach existing users (§8.1, accepted tradeoff).

This is why neither `seed_origin` nor `local_seed_version` is in v2.

### 3.4 What the seeder actually does

Inside a single Drizzle transaction:

```
BEGIN;
  FOR each exercise_definition_seed IN BUNDLE:
    repo.exerciseDefinitions.create({
      id: exercise_definition_seed.id,           // 'seed_bench_press', etc.
      name: exercise_definition_seed.name,
      // ... other typed columns from the bundle
      // The repo sets:
      //   local_dirty := 1
      //   local_updated_at_ms := nowMonotonic()
      //   created_at := nowMonotonic()
      //   updated_at := nowMonotonic()
      //   deleted_at := null
    });

  FOR each exercise_muscle_mapping_seed IN BUNDLE:
    repo.exerciseMuscleMappings.create({ ... });
COMMIT;
```

`muscle_groups` is not in this transaction — see §4.

The repo paths walk the topological order naturally because push
(t2 §3.4) picks up dirty rows layer by layer: `exercise_definitions`
in Layer 0, `exercise_muscle_mappings` in Layer 1. Deferrable FKs cover
any intra-batch dependency.

## 4. `muscle_groups` is client-only, separately seeded

### 4.1 Why it's separate

Per t1 §2.6.1, `muscle_groups` is **not** one of the 8 user-owned
entities. It is a client-only taxonomy with `is_editable = 0`. The
server has no `app_public.muscle_groups` table; the
`exercise_muscle_mappings.muscle_group_id` column is opaque text on the
server side (t1 §4.1 registers it as `untyped_text_references`).

The "sync first if synced" rule **does not apply** to `muscle_groups`.
There is no sync to wait for, no remote data to fetch. Every device
must seed its own copy of the taxonomy from the bundle.

### 4.2 When `muscle_groups` is seeded

On every fresh local install, **before sign-in is even attempted**, the
existing bootstrap path (`apps/mobile/src/data/bootstrap.ts`) seeds
`muscle_groups`:

```
IF count(rows in muscle_groups) == 0:
  insert every muscle_group from the bundle
```

This is independent of sync, independent of `bootstrap_completed_at`,
independent of auth state. It runs on the very first launch of the app.

`muscle_groups` rows do not get `local_dirty` (the column does not exist
on this table — only the 8 syncable entities carry `local_dirty` /
`local_updated_at_ms` per t2 §9). They never push, never pull, never
appear on the wire.

### 4.3 `is_editable` and the no-edit invariant

`muscle_groups.is_editable` is `0` for every seeded row and is guarded
by a CHECK constraint (`muscle_groups_non_editable_guard`,
`isEditable = 0`). The catalog UI does not expose mutation of muscle
groups; the seeder is the only writer. Muscle groups are a static,
read-only taxonomy on every device.

If a future product change introduces user-editable muscle groups, the
table will need to be promoted to a syncable entity (with a paired
server table per t1) and the `is_editable` guard relaxed. Out of scope
for v2.

## 5. Worked examples

T1, T2, … are wall-clock moments produced by `nowMonotonic()`. They are
strictly monotone: T1 < T2 < T3 < … always.

### 5.1 User renames "Bench Press" → "My Bench", syncs, reinstalls

**Setup.** User installs the app, signs in. The server has no rows for
this user (genuinely fresh account).

| # | Event | Server state for `seed_bench_press` | Local state on the device |
|---|---|---|---|
| 1 | App launches; bootstrap seeds `muscle_groups` (§4). | empty | `muscle_groups` populated. Entity tables empty. |
| 2 | User signs in. Bootstrapper runs. `bootstrap_completed_at` is NULL. | empty | unchanged |
| 3 | Bootstrapper invokes first full pull. All 4 layers drain. `rowsPulled = 0`. | empty | unchanged |
| 4 | Bootstrapper invokes seeder. Bundle is inserted via repo. | empty | `seed_bench_press`: `name='Bench Press', local_dirty=1, local_updated_at_ms=T1, created_at=T1, updated_at=T1, deleted_at=null` |
| 5 | Bootstrapper sets `bootstrap_completed_at := T_boot`. | empty | unchanged |
| 6 | Next cycle pushes dirty rows. | `name='Bench Press', client_updated_at_ms=T1` | `local_dirty=0` after push ack |
| 7 | User renames in UI. Repo writes via `nowMonotonic()`. | unchanged | `name='My Bench', local_dirty=1, local_updated_at_ms=T2, updated_at=T2` |
| 8 | Next cycle pushes. | `name='My Bench', client_updated_at_ms=T2` (LWW: `T2 > T1` ✓) | `local_dirty=0` |
| 9 | User uninstalls and reinstalls. Bootstrap re-seeds `muscle_groups`. Entity tables empty. `bootstrap_completed_at` is NULL again (lived in the now-deleted DB). User signs in. | unchanged | `muscle_groups` populated; entity tables empty. |
| 10 | Bootstrapper invokes first full pull. Layer 0 returns `seed_bench_press` with `client_updated_at_ms=T2`. `rowsPulled = 1`. | unchanged | `name='My Bench', local_dirty=0, local_updated_at_ms=T2, ...` |
| 11 | `rowsPulled > 0` → seeder no-ops. `bootstrap_completed_at := T_boot'`. | unchanged | unchanged |

**Net result after reinstall:** "My Bench" is restored. The seeder did
not touch the row.

### 5.2 User deletes "Bench Press", syncs, reinstalls — stays deleted

**Setup.** Continues from step 6 of §5.1 (server holds
`seed_bench_press` with name "Bench Press" and a clean push).

| # | Event | Server state for `seed_bench_press` | Local state on the device |
|---|---|---|---|
| 7' | User deletes "Bench Press" in UI. Repo soft-deletes. | `name='Bench Press', deleted_at=null, client_updated_at_ms=T1` | `name='Bench Press', deleted_at=T3, local_dirty=1, local_updated_at_ms=T3, updated_at=T3` |
| 8' | Next cycle pushes. | `name='Bench Press', deleted_at=T3, client_updated_at_ms=T3` (LWW: `T3 > T1` ✓) | `local_dirty=0` |
| 9' | User uninstalls and reinstalls. Bootstrap re-seeds `muscle_groups`. Entity tables empty. User signs in. | unchanged | `muscle_groups` populated; entity tables empty. |
| 10' | Bootstrapper invokes first full pull. Layer 0 returns the tombstoned `seed_bench_press`. `rowsPulled = 1` (tombstone counts). | unchanged | `name='Bench Press', deleted_at=T3, local_dirty=0, local_updated_at_ms=T3` |
| 11' | `rowsPulled > 0` → seeder no-ops. `bootstrap_completed_at := T_boot'`. | unchanged | unchanged |

**Net result:** Bench Press stays deleted. Local readers filter
`WHERE deleted_at IS NULL` (t2 §2.1) so the UI hides it. The row sits
on the device, fully populated, with `deleted_at` non-null. The seeder
never resurrects it because `rowsPulled` was non-zero.

> Why counting tombstones is correct: the user's intent — "Bench Press
> should not exist for me" — is expressed by the server holding a
> tombstone for that row. A device that pulls the tombstone sees
> `rowsPulled > 0` and concludes "the server has something for this
> user." Exactly the signal we want: the seeder must not re-create
> what the user has deleted.

### 5.3 Two devices rename the same seed within the same minute — LWW

**Setup.** Devices A and B are both signed in to the same account. The
server holds `seed_bench_press` with `name='Bench Press'`,
`client_updated_at_ms=T1` (from device A's earlier seed-push). Both
devices are caught up.

| # | Event | Device A | Server | Device B |
|---|---|---|---|---|
| 1 | User on A renames to "Bench A" at wall-clock T_A. | `name='Bench A', local_dirty=1, local_updated_at_ms=T_A` | unchanged | unchanged |
| 2 | User on B renames to "Bench B" at wall-clock T_B > T_A (~10ms later). | unchanged | unchanged | `name='Bench B', local_dirty=1, local_updated_at_ms=T_B` |
| 3 | Device A pushes. Server LWW: `T_A > T1` ✓. | `local_dirty=0` | `name='Bench A', client_updated_at_ms=T_A` | unchanged |
| 4 | Device B pushes. Server LWW: `T_B > T_A` ✓. | unchanged | `name='Bench B', client_updated_at_ms=T_B` | `local_dirty=0` |
| 5 | Device A's next cycle pulls. Local LWW per t2 §4.5 step 3: `T_B > T_A` ✓; UPDATE. | `name='Bench B', local_dirty=0, local_updated_at_ms=T_B` | unchanged | unchanged |
| 6 | Both devices converge. | `name='Bench B'` | `name='Bench B'` | `name='Bench B'` |

**Net result:** Both devices converge on "Bench B" (the later writer).
Device A's "Bench A" is lost. Accepted per `plan.md` purpose #2 and
t2 §1.1 LWW semantics.

The bootstrapper is irrelevant — it ran once on each device's first
sign-in and has not fired again. All subsequent reconciliation is pure
LWW.

> Edge case: same-millisecond writes. `nowMonotonic()` (t2 §8.1)
> guarantees uniqueness within a single device. Across devices, two
> writes *can* share a timestamp. Per t2 §1.1.1, `T_in <= T_db` is a
> no-op: whichever push the server commits second is the no-op; the
> first wins. The clock-skew clamp (t1 §1, ±5 minutes future-clock)
> bounds how bad this gets in practice.

### 5.4 App v3 adds `default_rest_sec` to all seeds — existing users do not see it

**Setup.** User has been on app v2 since the previous examples. Their
local DB has `seed_bench_press` with `name='Bench Press'`.
`bootstrap_completed_at` is set from the v2 sign-in. Server state
matches local.

App v3 ships. The bundle now includes `default_rest_sec: 90` on every
seed. A paired server migration (per t1 §4 + §9) adds the
`app_public.exercise_definitions.default_rest_sec bigint` column with
no default. The client schema also adds the column.

| # | Event | Server | Local |
|---|---|---|---|
| 1 | Server-first deploy: column lands. Existing rows have `default_rest_sec = NULL`. | `default_rest_sec = NULL` | unchanged |
| 2 | Client v3 ships. Local schema migration adds the column nullable. | unchanged | existing rows: `default_rest_sec = NULL` |
| 3 | User opens app v3. Still signed in. Sync runs. | unchanged | unchanged |
| 4 | **`bootstrap_completed_at` is already set from v2.** The bootstrapper does NOT run. The seeder never executes on this device. | unchanged | unchanged |

**Net result on this device:** `default_rest_sec` stays `NULL` on every
existing row. The user sees no rest-second default for any of their
exercises.

A user who installs app v3 for the first time *does* get
`default_rest_sec` populated, because the bootstrapper fires for the
first time and the seeder writes the bundle (which includes the new
field).

This is the explicit tradeoff (§8.1). If the field becomes important
enough to backfill, that is a one-off operational migration written
outside the seeder (e.g., a script that issues an UPDATE through the
normal repo path on every `seed_*` row).

## 6. Reinstall and sign-out semantics

### 6.1 Reinstall (same account on the same device)

Covered exhaustively by §5.1 and §5.2. Summary:

- A reinstall is, from the bootstrapper's perspective, the same as a
  fresh device — `bootstrap_completed_at` starts NULL because it lived
  in SQLite on the now-deleted DB.
- The first pull retrieves everything the server holds for this user
  (rows and tombstones). `rowsPulled` reflects that.
- If the user had ever pushed any row, `rowsPulled > 0` and the seeder
  no-ops.
- Only if the server holds nothing (a never-synced account) does the
  seeder run.

### 6.2 Sign-out / account switch

Per t2 §5, the auth layer is responsible for wiping local entity tables
on sign-out (or on a `auth.uid()` change). The bootstrapper sits
downstream of the wipe.

**t3 constraint emitted to the auth-wipe path.** The auth-layer's
sign-out / account-switch wipe MUST:

1. Clear all rows from the 8 syncable entity tables (t2 §5
   precondition).
2. Reset `sync_runtime_state.bootstrap_completed_at` to NULL.
3. Reset `sync_runtime_state.pull_cursor` to `{}` (the empty JSON map
   per t2 §9.2).
4. PRESERVE `sync_runtime_state.last_emitted_ms` (per t2 §8.3 — the
   counter is device-global).
5. PRESERVE `muscle_groups` rows (client-only taxonomy; no per-user
   ownership).

If #2 is forgotten, the bootstrapper never fires for the new account
and the new account will see an empty catalog after sync (broken if
they have no remote data).

### 6.3 Wipe-local-only (dev tool)

`resetLocalDataAndReseed` (in `apps/mobile/src/data/dev-reset.ts`) is
the dev affordance. For v2 it maps to:

1. Drop and recreate the 8 syncable entity tables, `muscle_groups`,
   and `sync_runtime_state`.
2. Re-run the `muscle_groups` bootstrap seed (§4).
3. Sign-in flow re-enters the bootstrapper from scratch.

This is identical to "uninstall and reinstall the app." No special
seed-loader path is needed.

## 7. Two-device semantics

### 7.1 Both devices fresh on the same account, simultaneous sign-in

Worst case for the seeder:

- Device A signs in at T_a. Bootstrapper starts; first pull returns
  `rowsPulled = 0`.
- Device B signs in at T_b ≈ T_a (within seconds). Bootstrapper starts;
  first pull also returns `rowsPulled = 0` (A's push has not yet
  committed).
- Both devices invoke the seeder. Both push.
- Server upserts by `(owner, id)`. Whichever push commits second is
  LWW-resolved per row: the slugs are identical, the
  `client_updated_at_ms` values differ by milliseconds, the row
  content is identical (the bundle is deterministic). Server ends up
  with one row per slug.
- Both devices pull on their next cycle. Each device's local row may
  be overwritten by the server's LWW winner (same content; visible
  no-op).

**Convergence is automatic. No duplicate rows. No special handling.**

### 7.2 User edits a seed on one device, second device hasn't seeded yet

- Device A: seeded `seed_bench_press` at T1, pushed. User renames to
  "My Bench" at T2, pushed.
- Device B: signs in for the first time. Bootstrapper's first pull
  returns `seed_bench_press` with `name='My Bench'`. `rowsPulled = 1`.
  Seeder no-ops. `bootstrap_completed_at` set.

Device B sees "My Bench" — the user's edit — not the bundle's
"Bench Press". Correct: the server is authoritative; the bundle is a
fallback for the truly-fresh case only.

## 8. Tradeoffs and forward notes

### 8.1 Bundle-field updates do not propagate to existing users

**Explicit tradeoff.** Once the bootstrapper has fired on a device, it
never fires again. Bundle changes after that — adding a field to an
existing seed, changing a canonical default, adding a new seed slug —
do not reach existing users on that device.

Acceptable because:

- Existing users have their local data, including any edits. Nothing
  they have done is lost.
- New users on app v3 get the v3 bundle on their first seed.
- The simplification cost (no `local_seed_version`, no per-row lineage,
  no additive-merge code path) is large enough that the tradeoff is
  preferred.
- If a specific bundle update is important enough to backfill, that is
  a one-off operational migration outside the seeder.

A future iteration could add an additive-merge pass without changing
this design's contract: the existing bootstrapper remains the
once-per-device path with the merge layered on top.

### 8.2 No "factory reset to seeds" UI

If a user soft-deletes every seed and later wants them restored, v2 has
no UI path. The bootstrapper is permanently inert and the server
returns the same tombstones.

**Workaround for v2:** dev-only `resetLocalDataAndReseed`. Not
user-facing.

**Future feature:** a "factory reset to bundle" UI that issues an
UPDATE through the normal repo path on every `seed_*` row, setting
`deleted_at = null` and resetting other fields to bundle canonical.
Sync-engine unchanged. Out of scope.

## 9. Constraints

### 9.1 Honoured from t1 / t2

| Constraint | Honoured in |
|---|---|
| t1 §1.1 — deletion is `deleted_at != null`, no `deleted boolean` | §3.1 (tombstones counted), §5.2 (worked example). |
| t1 §1.1.1 — row-level LWW on `client_updated_at_ms` | Inherited via the normal repo path; §5.1, §5.2, §5.3 are pure LWW. |
| t1 §4.1 — local-only columns registered in `sync-extras.json` | No new local-only columns introduced; inventory unchanged. |
| t2 §3.4 — push topological order | §3.4 — seeded rows go in via repo; push picks them up via `selectPushBatch()` walking the topological order. |
| t2 §5 — `bootstrap_completed_at` is the first-cycle flag | §1.1 — the bootstrapper sets the flag exactly once, after the first pull. |
| t2 §7 — dirty-bit lifecycle | §1.4 — seeder uses normal repo path; `local_dirty = 1` set, push ack clears. |
| t2 §8 — clock monotonicity | §1.4 — every seed write goes through `nowMonotonic()`. |
| t1 §2.6.1 — `muscle_groups` is not synced | §4 — `muscle_groups` seeded client-locally only. |

### 9.2 Emitted to other build tasks

| # | Constraint | Target |
|---|---|---|
| 1 | The auth-layer sign-out / account-switch wipe MUST reset `sync_runtime_state.bootstrap_completed_at` to NULL (alongside clearing entity rows per t2 §5 precondition). Without this the bootstrapper never fires for the new account on this device. | Auth-layer build task (out of scope for sync v2 build wave per t2 §5). |
| 2 | The bundle author MUST use stable `seed_*` slug IDs for `exercise_definitions` and `seed_map_*__*` slugs for `exercise_muscle_mappings`. The v2 bundle file drops the `sys_*` prefix and never reuses an old slug for a new seed. | t3 build wave. |
| 3 | The t3 build wave's local Drizzle migration MUST remove the `seedsAppliedAt` column from `sync_runtime_state` — it has no v2 analogue. | t3 build wave. |
| 4 | The bootstrapper MUST set `bootstrap_completed_at` **last**. A crash between pull and flag-set leaves the flag NULL so the next sign-in re-attempts cleanly. | t3 build wave. |
| 5 | The seeder MUST run inside a single Drizzle transaction. Partial seed states are not legal. | t3 build wave. |
| 6 | Dev mode MAY install a pre-seeded local DB and a mock auth session (the "dev profile fixture" of §1.3). The fixture sets `bootstrap_completed_at` so the bootstrapper does not run on top of it. Production builds (TestFlight, App Store) MUST NOT load the fixture — gate via `isDevMode()` per the no-`__DEV__` rule. | t3 build wave. |

## 10. What this design does not cover

Closed in companion design docs:

- **t1** — server schema, RLS, deferrable FKs, LWW semantics, drift checker.
- **t2** — push/pull wire shape, dirty-bit lifecycle, clock monotonicity, cycle interface.
- **t4** — iOS lifecycle triggers, the scheduler that runs cycles.

Out of scope here as deliberate boundaries:

- The bundle file format and the `sys_` → `seed_` rename diff. t3 build wave.
- The local Drizzle migration that removes `sync_runtime_state.seedsAppliedAt`. Build wave.
- The bootstrapper call-site wiring in the runtime. Build wave.
- The dev profile fixture (the snapshot file, the mock session, the
  `isDevMode()` gate). Build wave; the contract is in §1.3 / §9.2 #6.
- The auth-layer wipe behaviour (§6.2's constraint is emitted to
  whoever owns the wipe; a separate task).
- A "factory reset to bundle" UI affordance (§8.2). Future product feature.
- Tests. Build wave; the §5 worked examples are the test scenarios.
