# t3 — Sync v2: Seeded entities & user edits

> Design doc only. Plan ref: `docs/plans/sync-v2/plan.md` § "t3: Seeded
> entities & user edits".
>
> **Companion designs:**
>
> - t1 (merged): `designs/t1.md` — typed server schema, soft-delete via
>   `deleted_at`, RLS, deferrable FKs, drift checker. Authoritative.
> - t2 (merged): `designs/t2.md` — push/pull wire shape, the LWW rule,
>   dirty-bit lifecycle (§7), `nowMonotonic()` clock (§8), the cycle
>   interface (§6), `bootstrap_completed_at` (§5). Authoritative.
> - t4: iOS triggers. Only relevant insofar as "first cycle drains
>   under t4's scheduler" — t4's surface is not re-described here.
>
> **Scope shift from the original plan card.** The plan card's six
> "Direction" points were written before the user re-simplified the
> approach. The current direction (confirmed mid-design):
>
> 1. The seeder runs **at most once per device** — after the first sync
>    cycle completes, if every user-owned entity table is empty.
> 2. After that first seed, the seeder is permanently inert on this
>    device. Bundle updates in future app versions do not reach existing
>    users on this device. Accepted tradeoff.
> 3. The seeder uses the **normal repo path**: seed rows become
>    `local_dirty = 1` and are pushed by the next cycle like any other
>    write. There are no sync-invisible writes, no sentinel timestamps,
>    no `local_seed_version`, no `seed_origin`.
> 4. All deletion is soft (t1 §1.1).
> 5. Two-device LWW for seed renames is acceptable.
>
> Plan-card bullets the merged design supersedes:
>
> - "Seeder runs after every sync cycle, re-applying" → **once per device only**.
> - "Seeder never writes sync-relevant columns / sync-invisible writes" → **seeder writes are normal dirty writes**.
> - "`seed_origin` boolean" → **dropped**.
> - "`local_seed_version` integer" → **dropped**.
> - "Additive merge for new bundle fields on existing rows" → **dropped**.

## 1. When the seeder runs

### 1.1 The trigger

The seeder fires exactly once per device's lifetime as a **post-cycle hook**
on the first sync cycle that sets `sync_runtime_state.bootstrap_completed_at`
(t2 §5).

Concretely, the sequence on a fresh install:

```
1. User signs in.
2. The runtime starts the scheduler (t2 §5).
3. Cycle N (the first cycle):
     pull layer 0..3 (drains; cursors advance from null to the layer tail)
     push (nothing dirty yet)
     re-pull (drains again — empty in the steady state)
     all four layers reach has_more = false in the same cycle →
       sync_runtime_state.bootstrap_completed_at := now()
4. Post-cycle hook (the seeder) runs:
     IF bootstrap_completed_at WAS NULL before this cycle
        AND IS NOW NON-NULL
     THEN inspect local entity tables and decide (per §3).
5. Cycle N+1 runs as normal. If the seeder wrote rows, they push now.
```

The seeder is **not** triggered by:

- App launch (pre-sign-in).
- Subsequent cycles after the first (those don't re-set
  `bootstrap_completed_at`).
- A failed cycle (one that aborts on BG budget or error before
  `bootstrap_completed_at` is set). The next cycle that successfully
  drains all four layers is the one that fires the hook.

### 1.2 Why post-cycle, not pre-cycle

Two reasons:

- **It enforces "sync first" structurally.** If the user has remote data
  (re-install scenario, fresh second device), the pull populates the
  entity tables before the seeder can act. The seeder sees non-empty
  tables and does nothing.
- **It avoids the `local_updated_at_ms = 0` sentinel hack.** PR #57's
  earlier draft seeded *before* the pull and relied on a sentinel
  timestamp so the server's `T_in > 0` LWW would overwrite the seed.
  That hack required t2 to honour `local_updated_at_ms = 0` as a
  permanent loser, which t2 §8.3 does not promise (the monotonic clock
  guard treats `0` only as the cold-start default). Post-cycle ordering
  eliminates the need: real rows arrive from sync; the seeder fills only
  the truly-empty case.

### 1.3 Why exactly once per device

`bootstrap_completed_at` transitions from `NULL` to a non-null timestamp
exactly once per (device, account) pair (t2 §5: "the first time all four
layers drain to `has_more = false` in the same cycle"). The seeder
listens for that specific edge. Subsequent cycles cannot re-fire it
because the column is already non-null.

The seeder does not maintain its own state (no `seedsAppliedAt`, no
`local_seed_version`). The cycle-completion edge is the trigger; the
table-emptiness check is the gate.

### 1.4 What the seeder writes

The seeder uses the **same repo paths** the rest of the app uses for
inserts. Concretely, every row it produces is created via the existing
repo functions that wrap `nowMonotonic()` (t2 §8) and set
`local_dirty = 1`. The repos do not know — and do not need to know —
that the caller is the seeder.

Consequence: seed rows are dirty after the seeder completes; cycle N+1
picks them up via `selectPushBatch()` (t2 §3.4.2) and pushes them under
the topological order. The server stores them with
`server_received_at = now()` and the row's own
`client_updated_at_ms` (which was produced by `nowMonotonic()` and is
therefore strictly later than the dummy `0` baseline carried by
`sync_runtime_state.last_emitted_ms` on a fresh install — t2 §8).

This is the simplest possible mechanism. There is no sync-invisible
write path. There is no sentinel timestamp. There is no second seeder
pass.

### 1.5 The pre-sign-in / offline-on-first-launch cost

A user who:

- Installs the app and uses it without signing in, **or**
- Signs in but stays offline indefinitely

sees an empty exercise catalog. The seeder cannot fire because no cycle
has set `bootstrap_completed_at`. **Accepted cost.** The product
purposes in `plan.md` are all account-linked; the offline-pre-sign-in
flow is not a supported user journey for the catalog UI.

If the offline-pre-sign-in path becomes a product requirement later,
the seeder can be re-wired to fire on a different edge (e.g., a
"first launch + first auth attempt completed" signal) without changing
any other v2 contract. The trigger is local to this design.

`muscle_groups` is a separate story — see §4. The client-only taxonomy
is always seeded on every fresh install, before sign-in, because it
has no server side and the rest of the catalog UI depends on it.

## 2. Identity

### 2.1 Slug IDs

Seed rows have stable, deterministic IDs of the form
`seed_<snake_case_slug>`:

- `exercise_definitions.id`: `seed_bench_press`, `seed_barbell_back_squat`, etc.
- `exercise_muscle_mappings.id`: `seed_map_bench_press__chest`, where the
  `__` separates the exercise slug from the muscle group slug. The pair is
  also indexed for lookup performance (t1 §2.6); the slug PK is the
  authoritative identity.
- `muscle_groups.id`: bare slugs (`chest`, `back_lats`) — these are
  client-only taxonomy IDs (t1 §2.6.1) and don't need a prefix.

### 2.2 Why slugs, not UUIDs

If two devices on the same account both arrive at the post-cycle hook
with empty entity tables (a rare but real race — two fresh installs
signing in back-to-back, neither cycle having completed before the
other), both will seed and push.

- With slugs: both push `(owner, seed_bench_press)`. The server upserts
  by composite PK; LWW resolves on `client_updated_at_ms`. Whichever
  push commits last wins; the content (name="Bench Press", canonical
  defaults) is identical on both sides, so the LWW winner has the same
  visible content as the loser. Convergence is automatic.
- With UUIDs: both push `(owner, <random_a>)` and `(owner, <random_b>)`.
  The server now has two distinct rows for "Bench Press", and the
  client has to dedupe by name on pull. This is the v1 collision bug
  in slightly different clothing.

Slugs make the seed loader a **pure upsert** with no race window.

### 2.3 The `sys_` → `seed_` rename

The current seed bundle uses `sys_*` (e.g. `sys_barbell_bench_press`).
v2 renames to `seed_*` because "system" implies an ownership tier we
explicitly deleted from the v2 server schema (t1 §1: "no
`origin_scope_id`, every row is per-user from day one"). The prefix
should reflect *lineage*, not *ownership*.

The rename is a one-time bundle-file change in the t3 build wave. Since
the v2 migration drops every v1 server table (t1 §1, "hard cut from
v1") and the hosted DB is wiped post-merge, there are **no existing
`sys_*` rows on the server** to migrate. On the client side, the same
hard-cut applies: the v2 client-side migration that lands alongside
the schema wipe also drops and re-creates the entity tables. Every
device that upgrades to v2 enters with empty entity tables, signs in,
runs the first cycle (no remote data the first time around), and
re-seeds with `seed_*` IDs.

This sidesteps the entire "rename across reinstall" question that PR
#57 worried about. v1 IDs do not survive the v2 cutover anywhere.

## 3. The decision

### 3.1 The check

```
isFreshOnTheCloud =
  count(rows in exercise_definitions where owner_user_id = auth.uid()) == 0
  AND count(rows in exercise_muscle_mappings where owner_user_id = auth.uid()) == 0
  AND count(rows in gyms where owner_user_id = auth.uid()) == 0
  AND count(rows in sessions where owner_user_id = auth.uid()) == 0
  AND count(rows in session_exercises where owner_user_id = auth.uid()) == 0
  AND count(rows in exercise_sets where owner_user_id = auth.uid()) == 0
  AND count(rows in exercise_tag_definitions where owner_user_id = auth.uid()) == 0
  AND count(rows in session_exercise_tags where owner_user_id = auth.uid()) == 0
```

The check runs *after* the cycle that sets `bootstrap_completed_at`, so
every row the server holds for this user is already on the device.
"Empty after first sync" is equivalent to "the server holds nothing for
this user" — which is what we want to detect.

**The check counts all rows, including tombstoned rows
(`deleted_at IS NOT NULL`).** A tombstone is a row, not an absence; a
user who deleted everything on their other device still has the
tombstones pulled to this device, so the count is non-zero and the
seeder no-ops. This is the deletion-survives-reinstall guarantee
(plan.md outcomes #1, #2).

### 3.2 The decision table

| `isFreshOnTheCloud` | Action |
|---|---|
| `true` | Seed every row in the bundle via the normal repo path. Rows become `local_dirty = 1` with `local_updated_at_ms = nowMonotonic()`. Next cycle pushes them. |
| `false` | No-op. Permanently. |

There is no per-seed branch. There is no row-level inspection. The check
is global and one-shot.

### 3.3 Why no per-seed lineage tracking

The simpler model needs no per-row lineage because the seeder never
makes a per-row decision after the initial seed:

- "Did the user edit this seed?" — irrelevant; the seeder never visits
  this device again.
- "Is this row deleted?" — irrelevant; same reason.
- "Did the bundle add a new field to this row?" — irrelevant; bundle
  updates do not reach existing users (§8.1, accepted tradeoff).

This is why neither `seed_origin` nor `local_seed_version` is in v2.
Adding either would only pay off if the seeder needed to re-run
intelligently — which the once-per-device contract eliminates.

### 3.4 What the seeder actually does on `isFreshOnTheCloud = true`

Inside a single Drizzle transaction:

```
BEGIN;
  FOR each exercise_definition_seed IN BUNDLE:
    repo.exerciseDefinitions.create({
      id: exercise_definition_seed.id,           // 'seed_bench_press', etc.
      name: exercise_definition_seed.name,
      // ... other typed columns from the bundle
      // The repo sets:
      //   local_dirty := 1
      //   local_updated_at_ms := nowMonotonic()
      //   created_at := nowMonotonic()
      //   updated_at := nowMonotonic()
      //   deleted_at := null
    });

  FOR each exercise_muscle_mapping_seed IN BUNDLE:
    repo.exerciseMuscleMappings.create({ ... });
COMMIT;
```

`muscle_groups` is not in this transaction — see §4.

The repo paths are the same paths the rest of the app uses. They run
through `nowMonotonic()` (t2 §8.1), so the monotonic clock is properly
advanced and persisted. They set `local_dirty = 1`, so the next cycle's
`selectPushBatch()` (t2 §3.4.2) picks the rows up. They walk the
topological order naturally because the push-side batch builder walks
layers; `exercise_definitions` is in Layer 0, `exercise_muscle_mappings`
is in Layer 1, and the deferrable FKs in Layer 1 → Layer 0 are
satisfied by the time the batch is built.

There is no special "first seed pass uses this path, later passes use
that path" toggle (which the plan card's Builder brief speculated
about). The seeder runs once and uses one path: the normal one.

## 4. `muscle_groups` is client-only, separately seeded

### 4.1 Why it's separate

Per t1 §2.6.1, `muscle_groups` is **not** one of the 8 user-owned entities.
It is a client-only taxonomy with `is_editable = 0`. The server has no
`app_public.muscle_groups` table; the column on
`exercise_muscle_mappings.muscle_group_id` is opaque text on the server
side (t1 §2.6.1; t1 §4.1 registers it as
`untyped_text_references`).

The implication: the "sync first if synced" rule **does not apply** to
`muscle_groups`. There is no sync to wait for, no remote data to fetch.
Every device must seed its own copy of the taxonomy from the bundle, or
the catalog UI is broken.

### 4.2 When `muscle_groups` is seeded

On every fresh local install, **before sign-in is even attempted**, the
existing bootstrap path (`apps/mobile/src/data/bootstrap.ts`) seeds
`muscle_groups`. The seeder check is:

```
IF count(rows in muscle_groups) == 0:
  insert every muscle_group from the bundle
```

This is independent of sync, independent of `bootstrap_completed_at`,
independent of auth state. It runs on the very first launch of the app.

`muscle_groups` rows do not get `local_dirty` (there's no column for it,
because the table is not in t2 §9's two-column-per-entity-table list —
the `local_dirty`/`local_updated_at_ms` columns exist only on the 8
syncable entities). They never push, never pull, never appear on the
wire. This is correct: the server doesn't know about muscle groups as a
table.

### 4.3 `is_editable` and the no-edit invariant

`muscle_groups.is_editable` is `0` for every seeded row and is guarded by
a CHECK constraint (`muscle_groups_non_editable_guard`,
`isEditable = 0`). The catalog UI does not expose mutation of muscle
groups; the seeder is the only writer. The result: muscle groups are a
static, read-only taxonomy on every device.

If a future product change introduces user-editable muscle groups, the
table will need to be promoted to a syncable entity (with a paired
server table per t1) and the `is_editable` guard relaxed. That is out
of scope for v2.

## 5. Worked examples

The plan card mandates four examples (lines 289-299). Each is walked
through showing every column write that matters — `id`, `name`,
`local_dirty`, `local_updated_at_ms`, `deleted_at`, `created_at`,
`updated_at` — at each step.

For clarity, T1, T2, … are wall-clock moments produced by
`nowMonotonic()`. They are strictly monotone: T1 < T2 < T3 < … always.

### 5.1 User renames "Bench Press" → "My Bench", syncs, reinstalls

**Setup.** User installs the app, signs in. The server has no rows for
this user (genuinely fresh account).

| # | Event | Server state for `seed_bench_press` | Local state on the device |
|---|---|---|---|
| 1 | App launches; bootstrap seeds `muscle_groups` (§4). | empty | `muscle_groups` populated. Entity tables empty. |
| 2 | User signs in. Scheduler starts cycle 1. | empty | unchanged |
| 3 | Cycle 1 pulls — every layer empty. Push — nothing dirty. Re-pull — every layer empty. `bootstrap_completed_at := T_boot`. | empty | unchanged |
| 4 | Post-cycle hook runs. `isFreshOnTheCloud = true`. Seeder runs: inserts every bundle row via repo. | empty | `seed_bench_press`: `name='Bench Press', local_dirty=1, local_updated_at_ms=T1, created_at=T1, updated_at=T1, deleted_at=null` |
| 5 | Cycle 2 starts. `selectPushBatch()` picks up all dirty rows including the seeds. | (push lands) `name='Bench Press', client_updated_at_ms=T1` | `local_dirty=0` after push ack |
| 6 | User renames in UI. Repo writes via `nowMonotonic()`. | `name='Bench Press', client_updated_at_ms=T1` | `name='My Bench', local_dirty=1, local_updated_at_ms=T2, updated_at=T2` |
| 7 | Cycle 3 pushes. | `name='My Bench', client_updated_at_ms=T2` (LWW: `T2 > T1` ✓) | `local_dirty=0` after push ack |
| 8 | User uninstalls and reinstalls. Bootstrap re-seeds `muscle_groups`. Entity tables empty. User signs in. | `name='My Bench', client_updated_at_ms=T2` | `muscle_groups` populated; entity tables empty. |
| 9 | Cycle 1' pulls. Layer 0 returns `seed_bench_press` with `client_updated_at_ms=T2`. Pull applier writes the row per t2 §4.5 step 2. | unchanged | `name='My Bench', local_dirty=0, local_updated_at_ms=T2, ...` |
| 10 | All four layers drain. `bootstrap_completed_at := T_boot'`. | unchanged | unchanged |
| 11 | Post-cycle hook runs. `isFreshOnTheCloud = false` (entity tables non-empty after pull). Seeder no-ops. | unchanged | unchanged |

**Net result on device after reinstall:** "My Bench" is restored. The
seeder did not touch the row.

### 5.2 User deletes "Bench Press", syncs, reinstalls — stays deleted

**Setup.** Continues from step 5 of §5.1 (server holds
`seed_bench_press` with name "Bench Press" and a clean push).

| # | Event | Server state for `seed_bench_press` | Local state on the device |
|---|---|---|---|
| 6' | User deletes "Bench Press" in UI. Repo soft-deletes (sets `deleted_at = nowMonotonic()`). | `name='Bench Press', deleted_at=null, client_updated_at_ms=T1` | `name='Bench Press', deleted_at=T3, local_dirty=1, local_updated_at_ms=T3, updated_at=T3` |
| 7' | Cycle 3 pushes. | `name='Bench Press', deleted_at=T3, client_updated_at_ms=T3` (LWW: `T3 > T1` ✓; tombstone is just a row write per t1 §1.1) | `local_dirty=0` after push ack |
| 8' | User uninstalls and reinstalls. Bootstrap re-seeds `muscle_groups`. Entity tables empty. User signs in. | `deleted_at=T3, client_updated_at_ms=T3` | `muscle_groups` populated; entity tables empty. |
| 9' | Cycle 1' pulls. Layer 0 returns the tombstoned `seed_bench_press`. Pull applier INSERTs it (t2 §4.5 step 2) with the tombstone intact. | unchanged | `name='Bench Press', deleted_at=T3, local_dirty=0, local_updated_at_ms=T3` |
| 10' | All four layers drain. `bootstrap_completed_at := T_boot'`. | unchanged | unchanged |
| 11' | Post-cycle hook runs. **`isFreshOnTheCloud = false`** — `count(exercise_definitions) = 1` because the tombstoned row counts (§3.1). Seeder no-ops. | unchanged | unchanged |

**Net result:** Bench Press stays deleted. Local readers filter
`WHERE deleted_at IS NULL` (t2 §2.1) so the UI hides it. The row sits
on the device, fully populated, with `deleted_at` non-null. The seeder
never resurrects it because `isFreshOnTheCloud` is false.

> Why counting tombstones is correct: the user's intent — "Bench Press
> should not exist for me" — is expressed by the server holding a
> tombstone for that row. A device that pulls the tombstone and then
> consults a count-based emptiness check will see one row and conclude
> "the server has something for this user." That is exactly the signal
> we want: the seeder must not re-create what the user has deleted.

### 5.3 Two devices rename the same seed within the same minute — LWW

**Setup.** Devices A and B are both signed in to the same account. The
server holds `seed_bench_press` with `name='Bench Press'`,
`client_updated_at_ms=T1` (from device A's earlier seed-push). Both
devices are caught up at this point.

| # | Event | Device A | Server | Device B |
|---|---|---|---|---|
| 1 | User on A renames to "Bench A" at wall-clock T_A. `nowMonotonic()` returns T_A. | `name='Bench A', local_dirty=1, local_updated_at_ms=T_A` | unchanged | unchanged |
| 2 | User on B renames to "Bench B" at wall-clock T_B > T_A (~10ms later). | unchanged | unchanged | `name='Bench B', local_dirty=1, local_updated_at_ms=T_B` |
| 3 | Device A pushes. Server LWW: `T_A > T1` ✓. | `local_dirty=0` | `name='Bench A', client_updated_at_ms=T_A` | unchanged |
| 4 | Device B pushes. Server LWW: `T_B > T_A` ✓. | unchanged | `name='Bench B', client_updated_at_ms=T_B` | `local_dirty=0` |
| 5 | Device A's next cycle pulls. Pull returns `(name='Bench B', client_updated_at_ms=T_B)`. Local LWW per t2 §4.5 step 3: `T_B > T_A` ✓; UPDATE the row. | `name='Bench B', local_dirty=0, local_updated_at_ms=T_B` | unchanged | unchanged |
| 6 | Both devices converge. | `name='Bench B'` | `name='Bench B'` | `name='Bench B'` |

**Net result:** Both devices converge on "Bench B" (the later writer).
Device A's "Bench A" rename is lost. Accepted per `plan.md` purpose #2
(two-device support, low priority) and t2 §1.1 LWW semantics.

The seeder is irrelevant here — it ran once on each device's initial
post-cycle hook (before either rename) and has not fired again on
either device. All subsequent reconciliation is pure LWW.

> Edge case: same-millisecond writes. The `nowMonotonic()` guard in t2
> §8.1 (`max(Date.now(), last_emitted_ms + 1)`) guarantees that within a
> single device, no two writes share a timestamp. Across devices, two
> writes *can* share a timestamp (different clocks). If `T_A == T_B`,
> t2 §1.1.1's rule applies: `T_in <= T_db` is no-op. Whichever push
> server commits second is the no-op; the first wins. The clock-skew
> clamp (t1 §1, ±5 minutes future-clock) bounds how bad this gets in
> practice.

### 5.4 App v3 adds `default_rest_sec` to all seeds — existing users do not see it

**Setup.** User has been on app v2 since the previous examples. Their
local DB has `seed_bench_press` with `name='Bench Press'` (or whatever
they have edited it to). Server state matches.

App v3 ships. The bundle now includes `default_rest_sec: 90` on every
seed. A paired server migration (per t1 §4 + §9) adds the
`app_public.exercise_definitions.default_rest_sec bigint` column with
no default. The client schema also adds the column.

| # | Event | Server `seed_bench_press` | Local on user's device |
|---|---|---|---|
| 1 | Server-first deploy: column lands first. Existing rows on server have `default_rest_sec = NULL`. | `default_rest_sec = NULL` | unchanged |
| 2 | Client v3 ships. User upgrades, opens the app. Local schema migration adds the column. Existing rows have `default_rest_sec = NULL`. | `default_rest_sec = NULL` | `default_rest_sec = NULL` (existing rows) |
| 3 | User signs in (still signed in, actually — upgrade keeps session). Scheduler runs. | unchanged | unchanged |
| 4 | Cycle pulls — nothing new for this user. `bootstrap_completed_at` is **already set** from app v2, so the post-cycle hook does **not** re-fire. | unchanged | unchanged |
| 5 | App life proceeds. The seeder never runs on this device again. The bundle's `default_rest_sec: 90` value never lands on `seed_bench_press`. | `default_rest_sec = NULL` | `default_rest_sec = NULL` |

**Net result on this device:** `default_rest_sec` stays `NULL` on every
existing row. The user sees no rest-second default for any of their
exercises.

A user who installs app v3 for the first time *does* get `default_rest_sec`
populated, because their post-cycle hook fires for the first time and the
seeder writes the bundle (which includes the new field).

> This is the explicit tradeoff. The plan card called out a more refined
> additive-merge option; we are not taking it. The cost is that bundle
> field additions reach only new users. The benefit is that the seeder
> is one path, one trigger, one decision — and not a thing that
> repeatedly inspects per-row state forever.

If the field becomes important enough to backfill, that is a one-off
operational migration written outside the seeder (e.g., a script the
v3 developer ships that issues an UPDATE through the normal repo path
on every untouched `seed_*` row). Out of scope for this design.

## 6. Reinstall and sign-out semantics

### 6.1 Reinstall (same account on the same device)

Covered exhaustively by §5.1 and §5.2. Summary:

- A reinstall is, from the seeder's perspective, the same as a fresh
  device — `bootstrap_completed_at` starts NULL because it lived in
  SQLite on the now-deleted DB.
- The first cycle pulls everything the server holds for this user (rows
  and tombstones).
- If the user had ever pushed any row, the post-cycle hook sees
  non-empty entity tables and no-ops.
- Only if the server holds nothing (e.g., an account that has never
  successfully completed a sync cycle on any device) does the seeder
  re-seed.

### 6.2 Sign-out / account switch

Per t2 §5, the auth layer is responsible for wiping local entity tables
on sign-out (or on a `auth.uid()` change). The seeder does not handle
this; it sits downstream of the wipe.

After a wipe:

- `muscle_groups` is preserved (it's client-only taxonomy; no account
  ownership).
- The 8 syncable entity tables are empty.
- `sync_runtime_state.bootstrap_completed_at` is **reset to NULL** by
  the wipe (so the next sign-in's first cycle can re-fire the hook).
  This is a t2-emitted contract on the auth wipe: it must clear the
  bootstrap flag along with the entity rows, or the seeder will never
  fire for the new account. We assert this as a t3 → auth-wipe
  constraint below.
- `sync_runtime_state.last_emitted_ms` is **preserved** per t2 §8.3
  ("the counter is device-global, not per-user … any future
  local-wipe-on-account-switch path must leave `last_emitted_ms`
  alone").
- The four per-layer pull cursors are reset to null by the wipe (also
  per t2 — they reference an `owner_user_id` that no longer matches).

The next sign-in re-enters the §1.1 sequence as if the device were
fresh.

> **t3 constraint emitted to the auth-wipe path.** The auth-layer's
> sign-out / account-switch wipe MUST:
>
> 1. Clear all rows from the 8 syncable entity tables (already specified
>    by t2 §5 precondition).
> 2. Reset `sync_runtime_state.bootstrap_completed_at` to NULL.
> 3. Reset `sync_runtime_state.pull_cursor` to `{}` (the empty JSON map
>    per t2 §9.2).
> 4. PRESERVE `sync_runtime_state.last_emitted_ms` (per t2 §8.3).
> 5. PRESERVE `muscle_groups` rows (client-only taxonomy; no per-user
>    ownership).
>
> If 2 is forgotten, the seeder never fires for the new account and
> the new account will see an empty catalog after sync (which is fine
> if they have remote data, broken if they don't).
>
> This constraint is repeated in §9.2 of this doc as a
> machine-checkable item.

### 6.3 Wipe-local-only (dev tool)

The dev-only `resetLocalDataAndReseed` (in
`apps/mobile/src/data/dev-reset.ts`) is the dev affordance. For v2 it
maps to:

1. Drop and recreate the 8 syncable entity tables, `muscle_groups`,
   and `sync_runtime_state`.
2. Re-run the `muscle_groups` bootstrap seed (§4).
3. Sign-in flow re-enters §1.1 from scratch.

This is identical to "uninstall and reinstall the app." No special
seed-loader path is needed. (Compare PR #57 which carried a separate
"wipe-and-reseed" loader path; v2 drops it because the once-per-device
loader doesn't have multiple paths.)

## 7. Two-device semantics

### 7.1 Both devices fresh on the same account, simultaneous sign-in

Worst case for the seeder:

- Device A signs in at T_a. Cycle 1 starts; pull returns empty.
- Device B signs in at T_b ≈ T_a (within seconds). Cycle 1 starts; pull
  also returns empty (A's push has not yet committed).
- Both devices reach the post-cycle hook with empty entity tables.
  Both seed.
- Both push. Server upserts by `(owner, id)`. Whichever push commits
  second is LWW-resolved per row: the slugs are identical, the
  `client_updated_at_ms` values differ by milliseconds, the row content
  is identical (the bundle is deterministic). Server ends up with one
  row per slug.
- Both devices pull on cycle 2. Each device's local row may be
  overwritten by the server's LWW winner (same content; visible no-op).

**Convergence is automatic. No duplicate rows. No special handling.**

### 7.2 Why slugs are load-bearing here

If seeds had UUIDs, step 1 above would produce
`(owner_user_id, <uuid_a>)` from device A and
`(owner_user_id, <uuid_b>)` from device B for the same logical seed.
The server would have two rows for "Bench Press" because the PK
`(owner, id)` makes them distinct. The user would see "Bench Press"
twice in their catalog after pull.

Slug IDs collapse that into a single row by construction. The cost is a
naming discipline on bundle authors (each seed must have a stable,
unique slug); the benefit is no de-duplication logic anywhere.

### 7.3 User edits a seed on one device, second device hasn't seeded yet

- Device A: seeded `seed_bench_press` at T1, pushed. User renames to
  "My Bench" at T2, pushed.
- Device B: signs in for the first time. Cycle 1 pulls. Server returns
  `seed_bench_press` with `name='My Bench'`. Local row INSERTed (t2 §4.5
  step 2). Local row count = 1.
- Post-cycle hook on B: `isFreshOnTheCloud = false`. Seeder no-ops.

Device B sees "My Bench" — the user's edit — not the bundle's
"Bench Press". Correct: the server is authoritative; the bundle is a
fallback for the truly-fresh case only.

## 8. What this design does NOT do, and why

### 8.1 Does not propagate bundle-field updates to existing users

**Explicit, user-confirmed tradeoff.** Once the seeder has fired on a
device, it never fires again. Bundle changes after that — adding a
field to an existing seed, changing a canonical default, adding a
new seed slug — do not reach existing users on that device.

**Why this is acceptable for v2:**

- Existing users have their local data, including any user edits.
  Nothing they have done is lost.
- New users on app v3 get the v3 bundle on their first seed.
- The simplification cost — no `local_seed_version`, no per-row
  lineage, no additive-merge code path, no version-bump gate — is
  large enough that the tradeoff is preferred.
- If a specific bundle update is important enough to backfill, that
  is a one-off operational migration outside the seeder.

A future iteration could add an additive-merge pass (PR #57's
`local_seed_version` proposal) without changing the v2 ship's contract:
the column is purely additive, and the existing seeder remains the
once-per-device path with the merge layered on top. We are not
designing it now because we do not yet have evidence we need it.

### 8.2 Does not handle "the user wants their seeds back"

If a user soft-deletes every seed and later wants them restored, the
v2 design has no UI path for that. The seeder is permanently inert on
their device, and pulling from the server returns the same tombstones.

**Workaround for v2:** dev-only `resetLocalDataAndReseed`. Not
user-facing.

**Real solution for a future ship:** a "factory reset to bundle" UI
affordance that issues an UPDATE through the normal repo path on
every `seed_*` row, setting `deleted_at = null` and updating other
fields to the bundle's canonical values. This is a feature, not a
sync-engine change. Out of scope.

### 8.3 Does not introduce any local-only columns

Per the user-confirmed direction (and the override of the plan card),
no `seed_origin` and no `local_seed_version`. Therefore nothing is
added to `apps/mobile/src/data/schema/sync-extras.json` under
`exemptions.local_only_columns` for this design.

t2 §9.4's `local_only_columns: ["local_dirty", "local_updated_at_ms"]`
remains the complete inventory.

### 8.4 Does not change `muscle_groups`

`muscle_groups` is seeded unconditionally on every fresh install
(§4), independently of sync. This is the existing v1 behaviour
preserved verbatim into v2.

### 8.5 Does not introduce sentinel timestamps

The PR #57 draft proposed `local_updated_at_ms = 0` as a "loser
sentinel" so that seed-then-pull could resolve via LWW with the
pull winning. v2 does not need this because seed-then-pull never
happens: the pull always runs first (§1.2). Every write the seeder
issues uses `nowMonotonic()` like any other write.

### 8.6 Does not add a separate "seed has run" marker

The trigger and gate together (§1.1, §3.1) are sufficient. No
`seedsAppliedAt`, no `local_seed_done` boolean, no extra column on
`sync_runtime_state`. The existing `seedsAppliedAt` column on
`sync_runtime_state` (declared at
`apps/mobile/src/data/schema/sync-runtime-state.ts:13`) is **removed**
by the t3 build wave's local migration — it has no analogue in the v2
model.

### 9.1 Constraints from t1 / t2 honoured by this design

| Constraint | Honoured in |
|---|---|
| t1 §1.1 — deletion is `deleted_at != null`, no `deleted boolean` | §3.1 (the emptiness check counts tombstoned rows), §5.2 (worked example). |
| t1 §1.1.1 — row-level LWW on `client_updated_at_ms` | §5.1, §5.2, §5.3 use t2's normal LWW; the seeder writes a row that becomes a normal LWW participant. |
| t1 §4.1 — local-only columns registered in `sync-extras.json` | §8.3 — no local-only columns introduced. Inventory unchanged. |
| t2 §3.4 — push topological order | §3.4 — seeded rows go in via repo, which means push picks them up via `selectPushBatch()` walking the same topological order. `exercise_definitions` (Layer 0) and `exercise_muscle_mappings` (Layer 1) ship in the correct order. |
| t2 §5 — `bootstrap_completed_at` is the first-cycle flag | §1.1 makes this the exact seeder trigger. |
| t2 §7 — dirty-bit lifecycle | §1.4 — seeder uses normal repo path; `local_dirty = 1` is set, `local_updated_at_ms = nowMonotonic()`. Push ack clears `local_dirty` per t2 §7.2. |
| t2 §8 — clock monotonicity | §1.4 — every seed write goes through `nowMonotonic()`. No sentinel-zero hacks. |
| t1 §2.6.1 — `muscle_groups` is not synced | §4 — `muscle_groups` is seeded client-locally only. |

| # | Constraint | Target |
|---|---|---|
| 1 | The auth-layer sign-out / account-switch wipe MUST reset `sync_runtime_state.bootstrap_completed_at` to NULL (in addition to clearing entity rows per t2 §5 precondition). Otherwise the seeder never fires for the new account on this device. | auth-layer build task (out of scope for sync v2 build wave per t2 §5; called out as a flagged follow-up). |
| 2 | The bundle author MUST use stable `seed_*` slug IDs for `exercise_definitions` and `seed_map_*__*` slugs for `exercise_muscle_mappings`. The v2 build wave drops the `sys_*` rename in the bundle file and never reuses an old slug for a new seed. | t3 build wave (the bundle file rename). |

| 4 | The t3 build wave MUST implement the post-cycle hook as a deterministic edge-trigger on `bootstrap_completed_at NULL → non-NULL`, not as a per-cycle check. Re-firing on every cycle would re-seed every cycle after the server got wiped — which would surprise dev environments that wipe the hosted DB. | t3 build wave. |
| 5 | The seeder MUST run inside a single Drizzle transaction so the bundle either fully applies or fully rolls back. Partial seed states are not legal. | t3 build wave. |

### 9.3 Deviations from the plan card

The plan card's t3 task (as it stands at merge time of t1 and t2) gave
six numbered "Direction" points. This design honours #1 and #4 directly,
and **explicitly overrides** #2, #3, #5, and #6 per the user-confirmed
re-simplification. The overrides:

| Plan-card direction | This design's position |
|---|---|
| #1 — "Seeder runs AFTER every sync cycle" | **Adopted in a stricter form**: seeder runs after **exactly one** sync cycle (the one that sets `bootstrap_completed_at`). Not every cycle. |
| #2 — "Seeder NEVER writes sync-relevant columns" | **Overridden.** Seeder uses the normal repo path and writes `local_dirty = 1` and `local_updated_at_ms` like any other repo caller. The seeder fires only once; there is no later-pass interaction with the dirty-bit lifecycle to worry about. |
| #3 — "Sync ALWAYS overwrites seed data" | **Overridden in framing.** The seeder runs *after* sync, so there is nothing for sync to overwrite — the seeder fills only the truly-empty case. Sync's LWW remains authoritative for everything that follows. |
| #4 — "All deletes are soft" | **Adopted verbatim.** Inherits t1 §1.1. |
| #5 — "`seed_origin` is a boolean" | **Overridden.** Dropped entirely; no per-row lineage needed for a once-per-device seeder. |
| #6 — "Two-device LWW for seed renames is acceptable" | **Adopted verbatim.** §5.3 / §7 cover the case. |

The plan card's mention of `local_seed_version` and an additive-merge
path is similarly dropped. The card's prediction was that this would
be too complex to ship cleanly, and the design agrees: it is omitted
on simplicity grounds, with the tradeoff documented (§8.1).

## 10. What this design does not cover

Closed in companion design docs:

- **t1** — server schema, RLS, deferrable FKs, LWW semantics, drift checker.
- **t2** — push/pull wire shape, dirty-bit lifecycle, clock monotonicity, cycle interface.
- **t4** — iOS lifecycle triggers, the scheduler that runs cycles.

Out of scope here as deliberate boundaries:

- The bundle file format and the `sys_` → `seed_` rename diff.
  That belongs to the t3 build wave's loader rewrite.
- The local Drizzle migration that removes the
  `sync_runtime_state.seedsAppliedAt` column. Build wave.
- The repo-level call site that installs the post-cycle hook into the
  cycle runtime. Build wave; the hook contract is in §1.1.
- The auth-layer wipe behaviour (§6.2's constraint is emitted to
  whoever owns the wipe; a separate task, possibly outside sync v2).
- A "factory reset to bundle" UI affordance (§8.2). Future product
  feature.
- Tests. Build wave; the worked examples in §5 are the test scenarios.
