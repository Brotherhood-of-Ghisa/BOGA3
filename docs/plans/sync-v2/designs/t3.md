# t3 — Sync v2: seeded entities & user edits

> Status: design only. Builds on [plan.md](../plan.md) and [brainstorm.md](../brainstorm.md).
> No code changes ship with this PR.

## TL;DR

- **Slug IDs stay; rename the prefix from `sys_` to `seed_`.** Brainstorm's
  `seed:bench-press` notation maps to `seed_bench_press` once we drop the colon (SQLite
  text PKs are fine with colons, but every other repo treats `:` as a separator and we
  want one canonical shape across local + server). Confirmed: fixed slugs, not UUIDs.
- **Seed loader is idempotent, gated by `local_seed_version` + per-row `seed_origin`.**
  Loader runs every launch; the gate determines whether it acts.
- **Seeds are user-owned the moment they exist.** No separate "system catalog" on the
  server. The composite PK `(owner_user_id, id)` is what makes that safe.
- **User edits win, always.** Once a seeded row is touched (rename, soft-delete, custom
  field), the loader treats it as user-owned and never overwrites it. Tracked by
  `seed_origin` transitioning from `'seed'` → `'seed_modified'` and by the existing
  `local_dirty` bit.
- **Deletion is preserved across reinstall.** Re-seeding happens only on a truly empty
  local DB (count of every user-owned entity table is zero). After first sync, "absent"
  always means "deleted by user."
- **New fields on existing seeds reach existing users via additive merge** — but only
  for columns the user has not explicitly written. Renames, deletes, custom mappings
  are untouched.
- **Seed *removal* in a future app version is a no-op.** Bundled list is additive.
- **A new local column `seed_origin text` is added to every table that holds seedable
  entities** (`exercise_definitions`, `muscle_groups`, `exercise_muscle_mappings`).
  This column is **local-only**. The server doesn't see it; for the server, every row
  is just a user row.

## 2. Identity & schema

### 2.1 Slug IDs (confirmed from brainstorm)

Fixed slugs survive reinstall, allow LWW upserts across devices, and let the server's
composite PK `(owner_user_id, id)` absorb cross-user "collisions" naturally (they're
not collisions; same slug, different owner, different row).

**Concrete shape:**

- Exercise definitions: `seed_<snake_case_name>` — e.g. `seed_bench_press`,
  `seed_barbell_back_squat`. The migration from the existing `sys_*` prefix is a
  one-time rename in the seed bundle; existing on-device rows are touched by the
  upgrade-merge logic in §6.
- Muscle groups: bare slugs stay (`chest`, `delts_front`). They're a fixed taxonomy
  and the `is_editable = 0` check makes the prefix moot.
- Muscle mappings: `seed_map_<exercise_slug>__<muscle_slug>` — e.g.
  `seed_map_bench_press__chest`. Composite uniqueness is also enforced by the
  `(exercise_definition_id, muscle_group_id)` index, but the PK is the slug.

**Why not UUIDs:** UUIDs would force the loader to either query before inserting (a
race when two devices first-launch simultaneously, in the absence of sync) or accept
duplicates that need a downstream dedup pass. Slugs make the loader a pure upsert.

**Why rename `sys_` → `seed_`:** "system" implies a separate ownership tier we are
deliberately dropping in v2 (no system catalog on the server). The slug prefix should
reflect lineage, not ownership.

### 2.2 New local-only column: `seed_origin`

Add to each seedable client table (`exercise_definitions`, `muscle_groups`,
`exercise_muscle_mappings`):

```sql
seed_origin text not null default 'user'
  check (seed_origin in ('user', 'seed', 'seed_modified'))
```

Values:

| Value            | Meaning                                                          |
| ---------------- | ---------------------------------------------------------------- |
| `user`           | Row was created by the user (default for `INSERT … (id, name)`). |
| `seed`           | Row was inserted by the seed loader and has not been touched.    |
| `seed_modified`  | Row started as `seed`; user has since edited at least one field. |

**Local-only:** the server schema (per t1) does **not** include `seed_origin`. It's
not in the wire payload (because v2 push sends typed rows, not raw row dumps; the
column is just left out of the serializer for these tables). On pull, the column is
not present in the incoming payload and the client's pull-applier leaves it alone —
it only updates the columns that arrive.

**Why this column at all:** the seed loader needs to know "did the user touch this?"
to decide whether new bundled fields can safely merge in. Without it, the loader has
no way to tell `(seed inserted, never edited)` apart from `(seed inserted, user
deliberately reverted to seed defaults)`. The latter is a vanishingly rare case we
do not optimise for; the column tracks "ever modified," not "currently equal to
seed defaults."

The brainstorm's alternative — "distinguish by slug prefix" — works for *identity*
but not for *modification state*. The slug stays `seed_bench_press` even after rename
("My Bench"); the column is what flips. Both signals are useful; they answer
different questions.

### 2.3 Local seed-version marker

Replace the existing `seedsAppliedAt: timestamp_ms` on `sync_runtime_state` with two
columns:

```sql
alter table sync_runtime_state add column local_seed_version integer not null default 0;
alter table sync_runtime_state add column local_seed_applied_at integer; -- ms epoch
```

- `local_seed_version` is an integer monotonic counter. The bundle ships with a
  `BUNDLED_SEED_VERSION` constant. After the loader successfully applies version N to
  the local DB, it writes `local_seed_version = N`.
- `local_seed_applied_at` is informational (last successful seed pass). Useful for
  the Settings → Sync status screen.

The boolean-ish `seedsAppliedAt != null` from fix-sync T8 is reinterpreted as
`local_seed_version > 0`.

Bumping the version is **manual, by the developer who ships a bundle change**. The
seed bundle module exports `BUNDLED_SEED_VERSION = 5` (etc.); editing the bundle
without bumping the version is a footgun that the drift checker from t1 can be
extended to catch later. For v2's first ship, version starts at `1`.

## 3. Decision tree (per-seed, per-local-state)

The loader runs every app launch and after every bootstrap merge. For each seed in the
bundled list, it inspects the local row (or its absence) and one global signal (is
the local DB completely empty?).

### 3.1 Inputs the loader has, per pass

- `BUNDLED_SEED_VERSION` (constant from the bundle module).
- `local_seed_version` (from `sync_runtime_state`).
- For each seedable table, a count of all rows (used to decide "truly fresh install").
- For each seed `s`, the local row keyed by `s.id`:
  - **Absent**: no row with `id = s.id`.
  - **Present, `seed_origin = 'seed'`**: untouched seed row.
  - **Present, `seed_origin = 'seed_modified'`**: user has edited this row.
  - **Present, `seed_origin = 'user'`**: ID collision (extremely unlikely with slug
    prefix; treat as user-owned, no-op). Documented as defensive.
  - **Soft-deleted** (`deleted_at is not null`, `seed_origin in ('seed','seed_modified')`):
    user has deleted the seed.

### 3.2 Global precondition: "is local truly empty?"

```
isFreshInstall =
  count(exercise_definitions) == 0
  && count(muscle_groups) == 0
  && count(exercise_muscle_mappings) == 0
  && count(gyms) == 0
  && count(sessions) == 0
  && count(session_exercises) == 0
  && count(exercise_sets) == 0
  && count(exercise_tag_definitions) == 0
  && count(session_exercise_tags) == 0
  && local_seed_version == 0
```

The combination of "all data tables empty" *and* `local_seed_version == 0` is what
distinguishes "first launch ever" from "user deleted everything." The
`local_seed_version` check is the load-bearing one: a user who manually deletes every
row but has previously seeded is `local_seed_version > 0`, so we don't re-seed.

`local_seed_version` is **never reset to zero by user action**. Only the dev-only
`resetLocalDataAndReseed` resets it.

### 3.3 The table

| Local state                         | Bundle action               | `BUNDLED > local_seed_version`? | Result                                                                                |
| ----------------------------------- | --------------------------- | ------------------------------- | ------------------------------------------------------------------------------------- |
| Fresh install (per §3.2)            | seed present                | yes (0 < N)                     | **Insert** with `seed_origin='seed'`, `local_dirty=true`, `local_updated_at_ms=now`.  |
| Fresh install                       | seed absent from bundle     | n/a                             | no-op (nothing to do).                                                                |
| Row absent, not fresh install       | seed present                | irrelevant                      | **No-op.** Treat absence as intentional deletion (post-sync invariant).               |
| Row present, `seed`                 | seed present, same fields   | no                              | **No-op.** Fast path; loader exits before touching the row.                           |
| Row present, `seed`                 | seed present, new field added | yes                           | **Additive merge:** patch only the new column(s). `seed_origin` stays `seed`. `local_dirty=true`, `local_updated_at_ms=now`. |
| Row present, `seed`                 | seed present, *existing* field value changed in bundle | yes | **No-op on that field.** The bundle author meant to update content for new users; existing users keep their version. Logged once at debug. |
| Row present, `seed_modified`        | seed present, any change    | yes                             | **No-op.** User has touched this row; the loader must never write to it again.        |
| Row present, `user` (slug collision) | seed present               | any                             | **No-op.** Defensive: should never happen with `seed_` prefix discipline.             |
| Row soft-deleted, `seed*`           | seed present                | any                             | **No-op.** User deleted it; respect that.                                             |
| Row present, *any* origin           | seed absent from bundle     | n/a                             | **No-op.** Removal is additive-only; we never delete on the user's behalf.            |

### 3.4 Order of operations inside a single loader pass

```
1. Read BUNDLED_SEED_VERSION, local_seed_version, isFreshInstall.
2. If BUNDLED_SEED_VERSION == local_seed_version, return immediately. (Steady state.)
3. If isFreshInstall:
     3a. Insert every seed with seed_origin='seed'.
     3b. Write local_seed_version = BUNDLED_SEED_VERSION, local_seed_applied_at = now.
     3c. Mark every inserted row local_dirty=true (so next push uploads them).
4. Else (existing install, version bump):
     4a. For each seed in bundle, look up local row by id.
     4b. If absent: NO-OP (post-sync invariant says absence = deletion).
     4c. If present + seed_origin='seed' + bundle adds new column(s):
           additive UPDATE on new column(s) only; keep seed_origin='seed';
           local_dirty=true, local_updated_at_ms=now.
     4d. All other cases: no-op.
     4e. After processing all seeds, write local_seed_version = BUNDLED_SEED_VERSION,
         local_seed_applied_at = now.
```

Step 2 is the hot path; on the steady state (no version bump, no fresh install) the
loader exits after one indexed read. Step 4 is run-once-per-version-bump for existing
users.

The whole pass runs in one SQLite transaction so the version bump is atomic with the
field merges.

## 4. Deletion across reinstall

**Re-seed only on truly empty local DB.** Confirmed (matches the plan's likely answer).

The mechanic that makes "deleted" stick across reinstall is **the server, not the
loader**:

1. User deletes `seed_bench_press` on device A. Local soft-delete sets
   `deleted_at = now`, `local_dirty = true`. Sync v2 push uploads the row with
   `deleted = true`.
2. Server stores the row with `deleted = true`, latest payload retained (per
   brainstorm §3.1 tombstone semantics — though we are using typed schemas not blobs,
   the principle stands: a deleted row still exists with `deleted = true`).
3. User reinstalls. Local DB is empty; `local_seed_version = 0`. Loader inserts
   `seed_bench_press` with `seed_origin='seed'`, `local_dirty=true`. Catalog UI
   shows it for the brief window before the first pull.
4. Sync pull arrives. Server's deleted tombstone for `seed_bench_press` is applied:
   local row is soft-deleted (or hard-deleted, per t2's tombstone rules). `local_dirty`
   for that row is cleared because the pull's `client_updated_at_ms` for the tombstone
   is newer than the loader's `now` (the user's delete on device A is in the past, but
   it's monotonically after `now` for the seeded insert? — see below).

**Subtle: ordering of `local_updated_at_ms` on re-seed vs the tombstone.**

The reinstalled loader writes `local_updated_at_ms = now` (some time T2). The user's
original delete on device A had timestamp T1 < T2. Naive LWW would say "the re-seed
wins; un-delete it." That's wrong.

Resolution: the loader on a fresh install writes `local_updated_at_ms = 0` (or
`Number.MIN_SAFE_INTEGER + 1`, a sentinel meaningfully less than any real wall-clock
timestamp). The intent is "I am writing this row but I do not claim any
authoritative recency for it; if the server has a newer copy — including a tombstone
— that copy wins." The `local_dirty = true` still pushes the row, but the server's
LWW (`incoming_updated_at_ms > stored_updated_at_ms`) discards it because
`0 < T1`.

This is the load-bearing trick that makes deletion sticky. Spelling it out:

- **Fresh-install seed write:** `local_updated_at_ms = 0`, `local_dirty = true`.
- **Pull applies tombstone with T1 > 0:** local row is tombstoned. `local_dirty` is
  cleared (per t2's pull-cleared-dirty rule).
- **Push runs (could be before or after the pull):** if before, server rejects via LWW
  (`0 < T1`); push response says "rejected, no-op," client clears `local_dirty`. If
  after the pull, the row is already tombstoned locally and the push sends the
  tombstone, which is a no-op.

For the user who *hasn't* deleted anything, the seed never had a remote tombstone, so
the LWW-zero push is accepted into a never-before-seen row on the server and they're
back to steady state.

> Open question for t2 to confirm: t2 needs to spec that "client uploads `updated_at_ms
> = 0` for an authoritative-but-stale write means LWW will reject if the server has
> anything." This is a small contract addition that t3 is asserting; if t2 disagrees,
> we use a different sentinel (e.g., `1` or "skip push for `updated_at_ms == 0`"). The
> sentinel value itself isn't critical; the semantic of "loader writes do not claim
> recency" is.

### 4.1 Why the count-based fresh-install heuristic is correct

Consider the failure mode: user deletes ALL data (`Settings → Wipe local`), then opens
the app. Local tables are empty. Naively, `isFreshInstall = true` and we re-seed.

**`local_seed_version` saves us.** Wipe-local resets data tables but NOT
`sync_runtime_state`'s `local_seed_version`. So `isFreshInstall` is false on this
device, and the loader does nothing. The next pull restores from server (including
any tombstones).

The only thing that resets `local_seed_version` is `resetLocalDataAndReseed` (dev
only). Production users cannot accidentally trigger a re-seed.

## 5. New fields on existing seeds (P5 resolution)

**Picked: additive merge (option a), with hard guards against overwriting user edits.**

Reasoning:

- **Option b (only fresh installs get new fields)** strands the entire existing user
  base on stale data. If we add `default_rest_sec` to all seeds in v3.0, no v2.x user
  ever sees it without a full local reset. That's the v1 sync's "projection drift"
  problem all over again, just at the seed layer.
- **Option a (merge new fields into existing rows)** keeps the existing user base on
  the same footing as fresh installs for *bundle-shipped* fields, while explicitly
  preserving user-modified rows.

The mechanism is the version-bump path (§3.3 step 4):

- If `BUNDLED_SEED_VERSION > local_seed_version`, the loader iterates the bundle.
- For each present-and-untouched row (`seed_origin = 'seed'`), it diffs the bundle
  against the local row column-by-column. Columns that are NULL/default locally and
  non-NULL in the bundle are patched. Other columns are left alone.
- Rows with `seed_origin = 'seed_modified'` are skipped entirely.

The patched columns become `local_dirty = true` so the change propagates to the
server (so other devices owned by the same user converge). On the next sync, LWW
makes this safe: the loader's `local_updated_at_ms = now` is strictly later than
the previous server copy.

**Crucial constraint: bundle changes to *existing* fields on already-seeded rows are
not propagated.** If the bundle changes the canonical name from "Bench Press" to
"Barbell Bench Press" in v5, existing users with `seed_origin='seed'` still see
"Bench Press." Only new users get the new name. The decision-tree row "seed present,
existing field value changed" is no-op for this reason.

This is conservative on purpose. We're choosing "stale name on a few thousand users"
over "silent overwrite of someone's data we mistakenly thought was untouched." A
future minor (e.g., a `corrective_seed_version` companion that *does* overwrite
`seed_origin='seed'` rows) could change this, but is out of scope.

### 5.1 Detecting "new column added since last seed pass"

The loader doesn't need a SQL `information_schema` query — it has the bundle
in memory and the local row. The diff is:

```
for each column in BundledSeed:
  bundleValue = bundle[column]
  localValue  = localRow[column]
  if bundleValue is not null and localValue is null and seed_origin == 'seed':
    patch[column] = bundleValue
```

This handles the "added a new nullable field with a value in the bundle" case
exactly. If we ever add a non-nullable column with no default, the migration is
responsible for backfilling — outside the loader's remit.

## 6. Seed removal in a future app version

**Confirmed: no-op.** The bundled list is additive-only.

If v3 ships without `seed_bench_press` and the user has it locally:
- The loader iterates only the bundle. A bundled-absent seed is invisible to the
  loader. It does not delete anything.
- The user's row stays. If `seed_origin='seed_modified'`, it's the user's data.
  If `seed_origin='seed'`, it's a row whose canonical bundle entry is gone — still
  the user's data from the system's perspective; the slug prefix is a lineage marker
  only.

**Why not "if you remove from bundle, soft-delete locally":** auto-deleting data
the user can see is a major surprise factor and breaches the v2 invariant that
"absence after first sync = user-initiated deletion." A bundle author wanting a
seed gone should publish guidance for users to delete it themselves, or ship a
schema-version migration that explicitly does it (out of scope for the seed loader).

Documented expectation: bundle authors **never remove**, only **add** or **bump
existing canonical content for new installs**. The decision-tree row "seed absent
from bundle, row present" is permanent no-op.

## 7. Two-device LWW for user-edited seeds

Per the brainstorm, two-device support is purpose #2 (low priority). LWW on
`local_updated_at_ms` is the resolution mechanism. Two specific cases worth pinning:

### 7.1 Two devices, both un-modified seed, only A renames

- Device A: rename to "My Bench" at T=100. `seed_origin: seed → seed_modified`,
  `local_dirty=true`, `local_updated_at_ms=100`. Pushes to server.
- Device B: still has `seed_origin='seed'`, `name='Bench Press'`,
  `local_updated_at_ms=0` (sentinel from fresh-install seed write).
- Device B pulls. Server has `name='My Bench'`, `updated_at_ms=100`. LWW says
  remote wins (`100 > 0`). Local row updated to `name='My Bench'`,
  `local_updated_at_ms=100`, `local_dirty=false`.
- **Question: does device B's `seed_origin` become `seed_modified`?** Yes. The pull
  applier must promote `seed → seed_modified` whenever it writes a non-default value
  that originated from another device's edit. This requires the pull applier to know
  the row "came from outside this device's seed loader" — easy: any pull update is
  by definition external. Rule: **any pull-side write to a row whose local
  `seed_origin == 'seed'` promotes it to `seed_modified`**. This locks the row out of
  future additive-field merges, which is the correct behaviour (the bundle and the
  user-edited remote both think they have the canonical value; user-edited remote
  wins).

### 7.2 Both devices rename the same seed within the same minute

- Device A renames to "Bench A" at T=200, pushes.
- Device B renames to "Bench B" at T=210 (latest 10ms after A's commit; B has not yet
  pulled), pushes.
- Server LWW: `210 > 200`, B's name wins. Both devices end up with "Bench B" on next
  pull. A's "Bench A" is lost. Accepted tradeoff per brainstorm §5.
- `seed_origin` ends up `seed_modified` on both devices.

There is no user-visible warning. The brainstorm explicitly accepts this for
purpose #2.

### 7.3 Clock monotonicity guard

t2 specifies `local_updated_at_ms = max(Date.now(), last_emitted + 1)`. The seed
loader's fresh-install path bypasses this — it intentionally writes the sentinel `0`,
which is *less* than `last_emitted`. That's fine because `last_emitted` is per-device
and the sentinel write happens before any other write on a fresh install. The
loader's additive-merge path (existing user, version bump) uses the normal
monotonicity guard.

## 8. Worked examples

### 8.1 User renames "Bench Press" → "My Bench", syncs, reinstalls

**Setup.** User installs the app. `BUNDLED_SEED_VERSION = 1`,
`local_seed_version = 0`, all tables empty. Loader runs:

| Step                       | Action                                                                                                             |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| Launch 1, fresh install    | Loader inserts `(id='seed_bench_press', name='Bench Press', seed_origin='seed', local_dirty=true, local_updated_at_ms=0)`. Then `local_seed_version := 1`. |
| First sync push            | Pushes the row to server. Server stores it with `updated_at_ms=0` (acceptable; no prior row).                       |
| User renames in UI         | Repo writes `name='My Bench', local_dirty=true, local_updated_at_ms=Date.now()=T1000`. Trigger: `seed_origin: 'seed' → 'seed_modified'` (the repo's update path checks current origin and promotes on user-driven write). |
| Sync push at debounce      | Server receives `(name='My Bench', updated_at_ms=T1000, deleted=false)`. LWW: `T1000 > 0`, accepted.                |
| Sync pull (no-op)          | Nothing new for this user.                                                                                          |
| User deletes app, reinstalls. Loader runs on empty DB: |                                                                                          |
| Launch N, fresh install    | All tables empty, `local_seed_version=0`, so `isFreshInstall=true`. Loader inserts `(id='seed_bench_press', name='Bench Press', seed_origin='seed', local_dirty=true, local_updated_at_ms=0)`. |
| Initial pull               | Server returns the row with `(name='My Bench', updated_at_ms=T1000)`. Pull applier LWW: `T1000 > 0`, remote wins. Local row updated to `name='My Bench', local_updated_at_ms=T1000, local_dirty=false`. **`seed_origin` is promoted to `seed_modified`** because the pull touched a `seed` row. |
| Initial-sync modal         | Per t2: this user has remote data AND local data (`local_dirty=true` on the seed row from the fresh insert). Modal asks "Keep local and merge with cloud" / "Replace local with cloud." On "Merge" (default), the LWW above plays out. On "Replace," local is wiped before pull — same end state since "My Bench" is what's on the server. |
| Steady state               | UI shows "My Bench." `seed_origin='seed_modified'`. The slug is still `seed_bench_press`.                           |

**What the user sees:** "My Bench" is restored within ~1 minute of foreground per
plan outcome #1.

### 8.2 User deletes Bench Press, syncs, reinstalls

| Step                       | Action                                                                                                          |
| -------------------------- | --------------------------------------------------------------------------------------------------------------- |
| Launch 1, fresh install    | Loader inserts `(id='seed_bench_press', name='Bench Press', seed_origin='seed', local_dirty=true, local_updated_at_ms=0)`. `local_seed_version := 1`. |
| First sync push            | Row uploaded.                                                                                                   |
| User deletes via UI        | Repo writes `deleted_at = T500, local_dirty=true, local_updated_at_ms=T500`. (Repo soft-delete promotes origin: `seed → seed_modified`, since deletion is a user action.) |
| Push at debounce           | Server receives `(deleted=true, updated_at_ms=T500)`. Stored.                                                   |
| User reinstalls. Loader on empty DB: |                                                                                                       |
| Launch N, fresh install    | Loader inserts `(id='seed_bench_press', name='Bench Press', seed_origin='seed', local_dirty=true, local_updated_at_ms=0)`. |
| Initial pull               | Server returns `(deleted=true, updated_at_ms=T500)`. LWW `T500 > 0`. Local row tombstoned: `deleted_at=T500, name='Bench Press', local_updated_at_ms=T500, local_dirty=false, seed_origin='seed_modified'` (promoted by pull). |
| Loader subsequent passes   | `BUNDLED_SEED_VERSION == local_seed_version`, early return. The bench row stays tombstoned forever. |

**What the user sees:** Bench Press is not in the catalog. The deletion survived
reinstall — not because the loader refused to re-seed (it did re-seed transiently),
but because the server tombstone outranks the loader's sentinel write.

### 8.3 Two devices rename within the same minute (LWW)

| Time | Device A                                                                              | Device B                                                                              | Server                                                  |
| ---- | ------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------- | ------------------------------------------------------- |
| T0   | Fresh-installed; row seeded `(name='Bench Press', updated_at_ms=0, dirty=true)`       | Fresh-installed; same                                                                 | empty                                                   |
| T10  | Push: `name='Bench Press', updated_at_ms=0`                                            |                                                                                       | stores `(name='Bench Press', updated_at_ms=0)`          |
| T20  |                                                                                       | Push: `name='Bench Press', updated_at_ms=0`                                            | LWW: `0 ≤ 0`, no-op                                     |
| T100 | User renames to "Bench A". Local: `(name='Bench A', updated_at_ms=T100, dirty=true, seed_origin='seed_modified')` |                                                                                       |                                                         |
| T110 | Push: `name='Bench A', updated_at_ms=T100`                                             |                                                                                       | stores `(name='Bench A', updated_at_ms=T100)`           |
| T120 |                                                                                       | User renames to "Bench B". Local: `(name='Bench B', updated_at_ms=T120, dirty=true, seed_origin='seed_modified')` |                                                         |
| T125 |                                                                                       | Push: `name='Bench B', updated_at_ms=T120`                                             | LWW: `T120 > T100`, stores `(name='Bench B', updated_at_ms=T120)` |
| T200 | Pull. Server has `name='Bench B', updated_at_ms=T120`. LWW: `T120 > T100`, accept. Local: `name='Bench B', updated_at_ms=T120, dirty=false, seed_origin='seed_modified'` |                                                                                       |                                                         |
| T210 |                                                                                       | Pull: no change since last push                                                       |                                                         |

**Outcome:** Both devices converge to "Bench B." Device A's "Bench A" rename is lost.
This matches brainstorm §5's acceptance of LWW loss for two-device simultaneous edits.

**What about simultaneous identical timestamps?** If both pushes land at `T100`
exactly, LWW would be `T100 > T100 = false`, so whichever the server upserts second is
discarded. Server-side tiebreak per t2's pagination/cursor spec uses
`(owner, type, id)` as a secondary key — but for a single `(owner, type, id)` row,
the tiebreak is "whichever transaction the server commits second wins" since the
first one's value is what `updated_at_ms` is compared *against*. Acceptable for
purpose #2.

### 8.4 App upgrade adds `default_rest_sec` to all seeds

Setup: existing user on v2.0 with `BUNDLED_SEED_VERSION = 1`, has not modified any
seeds. Local rows:

```
exercise_definitions
  id='seed_bench_press' name='Bench Press' seed_origin='seed' default_rest_sec=NULL
  id='seed_squat'       name='Squat'       seed_origin='seed_modified' default_rest_sec=NULL  (user renamed to 'My Squat')
```

App ships v3.0 with:
- A migration adding a nullable `default_rest_sec integer` column.
- Bundle update: `BUNDLED_SEED_VERSION = 2`, and every seed in the bundle now has a
  `default_rest_sec` value (e.g. `90` for bench, `180` for squat).

User upgrades, opens the app. Loader runs:

| Step                           | Action                                                                                                       |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------ |
| Loader pre-check               | `BUNDLED_SEED_VERSION=2`, `local_seed_version=1`. Proceed to version-bump path.                              |
| `isFreshInstall`               | false (rows exist).                                                                                          |
| For `seed_bench_press`         | Present, `seed_origin='seed'`. Diff: bundle has `default_rest_sec=90`, local has NULL. Patch: `default_rest_sec=90`, `seed_origin` stays `seed`, `local_dirty=true, local_updated_at_ms=now=T9000`. |
| For `seed_squat`               | Present, `seed_origin='seed_modified'`. **Skip entirely.** User's `default_rest_sec` stays NULL.            |
| For new seeds in bundle        | None in this example, but if any: insert with `seed_origin='seed'`, dirty=true, `local_updated_at_ms=0` (sentinel — same treatment as fresh-install inserts, to preserve any remote tombstones for this slug). |
| End of pass                    | `local_seed_version := 2, local_seed_applied_at := T9000`.                                                   |
| Next sync push                 | Uploads the bench row with `default_rest_sec=90, updated_at_ms=T9000`. Squat is not dirty, not pushed.       |

**What the user sees:**
- "Bench Press" now has a rest-second default of 90.
- "My Squat" still has no default (the user's rename made it untouchable for this kind
  of merge; they'd need to set it themselves).

This is the deliberate trade-off in §5: bundle-provided defaults reach all unmodified
seeds, but never silently overwrite modifications.

**Edge case in 8.4: what if the user has a *second* device on v2.0 that hasn't
upgraded yet?**

- Device A (v3.0) writes `default_rest_sec=90, updated_at_ms=T9000`.
- Device B (v2.0) pulls. Its schema lacks the column entirely. **The pull applier
  silently drops unknown columns** (or, post-t1, stores them in `extras jsonb`). The
  `name` field is unchanged on B. `local_updated_at_ms=T9000`, `dirty=false`. No
  surprise.
- Device B upgrades. Its loader runs version 2 path. For
  `seed_bench_press`: local row is `seed_origin='seed'`, `default_rest_sec=NULL`
  (since the migration adds the column nullable). Patch applies `default_rest_sec=90`,
  `local_updated_at_ms=now=T10000`, `dirty=true`. Next push uploads — LWW says
  `T10000 > T9000`, server accepts the same value. Net effect: no change, one
  redundant push. Acceptable.

## 9. Confirmations / overrides vs brainstorm

| Brainstorm position                                                | This design                                       |
| ------------------------------------------------------------------ | ------------------------------------------------- |
| Fixed slug IDs (`seed:bench-press`)                                | **Confirmed**, with prefix renamed `sys_` → `seed_` and the colon dropped (no separator semantics). |
| Loader is idempotent, runs on every launch                         | **Confirmed**, gated by `local_seed_version` for the steady-state fast path. |
| Seeds are user-owned immediately                                   | **Confirmed.** No system catalog.                  |
| Not distinguished server-side                                      | **Confirmed.** Server schema has no `seed_origin`. |
| Locally distinguished via slug prefix only                         | **Override.** Slug prefix distinguishes lineage but not modification state. Adding `seed_origin text` solves the user-edit case directly. |
| Re-seed on every launch                                            | **Override** (already overridden in fix-sync T8): version-gated, not boolean-gated. The seed-once gate plus a version counter gives us both steady-state cheap-path AND future-bump path. |
| Single bool `seedsAppliedAt`                                       | **Override.** Replace with `local_seed_version` integer + `local_seed_applied_at`. Backwards-compatible: a non-null `seedsAppliedAt` maps to `local_seed_version=1` during the migration. |

## 10. Out of scope

- The actual bundle structure / module layout (data file vs many files vs generated).
- Server-side seed catalogue (we do not have one; not adding one).
- A "factory reset to seeds" user-facing affordance. The dev-only
  `resetLocalDataAndReseed` is sufficient for v2.
- A drift-checker that catches "bundle edited without `BUNDLED_SEED_VERSION` bump"
  (mention only; designed alongside t1's drift control).
- Anything about muscle groups beyond observing that the `is_editable = 0` check
  exempts them from the user-edit case; the loader still seeds them, and the version
  bump applies if their schema grows.
- Mappings (`exercise_muscle_mappings`) are seedable but follow the same rules as
  exercise definitions. No separate worked example because the mechanics are
  identical and the brainstorm did not flag them.

## 11. Open contract points for sibling design tasks

- **t1 (schema):** confirm `seed_origin text` is local-only, NOT in the server
  mapping, NOT in the drift checker's allowlist (it's allowed to be local-only by
  default, but the drift checker should know it's not a "missing server column"
  failure). Add a per-table opt-out: `local_only_columns: ['seed_origin']`.
- **t2 (protocol):** confirm `local_updated_at_ms = 0` semantics for fresh-install
  seed writes — the LWW must treat 0 as "stale; any positive remote value wins."
  Alternatively, t2 can specify "loader-origin writes are flagged
  `force_lww_loser`," which the server respects. Either works; t3 commits to "use a
  sentinel timestamp" because it's simpler.
- **t2 (initial-sync state machine):** the initial-sync modal's "Keep local and merge"
  branch must not regard the freshly-seeded rows as "local edits." Practically: the
  modal trigger condition is `count(rows where local_dirty=true and local_updated_at_ms > 0)`
  — i.e., dirty rows with a real timestamp. This excludes loader-origin seed rows.
  Document in t2.
