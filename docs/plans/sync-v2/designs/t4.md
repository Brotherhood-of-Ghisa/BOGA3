# t4 — iOS sync trigger reliability (design)

> Status: design doc only. No code or config is changed by this PR. A follow-up build
> task consumes this doc verbatim. iOS-only; Android is deferred per `plan.md` (current
> product ships via TestFlight as `com.phano.boga3.dev`).

## 0. TL;DR

- **Two-state scheduler.** At any instant the scheduler is in exactly one of two
  states: **running a cycle** or **scheduled to run a cycle in ≤ `SAFETY_INTERVAL`
  ms**. A cycle is `PULL → PUSH → PULL` repeated until the trailing `PULL` returns
  zero rows, then schedule the next cycle in `SAFETY_INTERVAL` ms. There is no
  separate "outbox flush" path — the cycle IS the flush.
- **Single trigger entry point: `requestSync()`.** If a cycle is running, no-op.
  Otherwise cancel any pending scheduled cycle and start one immediately. All
  events (foreground edge, NetInfo online, debounced mutation, manual "Sync now",
  background-task firing) route through `requestSync()`. There is no per-trigger
  branching downstream.
- **One cadence knob: `SAFETY_INTERVAL = 60s`.** No per-feature cadence (no
  `session-recorder` 10s, no dev 5s). The session recorder hits sync via
  push-on-edit + foreground re-entry instead — see §2.2. A dev override
  (`SYNC_DEV_SAFETY_INTERVAL_MS = 5_000`) is allowed but reachable only via
  `isDevMode()`; it is a verification aid, not a product cadence.
- **Push/pull interleave reasoning: PULL → PUSH → PULL.** The leading PULL
  cheaply discovers server-newer rows so LWW comparisons run against fresh state
  before we PUSH. The trailing PULL catches anything other clients wrote during
  our PUSH; loop it until empty.
- **Background mechanism — chosen:** **`expo-background-task`** (which wraps
  iOS `BGTaskScheduler`'s `BGAppRefreshTask`). Single task identifier
  `com.expo.modules.backgroundtask.processing` (the Expo-managed default). One task,
  one job: "run one full sync cycle, abort if BG budget exceeded." Not
  `BGProcessingTask`. Not silent push. Justification in §4.
- **Network state:** `@react-native-community/netinfo` with both `isConnected` and
  `isInternetReachable`. Pessimistic reachability (only consider us online when
  `isInternetReachable === true`) plus a request-level fallback that treats
  observed network failures as "go to offline state, schedule retry."
- **In-flight on suspend/resume:** any sync interrupted by suspend leaves dirty
  bits set. On resume, the foreground edge fires `requestSync()`, which starts a
  fresh cycle that re-pushes the same rows. Server-side LWW + idempotent push
  RPC (a t2 constraint) make retry safe. See §6.
- **"Sync now" UX:** lives on Settings under a new **Sync** section. Tapping
  calls `requestSync()`, which (because nothing is running) starts a cycle
  immediately. Disabled while offline or while a cycle is in flight.
- **Battery/data:** at idle with no edits, ~60 requests/hour (one cycle per
  `SAFETY_INTERVAL` tick = one short PULL when nothing is dirty). With sustained
  editing, push-on-edit debounce coalesces bursts so the upper bound stays in
  ~120 requests/hour. Background adds 1–4 cycles/day per iOS scheduling reality.

The rest of this doc walks every Outcomes bullet from `plan.md` t4 in order.

## 1. iOS app lifecycle states and which trigger fires in each transition

iOS exposes four [UI lifecycle states][apple-uistate] surfaced to React Native as
[`AppState`][rn-appstate]: `active`, `inactive`, `background`, and (implicit)
`terminated` (delivered via launch, not as an `AppState` event). `RN.AppState` emits
`active`, `inactive`, `background`. There is no public RN signal for "suspended"
distinct from "background" — iOS suspends a backgrounded app silently when its
remaining background time hits zero.

### 1.1 Transition table

| iOS transition | `AppState` event | Sync trigger | What runs |
|---|---|---|---|
| Cold launch (terminated → foreground) | initial `active` | **Bootstrap + `requestSync()`** | `startSyncRuntime()` → bootstrap if needed → `requestSync('cold-launch')` starts one cycle. |
| Warm resume (background → foreground) | `active` (from `background`) | **`requestSync('foreground')`** | Cancels the pending scheduled cycle (if any), starts a cycle immediately. After the cycle ends, the scheduler enters the "scheduled in SAFETY_INTERVAL" state per §2. |
| User pulls down notification center, swipes between apps, accepts incoming call (active → inactive) | `inactive` (from `active`) | **No-op** | The scheduler is in whichever of {running, scheduled} it already was. iOS does not throttle in-process JS work for `inactive`, so an in-flight cycle continues. |
| inactive → active (interruption resolves) | `active` (from `inactive`) | **No-op** | Avoids "double-fire" when the user briefly switches apps. The foreground edge trigger fires only on `background → active`, not on `inactive → active`. |
| active → background | `background` | **Background-prep best-effort** | If a cycle is in flight, it keeps running until iOS suspends the JS thread. If no cycle is in flight, we do **not** start one here — the OS's expiration-handler window is too narrow to reliably PULL→PUSH→PULL, and a half-done cycle just leaves dirty bits set anyway (the next BG firing or foreground edge will pick up). |
| background → suspended (no event) | (none) | **No-op** | The scheduler timer stops firing because the JS thread is paused. Any pending scheduled cycle is implicitly deferred until JS runs again. |
| suspended → terminated (no event) | (next launch fires `active`) | **No-op** | State will be reconstructed from local persistence on next launch. |
| Backgrounded long enough for `BGTaskScheduler` to grant time | (no `AppState` event; native task only) | **Background-task → `requestSync()`** (§4) | The Expo background task runs JS in a short-lived headless context. It calls `requestSync('background-task')` and awaits the cycle; the cycle aborts if the BG budget is exceeded. See §11 for the budget. |

### 1.2 Wiring vs today's code

Today, `apps/mobile/app/_layout.tsx` calls `startDefaultSyncScheduler()` once at
mount; the scheduler runs forever on the JS event loop and pauses *only* indirectly
when iOS suspends the app. There is no `AppState` listener in `sync/`. v2 adds one,
co-located with the scheduler, that translates `active`/`background` into
`schedulerOnAppActive()` / `schedulerOnAppBackground()` calls. The session-recorder
already has its own `AppState` listener at
`apps/mobile/src/session-recorder/lifecycle-helpers.ts`; the new sync listener is
independent and does not replace it.

[rn-appstate]: https://reactnative.dev/docs/appstate

## 2. Foreground triggers

### 2.1 The two-state scheduler

The scheduler has **two states only**:

- **`Running`** — a cycle is currently executing.
- **`Scheduled(t)`** — a cycle is scheduled to start at `t`, where `t - now()
  ≤ SAFETY_INTERVAL_MS`. Implemented as a single `setTimeout` handle.

There are no other states. There is no queue, no debounce timer that holds
state between events, no per-trigger branching. Invariant:

> At all times, either a cycle is `Running`, or there is exactly one
> `setTimeout` armed to fire in ≤ `SAFETY_INTERVAL_MS`.

### 2.2 The single entry point — `requestSync(reason)`

All triggers call exactly one function:

```ts
// Pseudocode — final placement is sync/scheduler.ts
function requestSync(reason: SyncReason): void {
  if (state.kind === 'running') {
    // A cycle is already running; let it complete. It will naturally
    // schedule the next cycle when it ends. We do NOT set a "rerun" flag —
    // the trailing PULL_until_empty in the running cycle already covers
    // anything that landed during it.
    return;
  }
  // state.kind === 'scheduled'
  clearTimeout(state.timer);
  startCycle(reason); // transitions state → 'running'
}
```

`startCycle()` is non-blocking; it kicks off the async cycle and registers
a `.finally()` that calls `scheduleNext()` to enter `Scheduled(now +
SAFETY_INTERVAL_MS)`.

#### Why no "rerun pending" flag

The cycle's trailing `PULL until empty` already coalesces any mutations or
remote changes that landed during it. Mutations that land during the
running cycle flip `local_dirty=true`; if the cycle's PUSH already started,
those rows are picked up by the **next** cycle (which the trailing PULL
will trigger or the SAFETY_INTERVAL tick will, whichever comes first via
the next mutation's `requestSync()`). The system is monotonic — every
dirty row eventually gets pushed in O(SAFETY_INTERVAL) bounded wall-clock
time.

### 2.3 Trigger sources — all routed through `requestSync()`

The trigger sources are simply *callers* of `requestSync()`. They do not
have their own state.

| Trigger | When it fires | Reason tag |
|---|---|---|
| Mutation (debounced) | Any repo write that flips `local_dirty=true`, debounced `500ms` to coalesce bursts (e.g. typing reps + weight + tap-complete on one set produces one call). | `'mutation'` |
| Foreground edge | `AppState` `background → active`. | `'foreground'` |
| Network online edge | NetInfo `isInternetReachable` transitions to `true`. Internally debounced `250ms` to absorb VPN handoffs (§3.2). | `'online'` |
| Manual "Sync now" | User taps the Settings → Sync button. | `'manual'` |
| Safety tick | The `setTimeout` from `Scheduled(t)` firing. | `'safety-tick'` |
| Background task | iOS fires the registered `BGAppRefreshTask` (§4). The task body awaits `requestSync('background-task')`. | `'background-task'` |

#### Debounce is local to the mutation source

The 500ms mutation debounce is **the only debounce** in the system. It
lives in the repo-write call site (or a small `debouncedRequestSync()`
wrapper used by mutation paths). All other triggers call `requestSync()`
directly. There is no per-reason cadence override, no `session-recorder`
special case, no remote-hint debounce.

### 2.4 The cycle itself — `PULL → PUSH → PULL_until_empty`

Inside `startCycle()`:

```ts
async function runOneCycle(reason: SyncReason): Promise<void> {
  await pull();                  // 1. Leading PULL: refresh server-side view.
  if (anyDirtyRows()) await push(); // 2. PUSH only if there is anything to send.
  while ((await pull()).rowCount > 0) {
    // 3. Trailing PULL loop: drain anything that landed while we were busy.
    // Continues until pull returns zero rows.
    // Bounded by the BG budget (§11) when running under background-task.
  }
}
```

The detailed reasoning for `PULL → PUSH → PULL` (why that order, why loop
the trailing PULL) is in §11.1.

### 2.5 No activity-specific cadence — the session recorder is covered by push-on-edit

v1 had a 10s `session-recorder` cadence and a 5s dev cadence. **Both are
gone.** The reasoning:

- Push-on-edit (debounced 500ms) already covers the user's edits during a
  workout. Every set/rep/weight commit fires `requestSync('mutation')`.
- The session recorder MUST persist edits to local SQLite frequently for
  push-on-edit to work. **Concretely:** the recorder commits to the local
  DB on every blur of a numeric input (reps, weight), on every tap of
  "complete set", on every navigation between sets, and on every dwell of
  ≥ 1s on a modified-but-uncommitted input. Whichever of these fires first
  triggers a `local_dirty=true` write and, via the debounce, a
  `requestSync('mutation')`. The recorder does NOT batch edits in memory
  across a multi-set entry; if the user closes the app mid-set, the
  partial state is on disk and dirty.
- The 60s `SAFETY_INTERVAL` is the only periodic cadence. The session
  recorder does not get a tighter one; it doesn't need one.

> **Audit note for the build task:** when porting the v1 scheduler, the
> existing 10s `SYNC_SESSION_RECORDER_CADENCE_MS` constant and the
> path-based `cadenceForContext(...)` logic in `scheduler.ts` are deleted
> outright. There is one cadence, full stop.

### 2.6 Concrete defaults

```ts
// proposed constants — final placement is sync/scheduler.ts
export const SYNC_MUTATION_PUSH_DEBOUNCE_MS = 500;
export const SYNC_ONLINE_EDGE_DEBOUNCE_MS = 250;
export const SYNC_SAFETY_INTERVAL_MS = 60_000;          // the ONLY cadence
export const SYNC_DEV_SAFETY_INTERVAL_MS = 5_000;       // dev-only override
```

The dev override path is **not** `__DEV__` — that is `false` in the dev
TestFlight build (`com.phano.boga3.dev`) where we most need the tight
cadence during manual testing. Implementation:

```ts
import { isDevMode } from '@/src/utils/isDevMode';

const safetyIntervalMs = () =>
  isDevMode() ? SYNC_DEV_SAFETY_INTERVAL_MS : SYNC_SAFETY_INTERVAL_MS;
```

The lint rule `no-restricted-syntax` (referenced in `isDevMode.ts`) already enforces
"no bare `__DEV__`." The dev override is a verification aid for manual
testing on TestFlight, not a product cadence.

### 2.7 State-machine diagram

```
                  ┌───────────────────────────────────────────────┐
                  │                                               │
                  │           requestSync() called                │
                  ▼                                               │
            ┌──────────┐                                          │
   ┌──────► │ Running  │                                          │
   │        └─────┬────┘                                          │
   │              │ cycle .finally()                              │
   │              ▼                                               │
   │      ┌───────────────┐  setTimeout fires (safety tick)       │
   └──────┤ Scheduled(t)  ├──────────────────────────────────────►│
          └───────┬───────┘                                        
                  │ requestSync() called                          
                  │   (any trigger — clears timer, runs now)      
                  └──────────────────────────────────────────────►Running
```

Transitions:
- `* → Running`: `requestSync()` while in `Scheduled(t)` — clears the
  timer, transitions to `Running`, starts cycle.
- `Running → Scheduled(now + SAFETY_INTERVAL_MS)`: cycle finishes
  (success, idle, transport failure, or BG-budget abort — all paths
  schedule the next cycle).
- `Running → Running`: re-entrant `requestSync()` is a no-op.
- `Scheduled(t) → Scheduled(t)`: re-entrant `requestSync()` does NOT
  happen; the first branch in `requestSync()` always either no-ops or
  transitions to `Running`.

Cancellation:
- The only cancellable thing is the `setTimeout` in `Scheduled(t)`.
  `requestSync()` clears it before starting a cycle.
- An in-flight cycle is NOT cancellable from JS. iOS suspension can
  interrupt it (§6); BG budget exhaustion can abort it (§11.3).

## 3. Network state

### 3.1 API choice — NetInfo

We adopt [`@react-native-community/netinfo`][netinfo-docs] (`NetInfo`). Reasons:

- It is the de facto network-state library across the React Native ecosystem and
  Expo SDK 54 supports it without prebuild (per Expo docs).
- It exposes both `isConnected` (link-layer / radio-up) and `isInternetReachable`
  (HEAD-probe-derived) — we need both.
- It emits `addEventListener` callbacks on radio changes; we wire it into
  `scheduler.setOnline(...)`.

We do **not** use `expo-network` for the live state surface because it does not emit
change events in the same form (it is a pull-based query API), only a one-shot
`getNetworkStateAsync()`. We may use `expo-network` opportunistically in the
background task (which runs outside of long-lived listeners) for a one-shot check
before performing work, since `Network.getNetworkStateAsync()` is also available
without a config plugin.

### 3.2 Known iOS pitfalls and how we handle each

| Pitfall | Behaviour | Our handling |
|---|---|---|
| **Airplane mode → off, slow Wi-Fi reassociation.** iOS sometimes reports `isConnected=true` ~1–3s before `isInternetReachable` flips. | NetInfo emits two events in quick succession. The first fires our online edge trigger; the actual HTTPS request then fails fast. | Our online-edge trigger fires on `isInternetReachable === true` specifically, not just `isConnected === true`. If we sync on a false-positive, the request will fail and we'll record a transport failure that schedules a backoff retry — but we'd rather avoid the noise. |
| **Captive portal (hotel Wi-Fi).** `isConnected=true`, `isInternetReachable` may be `true` or `null` depending on the probe URL. Real HTTPS to Supabase still 403/redirects. | Sync request fails with a 4xx the engine treats as transport failure. | Same backoff as any transport failure. The pull will fail until the user logs into the captive portal. No special UI needed — the status surface (§8) will show "Retry scheduled" and the cause "Network error." |
| **VPN handoff (briefly type=`vpn`, then `wifi`).** | Multiple connection-change events. | Debounce the online-edge trigger by 250ms inside the scheduler so we don't fire a flush on every NetInfo event. |
| **Cellular data with strict carrier limits / metering.** | NetInfo reports `cellular`. iOS does not surface "user disabled cellular data for this app" as a separate state. | Treat the same as offline at the request level once the OS returns the relevant error. No proactive disabling on cellular — sync payloads are small (typed JSON, <8 KB per typical batch). |
| **`isInternetReachable === null`** (NetInfo's initial state before first probe). | Until first probe, "online" is unknown. | Treat `null` as "unknown — try anyway." On failure, scheduler drops to offline state. |

### 3.3 Ambiguity handling — "seemingly online but requests fail"

NetInfo reports go-state lazily. Authoritative truth is "did my last RPC succeed?".
Concretely:

- The scheduler's `online` flag is updated by **both** NetInfo events **and** by
  the engine's own observed transport errors. The engine already calls
  `recordSyncTransportFailure` on fetch/network errors (see `engine.ts`); we extend
  this to also call `scheduler.setOnline(false)` on a transport error, even if
  NetInfo still claims `isConnected=true`.
- Backoff is driven by `outbox.calculateBackoffDelayMs` (already exponential with
  jitter, see exports in `sync/index.ts`). On the next NetInfo `isInternetReachable
  === true` *or* on the next safety tick (whichever first), we retry — the
  observed-error timestamp wins over NetInfo's stale "online" reading.

## 4. Background sync — chosen mechanism

### 4.1 The pick

**One mechanism: `expo-background-task` (wrapping iOS `BGAppRefreshTask`).**

Single registration, single task identifier, fires opportunistically while the app
is backgrounded.

### 4.2 Justification — why this, not the alternatives

- **Not `BGProcessingTask`.** BGProcessingTask is for heavy, long-running work
  (cleanup, ML training) scheduled by iOS during charging at night. Our sync
  payload is small and finishes in <2s on a normal connection. Apple's guidance
  is to use `BGAppRefreshTask` for "short tasks that keep your app's content up to
  date" — that is literally our use case. (Apple, "Choosing background strategies
  for your app.") Using BGProcessingTask would mean syncing much less often (often
  once a day) and is hostile to two-device freshness.
- **Not silent push.** Silent push (`content-available=1`) would give us
  "remote-changes-available" signalling and is technically the lowest-latency
  background path on iOS. We do not pursue it because:
  - Sync v2's purpose-#2 (two-device support) is explicitly low priority.
  - Silent push requires an APNs setup we do not have, and changes the Supabase
    backend story (need a notify-on-write trigger that pushes to APNs per device).
  - Apple **rate-limits** silent push: "Apple may throttle pushes that arrive at
    a high rate," and delivery is best-effort, often delayed by 30+ minutes
    under load. It's not a reliability win for our use case.
  - We can add it later (purpose #5 — groups — would benefit) without re-architecting.
- **Not `react-native-background-fetch`.** Third-party library, overlaps with
  what Expo SDK 54+ now provides natively via `expo-background-task` (added in
  SDK 50, stable in current versions). Adding a community package would mean an
  extra prebuild step and another moving piece in our `app.config.ts`. Expo's
  module is the lower-friction win.

### 4.3 What the task does

```ts
// proposed wiring — final placement is sync/background-task.ts
import * as BackgroundTask from 'expo-background-task';
import * as TaskManager from 'expo-task-manager';
import { runOneSyncCycle } from './scheduler';

const SYNC_BACKGROUND_TASK_NAME = 'sync-v2-foreground-refresh';

// Define in global scope (module side-effect; imported from app root).
TaskManager.defineTask(SYNC_BACKGROUND_TASK_NAME, async () => {
  try {
    // runOneSyncCycle returns SyncFlushResult-equivalent. It is responsible for
    // bailing if not signed in / disabled / offline. Internally guarded against
    // double-execution with the same single-in-flight slot as the foreground
    // scheduler.
    const result = await runOneSyncCycle({ reason: 'background-task' });
    return result.status === 'success' || result.status === 'idle'
      ? BackgroundTask.BackgroundTaskResult.Success
      : BackgroundTask.BackgroundTaskResult.Failed;
  } catch (err) {
    // Surface to logEvent — runOneSyncCycle should not throw normally.
    return BackgroundTask.BackgroundTaskResult.Failed;
  }
});

export const registerSyncBackgroundTask = async () => {
  if (!(await TaskManager.isTaskRegisteredAsync(SYNC_BACKGROUND_TASK_NAME))) {
    await BackgroundTask.registerTaskAsync(SYNC_BACKGROUND_TASK_NAME, {
      // minutes; iOS may run later than this but not sooner.
      minimumInterval: 15,
    });
  }
};
```

`registerSyncBackgroundTask()` is called from `app/_layout.tsx` next to
`startSyncRuntime()`. The task is auto-rescheduled by `expo-background-task` after
each run (per Expo docs).

### 4.4 iOS budget / Apple's stated rules

From Apple's [BGTaskScheduler documentation][bgtask] and the
[`UIApplication` background-execution docs][uiapp-bg]:

- `BGAppRefreshTask` execution time is **capped at ~30 seconds** of wall-clock
  CPU per invocation. Our sync cycle is well inside this — typical push+pull
  finishes in <2s.
- The system "chooses an optimal time to launch your background process based on
  device usage, battery state, and the conditions you set." There is **no
  guarantee** the task fires; on heavy/low-battery devices it may not fire for
  days. We commit to designing for "we'll get N opportunities a day," not "we
  pulled at 15:00:00." (Apple's docs: "Even in the best of circumstances,
  BGTaskScheduler isn't a precise timer.")
- The `minimumInterval` is a floor, not a ceiling.
- iOS suspends a backgrounded foreground app within seconds of entering background
  unless a `beginBackgroundTask` handler is active. The `AppState=background`
  transition gives us a few seconds at best to drain in-flight requests; we don't
  rely on it for correctness (§6).
- Background tasks require user permission via the system Background App Refresh
  toggle (Settings → General → Background App Refresh). If the user has it off
  globally or off for our app, `BackgroundTask.getStatusAsync()` returns
  `Restricted` and we get **zero** background firings. The Settings sync surface
  (§8) must call this out — "Background sync is off in iOS Settings; foreground
  sync continues to work."

### 4.5 Concrete registration block — `app.config.ts`

A future builder copies the following plugin entry verbatim into `app.config.ts`'s
`plugins:` array (between `"expo-router"` and `"expo-secure-store"` to keep the
order alphabetical-ish):

```ts

  "expo-background-task",
  {
    // Default identifier is "com.expo.modules.backgroundtask.processing".
    // We accept the default — no override needed. The identifier is registered
    // automatically into BGTaskSchedulerPermittedIdentifiers by the Expo plugin
    // during prebuild.

```

Per the Expo docs:

> "To enable background tasks on iOS, add the 'processing' value to the
> UIBackgroundModes array and register the 'com.expo.modules.backgroundtask.processing'
> identifier in the BGTaskSchedulerPermittedIdentifiers array. If using Continuous
> Native Generation (CNG), these configurations are applied automatically by
> prebuild."

Our project uses CNG (no `ios/` directory committed), so the Info.plist mutations
happen automatically during EAS build. The literal effect is:

```xml

  <string>processing</string>
</array>
<key>BGTaskSchedulerPermittedIdentifiers</key>

  <string>com.expo.modules.backgroundtask.processing</string>

```

If we later split identifiers (e.g. a separate "long pull on charging" task), this
expands; for v2 we keep it to one.

Also add `"expo-background-task"` and `"expo-task-manager"` to `apps/mobile/package.json`
dependencies in the build step — `expo install` will pick the SDK 54-compatible versions.

## 6. In-flight request behaviour on suspend / resume

### 6.1 What happens at the wire level

iOS will:

- Allow the JS thread to keep running for a brief window after `applicationDidEnterBackground:`,
  typically 5–30s depending on conditions.
- Suspend the app process when that window ends. In-flight `fetch()` calls are
  not cancelled, but the JS thread stops executing — so any `await` after a
  request completes never resolves until the app comes back to foreground (and
  often won't resolve at all, because iOS may terminate the suspended app).

### 6.2 How "try again" actually works — the concrete mechanism

The retry guarantee rests on a small set of invariants. There is **no outbox,
no resume token, no per-batch persistent state**. Recovery falls out of normal
operation.

#### 6.2.1 The two pieces of state that survive suspension

1. **Per-row `local_dirty=true` bit** in each entity table (set by the repo on
   every write; cleared only when the corresponding PUSH response arrives and
   is applied — see §6.2.2).
2. **Pull cursor** (a single bigint/timestamp in a local key/value table;
   advanced only when a PULL response is fully applied to local DB).

That's it. Both are durable in SQLite and survive any combination of suspend,
terminate, or device reboot.

#### 6.2.2 Step-by-step: what happens if a push is interrupted mid-flight

Suppose the user logs a set, the debounce fires, a cycle starts, the cycle's
PUSH leg sends a batch containing the set row, and then iOS suspends the app
before the response arrives.

1. **Before the request:** `exercise_sets.local_dirty = true` (set by the
   repo when the row was written).
2. **The cycle issues the PUSH:** `fetch(/sync/push, { ...batch })`. The
   `await fetch(...)` promise is now pending.
3. **iOS suspends the JS thread.** The promise is *still* pending — JS is
   frozen, not rejected. The TCP connection may live for some seconds in
   native land but is not observable from JS.
4. **(Possibly) iOS terminates the app.** The pending promise is gone. The
   dirty bit is still on disk.
5. **The user returns to the app.** `AppState` fires `active` (from
   `background`, or from initial launch if terminated). The foreground edge
   trigger calls `requestSync('foreground')`.
6. **`requestSync('foreground')` starts a new cycle.** The new cycle's PUSH
   leg reads `exercise_sets WHERE local_dirty = true`, sees the same row,
   and pushes it again.
7. **The response arrives.** The handler clears `local_dirty = false` and
   the row is now in sync.

The key property: **the dirty bit is the only thing that controls whether the
row gets pushed.** It is set by writes, cleared by acked pushes, and is never
touched by anything else. An interrupted push leaves the bit set, so the next
cycle re-pushes naturally.

#### 6.2.3 The "server committed but client never saw the ack" case

This is the case the reviewer specifically asked about. Concretely: the
server received the PUSH, committed the row, sent the response, and then
the network dropped (or iOS suspended the app) before the client could
apply the response.

The retry sends the same row again. The server sees a duplicate. **The push
RPC MUST be idempotent under retry** — same `(owner_user_id, entity_type, id,
client_updated_at_ms)` re-pushed yields server-state unchanged. Specifically,
LWW (`incoming.client_updated_at_ms <= stored.client_updated_at_ms → upsert
is a no-op`, per brainstorm.md §4.1) makes the duplicate a no-op on the
server. The server returns the same acked outcome; the client clears the
dirty bit.

> **t2 constraint:** the push RPC must be idempotent under retry. Re-pushing
> the same `(owner_user_id, entity_type, id, client_updated_at_ms, payload)`
> twice yields the same final server state and the same per-row outcome on
> the response. t4's recovery strategy depends on this.

> **t2 constraint:** the per-row response outcome for a push must be
> deterministic given the row's identity tuple. A retry of a previously-acked
> row must return an "applied" outcome (so the client clears the dirty bit),
> not an error.

#### 6.2.4 Pull is symmetric

Pull is idempotent by construction: the cursor advances only when the response
is fully applied. An interrupted pull leaves the cursor at its old value. The
next cycle re-fetches from the same cursor and re-applies the same rows. LWW
on the local-apply side makes the re-application a no-op for rows we already
have (assuming the local row's `local_updated_at_ms` is not stale — see t2 for
the local-apply semantics).

### 6.3 Why the in-flight outbox machinery becomes simpler

The v1 engine maintains a complex "in-flight flush promise" with single-slot
locking and an outbox of unacked events (`sync/engine.ts:35` —
`inFlightFlushPromise`). v2 keeps the single-cycle lock (still useful to
prevent overlap), but the underlying queue collapses: the queue *is* the table
of dirty rows. There is no separate outbox to compact or persist, no
resume token, no per-batch metadata.

Practical effect: a future builder deleting `sync/outbox.ts` and replacing it
with a per-table dirty-row reader is straightforward — the only "queue state"
that needs to survive a kill is already in the local DB.

### 6.4 Cycle resumption — concrete sequence

The path from "user returns to the app after a mid-cycle suspend" is:

```
[t=0]     User logs set → repo writes row, local_dirty=true,
          debounced requestSync('mutation') queued for t=500ms.
[t=500ms] Debounce fires → requestSync() → state was Scheduled(t1),
          clear timer, transition to Running.
[t=510ms] Cycle's PULL completes.
[t=520ms] Cycle's PUSH issues fetch(/sync/push).
[t=600ms] User presses home button → AppState 'background'.
[t=8s]    iOS suspends JS thread mid-fetch.
          (Possibly: iOS terminates app some seconds-to-minutes later.)
[t=...]   User opens app → AppState 'active' (from background, or initial
          'active' after terminate).
          Foreground edge fires requestSync('foreground').
          State is Scheduled(t2) (the post-bootstrap initial schedule).
          Clear timer, transition to Running.
          Cycle's PULL reads dirty rows → same set row is dirty.
          Cycle's PUSH succeeds, dirty bit cleared.
          Cycle's trailing PULL drains anything else.
          State → Scheduled(now + SAFETY_INTERVAL_MS).
```

No special "resume from where we left off" code path. The same `requestSync()`
that handles every other trigger handles resume too.

### 7.2 Tap behaviour

- **Button disabled** if any of: not signed in, sync disabled, currently in a
  running cycle, or `isInternetReachable === false`.
- **On tap:**
  1. Set button state to "Syncing…".
  2. Call `requestSync('manual')`. Because the user only tapped because the
     button was enabled (state was `Scheduled(t)`), this immediately clears
     the scheduled timer and starts a cycle. The cycle does the standard
     `PULL → PUSH → PULL_until_empty` (§2.4) — no special "force" mode is
     needed because the cycle is the only way to sync anything.
  3. Await the cycle completion via a one-shot subscription to the
     scheduler's state change. On result:
     - cycle ended with `success` / `idle` → toast "Sync complete" + refresh
       the Settings status block.
     - cycle ended with `failure_*` / `transport_failure` → toast "Sync
       failed: <message>".
     - request couldn't start (`disabled` / `offline`) → toast with the
       specific reason (these cases are already gated by the disabled check
       above, but a race with NetInfo can still hit them).
- **Feedback:** a small inline `UiText` showing the most recent result, persisted
  in the local Settings state for the duration the screen is mounted.

### 7.3 Why "Sync now" is just `requestSync('manual')`

In v1, "Sync now" needed a `force: true` flag to skip a per-trigger debounce.
In v2 there is no debounce at the trigger level (only the mutation source
debounces itself). "Sync now" just calls the same `requestSync()` everything
else calls. The two-state scheduler model (§2.1) collapses the special case.

The cycle order is `PULL → PUSH → PULL_until_empty` — the leading PULL means
the user always sees fresh server state in the response, and the PUSH leg
uploads any local changes they just made. This matches the user's mental
model: "make sure everything is in sync."

## 9. Battery / data tradeoffs

### 9.1 Quantification — foreground

**Baseline (no edits, app open):**
- `SAFETY_INTERVAL_MS = 60_000` → 60 cycles/hour at most.
- Each cycle when nothing is dirty = `PULL (empty) → no PUSH → PULL (empty,
  loop exits)` = **2 round-trips per cycle**. (The trailing PULL is needed to
  prove emptiness; in steady state both PULLs return zero rows.)
- Pull payload at steady state = empty body + 8-byte cursor + auth header.
  Response = empty rows + new cursor. ~1 KB per request.
- **~120 KB/hour** sustained while open and idle. (Doubled vs the previous
  estimate because each cycle now does two PULLs instead of one.)

> **Optimisation note:** the leading PULL of cycle N+1 and the trailing PULL of
> cycle N return the same empty result if no events fire between cycles. A
> simple optimisation — skip the leading PULL if the previous cycle's trailing
> PULL completed within the last `SAFETY_INTERVAL_MS / 2` and there are no
> dirty rows — would cut idle traffic in half. Out of scope for the design;
> flag as a follow-up perf opt for the build task.

**Active editing (logging a workout):**
- User edits trigger debounced `requestSync('mutation')` per set boundary.
- Workouts run 45–90 minutes; ~30 sets per workout → ~30 mutations → ~30 cycles
  (push-on-edit collapses bursts inside a single set via the 500ms debounce).
- Each non-idle cycle = `PULL → PUSH → PULL (empty)` = 3 round-trips.
- Plus the SAFETY_INTERVAL ticks during the workout: ~1 per minute × 60 min =
  60 idle cycles (most coalesced into the edit-driven cycles by the
  `Running → no-op` branch in `requestSync()`).
- Effective hourly rate during a workout: ~90–120 requests/hour.
- Payload: a typed exercise_set row JSON is ~200 bytes. 30 sets × 200 bytes =
  ~6 KB upload per workout. Pulls are similar order.
- **~50–100 KB per logged workout** of sync traffic, all over HTTPS to Supabase.

The session recorder NO LONGER has a tighter cadence — push-on-edit is the
mechanism that makes set logging feel "instant", not a faster periodic tick.

**Dev mode (`isDevMode() === true`):**
- `SYNC_DEV_SAFETY_INTERVAL_MS = 5_000` → up to 720 cycles/hour at idle.
  Acceptable on dev builds; not shipped to real users.

### 9.2 Quantification — background

`BGAppRefreshTask` opportunities: in practice, 1–4 per day for an actively-used app
on modern iOS (Apple does not document a number; user reports & WWDC sessions
converge here). Each task runs one cycle, identical cost to a foreground tick.

**Net background cost: negligible** (<10 KB/day).

### 9.3 Honest call-outs

- We do **not** pause sync on cellular (no reason to — payloads are tiny).
- We do **not** batch beyond `SYNC_BATCH_MAX_SIZE` (see §11.3 for the push-side
  cap and §11.4 for the pull-side pagination).
- We do **not** use compression beyond what HTTPS provides; Supabase enables gzip
  by default, payloads are small enough that this isn't a meaningful lever.
- Battery: the dominant cost is the radio wake, not the work. Foreground cadence
  of 60s aligns with the natural cadence of most messaging apps; we are not an
  outlier. Cadence is tunable — and now we have exactly **one** knob to pull
  (`SAFETY_INTERVAL_MS`), not three.

## 11. Anatomy of one sync cycle

This section is the runtime contract for `runOneCycle()`. It covers (1) what
happens in one cycle, (2) how a push batch handles FK relations, (3) how a
cycle fits inside the iOS background window, and (4) how server-to-client
batches are drained.

### 11.1 What happens in one cycle — step by step

```ts
async function runOneCycle(reason: SyncReason): Promise<CycleResult> {
  const budget = budgetFor(reason); // §11.3
  const t0 = nowMs();

  // 1. Leading PULL.
  //    Reads the local pull cursor, posts to /sync/pull, applies the response
  //    rows to local SQLite under a single transaction per page (LWW on the
  //    client-apply side). Advances the cursor only after the apply commits.
  await drainPull(budget, t0); // §11.4

  // 2. PUSH if anything is dirty.
  //    Reads every row with local_dirty=true across all entity tables (single
  //    SELECT per table, UNION of identity tuples + payloads). Caps the batch
  //    at SYNC_BATCH_MAX_PUSH_ROWS. Posts to /sync/push. On per-row "applied"
  //    outcome, clears local_dirty=false in the same local transaction that
  //    applies any returned canonical values from the server (rare; mostly
  //    server-side defaults).
  if (anyDirtyRows()) {
    await runPushBatches(budget, t0); // §11.2, §11.3
  }

  // 3. Trailing PULL loop.
  //    Drain until /sync/pull returns zero rows. Bounded by budget.
  while (true) {
    if (overBudget(budget, t0)) {
      return { kind: 'aborted-budget' };
    }
    const res = await drainPullOnePage(); // §11.4
    if (res.rowCount === 0) break;
  }

  return { kind: 'success', durationMs: nowMs() - t0 };
}
```

**Why `PULL → PUSH → PULL`, not `PUSH → PULL`?**

- The **leading PULL** is cheap (empty body, ~1 KB) and discovers if the
  server has rows newer than our cursor. If a remote write has happened since
  our last cycle, applying it first means our subsequent PUSH compares against
  fresh server state. In LWW, this can flip a locally-dirty row to
  "actually-not-dirty-anymore" (e.g. the server has a newer `client_updated_at_ms`
  for the same row from another device), saving a wasted PUSH round-trip.
- The **PUSH** runs after the leading PULL so it uploads against fresh state.
  Per t2, the push RPC is LWW on the server side, so even if we skipped the
  leading PULL we'd be safe — the leading PULL is a latency/efficiency win,
  not a correctness requirement.
- The **trailing PULL** catches anything other clients wrote while we were
  PUSHing. It's looped because, in two-device scenarios, another client may
  write while our trailing PULL is in flight; one more PULL after that
  guarantees we drained everything visible at this instant.
- The loop terminates because each PULL advances the cursor; the server only
  returns rows with `server_received_at > cursor`. In steady state with no
  other writers, the loop exits after one empty PULL.

### 11.2 FK compatibility within a push batch

> **Context:** A push batch can contain rows from multiple tables (e.g. a
> Session and its child Exercise Sets). The server schema declares FKs
> between those tables (per t1). If the FK is enforced row-by-row at insert
> time, the order of rows in the batch matters — the parent must be inserted
> before the child. If the FK is `DEFERRABLE INITIALLY DEFERRED`, the FK is
> checked at COMMIT, so order within the batch is irrelevant.

The t4 client must NOT attempt topological ordering. Instead:

1. **The push batch always contains every currently-dirty row, no aggregate
   grouping, no ordering.** The client reads `WHERE local_dirty=true` across
   all entity tables and concatenates the results into one batch (subject to
   the size cap in §11.3).
2. **t2 owns the wire-side FK story.** Specifically:

> **t2 constraint:** the push RPC must wrap the entire batch in a single
> server transaction with FKs `DEFERRABLE INITIALLY DEFERRED`. Order of
> rows within the batch is not significant to correctness.

3. **What if a child row's parent isn't in the batch and isn't yet on the
   server?** This is the genuinely ambiguous case. Per t2, the server has two
   reasonable behaviours and t4 does not pick:
   - **Whole-batch reject** with a per-row outcome `{ status:
     'parent_missing' }` → t4 keeps the dirty bits set, waits for the next
     cycle (which will, presumably, include the parent because it was also
     dirty and will be selected on the next read).
   - **Partial accept** of orderable subgraphs → t4 clears dirty bits for
     applied rows only, leaves the rest dirty.

   Either way t4 behaves correctly because the dirty bits are the source of
   truth and the next cycle re-pushes whatever didn't apply.

> **t2 constraint:** the push RPC's response must include a per-row outcome
> tagged with the input identity tuple. t4 uses the outcome to clear or
> retain the local `local_dirty` bit on a per-row basis. Whole-batch reject
> is acceptable (every row gets a `failed` outcome); partial accept is also
> acceptable.

#### Why not "smart batching" (group by aggregate)?

The brainstorm flirted with per-aggregate batching (one Session + its
children in one batch). t4 rejects this because:

- It re-introduces the v1 complexity of "knowing which rows are related" on
  the client.
- With deferrable FKs, it solves a problem that doesn't exist.
- The push batch is already capped at `SYNC_BATCH_MAX_PUSH_ROWS` (§11.3), so
  the size argument doesn't apply — we just send up to that many dirty rows
  in arbitrary order.

The simplest rule that works: **client always pushes the current set of
dirty rows, no aggregate-grouping, no ordering, no foreign-key awareness on
the client side.** t2 commits with deferrable FKs and reports per-row
outcomes; t4 acts on those outcomes.

### 11.3 Background-window budgeting

Apple's `BGAppRefreshTask` gives the app **~30 seconds** of wall-clock
execution time per invocation (§4.4). To stay well inside this, t4 imposes
a **soft budget** that is honoured both in the BG path and (for symmetry) in
the foreground path.

**Budgets:**

```ts
// proposed constants — final placement is sync/scheduler.ts
export const SYNC_BUDGET_MS_FOREGROUND = Infinity;  // foreground has no hard cap
export const SYNC_BUDGET_MS_BACKGROUND = 20_000;    // 20s of 30s BG window
export const SYNC_BATCH_MAX_PUSH_ROWS = 500;        // cap one push at 500 rows
```

**Enforcement:**

- `runOneCycle()` snapshots `t0 = nowMs()` at start.
- Before each network call (the leading PULL, each push batch, each pull
  page in the trailing loop), check `nowMs() - t0 >= budget`. If yes,
  abort: return `{ kind: 'aborted-budget' }` immediately.
- "Abort" means: dirty bits remain set, cursor remains at its last
  successfully-applied value, the next cycle picks up where we left off.
- The cycle's `.finally()` always schedules the next cycle in
  `SAFETY_INTERVAL_MS`, including after a budget abort.

**Push-side cap (`SYNC_BATCH_MAX_PUSH_ROWS = 500`):**

- Reasoning: a 500-row push of typed exercise_set JSON (~200 bytes/row) is
  ~100 KB upload, well within any reasonable bandwidth budget. Round-trip
  latency on 4G is dominated by the request setup, not the payload, so a
  larger batch is no slower than a smaller one until we hit server-side
  processing time.
- If there are >500 dirty rows, the cycle issues multiple push batches in
  series, each within the budget check. This handles the "user came back
  after a long offline period with a lot of pending edits" case.
- Each batch is independent — its own request, its own per-row outcomes,
  its own dirty-bit-clearing transaction.

**Why is foreground budget `Infinity`?** Foreground sync has no hard
deadline. The JS event loop runs as long as the user has the app open. If
a foreground cycle takes 2 minutes because the user has 10K dirty rows
after a long offline, that's fine — the next cycle is just scheduled later.
The budget exists to ensure background cycles don't get killed mid-flight
by iOS; it's not a UX cap.

> **t2 constraint:** the push RPC must accept partial batches (i.e. it
> must not require the client to send a complete "session aggregate" in one
> request). t4 may issue multiple push batches in series; each is
> independent. Wire example: pushing 1200 dirty rows produces three
> requests of 500/500/200 rows in series, each with its own response.

### 11.4 Server-to-client batches (pull pagination)

Pull is server-paginated. Per the brainstorm and t2:

```
POST /sync/pull
{ since: <cursor>, limit: 500 }
→ { entities: [...], next_cursor, has_more }
```

t4's responsibilities:

1. **Persist the cursor** in a local key/value table. Read on cycle start,
   write only after the response is fully applied to local SQLite under a
   single transaction. An interrupted apply leaves the cursor at its last
   successfully-applied value.
2. **Drain pages** until `has_more === false` OR the budget is exhausted
   (§11.3). Each page is a separate HTTP request; the loop is:
   ```ts
   async function drainPull(budget, t0) {
     while (true) {
       if (overBudget(budget, t0)) return;
       const res = await drainPullOnePage();
       if (!res.hasMore) return;
     }
   }
   ```
3. **Apply LWW on receive.** For each pulled row, compare server's
   `client_updated_at_ms` to the local row's `local_updated_at_ms` (per
   t2). Newer server row replaces local; older server row is ignored.

> **t2 constraint:** the pull RPC must paginate via a cursor that
> monotonically advances. The cursor format is opaque to t4; t4 only stores
> and forwards it. The cursor's tiebreak rule (per t2: `(server_received_at,
> owner_user_id, type, id)`) is t2's concern — t4 just needs `has_more` and
> the next cursor.

> **t2 constraint:** the pull response must be commutatively applicable
> within a single response page (no inter-row dependencies inside one page).
> If two rows in the same page reference each other via FK (e.g. a Session
> and one of its Exercise Sets), the local-apply transaction must succeed
> regardless of the row order in the response array. This is satisfied by
> the local schema also having deferrable FKs, or by t2 ordering parents
> before children — either is acceptable from t4's perspective, but t2 must
> pick one and document it.

#### Resumption across cycles

If a budget abort fires mid-drain (say, after page 3 of 10), the cursor is
already advanced to the end of page 3 (because each page is applied and
committed before the next is fetched). The next cycle starts a fresh
`drainPull()` from cursor-at-end-of-page-3 and continues from page 4. No
resume token, no per-cycle state beyond the cursor.

## 13. t2 constraints emitted by t4

This is the consolidated index of every constraint t4's design places on the
t2 (sync protocol) design. When t2 is (re)written, the agent should treat
each of these as a hard input. They are scattered through the doc as
"**t2 constraint:**" call-outs; this section is the single index.

### 13.1 Push RPC

1. **Idempotent under retry.** Re-pushing the same `(owner_user_id,
   entity_type, id, client_updated_at_ms, payload)` twice yields the same
   final server state and the same per-row outcome. (§6.2.3)
2. **Deterministic per-row outcome.** A retry of a previously-acked row
   returns an "applied" outcome (so the client clears the dirty bit), not
   an error. (§6.2.3)
3. **Deferrable FKs in the server transaction.** The push RPC must wrap
   the batch in a single server transaction with FKs `DEFERRABLE INITIALLY
   DEFERRED`. Order of rows within the batch is not significant to
   correctness. (§11.2)
4. **Per-row outcomes in the response.** The push RPC's response must
   include a per-row outcome tagged with the input identity tuple. Whole-
   batch reject is acceptable (every row gets a `failed` outcome); partial
   accept is also acceptable. (§11.2)
5. **Accepts partial batches.** The push RPC must NOT require a complete
   "session aggregate" or any other relational grouping. t4 issues
   batches of arbitrary mixed rows, capped at 500 rows per batch. (§11.3)

### 13.2 Pull RPC

6. **Monotonically advancing cursor.** The pull RPC paginates via a cursor
   whose format is opaque to t4. The cursor's tiebreak rule is t2's choice
   (likely `(server_received_at, owner_user_id, type, id)`). t4 stores
   `next_cursor` and forwards it on the next request. (§11.4)
7. **`has_more` flag.** Each response carries `has_more: boolean` so t4
   knows whether to fetch another page. (§11.4)
8. **Commutatively applicable within one page.** The set of rows in a
   single pull response must be applicable to local SQLite in a single
   transaction regardless of intra-page order. Either t2 orders parents
   before children, or the local schema also has deferrable FKs (t1's
   concern). Either is acceptable; t2 must pick and document. (§11.4)
9. **LWW comparison field.** Each pulled row carries
   `client_updated_at_ms`. t4 compares this to the local row's
   `local_updated_at_ms` to decide whether to apply. (§11.4)

### 13.3 Out of t4's scope but t2 must own

- The exact JSON shapes of `/sync/push` and `/sync/pull` requests and
  responses.
- How partial-accept maps to per-row outcomes on the wire.
- Whether the server commits push transactions per batch or per row (t4
  assumes per batch; if t2 picks per row, the deferrable-FK requirement
  goes away but the partial-accept requirement remains).
- Tombstone semantics (per the plan.md t2 outcomes, tombstones are t2's
  concern; t4 just treats `deleted=true` rows as ordinary dirty rows on
  push and ordinary rows on pull).
