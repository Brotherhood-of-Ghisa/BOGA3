# t4 ÔÇö iOS sync trigger reliability (design)

> Status: design doc only. No code or config is changed by this PR. A follow-up build
> task consumes this doc verbatim. iOS-only; Android is deferred per `plan.md` (current
> product ships via TestFlight as `com.phano.boga3.dev`).
>
> This is a **requirements document**, not a build doc. Mechanism details for the wire
> protocol, the push/pull RPCs, and the cycle's internal shape are t2's concern; t4
> only states the requirements it places on t2 (see ┬º11).

## 0. TL;DR

- **Two-state scheduler.** At any instant the scheduler is in exactly one of two
  states: **`Running`** (a cycle is executing) or **`Scheduled(t)`** (a cycle is
  scheduled to start at time `t`, where `t - now() Ôëñ SAFETY_INTERVAL`). One
  `setTimeout` handle; no queue; no separate "outbox flush" path.
- **Single trigger entry point: `requestSync(reason)`.** Every event ÔÇö foreground
  edge, online edge, edit, manual "Sync now", safety tick, background-task fire ÔÇö
  routes through this one function. There is no per-trigger branching downstream.
- **Edit events use a MIN-debounce trick inside `requestSync()`.** An edit never
  starts a cycle immediately. If `Scheduled(t)`, it sets
  `Scheduled(MIN(t, now() + DEBOUNCE_INTERVAL))`. Bursts of edits within
  `DEBOUNCE_INTERVAL` naturally collapse to one cycle. Other triggers (foreground,
  online, manual) start a cycle immediately. See ┬º2.
- **One cadence knob: `SAFETY_INTERVAL = 60s`.** No per-feature cadence. No
  `__DEV__` override, no `isDevMode()` override ÔÇö **dev reflects production.**
  Verification on TestFlight uses the same 60s cadence as users see.
- **Edit debounce: `DEBOUNCE_INTERVAL = 500ms`.** This is also the rate ceiling
  under continuous editing.
- **Rate ceilings.** Idle: at most 1 cycle per `SAFETY_INTERVAL` (~60/hour).
  Active editing: at most 1 cycle per `DEBOUNCE_INTERVAL` (~7200/hour in the
  worst case of one edit every 500ms; real workouts produce far fewer edits).
  Per-request payloads are small (single-digit KB).
- **Background mechanism: `expo-background-task`** wrapping iOS
  `BGAppRefreshTask`. One task, one job: run one cycle, abort cleanly if budget
  exceeded. Justified in ┬º4.
- **Network state:** `@react-native-community/netinfo` with both `isConnected`
  and `isInternetReachable`. Pessimistic reachability plus a request-level
  fallback that treats observed network failures as "offline, schedule retry."
- **In-flight on suspend/resume:** any cycle interrupted by suspend leaves dirty
  bits set. On resume, the foreground edge fires `requestSync()` and the next
  cycle re-pushes. Requirements on t2 (idempotency, batched commits, durable
  state) are in ┬º6 and ┬º11.
- **"Sync now" UX:** new **Sync** section in Settings. Tapping calls
  `requestSync('manual')` which cancels the scheduled timer and starts a cycle
  immediately. Disabled while offline or while a cycle is in flight.

The rest of this doc walks every Outcomes bullet from `plan.md` t4 in order.

## 1. iOS app lifecycle states and which trigger fires in each transition

iOS exposes four [UI lifecycle states][apple-uistate] surfaced to React Native as
[`AppState`][rn-appstate]: `active`, `inactive`, `background`, and (implicit)
`terminated` (delivered via launch, not as an `AppState` event). `RN.AppState` emits
`active`, `inactive`, `background`. There is no public RN signal for "suspended"
distinct from "background" ÔÇö iOS suspends a backgrounded app silently when its
remaining background time hits zero.

### 1.1 Transition table

| iOS transition | `AppState` event | Sync trigger | What runs |
|---|---|---|---|
| Cold launch (terminated ÔåÆ foreground) | initial `active` | **Bootstrap + `requestSync('cold-launch')`** | Bootstrap if needed, then immediate cycle. |
| Warm resume (background ÔåÆ foreground) | `active` (from `background`) | **`requestSync('foreground')`** | Immediate cycle. Cancels any pending scheduled timer. |
| active ÔåÆ inactive (notification center, app-switcher, call) | `inactive` (from `active`) | **No-op** | The scheduler is in whichever of `{Running, Scheduled}` it already was. iOS does not throttle in-process JS for `inactive`. |
| inactive ÔåÆ active (interruption resolves) | `active` (from `inactive`) | **No-op** | Avoids "double-fire" on brief inactive intervals. The foreground edge fires only on `background ÔåÆ active`, not on `inactive ÔåÆ active`. |
| active ÔåÆ background | `background` | **Best-effort drain** | If a cycle is in flight, it keeps running until iOS suspends the JS thread. If no cycle is in flight, we do **not** start one here ÔÇö the OS's expiration-handler window is too narrow and a half-done cycle just leaves dirty bits set anyway (the next BG firing or foreground edge picks up). |
| background ÔåÆ suspended (no event) | (none) | **No-op** | The scheduler timer stops firing because the JS thread is paused. Any pending scheduled cycle is implicitly deferred until JS runs again. |
| suspended ÔåÆ terminated (no event) | (next launch fires `active`) | **No-op** | State reconstructs from local persistence on next launch. |
| Backgrounded long enough for `BGTaskScheduler` to grant time | (no `AppState` event; native task only) | **Background-task ÔåÆ `requestSync('background-task')`** (┬º4) | The Expo background task runs JS in a short-lived headless context, awaits one cycle, returns Success/Failed. |

### 1.2 Wiring vs today's code

Today, `apps/mobile/app/_layout.tsx` calls `startDefaultSyncScheduler()` once at
mount; the scheduler runs forever on the JS event loop and pauses *only* indirectly
when iOS suspends the app. There is no `AppState` listener in `sync/`. v2 adds one,
co-located with the scheduler, that translates `active`/`background` into
`schedulerOnAppActive()` / `schedulerOnAppBackground()` calls. The session-recorder
already has its own `AppState` listener at
`apps/mobile/src/session-recorder/lifecycle-helpers.ts`; the new sync listener is
independent and does not replace it.

[rn-appstate]: https://reactnative.dev/docs/appstate

## 2. Foreground triggers

### 2.1 The two-state scheduler

The scheduler has exactly two states:

- **`Running`** ÔÇö a cycle is currently executing.
- **`Scheduled(t)`** ÔÇö a cycle is scheduled to start at `t`, where
  `t - now() Ôëñ SAFETY_INTERVAL`. Implemented as a single `setTimeout` handle.

Invariant:

> At all times, either a cycle is `Running`, or there is exactly one `setTimeout`
> armed to fire in Ôëñ `SAFETY_INTERVAL`.

There is no queue, no per-trigger debounce timer that lives between events, no
per-trigger branching.

### 2.2 The single entry point ÔÇö `requestSync(reason)`

Every trigger calls `requestSync(reason)`. The function's behaviour depends on
the reason:

**Edit / mutation events** (the only "rate-limited" trigger):

1. If `Running`: no-op. The running cycle (and the one that will be scheduled
   when it finishes) will pick up the dirty row.
2. If `Scheduled(t)`: set `Scheduled(MIN(t, now() + DEBOUNCE_INTERVAL))`.

This is the **MIN-debounce trick.** It is the only debounce mechanism in the
system. It lives inside `requestSync()`, not at the call site. Properties:

- An edit never starts a cycle immediately; it only ever pulls the scheduled
  time forward to `now() + DEBOUNCE_INTERVAL`.
- The `MIN` ensures we never *delay* a sooner-scheduled cycle. If a safety
  tick is already due in 200ms and an edit fires, the schedule stays at
  200ms, not 500ms.
- A burst of N edits within `DEBOUNCE_INTERVAL` collapses to one rescheduling
  ÔÇö each edit pulls the schedule forward by the same amount, and `MIN`
  clamps it. The first edit in the burst sets the time; subsequent edits in
  the burst are MIN-no-ops.
- Rate ceiling under continuous editing: at most one cycle per
  `DEBOUNCE_INTERVAL` (~7200/hour worst case at 500ms; real workouts produce
  far fewer edits).

**Foreground edge** (`background ÔåÆ active`):

- If `Running`: no-op.
- If `Scheduled(t)`: clear timer, start a cycle immediately.

**Online edge** (NetInfo `isInternetReachable` flips to `true`):

- Same as foreground edge: if `Scheduled(t)`, clear timer and start immediately.
- Justification: an online edge is a high-information event ("we were offline,
  we're back") ÔÇö sync state is likely stale and the user is observably
  reconnected. We want to converge fast. The "VPN handoff" noise concern is
  handled at the NetInfo layer (┬º3) by debouncing the reachability signal
  itself; what reaches `requestSync('online')` is a definitive transition.

**Manual "Sync now"**:

- If `Running`: button is disabled (see ┬º7), so this case should not occur.
  Defensive: no-op.
- If `Scheduled(t)`: clear timer, start a cycle immediately.

**Safety tick** (`Scheduled(t)`'s `setTimeout` fires):

- Always transitions to `Running` and starts a cycle. This is the timer's
  whole purpose; no decision logic.

**Background task** (iOS `BGAppRefreshTask` fires; see ┬º4):

- Enters via its own path. The task body awaits `requestSync('background-task')`
  which, if `Scheduled(t)`, clears the timer and starts a cycle. The task
  awaits the cycle's completion (or budget-abort) before returning to iOS.

**Summary:** every trigger except an edit starts a cycle immediately. Only
edits use the MIN-debounce trick. There is no separate `debouncedRequestSync()`
wrapper; the debounce is inside `requestSync()` itself, applied only to the
`'mutation'` reason.

### 2.3 Why edits use MIN-debounce, not immediate-start

Edits arrive in bursts (typing reps, weight, tapping "complete set" ÔÇö all within
a second). If each edit started a cycle immediately, we'd thrash. The
MIN-debounce trick coalesces bursts into one cycle that fires
`DEBOUNCE_INTERVAL` after the first edit in the burst, without losing
correctness: every edit flips `local_dirty = true` on its row, and the cycle
that fires picks up every dirty row.

Foreground and online edges, by contrast, are rare (Ôëñ a few times per hour
typically) and high-information. Debouncing them buys nothing and delays
convergence; immediate start is the right call.

### 2.4 What a cycle does ÔÇö pointer to ┬º6 and ┬º11

The internal shape of a cycle (`PULL ÔåÆ PUSH ÔåÆ PULL`, batching, FK handling, BG
budget, cursor management) is a t2 concern. t4 places two requirements on it
(forwarded to t2 in ┬º11):

1. A cycle continues until there are no new rows to sync, either on the client
   or on the server.
2. A cycle internally operates in batches that commit independently, so a cycle
   interrupted mid-flight still makes forward progress.

Detail belongs in t2.

### 2.5 No activity-specific cadence

v1 had a 10s `session-recorder` cadence. **Gone.** Push-on-edit (the
MIN-debounce trick above) already covers the user's edits during a workout.
Every set/rep/weight commit fires `requestSync('mutation')` and the cycle
fires within `DEBOUNCE_INTERVAL` after the last edit.

For push-on-edit to work, the session recorder must persist edits to local
SQLite promptly: on every numeric-input blur, on every "complete set" tap, on
every set-navigation, and after ÔëÑ 1s dwell with unsaved changes. Each of those
writes flips `local_dirty = true` and triggers `requestSync('mutation')`.

> **Audit note for the build task:** when porting the v1 scheduler, the
> existing 10s `SYNC_SESSION_RECORDER_CADENCE_MS` constant and the path-based
> `cadenceForContext(...)` logic in `scheduler.ts` are deleted outright. There
> is one cadence, full stop.

### 2.6 Concrete defaults

- `SAFETY_INTERVAL` = **60 seconds**. The only periodic cadence. Same in dev
  and prod. No `__DEV__` override. No `isDevMode()` override.
- `DEBOUNCE_INTERVAL` = **500 ms**. The MIN-debounce window for edit events.
  Same in dev and prod.

(The `isDevMode()` rule from the user's memory still applies to other dev
affordances elsewhere in the app ÔÇö e.g. dev-only buttons. It does NOT apply
to scheduler cadence: dev reflects production here.)

## 3. Network state

### 3.1 API choice ÔÇö NetInfo

We adopt [`@react-native-community/netinfo`][netinfo-docs] (`NetInfo`). Reasons:

- It is the de facto network-state library across the React Native ecosystem and
  Expo SDK 54 supports it without prebuild.
- It exposes both `isConnected` (link-layer / radio-up) and `isInternetReachable`
  (HEAD-probe-derived) ÔÇö we need both.
- It emits `addEventListener` callbacks on radio changes; we wire it into
  `scheduler.setOnline(...)`.

We do **not** use `expo-network` for the live foreground state surface because it
does not emit change events in the same form (it is a pull-based query API). We
do use `expo-network` opportunistically in the background task path for a
one-shot pre-flight check (see ┬º4).

### 3.2 Known iOS pitfalls and how we handle each

| Pitfall | Behaviour | Our handling |
|---|---|---|
| **Airplane mode ÔåÆ off, slow Wi-Fi reassociation.** iOS sometimes reports `isConnected=true` ~1ÔÇô3s before `isInternetReachable` flips. | NetInfo emits two events in quick succession. | Our online-edge trigger fires on `isInternetReachable === true` specifically, not on `isConnected === true`. |
| **Captive portal (hotel Wi-Fi).** `isConnected=true`, `isInternetReachable` may be `true` or `null`. Real HTTPS to Supabase still 403/redirects. | Sync request fails with a transport error. | Standard backoff. Status surface (┬º8) shows "Retry scheduled." |
| **VPN handoff (briefly type=`vpn`, then `wifi`).** | Multiple connection-change events. | Debounce the online-edge signal by 250ms **at the NetInfo listener layer** (not inside `requestSync()`). What `requestSync('online')` sees is a single definitive transition. |
| **Cellular data with strict carrier limits.** | NetInfo reports `cellular`. | Treat the same as any connection. Payloads are small (<8 KB typical batch). No proactive disabling. |
| **`isInternetReachable === null`** (NetInfo's initial state before first probe). | Until first probe, "online" is unknown. | Treat `null` as "unknown ÔÇö try anyway." On failure, scheduler drops to offline state. |

### 3.3 Ambiguity handling ÔÇö "seemingly online but requests fail"

NetInfo reports go-state lazily. Authoritative truth is "did my last RPC
succeed?". Concretely:

- The scheduler's `online` flag is updated by **both** NetInfo events **and** by
  the engine's own observed transport errors. The engine already calls
  `recordSyncTransportFailure` on fetch/network errors (see `engine.ts`); v2
  extends this to also call `scheduler.setOnline(false)` on a transport error,
  even if NetInfo still claims `isConnected=true`.
- Backoff is driven by `outbox.calculateBackoffDelayMs` (exponential with
  jitter). On the next `isInternetReachable === true` *or* on the next safety
  tick, we retry ÔÇö the observed-error timestamp wins over NetInfo's stale
  "online" reading.

## 4. Background sync ÔÇö chosen mechanism

### 4.1 The pick

**`expo-background-task`** (wrapping iOS `BGAppRefreshTask`). One task, one
identifier, fires opportunistically while the app is backgrounded.

We do **not** use:

- **`BGProcessingTask`** ÔÇö designed for long-running overnight work; Apple's
  scheduling pushes it to charging windows. We want frequent short refreshes,
  which is exactly what `BGAppRefreshTask` is for.
- **Silent push** ÔÇö requires APNs infrastructure we don't have, Apple
  rate-limits aggressively, and two-device support is explicitly low priority.
  Can be added later for purpose #5 (groups) without re-architecting.
- **`react-native-background-fetch`** ÔÇö third-party, overlaps with what Expo
  already provides natively. Adds prebuild friction with no offsetting win.

### 4.2 Apple's stated rules

- `BGAppRefreshTask` execution time is **capped at ~30 seconds** of wall-clock
  CPU per invocation.
- iOS schedules tasks **opportunistically** based on device usage, battery,
  and network conditions. Typical firings for an actively-used app: 1ÔÇô4 per
  day. There is no guarantee.
- `minimumInterval` is a **floor, not a ceiling.** Setting it to 15 minutes
  means "no sooner than 15 minutes;" it does not mean "every 15 minutes."
- Background tasks require user permission via **Settings ÔåÆ General ÔåÆ
  Background App Refresh**. If disabled globally or for our app,
  `BackgroundTask.getStatusAsync()` returns `Restricted` and we get **zero**
  background firings. The Settings sync surface (┬º8) calls this out
  explicitly.

### 4.3 The critical constraint: cycle must fit in 30s

A BG cycle has at most ~30s of wall-clock budget. We target **Ôëñ 20s** for the
background path (a 10s margin). This is only safe because of one
requirement on t2:

> **t2 constraint:** A sync cycle internally operates in batches that commit
> independently. An interruption mid-cycle (budget exceeded, network drop,
> iOS suspend) leaves the database in a consistent state and the next cycle
> picks up where this one left off.

Without batched-independent-commits, a BG cycle that hit the 30s cap would
either (a) corrupt state by committing partially, or (b) waste the entire
budget if forced to be all-or-nothing. With it, we always make progress.

### 4.4 BG task Ôåö network detection

When the BG task fires, the app's NetInfo listener is **not** running ÔÇö there
is no foreground JS context with long-lived subscriptions. The BG task must:

1. **Do a one-shot network check synchronously** before starting a cycle.
   `expo-network`'s `getNetworkStateAsync()` (or equivalent) returns the
   current state without subscribing.
2. **Abort cleanly if offline.** Starting a cycle that immediately fails the
   first request wastes the BG budget. Returning `Success` (i.e. "nothing to
   do") preserves Apple's scheduling preference for this task.
3. **Do NOT try to subscribe to NetInfo events inside the BG task.** The task
   lifetime is too short to receive any. NetInfo's long-lived listener is a
   foreground-only world.

The foreground NetInfo listener (┬º3) and the BG task's one-shot check are
**separate code paths** that don't share state. The BG task is stateless from
NetInfo's perspective.

### 4.5 Configuration (prose only)

The build task adds the following to `apps/mobile/app.config.ts`:

- An entry in the `plugins` array for `expo-background-task`. The Expo plugin
  accepts default options; the default task identifier
  (`com.expo.modules.backgroundtask.processing`) is fine ÔÇö no override.
- Dependencies `expo-background-task` and `expo-task-manager` added to
  `apps/mobile/package.json` via `expo install` (picks SDK 54-compatible
  versions).

During prebuild (CNG; we don't commit an `ios/` directory), Expo's plugin
writes the following into the generated `Info.plist`:

- `UIBackgroundModes` array gains the `"processing"` entry.
- `BGTaskSchedulerPermittedIdentifiers` array gains
  `"com.expo.modules.backgroundtask.processing"`.

No literal `.plist` or `app.config.ts` snippets in this design doc ÔÇö the build
task copies from the [Expo `expo-background-task` docs][expo-bg-task]. The
critical requirement is the identifier match between the plugin registration
and the runtime `TaskManager.defineTask` call.

[expo-bg-task]: https://docs.expo.dev/versions/latest/sdk/background-task/

## 6. In-flight on suspend / resume ÔÇö requirements

This section states what the sync protocol (t2) must provide so that t4's
trigger-and-cycle model is correct under iOS suspension and termination. It
does NOT prescribe a mechanism ÔÇö that is t2's design.

### 6.1 Cycle duration target

- **Background path:** cycle must complete (or cleanly abort with progress)
  within **20 seconds** of wall-clock time. This is the soft budget; the
  hard ceiling is Apple's ~30s BG cap (┬º4).
- **Foreground path:** **soft target Ôëñ 5 seconds** under normal conditions
  (typical PULL+PUSH+PULL with small batches). No hard ceiling ÔÇö foreground
  can run as long as the JS event loop is available. The 5s target is for
  UX responsiveness; missing it is not a correctness issue.

These targets feed back into t2's choice of batch size and per-request
timeouts.

### 6.2 Per-batch atomicity

Each batch (push batch or pull page) commits independently. An interruption
mid-cycle ÔÇö JS thread suspended, network drop, budget abort, app termination
ÔÇö leaves the database in a consistent state for every batch that completed.
The next cycle resumes from the next un-committed batch.

> **t2 constraint:** push batches and pull pages each commit independently
> on both ends (server and client). No "cycle-wide transaction" semantics;
> no all-or-nothing per cycle.

### 6.3 Idempotency under interruption

A cycle that re-runs after interruption must **converge, not duplicate or
corrupt.** Specifically: the same set of dirty rows re-pushed, the same pull
cursor re-fetched, must produce the same final state as if the original
cycle had completed.

> **t2 constraint:** push is idempotent. Re-pushing the same row
> (`(owner_user_id, entity_type, id, client_updated_at_ms, payload)`) twice
> yields the same final server state and the same per-row outcome on the
> response.
>
> **t2 constraint:** pull is idempotent by cursor. The cursor advances only
> when a page is fully applied to local SQLite; an interrupted apply leaves
> the cursor at its pre-page value and the next cycle re-fetches the same
> page.

### 6.4 Surviving state

The only state that survives iOS suspension / termination is what's
durably written to local SQLite. Specifically:

- **Per-row `local_dirty` bit** in each entity table. Set by repo writes;
  cleared only when the corresponding PUSH response is applied.
- **Pull cursor** in a single key/value row. Advanced only when a pull page
  is fully applied.

No in-memory queue, no separate "outbox" persistence, no resume token.

> **t2 constraint:** the only state surfaces that survive app suspension
> are the per-row `local_dirty` bits and the single pull cursor. Any
> additional state t2 introduces must either (a) be reconstructible from
> these on next cycle, or (b) be persisted in SQLite with equivalent
> survival guarantees.

### 6.5 Resumption is just the foreground edge

On `background ÔåÆ active`, `requestSync('foreground')` fires (┬º1). The cycle
that runs reads dirty rows (same set the interrupted cycle was working on)
and the current cursor (same one the interrupted cycle had advanced to).
There is no separate "resume" code path.

### 7.2 Tap behaviour

- **Button disabled** if any of: not signed in, sync disabled, currently in a
  running cycle, or `isInternetReachable === false`.
- **On tap:** call `requestSync('manual')`. Because the button was enabled
  only when `Scheduled(t)`, this clears the timer and starts a cycle
  immediately.
- **Feedback:** await the scheduler's next state transition out of `Running`.
  Toast "Sync complete" on success/idle; "Sync failed: <message>" on
  transport or server failure.

### 7.3 No special "force" mode

In v1, "Sync now" needed `force: true` to skip per-trigger debounces. In v2
there is no debounce on non-edit triggers; "Sync now" just calls the same
`requestSync()` everything else calls.

## 8. Sync status surface ÔÇö what changes vs `profile-status.ts`

### 8.1 What v1 already provides

`apps/mobile/src/sync/profile-status.ts:SyncProfileStatusSnapshot` exposes:

- `isEnabled`, `kind`, `statusLabel`, `statusHint`
- `pendingCount`, `pendingCountCapped`
- `lastSuccessfulSyncAt`, `nextAttemptAt`
- `errorMessage`, `retryHint`
- `isOnline`

Consumed by `app/profile.tsx` and tested in
`app/__tests__/sync-profile-status.test.ts` +
`settings-profile-navigation.test.tsx`.

### 8.2 What v2 changes

1. **`pendingCount` source** changes from "rows in the outbox table" to
   "count of rows where `local_dirty = true` across all entity tables."
   Bounded by `SYNC_PROFILE_PENDING_SCAN_LIMIT` (currently 1000) for cap
   behaviour. Keeps `pendingCountCapped` semantics.
2. **New fields:**
   - `networkType: 'wifi' | 'cellular' | 'other' | 'none' | 'unknown'` ÔÇö
     from NetInfo. Renders as "Wi-Fi ┬À online" / "Offline".
   - `backgroundTaskStatus: 'available' | 'restricted' | 'denied' | 'unknown'`
     ÔÇö from `BackgroundTask.getStatusAsync()`. Renders "Background sync:
     Available / Disabled in iOS Settings".
   - `lastSyncDurationMs: number | null` ÔÇö for honest power-user telemetry.
3. **`kind` enum extension** to add `'background_disabled'` (info-level,
   not blocking) when `backgroundTaskStatus === 'restricted'` and no
   higher-priority state. Suggests enabling Background App Refresh; not an
   error ÔÇö foreground sync still works.

### 8.3 What stays the same

`deriveSyncProfileStatus` keeps its pure-function shape; new fields slot
into existing branches without rewriting decision logic.

### 8.4 What gets deleted

Outbox-specific fields (`deliveryState.retryBlocked`, etc.) come from a
thinner `dirty-row-flush-state` struct in v2 ÔÇö different source, same wire
shape.

## 9. Testability

### 9.1 Unit tests (Jest, `apps/mobile/app/__tests__/`)

Keep existing `app/__tests__/sync-scheduler.test.ts` and extend:

- **Two-state invariant:** assert the scheduler is always in exactly one of
  `{Running, Scheduled(t)}` after `start()`. Use a state-probe helper.
- **MIN-debounce on edit:** fire `requestSync('mutation')` while
  `Scheduled(now + 30000)` ÔåÆ schedule becomes `MIN(30000, 500) = 500ms`.
  Fire `requestSync('mutation')` again 100ms later ÔåÆ schedule stays at
  ~400ms remaining (MIN held). Single cycle fires after the burst.
- **Edit does not preempt sooner schedule:** scheduler in `Scheduled(now +
  200ms)` (e.g. safety tick imminent). Fire `requestSync('mutation')` ÔåÆ
  schedule stays at 200ms, NOT 500ms.
- **Edit while Running is no-op:** during a running cycle, fire
  `requestSync('mutation')` ÔåÆ no state change, no new timer, no rerun
  flag.
- **Foreground edge:** `AppState` `background ÔåÆ active` while
  `Scheduled(t)` ÔåÆ timer cancelled, cycle starts immediately.
- **Online edge:** NetInfo offline ÔåÆ online while signed in ÔåÆ immediate
  cycle.
- **Manual "Sync now":** `requestSync('manual')` while `Scheduled(t)` ÔåÆ
  immediate cycle.
- **Safety tick fires:** timer expiry transitions to `Running`.
- **Cycle .finally() schedules next:** after a cycle ends (success, failure,
  budget abort), state becomes `Scheduled(now + SAFETY_INTERVAL)`.
- **Background-task wrapper:** the `TaskManager.defineTask` callback awaits
  `requestSync('background-task')`; returns `Success` on clean result,
  `Failed` on failure status. Pre-flight network check aborts cleanly
  (returns `Success`, no cycle started) when offline.

### 9.2 Sync engine tests (existing ÔÇö refactor in place)

`app/__tests__/sync-flush-partial-failure.test.ts`,
`sync-runtime-bootstrap.test.ts`, `sync-bootstrap-seed-survival.test.ts`
survive the v2 transition ÔÇö they test convergence behaviour. Update mocks

### 9.3 Maestro flows

Add `apps/mobile/.maestro/flows/sync-trigger-foreground.yaml`:

1. Opens via `boga3://maestro-harness?reset=data&teleport=session-recorder`.
2. Logs one set.

## 11. t2 constraints emitted by t4

This is the consolidated index of every constraint t4's design places on the
t2 (sync protocol) design. When t2 is (re)written, the agent should treat
each of these as a hard input. They are scattered through the doc as
"**t2 constraint:**" call-outs; this section is the single index.

### 11.1 Cycle shape

1. **Cycle drains until empty.** A cycle continues issuing PULLs until the
   server has no new rows beyond the cursor, and PUSHes until no local rows
   are dirty. The exact PULL/PUSH interleave is t2's choice; the requirement
   is convergence to "nothing left on either side" before the cycle ends.
   (┬º2.4)
2. **Cycle duration target.** Background cycles must complete or cleanly
   abort with progress within **20 seconds** of wall-clock (10s margin under
   Apple's ~30s BG cap). Foreground cycles target Ôëñ 5 seconds for
   responsiveness (soft, not hard). (┬º6.1)

### 11.2 Per-batch atomicity

3. **Batched independent commits.** A cycle internally operates in batches
   (push batches, pull pages) that commit independently on both server and
   client. No cycle-wide transaction; no all-or-nothing per cycle.
   Interruption mid-cycle (budget abort, suspend, network drop) leaves the
   database consistent through the last committed batch. The next cycle
   picks up from the next un-committed batch. (┬º4.3, ┬º6.2)

### 11.3 Idempotency under interruption

4. **Push is idempotent.** Re-pushing the same row tuple
   `(owner_user_id, entity_type, id, client_updated_at_ms, payload)` twice
   yields the same final server state and the same per-row outcome on the
   response. A retry of a previously-acked row returns an "applied"
   outcome (so the client clears the dirty bit), not an error. (┬º6.3)
5. **Pull is idempotent by cursor.** The cursor advances only when a page
   is fully applied to local SQLite. An interrupted apply leaves the
   cursor at its pre-page value; the next cycle re-fetches the same page.
   (┬º6.3)

### 11.4 Surviving state

6. **Only dirty bits and the pull cursor survive suspension.** The per-row
   `local_dirty` bit and the single pull cursor are the only state surfaces
   that must survive iOS suspension or termination. Any additional state
   t2 introduces must be either (a) reconstructible from these on the next
   cycle, or (b) persisted in SQLite with equivalent durability. No
   in-memory queue, no separate outbox persistence, no resume token.
   (┬º6.4)

### 11.5 Out of t4's scope but t2 must own

- The exact JSON shapes of `/sync/push` and `/sync/pull` requests and
  responses.
- The PULL/PUSH ordering within a cycle (we recommend `PULL ÔåÆ PUSH ÔåÆ
  PULL_until_empty` but do not require it; convergence is what matters).
- Push batching policy (which rows go in which batch; FK closure;
  per-batch row caps).
- The dirty-bit lifecycle (when set, when cleared, how the clear
  interacts with concurrent writes).
- Tombstone semantics (per `plan.md` t2 outcomes).
- Server-side FK strategy (deferrable FKs vs ordered batches).
- Per-row response outcomes (whole-batch reject vs partial accept).
