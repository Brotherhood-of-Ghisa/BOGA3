# t4 — iOS sync trigger reliability (design)

> Status: design doc only. No code or config is changed by this PR. A follow-up build
> task consumes this doc verbatim. iOS-only; Android is deferred per `plan.md` (current
> product ships via TestFlight as `com.phano.boga3.dev`).

## 0. TL;DR

- **Foreground is the primary sync surface.** v2 runs an in-process scheduler very
  similar to today's `scheduler.ts`. The shape changes from "scheduler periodically
  flushes the outbox" to "scheduler periodically runs `push → pull` against rows
  flagged `local_dirty=true`, plus a pull cursor advance."
- **Triggers fire on:** debounced mutation (`500ms` push debounce, `1500ms` pull
  debounce), app foreground (`active`), explicit network online edge, a "sync now"
  manual control, and a safety interval (default `60s` foreground; `10s` while the
  user is in `session-recorder`; `5s` only in dev builds, gated by `isDevMode()`).
- **Background mechanism — chosen:** **`expo-background-task`** (which wraps
  iOS `BGTaskScheduler`'s `BGAppRefreshTask`). Single task identifier
  `com.expo.modules.backgroundtask.processing` (the Expo-managed default). One task,
  one job: "run one push+pull cycle if anything is dirty or the cursor is stale."
  Not `BGProcessingTask`. Not silent push. Justification in §4.
- **Network state:** `@react-native-community/netinfo` with both `isConnected` and
  `isInternetReachable`. Pessimistic reachability (only consider us online when
  `isInternetReachable === true`) plus a request-level fallback that treats
  observed network failures as "go to offline state, schedule retry."
- **In-flight on suspend/resume:** any push interrupted by suspend is treated as
  failed-but-idempotent. On resume, dirty bits are still set, the same rows are
  re-pushed; LWW on the server makes the retry safe. The client retains no
  per-batch persistent state beyond the dirty bits and pull cursor.
- **"Sync now" UX:** lives on Settings under a new **Sync** section, distinct from
  the Profile screen's existing sync block. Tapping forces an immediate
  `push → pull` ignoring debounce, with a result toast. Disabled while offline or
  while a sync is in flight.
- **Battery/data:** at the default cadence with no edits, foreground costs at most
  ~60 requests/hour (one safety tick per minute; each tick is a single pull RPC if
  nothing is dirty). With sustained editing it's bounded by debounce + cadence to
  ~120 requests/hour. Background adds at most a handful per day per iOS scheduling
  reality.

The rest of this doc walks every Outcomes bullet from `plan.md` t4 in order.

## 1. iOS app lifecycle states and which trigger fires in each transition

iOS exposes four [UI lifecycle states][apple-uistate] surfaced to React Native as
[`AppState`][rn-appstate]: `active`, `inactive`, `background`, and (implicit)
`terminated` (delivered via launch, not as an `AppState` event). `RN.AppState` emits
`active`, `inactive`, `background`. There is no public RN signal for "suspended"
distinct from "background" — iOS suspends a backgrounded app silently when its
remaining background time hits zero.

### 1.1 Transition table

| iOS transition | `AppState` event | Sync trigger | What runs |
|---|---|---|---|
| Cold launch (terminated → foreground) | initial `active` | **Bootstrap + immediate flush** | `startSyncRuntime()` → bootstrap if needed → `push → pull` |
| Warm resume (background → foreground) | `active` (from `background`) | **Foreground trigger** | One immediate `push → pull`, then resume safety-interval cadence |
| User pulls down notification center, swipes between apps, accepts incoming call (active → inactive) | `inactive` (from `active`) | **No-op** | Scheduler keeps ticking; flushes continue if running. We do *not* pause the cadence during `inactive`. iOS does not throttle in-process work for `inactive`. |
| inactive → active (interruption resolves) | `active` (from `inactive`) | **No-op** | Cadence already running. No extra flush — avoids "double-fire" when the user briefly switches apps. |
| active → background | `background` | **Background-prep trigger** | Best-effort: kick off one synchronous `push → pull` attempt using the OS's standard ~5s/up-to-30s "expiration handler" window iOS grants apps entering background. Implemented as a fire-and-forget call inside the `AppState` listener; we do NOT use `beginBackgroundTask` directly (Expo doesn't surface it). The in-flight request continues until iOS suspends us — if interrupted, the dirty bits stay set; see §6. |
| background → suspended (no event) | (none) | **No-op** | The scheduler timer stops firing because the JS thread is paused. |
| suspended → terminated (no event) | (next launch fires `active`) | **No-op** | State will be reconstructed from local persistence on next launch. |
| Backgrounded long enough for `BGTaskScheduler` to grant time | (no `AppState` event; native task only) | **Background-task trigger** (§4) | The Expo background task runs JS in a short-lived headless context. Performs one `push → pull` cycle. |

### 1.2 Wiring vs today's code

Today, `apps/mobile/app/_layout.tsx` calls `startDefaultSyncScheduler()` once at
mount; the scheduler runs forever on the JS event loop and pauses *only* indirectly
when iOS suspends the app. There is no `AppState` listener in `sync/`. v2 adds one,
co-located with the scheduler, that translates `active`/`background` into
`schedulerOnAppActive()` / `schedulerOnAppBackground()` calls. The session-recorder
already has its own `AppState` listener at
`apps/mobile/src/session-recorder/lifecycle-helpers.ts`; the new sync listener is
independent and does not replace it.

[apple-uistate]: https://developer.apple.com/documentation/uikit/managing-your-app-s-life-cycle
[rn-appstate]: https://reactnative.dev/docs/appstate

## 2. Foreground triggers

### 2.1 Trigger set

The scheduler is event-driven plus a periodic safety net. **Five trigger sources**:

1. **Debounced mutation push.** A repo write (anything that flips `local_dirty=true`)
   schedules a push `500ms` later. A second mutation inside the window restarts the
   timer (coalescing). This bounds the burst rate during multi-set logging: typing
   8 reps then 135 lb then tapping "complete set" produces one push, not three.
2. **Debounced pull on remote-hint signal.** Push responses *will* carry a "server
   has newer rows than your cursor" hint (per t2). Receiving that hint schedules a
   pull `1500ms` later (longer than push debounce — the user's pending edits should
   land first). On its own this is also useful for a future silent-push integration.
3. **App foreground (`active` from `background`).** Immediate `push → pull` (no
   debounce). The user has just returned, they want freshness.
4. **Network online edge.** When NetInfo transitions to "we believe we are online,"
   immediate `push → pull`. This already exists in scheduler.ts as
   `setOnline(true)` → fires `flush()`. We keep that semantics and add the pull leg.
5. **Safety-interval tick.** Plain interval, default `60s`. Each tick does
   `push` if `local_dirty` rows exist, `pull` always (server-time cursor is cheap
   to advance, and it covers two-device freshness for purpose #2). When the
   pathname is `session-recorder`, the interval tightens to `10s`. In dev
   (`isDevMode() === true`) it tightens further to `5s` — useful for manual
   verification of cadence behaviour on a TestFlight dev build.

### 2.2 Coalescing rules

The scheduler exposes a single `requestSync(reason)` API. All triggers route through
it. Internally:

- If a sync is in-flight, the new request **sets a "pending" flag** that runs one
  more cycle when the current one finishes. The flag is not a queue — N requests
  while in-flight collapse to one re-run.
- Pull is always done after push within one cycle. A "pull-only" trigger (e.g.
  remote-hint debounce) still goes through `requestSync('pull')` which internally
  becomes `(push-if-dirty, then pull)`.
- The debounce timer is independent per `(reason)`. Two close mutations collapse
  via the same timer; a mutation and a remote-hint pull don't compete because they
  resolve to the same `requestSync` call anyway.

### 2.3 Concrete defaults

```ts
// proposed constants — final placement is sync/scheduler.ts
export const SYNC_MUTATION_PUSH_DEBOUNCE_MS = 500;
export const SYNC_REMOTE_HINT_PULL_DEBOUNCE_MS = 1500;
export const SYNC_GENERAL_CADENCE_MS = 60_000;            // unchanged from v1
export const SYNC_SESSION_RECORDER_CADENCE_MS = 10_000;   // unchanged from v1
export const SYNC_DEV_CADENCE_MS = 5_000;                 // new — only used when isDevMode()
```

The dev override path is **not** `__DEV__` — that is `false` in the dev
TestFlight build (`com.phano.boga3.dev`) where we most need the tight cadence
during manual testing. Implementation:

```ts
import { isDevMode } from '@/src/utils/isDevMode';

const cadenceForContext = (context: SyncCadenceContext) => {
  if (isDevMode()) return SYNC_DEV_CADENCE_MS;
  return context === 'session-recorder'
    ? SYNC_SESSION_RECORDER_CADENCE_MS
    : SYNC_GENERAL_CADENCE_MS;
};
```

The lint rule `no-restricted-syntax` (referenced in `isDevMode.ts`) already enforces
"no bare `__DEV__`."

## 3. Network state

### 3.1 API choice — NetInfo

We adopt [`@react-native-community/netinfo`][netinfo-docs] (`NetInfo`). Reasons:

- It is the de facto network-state library across the React Native ecosystem and
  Expo SDK 54 supports it without prebuild (per Expo docs).
- It exposes both `isConnected` (link-layer / radio-up) and `isInternetReachable`
  (HEAD-probe-derived) — we need both.
- It emits `addEventListener` callbacks on radio changes; we wire it into
  `scheduler.setOnline(...)`.

We do **not** use `expo-network` for the live state surface because it does not emit
change events in the same form (it is a pull-based query API), only a one-shot
`getNetworkStateAsync()`. We may use `expo-network` opportunistically in the
background task (which runs outside of long-lived listeners) for a one-shot check
before performing work, since `Network.getNetworkStateAsync()` is also available
without a config plugin.

### 3.2 Known iOS pitfalls and how we handle each

| Pitfall | Behaviour | Our handling |
|---|---|---|
| **Airplane mode → off, slow Wi-Fi reassociation.** iOS sometimes reports `isConnected=true` ~1–3s before `isInternetReachable` flips. | NetInfo emits two events in quick succession. The first fires our online edge trigger; the actual HTTPS request then fails fast. | Our online-edge trigger fires on `isInternetReachable === true` specifically, not just `isConnected === true`. If we sync on a false-positive, the request will fail and we'll record a transport failure that schedules a backoff retry — but we'd rather avoid the noise. |
| **Captive portal (hotel Wi-Fi).** `isConnected=true`, `isInternetReachable` may be `true` or `null` depending on the probe URL. Real HTTPS to Supabase still 403/redirects. | Sync request fails with a 4xx the engine treats as transport failure. | Same backoff as any transport failure. The pull will fail until the user logs into the captive portal. No special UI needed — the status surface (§8) will show "Retry scheduled" and the cause "Network error." |
| **VPN handoff (briefly type=`vpn`, then `wifi`).** | Multiple connection-change events. | Debounce the online-edge trigger by 250ms inside the scheduler so we don't fire a flush on every NetInfo event. |
| **Cellular data with strict carrier limits / metering.** | NetInfo reports `cellular`. iOS does not surface "user disabled cellular data for this app" as a separate state. | Treat the same as offline at the request level once the OS returns the relevant error. No proactive disabling on cellular — sync payloads are small (typed JSON, <8 KB per typical batch). |
| **`isInternetReachable === null`** (NetInfo's initial state before first probe). | Until first probe, "online" is unknown. | Treat `null` as "unknown — try anyway." On failure, scheduler drops to offline state. |

### 3.3 Ambiguity handling — "seemingly online but requests fail"

NetInfo reports go-state lazily. Authoritative truth is "did my last RPC succeed?".
Concretely:

- The scheduler's `online` flag is updated by **both** NetInfo events **and** by
  the engine's own observed transport errors. The engine already calls
  `recordSyncTransportFailure` on fetch/network errors (see `engine.ts`); we extend
  this to also call `scheduler.setOnline(false)` on a transport error, even if
  NetInfo still claims `isConnected=true`.
- Backoff is driven by `outbox.calculateBackoffDelayMs` (already exponential with
  jitter, see exports in `sync/index.ts`). On the next NetInfo `isInternetReachable
  === true` *or* on the next safety tick (whichever first), we retry — the
  observed-error timestamp wins over NetInfo's stale "online" reading.

[netinfo-docs]: https://github.com/react-native-netinfo/react-native-netinfo

## 4. Background sync — chosen mechanism

### 4.1 The pick

**One mechanism: `expo-background-task` (wrapping iOS `BGAppRefreshTask`).**

Single registration, single task identifier, fires opportunistically while the app
is backgrounded.

### 4.2 Justification — why this, not the alternatives

- **Not `BGProcessingTask`.** BGProcessingTask is for heavy, long-running work
  (cleanup, ML training) scheduled by iOS during charging at night. Our sync
  payload is small and finishes in <2s on a normal connection. Apple's guidance
  is to use `BGAppRefreshTask` for "short tasks that keep your app's content up to
  date" — that is literally our use case. (Apple, "Choosing background strategies
  for your app.") Using BGProcessingTask would mean syncing much less often (often
  once a day) and is hostile to two-device freshness.
- **Not silent push.** Silent push (`content-available=1`) would give us
  "remote-changes-available" signalling and is technically the lowest-latency
  background path on iOS. We do not pursue it because:
  - Sync v2's purpose-#2 (two-device support) is explicitly low priority.
  - Silent push requires an APNs setup we do not have, and changes the Supabase
    backend story (need a notify-on-write trigger that pushes to APNs per device).
  - Apple **rate-limits** silent push: "Apple may throttle pushes that arrive at
    a high rate," and delivery is best-effort, often delayed by 30+ minutes
    under load. It's not a reliability win for our use case.
  - We can add it later (purpose #5 — groups — would benefit) without re-architecting.
- **Not `react-native-background-fetch`.** Third-party library, overlaps with
  what Expo SDK 54+ now provides natively via `expo-background-task` (added in
  SDK 50, stable in current versions). Adding a community package would mean an
  extra prebuild step and another moving piece in our `app.config.ts`. Expo's
  module is the lower-friction win.

### 4.3 What the task does

```ts
// proposed wiring — final placement is sync/background-task.ts
import * as BackgroundTask from 'expo-background-task';
import * as TaskManager from 'expo-task-manager';
import { runOneSyncCycle } from './scheduler';

const SYNC_BACKGROUND_TASK_NAME = 'sync-v2-foreground-refresh';

// Define in global scope (module side-effect; imported from app root).
TaskManager.defineTask(SYNC_BACKGROUND_TASK_NAME, async () => {
  try {
    // runOneSyncCycle returns SyncFlushResult-equivalent. It is responsible for
    // bailing if not signed in / disabled / offline. Internally guarded against
    // double-execution with the same single-in-flight slot as the foreground
    // scheduler.
    const result = await runOneSyncCycle({ reason: 'background-task' });
    return result.status === 'success' || result.status === 'idle'
      ? BackgroundTask.BackgroundTaskResult.Success
      : BackgroundTask.BackgroundTaskResult.Failed;
  } catch (err) {
    // Surface to logEvent — runOneSyncCycle should not throw normally.
    return BackgroundTask.BackgroundTaskResult.Failed;
  }
});

export const registerSyncBackgroundTask = async () => {
  if (!(await TaskManager.isTaskRegisteredAsync(SYNC_BACKGROUND_TASK_NAME))) {
    await BackgroundTask.registerTaskAsync(SYNC_BACKGROUND_TASK_NAME, {
      // minutes; iOS may run later than this but not sooner.
      minimumInterval: 15,
    });
  }
};
```

`registerSyncBackgroundTask()` is called from `app/_layout.tsx` next to
`startSyncRuntime()`. The task is auto-rescheduled by `expo-background-task` after
each run (per Expo docs).

### 4.4 iOS budget / Apple's stated rules

From Apple's [BGTaskScheduler documentation][bgtask] and the
[`UIApplication` background-execution docs][uiapp-bg]:

- `BGAppRefreshTask` execution time is **capped at ~30 seconds** of wall-clock
  CPU per invocation. Our sync cycle is well inside this — typical push+pull
  finishes in <2s.
- The system "chooses an optimal time to launch your background process based on
  device usage, battery state, and the conditions you set." There is **no
  guarantee** the task fires; on heavy/low-battery devices it may not fire for
  days. We commit to designing for "we'll get N opportunities a day," not "we
  pulled at 15:00:00." (Apple's docs: "Even in the best of circumstances,
  BGTaskScheduler isn't a precise timer.")
- The `minimumInterval` is a floor, not a ceiling.
- iOS suspends a backgrounded foreground app within seconds of entering background
  unless a `beginBackgroundTask` handler is active. The `AppState=background`
  transition gives us a few seconds at best to drain in-flight requests; we don't
  rely on it for correctness (§6).
- Background tasks require user permission via the system Background App Refresh
  toggle (Settings → General → Background App Refresh). If the user has it off
  globally or off for our app, `BackgroundTask.getStatusAsync()` returns
  `Restricted` and we get **zero** background firings. The Settings sync surface
  (§8) must call this out — "Background sync is off in iOS Settings; foreground
  sync continues to work."

### 4.5 Concrete registration block — `app.config.ts`

A future builder copies the following plugin entry verbatim into `app.config.ts`'s
`plugins:` array (between `"expo-router"` and `"expo-secure-store"` to keep the
order alphabetical-ish):

```ts
// In apps/mobile/app.config.ts, plugins array:
[
  "expo-background-task",
  {
    // Default identifier is "com.expo.modules.backgroundtask.processing".
    // We accept the default — no override needed. The identifier is registered
    // automatically into BGTaskSchedulerPermittedIdentifiers by the Expo plugin
    // during prebuild.
  }
],
```

Per the Expo docs:

> "To enable background tasks on iOS, add the 'processing' value to the
> UIBackgroundModes array and register the 'com.expo.modules.backgroundtask.processing'
> identifier in the BGTaskSchedulerPermittedIdentifiers array. If using Continuous
> Native Generation (CNG), these configurations are applied automatically by
> prebuild."

Our project uses CNG (no `ios/` directory committed), so the Info.plist mutations
happen automatically during EAS build. The literal effect is:

```xml

<key>UIBackgroundModes</key>
<array>
  <string>processing</string>
</array>
<key>BGTaskSchedulerPermittedIdentifiers</key>
<array>
  <string>com.expo.modules.backgroundtask.processing</string>
</array>
```

If we later split identifiers (e.g. a separate "long pull on charging" task), this
expands; for v2 we keep it to one.

Also add `"expo-background-task"` and `"expo-task-manager"` to `apps/mobile/package.json`
dependencies in the build step — `expo install` will pick the SDK 54-compatible versions.

## 6. In-flight request behaviour on suspend / resume

### 6.1 What happens at the wire level

iOS will:

- Allow the JS thread to keep running for a brief window after `applicationDidEnterBackground:`,
  typically 5–30s depending on conditions.
- Suspend the app process when that window ends. In-flight `fetch()` calls are
  not cancelled, but the JS thread stops executing — so any `await` after a
  request completes never resolves until the app comes back to foreground (and
  often won't resolve at all, because iOS may terminate the suspended app).

### 6.2 What the v2 client does

**The client treats any sync interrupted by suspend as a no-op-and-retry.** The
only persistent state that matters is:

1. Per-row `local_dirty=true` bit, set when the row is written (§ plan.md t1).
2. The pull cursor (a single bigint/timestamp stored in a local key/value table).

If a push interrupted by suspend never received its server response, the dirty
bits remain set (we clear them in the response handler, which never ran). On
resume, the next trigger pushes the same rows again. The server's LWW upsert
is idempotent: pushing the same `(owner, type, id, client_updated_at_ms,
payload)` twice produces the same final row state.

Pull is similarly idempotent: the cursor only advances when the response is
processed locally. An interrupted pull means we keep the old cursor and try
again, possibly re-receiving rows we already had — LWW makes that safe.

### 6.3 Why this is correct (and why the in-flight outbox machinery becomes simpler)

The v1 engine maintains a complex "in-flight flush promise" with single-slot
locking and an outbox of unacked events (`sync/engine.ts:35` —
`inFlightFlushPromise`). v2 keeps the single-in-flight lock (still useful to
prevent overlap), but the underlying queue collapses: the queue *is* the table
of dirty rows. There is no separate outbox to compact or persist.

Practical effect: a future builder deleting `sync/outbox.ts` and replacing it with
a per-table dirty-row reader is straightforward — the only "queue state" that
needs to survive a kill is already in the local DB.

### 6.4 What if the server *did* commit but the client never saw the ack?

LWW guarantees re-sending the same payload with the same `client_updated_at_ms`
is a no-op on the server (`updated_at_ms <= stored` → reject upsert, per
brainstorm.md §4.1). The second client push just clears the dirty bit. Net result:
no data loss, slight wasted bandwidth (one duplicate request).

## 7. "Sync now" UX

### 7.1 Placement

A new **Sync** section in `app/(tabs)/settings.tsx`, separate from the existing
Profile sync widget (the existing widget on `app/profile.tsx` is a status surface;
it stays). The Settings entry is the user-action surface.

Layout (text-only mock; UI builder owns the visual styling):

```
Settings → Sync
┌─────────────────────────────────────────────┐
│ Sync status                                 │
│ Up to date                                  │
│ Last sync: 2 minutes ago                    │
│                                             │
│ [ Sync now ]   ← primary button             │
│                                             │
│ Pending changes: 0                          │
│ Network: Wi-Fi · online                     │
│ Background sync: Available                  │
│                                             │
│ ─────────────────────────────────────────── │
│ Sync is enabled                       [ON]  │
│                                             │
│ [ Wipe local data ]   (dev only)            │
│ [ Wipe my remote data ]   (dev only)        │
└─────────────────────────────────────────────┘
```

### 7.2 Tap behaviour

- **Button disabled** if any of: not signed in, sync disabled, currently in-flight,
  or `isInternetReachable === false`.
- **On tap:**
  1. Set button state to "Syncing…".
  2. Call `runOneSyncCycle({ reason: 'manual', force: true })`.
     - `force: true` skips the debounce window — but it still respects the
       single-in-flight lock; if a sync is already running, this becomes a wait
       on the in-flight promise.
     - Push first, then pull, regardless of cursor staleness or dirty count.
  3. On result:
     - `success` / `idle` → toast "Sync complete" + refresh the Settings status
       block.
     - `failure_*` / `transport_failure` → toast "Sync failed: <message>".
     - `disabled` / `offline` / `blocked` → toast with the specific reason.
- **Feedback:** a small inline `UiText` showing the most recent result, persisted
  in the local Settings state for the duration the screen is mounted.

### 7.3 Why force-push-then-pull (not pull-only)

The user reaches for "Sync now" usually because they just made a change and want
to confirm it's safe, or because they're switching to another device. In either
case they want their own writes uploaded before they see what the server thinks.
The cost of running push when nothing is dirty is one zero-row request — cheap.

## 8. Sync status surface — what changes vs `profile-status.ts`

### 8.1 What v1 already provides

`apps/mobile/src/sync/profile-status.ts:SyncProfileStatusSnapshot` exposes:

- `isEnabled`, `kind`, `statusLabel`, `statusHint`
- `pendingCount`, `pendingCountCapped`
- `lastSuccessfulSyncAt`, `nextAttemptAt`
- `errorMessage`, `retryHint`
- `isOnline`

Consumed by `app/profile.tsx` (Profile screen) and tested in
`app/__tests__/sync-profile-status.test.ts` + `settings-profile-navigation.test.tsx`.

### 8.2 What v2 changes

The shape stays mostly intact — it's a good abstraction. Three changes:

1. **`pendingCount` source** changes from "rows in the outbox table" to "count of
   rows where `local_dirty = true` across all local entity tables." The query
   becomes `SELECT SUM(c) FROM (SELECT count(*) AS c FROM <table> WHERE local_dirty
   GROUP BY ALL) t` — bounded by `SYNC_PROFILE_PENDING_SCAN_LIMIT` (currently
   `1000`) for cap behaviour. Keeps `pendingCountCapped` semantics.
2. **New fields added** to `SyncProfileStatusSnapshot`:
   - `networkType: 'wifi' | 'cellular' | 'other' | 'none' | 'unknown'` — from
     NetInfo. Renders in Settings as "Wi-Fi · online" / "Cellular · online" / "Offline".
   - `backgroundTaskStatus: 'available' | 'restricted' | 'denied' | 'unknown'` —
     from `BackgroundTask.getStatusAsync()`. Renders the "Background sync:
     Available / Disabled in iOS Settings" line.
   - `lastSyncDurationMs: number | null` — for honest power-user telemetry on
     Settings.
3. **`kind` enum extension** to add `'background_disabled'` (info-level, not
   blocking) — fires when `backgroundTaskStatus === 'restricted'` AND there's no
   higher-priority state. Suggests "Enable Background App Refresh in iOS
   Settings." This is a hint, not an error — foreground sync still works.

### 8.3 What stays the same

`deriveSyncProfileStatus` keeps its pure-function shape; the new fields slot into
existing branches without rewriting any decision logic. The Profile screen's
display continues to work without changes. Settings reads the same snapshot via
`loadSyncProfileStatus({ isSignedIn })`.

### 8.4 What gets deleted

`runtimeState.bootstrapCompletedAt` and `lastBootstrapError` survive (v2 still has
a bootstrap step per t2). The outbox-specific fields (`deliveryState.retryBlocked`,
`deliveryState.nextAttemptAt`, etc.) come from a thinner `dirty-row-flush-state`
struct in v2 — different source, same shape on the wire.

## 9. Battery / data tradeoffs

### 9.1 Quantification — foreground

**Baseline (no edits, app open):**
- Safety interval = 60s → 60 syncs/hour at most.
- Each safety tick = 1 pull RPC (no dirty rows → no push). Pull payload at steady
  state = empty body + 8-byte cursor + auth header. Response = empty rows + new
  cursor. Estimate <1 KB total round-trip per request.
- **~60 KB/hour** sustained while open and idle.

**Active editing (logging a workout):**
- User in `session-recorder` → safety interval = 10s → up to 360/hour.
- Plus debounced mutations: typing ~30 sets / hour with 500ms debounce → at most
  one push per set boundary → ~30 pushes/hour.
- Realistic: workouts are 45–90 minutes; effective hourly rate during a workout
  is ~120–150 requests/hour.
- Payload: a typed exercise_set row JSON is ~200 bytes. 30 sets × 200 bytes =
  ~6 KB upload per workout. Pulls are similar order.
- **~50–100 KB per logged workout** of sync traffic, all over HTTPS to Supabase.

**Dev mode (`isDevMode() === true`):**
- 5s safety interval → up to 720/hour. Acceptable on dev builds; not shipped to
  real users.

### 9.2 Quantification — background

`BGAppRefreshTask` opportunities: in practice, 1–4 per day for an actively-used app
on modern iOS (Apple does not document a number; user reports & WWDC sessions
converge here). Each task does one push+pull, identical cost to a foreground
safety tick.

**Net background cost: negligible** (<10 KB/day).

### 9.3 Honest call-outs

- We do **not** pause sync on cellular (no reason to — payloads are tiny).
- We do **not** batch beyond what `SYNC_BATCH_MAX_SIZE` (already in code) imposes.
- We do **not** use compression beyond what HTTPS provides; Supabase enables gzip
  by default, payloads are small enough that this isn't a meaningful lever.
- Battery: the dominant cost is the radio wake, not the work. Foreground cadence
  of 60s aligns with the natural cadence of most messaging apps; we are not an
  outlier. Cadence is tunable — if a user reports battery pain we have one knob
  to pull.

## 10. Testability

### 10.1 Unit tests (Jest, `apps/mobile/app/__tests__/`)

Keep the existing scheduler unit tests (`app/__tests__/sync-scheduler.test.ts`)
and extend to cover:

- **Debounced mutation:** schedule `requestSync('mutation')` twice within
  500ms → exactly one push fires. Use Jest fake timers.
- **Foreground edge:** simulate `AppState` `background → active` → exactly
  one immediate sync fires regardless of safety-tick phase.
- **Online edge:** simulate NetInfo offline → online while signed in → one
  immediate sync.
- **Single-in-flight lock:** trigger two `requestSync()` calls while the first is
  awaiting → second runs once, exactly once, after the first resolves.
- **`isDevMode()` cadence override:** mock `isDevMode()` → true, assert the
  cadence is 5000ms; mock → false, assert 60000ms.
- **Background-task wrapper:** `defineTask` callback runs `runOneSyncCycle` once
  and returns `Success` on a clean result, `Failed` on any failure status.

### 10.2 Sync engine tests (existing — refactor in place)

`app/__tests__/sync-flush-partial-failure.test.ts`,
`sync-runtime-bootstrap.test.ts`, `sync-bootstrap-seed-survival.test.ts` survive
the v2 transition — they test convergence behaviour, which the new push/pull
model needs to preserve. Update mocks for the new push/pull RPC shape; assertions
unchanged.

### 10.3 Maestro flows

Add a new flow `apps/mobile/.maestro/flows/sync-trigger-foreground.yaml` that:

1. Opens the app via `boga3://maestro-harness?reset=data&teleport=session-recorder`.
2. Logs one set.
3. Asserts (via a test-only `Sync now` button or a status text on Settings)
   that pending count goes to 0 within 5s (dev cadence helps here).
4. Backgrounds the app (Maestro: `pressKey: HOME` or equivalent), waits ~3s,
   foregrounds (`launchApp`).
5. Asserts the status reads "Up to date" and the last sync timestamp is recent.

Add an entry to `apps/mobile/package.json:scripts`:

```
"test:e2e:ios:sync-foreground": "./scripts/maestro-ios-sync-foreground.sh"
```

The orchestrator runs `test:e2e:ios:smoke` and `test:e2e:ios:data-smoke` by
default; per the user's memory rule, after UI changes those two are mandatory.
The new flow is an optional add-on the builder runs locally before opening the
PR, then wires into CI alongside the others.

### 10.4 What we do **not** automate

- **Real `BGAppRefreshTask` firing.** Apple's scheduler isn't programmable from
  CI. We rely on `BackgroundTask.triggerTaskWorkerForTestingAsync()` (dev-only)
  for a manual smoke-test loop during dev — invoked from a hidden dev-only
  button in Settings (gated by `isDevMode()`), wired to the new
  `runSyncBackgroundTaskForTesting()` helper.
- **Captive-portal scenarios.** No automated coverage; documented in §3 so a
  reviewer / SRE knows the expected behaviour.
- **Cross-device LWW races.** Out of scope for t4 (t2's concern).
