# t4 — Sync v2: iOS sync trigger reliability (design)

> Design doc only. No code or config is changed by this PR. A follow-up build
> task consumes this doc verbatim. iOS-only; Android is deferred per `plan.md`.
>
> **Companion designs:**
>
> - t1 (merged): typed server schema, RLS, deferrable FKs, LWW semantics.
> - t2 (merged): wire protocol, cycle interface (t2 §6), dirty-bit lifecycle
>   (t2 §7), the four per-layer pull cursors (t2 §9.2). t4 schedules calls
>   into the cycle t2 defines.
> - t3 (deferred): seed loader. Not in scope here.

## 0. TL;DR

The scheduler is a 4-state machine:
**OFFLINE / LONG_TIMEOUT / SHORT_TIMEOUT / RUNNING**. Two intervals,
three external inputs, two internal events. The full transition
tables are in §2.2 and §2.3.

- **`SHORT_INTERVAL = 1000ms`** — the debounce window. A "request
  sync" input in LONG_TIMEOUT pulls the timer forward into
  SHORT_TIMEOUT; in SHORT_TIMEOUT it's a no-op (already short-armed).
- **`LONG_INTERVAL = 60_000ms`** — the safety backstop. Armed on
  every cycle-ends transition (when still online). The only retry
  path: a failed cycle ends, LONG_TIMEOUT re-arms, the next tick
  re-pushes.
- **Network state** flips OFFLINE ↔ {LONG, SHORT}_TIMEOUT exclusively
  via NetInfo (§3). v1's engine-emitted `setOnline(false)` on
  transport errors is gone.

There is one `setTimeout` handle in the system. There is no queue,
no backoff table, no `nextAttemptAt` bookkeeping, no per-trigger
code path.

A cycle with nothing to do is cheap (one `sync_pull` RPC returning
empty layer-0, no SQLite writes), so we run them freely. This is
what justifies the simplicity: we don't need elaborate gating
because cycles don't cost much.

Background mechanism: **`expo-background-task`** wrapping iOS
`BGAppRefreshTask`. The task body does a one-shot network check,
runs one cycle directly (outside the four-state machine), and
returns.

The rest of this doc walks every Outcomes bullet from `plan.md` t4
in order.

## 1. iOS lifecycle states and which trigger fires in each transition

iOS exposes four [UI lifecycle states][apple-uistate] surfaced to React
Native as [`AppState`][rn-appstate]: `active`, `inactive`, `background`,
plus the implicit `terminated` state delivered as a launch (not as an
`AppState` event). There is no public RN signal for "suspended"
distinct from "background" — iOS suspends a backgrounded app silently
when its remaining background time hits zero.

### 1.1 Transition table

| iOS transition | `AppState` event | Sync action |
|---|---|---|
| Cold launch (terminated → foreground) | initial `active` | Bootstrap (auth/runtime), then emit "request sync". |
| Warm resume (background → foreground) | `active` (from `background`) | Emit "request sync". |
| active → inactive (notification center, app-switcher, call) | `inactive` (from `active`) | **No-op.** The scheduler keeps its current deadline. iOS does not throttle in-process JS for `inactive`. |
| inactive → active (interruption resolves) | `active` (from `inactive`) | **No-op.** The foreground edge fires only on `background → active`. |
| active → background | `background` | **No-op.** If a cycle is running it keeps running until iOS suspends the JS thread; if not, we do not start one — the OS's background-time grace is too narrow to be worth chasing. The next foreground edge or background-task firing picks up. |
| background → suspended (no event) | (none) | The JS thread is paused; the timer stops firing. Any pending deadline is implicitly deferred until JS runs again. |
| suspended → terminated (no event) | (next launch fires `active`) | State reconstructs from local SQLite on next launch. |
| iOS grants `BGAppRefreshTask` time | (no `AppState` event; native task only) | **Background-task path** (§4) runs one cycle directly. |

### 1.2 Wiring vs today's code

Today, `apps/mobile/app/_layout.tsx` calls `startDefaultSyncScheduler()`
once at mount and the scheduler runs forever on the JS event loop,
pausing only indirectly when iOS suspends the app. There is no
`AppState` listener at the sync layer. v2 adds one, co-located with
the scheduler, that translates `background → active` into a "request
sync" input. The session-recorder already has its own `AppState`
listener at `apps/mobile/src/session-recorder/lifecycle-helpers.ts`;
the sync listener is independent and does not replace it.

[apple-uistate]: https://developer.apple.com/documentation/uikit/managing-your-app-s-life-cycle
[rn-appstate]: https://reactnative.dev/docs/appstate

## 2. The scheduler

The scheduler is a 4-state machine. Two intervals (`SHORT_INTERVAL`,
`LONG_INTERVAL`); three external inputs (request sync, go online, go
offline); two internal events (timer fires, cycle ends). Everything
the scheduler does is captured by the transition tables in §2.2 and
§2.3.

### 2.1 The four states

| State | Meaning | Timer armed? |
|---|---|---|
| **OFFLINE** | No usable network. We do not sync. | No |
| **LONG_TIMEOUT** | Online and idle; the safety backstop is ticking. | Yes, `LONG_INTERVAL` |
| **SHORT_TIMEOUT** | Online and something nudged us; the debounce window is ticking. | Yes, `SHORT_INTERVAL` |
| **RUNNING** | A cycle is in flight. | No |

Initial state on app launch is **OFFLINE** (per §3 — we wait for
NetInfo's first definitive `isInternetReachable === true` before
considering ourselves online).

### 2.2 External inputs

Three external inputs reach the scheduler:

- **request sync** — an edit, a foreground edge, a "Sync now" tap, or
  the cold-launch trigger. All call the same entry point. The call
  sites are listed in §2.5.
- **go online** — NetInfo emits `isInternetReachable === true` from a
  previously not-true value.
- **go offline** — NetInfo emits `isInternetReachable !== true`
  (false or null) from a previously true value.

| From → | request sync | go online | go offline |
|---|---|---|---|
| **OFFLINE** | no-op | → **SHORT_TIMEOUT**, arm `SHORT_INTERVAL` | no-op |
| **LONG_TIMEOUT** | cancel timer, → **SHORT_TIMEOUT**, arm `SHORT_INTERVAL` | no-op (already online) | cancel timer, → **OFFLINE** |
| **SHORT_TIMEOUT** | no-op (already short-armed — don't restart and delay it) | no-op (already online) | cancel timer, → **OFFLINE** |
| **RUNNING** | no-op (the cycle drains both ends — t2 §6) | no-op | no-op (let the in-flight cycle complete or error; the cycle-ends transition in §2.3 takes us to OFFLINE) |

The SHORT_TIMEOUT row is the heart of the design. "request sync" never
restarts an already-armed short timer — that would *delay* the cycle.
It only ever pulls a LONG_TIMEOUT forward into a SHORT_TIMEOUT.

### 2.3 Internal transitions

Two internal events occur as a consequence of state, not as external
inputs:

- **timer fires** — the armed `setTimeout` reaches its deadline.
- **cycle ends** — the running cycle's promise settles (success or
  thrown error; the scheduler does not distinguish).

| From → | timer fires | cycle ends |
|---|---|---|
| **OFFLINE** | — (no timer armed) | — (no cycle running) |
| **LONG_TIMEOUT** | → **RUNNING**, start cycle | — |
| **SHORT_TIMEOUT** | → **RUNNING**, start cycle | — |
| **RUNNING** | — (no timer armed) | if currently online → **LONG_TIMEOUT**, arm `LONG_INTERVAL`. If currently offline → **OFFLINE**. |

The "currently online" check at cycle-end consults the latest NetInfo
projection — the cycle may have started online and ended offline (or
vice versa) if NetInfo events landed during the cycle. The state
machine handles both branches symmetrically.

### 2.4 Why this shape

- **SHORT_TIMEOUT and LONG_TIMEOUT are distinct states**, not a
  single state with a deadline number. The two states have different
  no-op rules under "request sync" (LONG_TIMEOUT shrinks; SHORT_TIMEOUT
  doesn't), which is awkward to express as a number comparison and
  natural as a state.
- **OFFLINE has no timer.** When network drops we cancel the timer.
  No safety tick fires while offline — it would just be a wasted RPC.
- **RUNNING is sticky.** Inputs during a running cycle are no-ops; the
  cycle drains both ends to empty (per t2 §6), so an edit landing
  mid-cycle is picked up by the cycle's own iteration. cycle-ends
  always re-arms the appropriate state.
- **The transition tables are total.** Every (state, event) cell is
  filled in — no implicit branches, no defensive guards, no edge
  cases. The build wave's tests in §10.1 walk this table cell-by-cell.

The whole machine has 4 states × 5 events = 20 cells. Compared to v1's
outbox-driven scheduler with retry counters, backoff state, per-route
cadence context, and engine-emitted online flips, this is materially
smaller.

### 2.5 What feeds "request sync"

Every external signal that should cause sync activity routes through
"request sync":

| Source | Notes |
|---|---|
| Repo write (any mutation flipping `local_dirty = 1`) | Fired from the repo layer after the SQLite transaction commits. |
| `AppState` `background → active` | The foreground edge. `inactive ↔ active` does NOT fire. |
| "Sync now" button tap | See §7 for UX. |
| Cold launch after bootstrap | Fired once at the end of the boot sequence; if still in OFFLINE the input is a no-op, and the first NetInfo "go online" will take us into SHORT_TIMEOUT instead. |

NetInfo `isInternetReachable` transitions feed "go online" / "go
offline" (§3.2), not "request sync".

The background-task path (§4) is separate. It does not feed any of
these inputs — it runs its own cycle directly in a short-lived native
context, outside the four-state machine entirely.

### 2.6 Constants

| Name | Value | Notes |
|---|---|---|
| `SHORT_INTERVAL` | **1000 ms** | The debounce window. The wait between "request sync" arriving and the cycle starting. |
| `LONG_INTERVAL` | **60_000 ms** | The safety backstop. The wait between a cycle ending and the next idle-state cycle starting. |

Both same in dev and prod. **No `__DEV__` override. No `isDevMode()`
override.** Dev reflects production for both intervals; TestFlight
verification uses the same cadence as users see. (The `isDevMode()`
rule from the user's memory still applies to other dev affordances
elsewhere — e.g. the "Wipe local data" button — it does not apply to
scheduler cadence.)

### 2.7 What v1 had that v2 doesn't

When porting the v1 scheduler the build task deletes:

- `SYNC_SESSION_RECORDER_CADENCE_MS = 10_000` — no per-route cadence.
- `SESSION_RECORDER_ROUTE_SEGMENT` and
  `syncCadenceContextFromPathname` — no path-based context.
- `setContext` / `getContext` on the scheduler, and the
  `setDefaultSyncCadenceContextFromPathname` invocations across
  `apps/mobile/app/` — cadence context is no longer plumbed.
- `outbox.calculateBackoffDelayMs` and any `nextAttemptAt` /
  `retryBlocked` columns and code paths — no backoff bookkeeping.
- `recordSyncTransportFailure → setOnline(false)` in `engine.ts` —
  NetInfo is the sole emitter of "go offline" (§3.4).

For push-on-edit to work, the session recorder must persist edits to
local SQLite promptly: on every numeric-input blur, every "complete
set" tap, every set-navigation, and after ≥ 1s dwell with unsaved
changes. Each of those writes flips `local_dirty = 1` and fires
"request sync". This is an audit item for the build wave, not a new
requirement — the recorder already persists at these points; v2 just
removes the polling cadence safety net.

## 3. Network state

### 3.1 API choice — NetInfo

Adopt [`@react-native-community/netinfo`][netinfo-docs]. Reasons:

- De facto network-state library across the React Native ecosystem;
  Expo SDK 54 supports it without prebuild.
- Exposes both `isConnected` (link-layer / radio-up) and
  `isInternetReachable` (HEAD-probe-derived). We use the latter.
- Emits `addEventListener` callbacks on network changes; wires into
  `setOnline(...)` directly.

We do not use `expo-network` for the foreground state surface
(pull-based query API, no change events). We do use `expo-network`
for the background-task one-shot pre-flight (§4).

### 3.2 Mapping NetInfo events to scheduler inputs

A single listener on the NetInfo subscription projects
`isInternetReachable` to a boolean (`true` → online; `false` or
`null` → offline). On a change in that boolean projection — and only
on a change — it emits one of the §2.2 external inputs:

- Projection flips `offline → online` → emit **go online**.
- Projection flips `online → offline` → emit **go offline**.

No change → no input. The state machine in §2 then decides what to do
(`go online` from OFFLINE → SHORT_TIMEOUT; from any other state → no-op).

The listener does not call any other scheduler entry point. In
particular, reconnection is *not* a "request sync" — `go online` lands
in SHORT_TIMEOUT directly (see §2.2's OFFLINE-row "go online" cell),
without going through the "request sync" path.

VPN handoff / airplane-mode reassociation can produce multiple NetInfo
events in quick succession. Each is filtered by the
projection-changed check, so a "wifi → vpn → wifi" sequence that never
flips `isInternetReachable` away from `true` produces zero scheduler
inputs. Sequences that *do* flip — `online → offline → online` within
a few hundred ms — produce a clean OFFLINE → SHORT_TIMEOUT pair, which
is exactly what we want.

### 3.3 Pitfalls and how we handle each

| Pitfall | Behaviour | Our handling |
|---|---|---|
| **Airplane mode → off, slow Wi-Fi reassociation.** iOS sometimes reports `isConnected=true` ~1–3s before `isInternetReachable` flips. | NetInfo emits two events. | We project on `isInternetReachable === true`, not on `isConnected === true`. The first event (link up, no internet) doesn't flip the projection, so no scheduler input is emitted. |
| **Captive portal (hotel Wi-Fi).** `isConnected=true`; `isInternetReachable` may be `true` or `null`. Real HTTPS still fails. | A cycle's RPC fails with a transport error. | The cycle ends; the state machine arms LONG_TIMEOUT; the next safety tick (60s later) retries. Status surface (§7) shows "Up to date as of N minutes ago" — no special "captive portal" branch. |
| **VPN handoff** (briefly type=`vpn`, then `wifi`). | Multiple connection-change events. | Most don't flip `isInternetReachable`. Of those that do, the SHORT_TIMEOUT no-op rule (§2.2 SHORT_TIMEOUT row, "request sync" cell — and same logic applies to a "go online" arriving in OFFLINE → SHORT_TIMEOUT then a flicker → OFFLINE → SHORT_TIMEOUT pair) collapses repeated short-armings so the cycle fires at most once per `SHORT_INTERVAL`. |
| **Cellular data with strict carrier limits.** | NetInfo reports `cellular`. | Treated the same as any connection. Payloads are small (see §9). |
| **`isInternetReachable === null`** (NetInfo's initial state before the first probe). | The boolean projection (§3.2) is `false`. | We stay in OFFLINE until the first definitive `true` lands. Cold launch is mildly delayed by the time-to-first-NetInfo-probe (~hundreds of ms). The simpler alternative ("treat null as online") would have the first cycle race the NetInfo probe; the failure would still self-heal on the next LONG_TIMEOUT, but the launch path is cleaner if we wait for a definitive answer. |

### 3.4 NetInfo is the sole emitter of "go offline"

v1's pattern (`recordSyncTransportFailure` in `engine.ts` → flipping
the scheduler's online flag false on observed transport errors) is
dropped. NetInfo is the sole source of the "go offline" input. A
transport error inside a cycle is just an error: the cycle ends, the
state machine arms LONG_TIMEOUT (per §2.3, "if currently online"),
and the next safety tick retries.

This costs one wasted retry per `LONG_INTERVAL` while requests are
failing for reasons NetInfo doesn't see (captive portal, Supabase
outage, intermittent 500s). The retry itself is cheap — one
round-trip that fails fast. Acceptable cost for not having a second
source of truth to keep in sync with NetInfo.

[netinfo-docs]: https://github.com/react-native-netinfo/react-native-netinfo

## 4. Background sync — chosen mechanism

### 4.1 The pick

**`expo-background-task`** wrapping iOS `BGAppRefreshTask`. One task,
one identifier. The task body runs one cycle and returns.

We do **not** use:

- **`BGProcessingTask`** — designed for long overnight work; iOS
  pushes it to charging windows. We want frequent short refreshes,
  which is exactly `BGAppRefreshTask`'s purpose.
- **Silent push** — needs APNs infrastructure we don't have. Apple
  rate-limits aggressively. Two-device support is explicitly low
  priority (`plan.md` purpose #2). Can be added later for purpose #5
  (groups) without re-architecting the trigger surface.
- **`react-native-background-fetch`** — third-party, overlaps with
  what Expo provides natively. Prebuild friction with no offsetting
  win.

### 4.2 Apple's stated rules

- `BGAppRefreshTask` execution time is capped at **~30 seconds** of
  wall-clock per invocation (Apple's [BGTaskScheduler docs][apple-bgtask]).
- iOS schedules tasks **opportunistically** based on device usage,
  battery, and network conditions. Typical firings for an actively-used
  app: 1–4 per day. No guarantee.
- `minimumInterval` is a **floor, not a ceiling.** Setting 15 minutes
  means "no sooner than 15 minutes;" it does not mean "every 15
  minutes."
- Background tasks require **Settings → General → Background App
  Refresh** to be on globally and for our app. If disabled,
  `BackgroundTask.getStatusAsync()` returns `Restricted` and we get
  zero background firings. The sync status surface (§7) calls this
  out.

[apple-bgtask]: https://developer.apple.com/documentation/backgroundtasks/bgtaskscheduler

### 4.3 The critical constraint: cycle must fit in ~30s

A BG cycle has at most ~30s of wall-clock budget. We target **≤ 20s**
for the background path (10s margin). This is safe because of t2 §6.2's
"Cycle duration target" and "Per-batch commits" guarantees — a cycle
interrupted on budget makes forward progress, and the next firing
picks up. See §11 #2 / #3.

### 4.4 BG task pre-flight and body

The task body (registered via `TaskManager.defineTask` at module-load
time in `apps/mobile/src/sync/background-task.ts`) does two things,
in order:

1. **Pre-flight network check** via `expo-network`'s
   `getNetworkStateAsync()`. NetInfo's long-lived listener is not
   running in this short-lived native context, so we query
   synchronously. If `isInternetReachable` is not `true`, return
   `BackgroundTaskResult.Success` without starting a cycle —
   returning `Success` (rather than `Failed`) preserves Apple's
   scheduling preference for the task.
2. **Run one cycle directly.** Bypasses the four-state machine in
   §2 — the BG task IS the trigger; there's nothing to coalesce.
   Return `BackgroundTaskResult.Success` on success;
   `BackgroundTaskResult.Failed` on thrown error.

Registration via `BackgroundTask.registerTaskAsync` happens once
during app init (same module that starts the foreground scheduler)
with `minimumInterval: 15` (minutes; Apple ignores smaller values).

The task body does NOT consult or mutate the foreground scheduler's
state. It runs its own cycle in a stateless context. The foreground
scheduler's timer is irrelevant here — the BG task is woken by iOS,
not by a JS setTimeout.

### 4.5 Configuration (the `app.config.ts` plugin block)

The build task adds to `apps/mobile/app.config.ts`:

```ts
// In the plugins array, append:
"expo-background-task",
```

That's the entire plugin entry — no options needed. The plugin's
default identifier matches what `TaskManager.defineTask` registers.

Dependencies added to `apps/mobile/package.json` via `expo install
expo-background-task expo-task-manager @react-native-community/netinfo
expo-network` (picks SDK-compatible versions).

During prebuild (CNG; we don't commit `ios/`), Expo's plugin writes
into the generated `Info.plist`:

- `UIBackgroundModes` array gains the `processing` entry.
- `BGTaskSchedulerPermittedIdentifiers` array gains the task
  identifier.

The build wave verifies the identifier match (the same constant used
in `TaskManager.defineTask` AND
`BackgroundTask.registerTaskAsync` AND the plist entry) before merging.
Identifier mismatch is the most common failure mode for this API and
produces zero firings with no diagnostic.

## 6. In-flight on suspend / resume

### 6.1 What happens when iOS suspends mid-cycle

A cycle in flight when the app is backgrounded continues to run on
the JS thread until iOS decides to suspend (typically within 5–30
seconds of `background`, depending on which background modes are
declared). When suspension hits, the JS thread is paused mid-execution.

Three sub-cases:

1. **Suspension hits between batches.** The last committed batch's
   dirty bits are cleared (push) or the cursor is advanced (pull).
   Per-batch commits (t2 §6.2, t4 §11 #3) make this safe.
2. **Suspension hits during an HTTP request.** The request is
   abandoned by the OS. On resume, the foreground edge
   (`background → active`) emits a "request sync" input, which lands
   the scheduler in SHORT_TIMEOUT; one second later the cycle starts,
   reads the still-set dirty bits, and re-pushes. The server's LWW
   handles the duplicate idempotently (t2 §3.6).
3. **Suspension hits during the page-apply transaction.** SQLite's
   transaction is rolled back when the process is paused mid-write
   (commit log not yet flushed); the cursor is unchanged. On resume,
   the next cycle re-fetches the same page. Idempotent (t2 §4.7).

In all three cases, **no separate "resume" code path is needed.** The
foreground edge fires "request sync" like any other source (§2.5),
and the cycle reconstructs from local SQLite state (dirty bits + four
per-layer cursors + `last_emitted_ms`, per t2 §9).

### 6.2 Surviving state — concrete inventory

Per t2 §9, the persisted state surfaces are:

- Per-entity: `local_dirty`, `local_updated_at_ms` on every row.
- Singleton on `sync_runtime_state`: `pull_cursor` (JSON map of four
  per-layer cursors), `last_emitted_ms`, `bootstrap_completed_at`.

No in-memory queue, no separate outbox table (v1's
`sync_outbox_events` is deleted in the build wave), no resume token,
no `nextAttemptAt`.

The scheduler's own state (which of the four §2.1 states we're in,
plus the live `setTimeout` handle if any) is in-memory and reconstructs
on next launch: start in OFFLINE, and the first NetInfo "go online"
input plus the `AppState` initial `active` event take it from there.

## 7. "Sync now" UX

### 7.1 Placement

A new **Sync** section in `app/(tabs)/settings.tsx`, separate from the
existing Profile sync widget on `app/profile.tsx` (the Profile widget
stays — it's a status surface, not a user-action surface).

Text-only mock:

```
Settings → Sync
┌─────────────────────────────────────────────┐
│ Sync status                                 │
│ Up to date                                  │
│ Last sync: 2 minutes ago                    │
│                                             │
│ [ Sync now ]                                │
│                                             │
│ Pending changes: 0                          │
│ Network: Wi-Fi · online                     │
│ Background sync: Available                  │
│                                             │
│ ─────────────────────────────────────────── │
│ Sync is enabled                       [ON]  │
│                                             │
│ [ Wipe local data ]   (dev only)            │
│ [ Wipe my remote data ]   (dev only)        │
└─────────────────────────────────────────────┘
```

### 7.2 Tap behaviour

- **Button disabled** if any of: not signed in, sync disabled,
  scheduler is in OFFLINE, scheduler is in RUNNING.
- **On tap:** emit a "request sync" input. The button was enabled
  only when in LONG_TIMEOUT or SHORT_TIMEOUT; either way the
  transition table in §2.2 lands us in SHORT_TIMEOUT and the cycle
  fires within `SHORT_INTERVAL` (1s).
- **Feedback:** show a toast "Syncing…" immediately on tap (so the
  button feels responsive). When the scheduler transitions out of
  RUNNING, show "Sync complete" (success path) or "Sync failed:
  <message>" (cycle threw).

### 7.3 No "force" mode

In v1, "Sync now" needed `force: true` to bypass per-trigger
debounces. In v2 there is one debounce path and the button takes it.
No force mode; the worst-case wait is `SHORT_INTERVAL` (1s) and
the toast-on-tap covers the UX.

## 8. Sync status surface — what changes vs `profile-status.ts`

### 8.1 What v1 provides

`apps/mobile/src/sync/profile-status.ts:SyncProfileStatusSnapshot`
currently exposes: `isEnabled`, `kind`, `statusLabel`, `statusHint`,
`pendingCount`, `pendingCountCapped`, `lastSuccessfulSyncAt`,
`nextAttemptAt`, `errorMessage`, `retryHint`, `isOnline`.

Consumed by `app/profile.tsx` and tested in
`app/__tests__/sync-profile-status.test.ts` plus
`settings-profile-navigation.test.tsx`.

### 8.2 What v2 changes

1. **`pendingCount` source** changes from "rows in
   `sync_outbox_events`" to "count of rows where `local_dirty = 1`
   across the eight entity tables." Bounded by
   `SYNC_PROFILE_PENDING_SCAN_LIMIT` (currently 1000) for the cap
   semantics. `pendingCountCapped` is preserved.
2. **`nextAttemptAt` source** changes from the outbox's
   `next_attempt_at` to the deadline of the scheduler's current armed
   timer (the SHORT_TIMEOUT or LONG_TIMEOUT state in §2.1). The field
   is now "the next time a cycle is scheduled to start" — could be
   `SHORT_INTERVAL` out (debounced) or `LONG_INTERVAL` out (safety
   tick), or `null` in OFFLINE / RUNNING.
3. **New fields:**
   - `networkType: 'wifi' | 'cellular' | 'other' | 'none' | 'unknown'`
     — from NetInfo. Renders as "Wi-Fi · online" / "Offline".
   - `backgroundTaskStatus: 'available' | 'restricted' | 'denied' | 'unknown'`
     — from `BackgroundTask.getStatusAsync()`. Renders "Background
     sync: Available" or "Background sync: Disabled in iOS Settings".
   - `lastSyncDurationMs: number | null` — for honest power-user
     telemetry; populated by wrapping the cycle in a timer.
4. **`kind` enum extension** adds `'background_disabled'`
   (info-level, not blocking) when `backgroundTaskStatus ===
   'restricted'` and no higher-priority state is active. Suggests
   enabling Background App Refresh; not an error.
5. **`kind: 'retry_scheduled'` is dropped.** The v1 retry-scheduled
   state was driven by `outbox.calculateBackoffDelayMs`. v2 has no
   backoff state; a cycle that fails is indistinguishable in the
   status surface from a cycle that hasn't run yet. The user sees
   either "Up to date" (last cycle succeeded) or "Syncing…" (cycle
   in flight). Persistent failures show as a growing
   `lastSuccessfulSyncAt` staleness — the value the user actually
   cares about.
6. **`deliveryState.retryBlocked` path is dropped.** v1's
   "non-retryable backend response → block all sync" gate is gone
   in v2 because the only protocol-level non-retriable error is
   `FK_VIOLATION` (t2 §2.2), which is a structural bug fixed by app
   update; there is no class of normal errors we'd want to block
   on. `kind: 'action_required'` is reserved for the bootstrap-
   failed case (t2 §5 follow-up) and for `'background_disabled'`.

### 8.3 What stays the same

`deriveSyncProfileStatus` keeps its pure-function shape; new fields
slot into existing branches without rewriting decision logic. The
Profile widget and Settings widget can share one
`SyncProfileStatusSnapshot` source.

### 8.4 What gets deleted

`apps/mobile/src/sync/outbox.ts:getSyncDeliveryState` and its
`SyncDeliveryStateSnapshot` shape come out entirely when the outbox
table is dropped in the v2 build wave. The remaining "delivery"
information (last success, last error, pending count) moves to a
thinner `sync_runtime_state`-derived snapshot.

## 9. Battery / data tradeoffs

Honest quantification at the v2 default cadence.

### 9.1 Idle foreground (user has the app open but isn't editing)

- 1 cycle per `LONG_INTERVAL` = 60 cycles/hour.
- Per cycle, idle: 4 `sync_pull` RPCs (one empty page per layer),
  zero pushes. Each empty pull is ~300 bytes request + ~150 bytes
  response over TLS.
- ~60 cycles × ~2 KB ≈ **~120 KB/hour** of idle network traffic.
- See t2 §4.4's open follow-up about collapsing the 4 idle-layer
  probes into one — when that lands, idle drops to ~30 KB/hour.

### 9.2 Active editing (mid-workout)

- Edits arrive in bursts; the debounce coalesces them.
  Conservatively: 1 set logged ≈ 1 cycle (the 1s of typing collapses
  to one debounce window).
- A typical 1-hour workout has ~40 sets logged. **~40 cycles** in the
  hour.
- Per push cycle, a fresh set adds one row to `exercise_sets`
  (~200 bytes payload). The 4 pull probes still happen.
- ~40 cycles × (~2 KB pull + ~300 bytes push) ≈ **~100 KB/hour** of
  edit-driven traffic.

### 9.3 Background

- 1–4 `BGAppRefreshTask` firings per day for an actively-used app
  (Apple's typical figure; not a guarantee).
- Each firing is one cycle. Idle cost: ~2 KB. With dirty rows:
  ~2 KB + (payload).
- **Negligible** compared to foreground use.

### 9.4 Battery

The dominant cost is the radio waking for HTTPS requests, which
batches with the OS's general network activity (other apps' polling,
push, etc.) — iOS coalesces these. The marginal battery cost of our
~60 idle cycles/hour is small; the cellular-radio wake cost is the
real number and it amortises across the device's existing background
activity.

The 4-empty-pull idle waste (§9.1) is the only obviously-reducible
cost, tracked as a t2 follow-up. Until then, this is the v2 default.

### 9.5 Where this could go wrong

The numbers above assume "idle" means "no remote changes." A
two-device scenario where Device A is actively editing while Device
B is foregrounded but idle would have Device B's idle cycles
returning non-empty pull pages. With small payloads (typical
workout edits are <1 KB per row) the traffic stays within the same
order of magnitude. With pathological churn (one device looping
edits on hundreds of rows), Device B's pulls would amplify
accordingly — outside v2's scope (purpose #2 is explicitly low
priority).

## 10. Testability

### 10.1 Unit tests (Jest, `apps/mobile/app/__tests__/`)

Replace the existing `app/__tests__/sync-scheduler.test.ts` (v1's
tests assume the path-based cadence which is gone). The new tests
walk the §2.2 / §2.3 transition tables cell-by-cell.

Per-cell coverage:

- **OFFLINE × request sync → no-op.** Start in OFFLINE; emit "request
  sync". Assert no state change, no timer armed.
- **OFFLINE × go online → SHORT_TIMEOUT.** Emit "go online" in OFFLINE.
  Assert SHORT_TIMEOUT, timer armed at `SHORT_INTERVAL`.
- **LONG_TIMEOUT × request sync → SHORT_TIMEOUT.** Arm LONG_TIMEOUT;
  emit "request sync" at t+30s. Assert the LONG timer is cancelled,
  state becomes SHORT_TIMEOUT, new timer armed for `SHORT_INTERVAL`
  (so cycle fires at ~t+31s, not t+60s).
- **SHORT_TIMEOUT × request sync → no-op.** In SHORT_TIMEOUT with the
  timer due in 500ms, emit five more "request sync" inputs. Assert
  the timer is *not* reset (cycle still fires at the original 500ms
  mark, not 1s after the last input).
- **SHORT_TIMEOUT × go offline → OFFLINE.** Arm SHORT_TIMEOUT; emit
  "go offline". Assert OFFLINE, timer cancelled.
- **LONG_TIMEOUT × go offline → OFFLINE.** Same pattern.
- **{LONG, SHORT}_TIMEOUT × timer fires → RUNNING.** Let the timer
  expire; assert the cycle starts and state is RUNNING.
- **RUNNING × cycle ends (online) → LONG_TIMEOUT.** Stub the cycle to
  resolve; assert post-cycle state is LONG_TIMEOUT with a fresh
  `LONG_INTERVAL` timer.
- **RUNNING × cycle ends (offline) → OFFLINE.** Stub the cycle to
  resolve while a "go offline" landed during execution; assert
  post-cycle state is OFFLINE with no timer armed.
- **RUNNING × cycle ends after error → LONG_TIMEOUT (no backoff).**
  Stub the cycle to reject; assert post-cycle state is LONG_TIMEOUT,
  timer armed at `LONG_INTERVAL` (not at some backoff value). Assert
  no `nextAttemptAt` is exposed anywhere on the public surface.
- **RUNNING × request sync → no-op.** Start a cycle (unresolved);
  emit "request sync". Assert state stays RUNNING, no extra timer
  armed.

Higher-level integration:

- **Foreground edge:** the `AppState` listener emits "request sync"
  on `background → active` and nothing on `inactive` transitions.
  Walked through the appropriate §2.2 cells via the per-cell tests.
- **NetInfo projection:** the listener emits "go online" only when
  `isInternetReachable === true` (not on `isConnected` flips, not on
  `null`). Multiple events with the same projection produce no input.
- **Background-task wrapper:** define the task with a stub cycle.
  With network online, assert the task awaits the cycle and returns
  `Success`. With network offline (pre-flight fails), assert no
  cycle runs and returns `Success`. With cycle throwing, assert
  returns `Failed`. None of this touches the four-state machine.

### 10.2 Engine tests (existing)

`app/__tests__/sync-flush-partial-failure.test.ts` and friends
survive the v2 transition — they exercise convergence behaviour
under errors. Update mocks for the new `sync_push` / `sync_pull`

### 10.3 Maestro flows

Add `apps/mobile/.maestro/flows/sync-trigger-foreground.yaml`:

1. Opens via `boga3://maestro-harness?reset=data&teleport=session-recorder`.
2. Logs one set.

## 11. t2 constraints emitted by t4

The consolidated index of every constraint t4's design places on t2.
These are scattered through the doc as the requirements t4 needs the
cycle to satisfy.

### 11.1 Cycle shape

1. **Cycle drains until empty.** A cycle continues issuing PULLs
   until the server has no new rows beyond the cursor (per-layer),
   and PUSHes until no local rows are dirty. Convergence to
   "nothing left on either side" is the cycle's job; the exact
   PULL/PUSH interleave is t2's choice (t2 §6.1). (§2.5)
2. **Cycle duration target.** Background cycles complete or cleanly
   abort with progress within **20 seconds** of wall-clock (10s
   margin under Apple's ~30s BG cap). Foreground cycles target
   ≤ 5 seconds for responsiveness (soft, not hard). (§4.3, §6.1)

### 11.2 Per-batch atomicity

3. **Batched independent commits.** A cycle internally operates in
   batches (push batches, pull pages) that commit independently on
   both server and client. No cycle-wide transaction; no
   all-or-nothing per cycle. Interruption mid-cycle (budget abort,
   suspend, network drop) leaves the database consistent through
   the last committed batch. The next cycle picks up from the next
   un-committed batch. (§4.3, §6.1)

### 11.3 Idempotency under interruption

4. **Push is idempotent.** Re-pushing the same row tuple
   `(owner_user_id, entity_type, id, client_updated_at_ms,
   payload)` twice yields the same final server state and clears
   the dirty bit either way. A retry of a previously-acked row
   succeeds (no error). (§6.1 #2, t2 §3.6)
5. **Pull is idempotent by cursor.** The cursor advances only when
   a page is fully applied to local SQLite. An interrupted apply
   leaves the cursor at its pre-page value; the next cycle
   re-fetches the same page. (§6.1 #3, t2 §4.7)

### 11.4 Surviving state

6. **Only dirty bits, the four per-layer pull cursors, and
   `last_emitted_ms` survive suspension.** Per t2 §9, these are
   the local SQLite surfaces. Any additional state t2 introduces
   must be either (a) reconstructible from these on the next cycle,
   or (b) persisted in SQLite with equivalent durability. No
   in-memory queue, no separate outbox persistence, no resume
   token, no `nextAttemptAt`. (§6.2)

### 11.5 Out of t4's scope but t2 owns

- The exact JSON shapes of `sync_push` and `sync_pull` requests
  and responses (t2 §3, §4).
- The PULL/PUSH ordering within a cycle (t2 §6.1).
- Push batching policy — topological order, FK closure, per-batch
  caps (t2 §3.4).
- The dirty-bit lifecycle including the push-in-flight race (t2 §7).
- Tombstone semantics (`deleted_at` only, no separate flag — t1 §1.1,
  t2 §2.1).
- The four per-layer pull-cursor semantics (t2 §4.4, §9.2).
- The clock-monotonicity guard (t2 §8).
