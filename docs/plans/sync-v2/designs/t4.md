> Design doc only. No code or config is changed by this PR. A follow-up build
> task consumes this doc verbatim. iOS-only; Android is deferred per `plan.md`.
>
> **Companion designs:**

> - t2 (merged): wire protocol, cycle interface (t2 §6), dirty-bit lifecycle

>   into the cycle t2 defines.
> - t3 (deferred): seed loader. Not in scope here.

## 0. TL;DR

The scheduler is intentionally tiny. There are two scheduling rules:

1. **Debounce (`DEBOUNCE_INTERVAL = 1000ms`).** Any external signal —
   user edit, app foregrounded, network reconnected, "Sync now" tap —
   calls `requestSync()`, which moves the next-cycle deadline to
   `min(currentDeadline, now + DEBOUNCE_INTERVAL)`. Bursts coalesce
   into one cycle.
2. **Safety (`SAFETY_INTERVAL = 60_000ms`).** After every cycle's
   `.finally()`, the scheduler arms a deadline at `now + SAFETY_INTERVAL`.
   This is the only retry path: a failed cycle leaves dirty bits set and
   the next safety tick re-pushes.

There is one `setTimeout` handle in the system. There is no queue, no
backoff table, no `nextAttemptAt` bookkeeping, no per-trigger code
path. Network state purely gates whether anything is scheduled — when
offline the timer is cancelled; on reconnect `requestSync()` is called.

A cycle with nothing to do is cheap (one `sync_pull` RPC returning an
empty layer-0 page, no SQLite writes), so we run them freely. This is
what justifies the simplicity above: we don't need elaborate gating
because cycles don't cost much.

Background mechanism: **`expo-background-task`** wrapping iOS
`BGAppRefreshTask`. The task body does a one-shot network check, runs
one cycle directly (bypassing the debounce), and returns.

The rest of this doc walks every Outcomes bullet from `plan.md` t4 in
order.

## 1. iOS lifecycle states and which trigger fires in each transition

iOS exposes four [UI lifecycle states][apple-uistate] surfaced to React
Native as [`AppState`][rn-appstate]: `active`, `inactive`, `background`,
plus the implicit `terminated` state delivered as a launch (not as an
`AppState` event). There is no public RN signal for "suspended"
distinct from "background" — iOS suspends a backgrounded app silently
when its remaining background time hits zero.

### 1.1 Transition table

| iOS transition | `AppState` event | Sync action |
|---|---|---|
| Cold launch (terminated → foreground) | initial `active` | Bootstrap (auth/runtime), then **`requestSync()`**. |
| Warm resume (background → foreground) | `active` (from `background`) | **`requestSync()`**. |
| active → inactive (notification center, app-switcher, call) | `inactive` (from `active`) | **No-op.** The scheduler keeps its current deadline. iOS does not throttle in-process JS for `inactive`. |
| inactive → active (interruption resolves) | `active` (from `inactive`) | **No-op.** The foreground edge fires only on `background → active`. |
| active → background | `background` | **No-op.** If a cycle is running it keeps running until iOS suspends the JS thread; if not, we do not start one — the OS's background-time grace is too narrow to be worth chasing. The next foreground edge or background-task firing picks up. |
| background → suspended (no event) | (none) | The JS thread is paused; the timer stops firing. Any pending deadline is implicitly deferred until JS runs again. |
| suspended → terminated (no event) | (next launch fires `active`) | State reconstructs from local SQLite on next launch. |
| iOS grants `BGAppRefreshTask` time | (no `AppState` event; native task only) | **Background-task path** (§4) runs one cycle directly. |

### 1.2 Wiring vs today's code

Today, `apps/mobile/app/_layout.tsx` calls `startDefaultSyncScheduler()`
once at mount and the scheduler runs forever on the JS event loop,
pausing only indirectly when iOS suspends the app. There is no
`AppState` listener at the sync layer. v2 adds one, co-located with
the scheduler, that translates `background → active` into
`requestSync()`. The session-recorder already has its own `AppState`
listener at `apps/mobile/src/session-recorder/lifecycle-helpers.ts`;
the sync listener is independent and does not replace it.

[apple-uistate]: https://developer.apple.com/documentation/uikit/managing-your-app-s-life-cycle
[rn-appstate]: https://reactnative.dev/docs/appstate

## 2. The scheduler

### 2.1 State

```ts
{
  running: boolean;         // true while a cycle is in flight
  online: boolean;          // NetInfo's last known isInternetReachable === true
  timerDeadline: number | null;  // epoch-ms of the next scheduled cycle, or null
  timerHandle: NodeJS.Timeout | null;  // the one setTimeout in the system
}
```

There is no queue, no per-trigger debounce timer that lives between
events, no per-trigger branching, no `reason` parameter passed
anywhere.

### 2.2 The single entry point — `requestSync()`

```ts
function requestSync(): void {
  if (!online) return;          // offline: do nothing (§3)
  if (running) return;          // cycle in flight; its .finally() will arm next
  const target = now() + DEBOUNCE_INTERVAL;
  if (timerDeadline === null || target < timerDeadline) {
    clearTimer();
    timerDeadline = target;
    timerHandle = setTimeout(fireCycle, DEBOUNCE_INTERVAL);
  }
  // else: there is already a sooner-or-equal deadline; leave it alone.
}
```

The `target < timerDeadline` guard is the **MIN-debounce trick**: any
event can pull the deadline forward but never pushes it back. Bursts
of N edits within `DEBOUNCE_INTERVAL` produce one cycle.

This single function handles every external signal — see §2.4. No
caller passes a "reason" because the scheduler does not branch on
one.

### 2.3 The safety tick

```ts
function armSafetyTick(): void {
  if (!online) {
    timerDeadline = null;
    timerHandle = null;
    return;
  }
  clearTimer();
  timerDeadline = now() + SAFETY_INTERVAL;
  timerHandle = setTimeout(fireCycle, SAFETY_INTERVAL);
}
```

Called from `fireCycle().finally()`. If a `requestSync()` lands during
the safety window, the MIN-debounce trick in §2.2 shrinks the deadline
to `now + DEBOUNCE_INTERVAL`. Otherwise the safety tick fires at
`SAFETY_INTERVAL`.

The safety tick is the **only retry path**. There is no exponential
backoff, no `nextAttemptAt` row, no separate retry timer. A cycle that
failed leaves dirty bits set and the next safety tick (60s later) tries
again. If still failing, again 60s later. Etc.

### 2.4 The entry points (all call `requestSync()`)

| Source | Trigger |
|---|---|
| Repo write (any mutation flips `local_dirty = 1` on a row) | `requestSync()` |
| `AppState` `background → active` | `requestSync()` |
| NetInfo `isInternetReachable` flips `false|null → true` | `requestSync()` |
| "Sync now" button tap | `requestSync()` (see §7 for UX) |
| Cold launch after bootstrap | `requestSync()` |

The background-task path (§4) is separate — it bypasses the debounce
because it runs synchronously in a short-lived native context. Every
other entry point routes through `requestSync()`.

### 2.5 The cycle

```ts
async function fireCycle(): Promise<void> {
  timerHandle = null;
  timerDeadline = null;
  if (running || !online) {
    armSafetyTick();
    return;
  }
  running = true;
  try {
    await runSyncCycle();           // t2 §6.1
  } catch (err) {
    logSyncError(err);              // observability only; no retry bookkeeping
  } finally {
    running = false;
    armSafetyTick();
  }
}
```

`runSyncCycle()` is owned by t2 (§6). It drains all four pull layers,
pushes dirty rows in topological order, re-drains, and returns when
both ends are quiet (or budget aborts, or an error). Whatever it does,
the scheduler treats the call as opaque.

### 2.6 Constants

| Name | Value | Notes |
|---|---|---|
| `DEBOUNCE_INTERVAL` | **1000 ms** | The window edits collapse into. Same in dev and prod. |
| `SAFETY_INTERVAL` | **60_000 ms** | The steady backstop after every cycle. Same in dev and prod. |

**No `__DEV__` override. No `isDevMode()` override.** Dev reflects
production for both constants. Verification on TestFlight uses the
same cadence as users see. (The `isDevMode()` rule from the user's
memory still applies to other dev affordances elsewhere — e.g. the
"Wipe local data" button in Settings — it does not apply to scheduler
cadence.)

### 2.7 Why edits are debounced but everything else isn't (separately)

Edits arrive in bursts (typing reps, weight, tapping "complete set" —
all within a second). The debounce makes a burst into one cycle.
Foreground / online / manual events also arrive close together
sometimes (resume from background → NetInfo "reconnected" event
within 200ms), so routing them all through the same MIN-debounce
collapses the noise without any extra logic. The user pays 1s of
latency at most; the cycle itself is cheap regardless of trigger.

The only place a 1s-latency-on-tap might matter is the "Sync now"
button. §7 addresses the UX (toast confirms the button registered;
the cycle starts within 1s).

### 2.8 No activity-specific cadence

v1 had a 10s `session-recorder` cadence
(`SYNC_SESSION_RECORDER_CADENCE_MS`). Gone. Push-on-edit (the
debounce above) already covers the workout case. Every set/rep/weight
commit flips `local_dirty = 1` and calls `requestSync()`; the cycle
fires within 1s of the last edit.

> **Audit note for the build task:** when porting the v1 scheduler,
> the existing `SYNC_SESSION_RECORDER_CADENCE_MS = 10_000` constant,
> the `SESSION_RECORDER_ROUTE_SEGMENT` constant, the
> `syncCadenceContextFromPathname()` helper, the `setContext()` /
> `getContext()` methods on the scheduler, and the path-based
> `setDefaultSyncCadenceContextFromPathname` invocations across
> `apps/mobile/app/` are deleted outright. There is one cadence, full
> stop.

For push-on-edit to work, the session recorder must persist edits to
local SQLite promptly: on every numeric-input blur, every "complete
set" tap, every set-navigation, and after ≥ 1s dwell with unsaved
changes. Each of those writes flips `local_dirty = 1` and calls
`requestSync()`. This is an audit item for the build wave, not a new
requirement (the recorder already persists at these points; v2 just
removes the polling cadence safety net).

## 3. Network state

### 3.1 API choice — NetInfo

Adopt [`@react-native-community/netinfo`][netinfo-docs]. Reasons:

- De facto network-state library across the React Native ecosystem;
  Expo SDK 54 supports it without prebuild.
- Exposes both `isConnected` (link-layer / radio-up) and
  `isInternetReachable` (HEAD-probe-derived). We use the latter.
- Emits `addEventListener` callbacks on network changes; wires into
  `setOnline(...)` directly.

We do not use `expo-network` for the foreground state surface
(pull-based query API, no change events). We do use `expo-network`
for the background-task one-shot pre-flight (§4).

### 3.2 The online edge is treated like an edit

```ts
// In the NetInfo listener:
function onNetInfoChange(state: NetInfoState) {
  const nextOnline = state.isInternetReachable === true;
  if (nextOnline === online) return;
  online = nextOnline;
  if (nextOnline) {
    requestSync();              // reconnect = treat like an edit
  } else {
    clearTimer();               // offline: cancel any pending cycle
    timerDeadline = null;
  }
}
```

Reconnect goes through the same MIN-debounce path. VPN handoff /
airplane-mode reassociation produces multiple events in quick
succession; the debounce naturally coalesces them into one cycle. No
NetInfo-layer debounce is needed.

### 3.3 Pitfalls and how we handle each

| Pitfall | Behaviour | Our handling |
|---|---|---|
| **Airplane mode → off, slow Wi-Fi reassociation.** iOS sometimes reports `isConnected=true` ~1–3s before `isInternetReachable` flips. | NetInfo emits two events. | We listen on `isInternetReachable === true` specifically, not on `isConnected === true`. The first event (link up, no internet) is ignored. |
| **Captive portal (hotel Wi-Fi).** `isConnected=true`; `isInternetReachable` may be `true` or `null`. Real HTTPS still fails. | The cycle's RPC fails with a transport error. | The cycle returns; dirty bits stay set; the next safety tick (60s later) retries. Status surface (§7) shows "Up to date as of N minutes ago" — no special "captive portal" branch. |
| **VPN handoff** (briefly type=`vpn`, then `wifi`). | Multiple connection-change events. | The 1s debounce in `requestSync()` coalesces them. Only one cycle fires. |
| **Cellular data with strict carrier limits.** | NetInfo reports `cellular`. | Treat the same as any connection. Payloads are small (see §8). |
| **`isInternetReachable === null`** (NetInfo's initial state before the first probe). | Online state is unknown for ~hundreds of ms after launch. | Treat `null` as **offline** until the first definitive probe lands. This is mildly pessimistic — the first cycle is delayed by the time-to-first-NetInfo-event, typically <500ms. The simpler alternative ("treat null as online") would have the first cycle race the NetInfo probe; the failure would still self-heal on the next safety tick, but the launch path is cleaner if we wait for a definitive answer. |

### 3.4 No engine-level setOnline(false) on transport errors

v1's pattern (`recordSyncTransportFailure` in `engine.ts` → flipping
the scheduler's `online` flag false on observed transport errors) is
dropped. **NetInfo is the only authority on online state.** A
transport error during a cycle is just an error: the cycle returns,
dirty bits stay set, the next safety tick retries.

This costs one wasted retry per 60s while a request is failing for
reasons NetInfo doesn't see (e.g. captive portal, Supabase outage,
intermittent server 500s). The retry itself is cheap — one round-trip
that fails fast. Acceptable cost for not having a second source of
truth to keep in sync with NetInfo.

[netinfo-docs]: https://github.com/react-native-netinfo/react-native-netinfo

## 4. Background sync — chosen mechanism

### 4.1 The pick

**`expo-background-task`** wrapping iOS `BGAppRefreshTask`. One task,
one identifier. The task body runs one cycle and returns.

We do **not** use:

- **`BGProcessingTask`** — designed for long overnight work; iOS
  pushes it to charging windows. We want frequent short refreshes,
  which is exactly `BGAppRefreshTask`'s purpose.
- **Silent push** — needs APNs infrastructure we don't have. Apple
  rate-limits aggressively. Two-device support is explicitly low
  priority (`plan.md` purpose #2). Can be added later for purpose #5
  (groups) without re-architecting the trigger surface.
- **`react-native-background-fetch`** — third-party, overlaps with
  what Expo provides natively. Prebuild friction with no offsetting
  win.

### 4.2 Apple's stated rules

- `BGAppRefreshTask` execution time is capped at **~30 seconds** of
  wall-clock per invocation (Apple's [BGTaskScheduler docs][apple-bgtask]).
- iOS schedules tasks **opportunistically** based on device usage,
  battery, and network conditions. Typical firings for an actively-used
  app: 1–4 per day. No guarantee.
- `minimumInterval` is a **floor, not a ceiling.** Setting 15 minutes
  means "no sooner than 15 minutes;" it does not mean "every 15
  minutes."
- Background tasks require **Settings → General → Background App
  Refresh** to be on globally and for our app. If disabled,
  `BackgroundTask.getStatusAsync()` returns `Restricted` and we get
  zero background firings. The sync status surface (§7) calls this
  out.

[apple-bgtask]: https://developer.apple.com/documentation/backgroundtasks/bgtaskscheduler

### 4.3 The critical constraint: cycle must fit in ~30s

A BG cycle has at most ~30s of wall-clock budget. We target **≤ 20s**
for the background path (10s margin). This is safe because of t2 §6.2's
"Cycle duration target" and "Per-batch commits" guarantees — a cycle
interrupted on budget makes forward progress, and the next firing
picks up. See §11 #2 / #3.

### 4.4 BG task pre-flight and body

```ts
// apps/mobile/src/sync/background-task.ts (build wave)
TaskManager.defineTask(BG_TASK_IDENTIFIER, async () => {
  // 1. One-shot network check. NetInfo's long-lived listener is not
  // running in this short-lived native context; expo-network gives
  // us a synchronous-style query.
  const netState = await Network.getNetworkStateAsync();
  if (!netState.isInternetReachable) {
    return BackgroundTaskResult.Success;   // nothing to do; preserve scheduling preference
  }

  // 2. Run one cycle directly. Bypasses requestSync()'s debounce —
  // the BG task IS the trigger; there's nothing to coalesce.
  try {
    await runSyncCycle();
    return BackgroundTaskResult.Success;
  } catch (err) {
    logSyncError(err);
    return BackgroundTaskResult.Failed;
  }
});
```

Registration happens once during app init (e.g. in the same module
that starts the foreground scheduler):

```ts
await BackgroundTask.registerTaskAsync(BG_TASK_IDENTIFIER, {
  minimumInterval: 15,   // minutes; Apple ignores values < ~15
});
```

The task body does NOT use the foreground scheduler's state. It runs
its own cycle in a stateless context. The foreground scheduler's
timer is irrelevant here — the BG task is woken by iOS, not by a JS
setTimeout.

### 4.5 Configuration (the `app.config.ts` plugin block)

The build task adds to `apps/mobile/app.config.ts`:

```ts
// In the plugins array, append:
"expo-background-task",
```

That's the entire plugin entry — no options needed. The plugin's
default identifier matches what `TaskManager.defineTask` registers.

Dependencies added to `apps/mobile/package.json` via `expo install
expo-background-task expo-task-manager @react-native-community/netinfo
expo-network` (picks SDK-compatible versions).

During prebuild (CNG; we don't commit `ios/`), Expo's plugin writes
into the generated `Info.plist`:

- `UIBackgroundModes` array gains the `processing` entry.
- `BGTaskSchedulerPermittedIdentifiers` array gains the task
  identifier.

The build wave verifies the identifier match (the same constant used
in `TaskManager.defineTask` AND
`BackgroundTask.registerTaskAsync` AND the plist entry) before merging.
Identifier mismatch is the most common failure mode for this API and
produces zero firings with no diagnostic.

## 6. In-flight on suspend / resume

### 6.1 What happens when iOS suspends mid-cycle

A cycle in flight when the app is backgrounded continues to run on
the JS thread until iOS decides to suspend (typically within 5–30
seconds of `background`, depending on which background modes are
declared). When suspension hits, the JS thread is paused mid-execution.

Three sub-cases:

1. **Suspension hits between batches.** The last committed batch's
   dirty bits are cleared (push) or the cursor is advanced (pull).
   Per-batch commits (t2 §6.2, t4 §11 #3) make this safe.
2. **Suspension hits during an HTTP request.** The request is
   abandoned by the OS. On resume, NetInfo emits no special event,
   but the foreground edge (`background → active`) calls
   `requestSync()`, which fires a cycle within 1s. That cycle reads
   the still-set dirty bits and re-pushes; the server's LWW handles
   the duplicate idempotently (t2 §3.6).
3. **Suspension hits during the page-apply transaction.** SQLite's
   transaction is rolled back when the process is paused mid-write
   (commit log not yet flushed); the cursor is unchanged. On resume,
   the next cycle re-fetches the same page. Idempotent (t2 §4.7).

In all three cases, **no separate "resume" code path is needed.** The
foreground edge in §2.4 fires `requestSync()` like any other trigger,
and the cycle reconstructs from local SQLite state (dirty bits + four
per-layer cursors + `last_emitted_ms`, per t2 §9).

### 6.2 Surviving state — concrete inventory

Per t2 §9, the persisted state surfaces are:

- Per-entity: `local_dirty`, `local_updated_at_ms` on every row.
- Singleton on `sync_runtime_state`: `pull_cursor` (JSON map of four
  per-layer cursors), `last_emitted_ms`, `bootstrap_completed_at`.

No in-memory queue, no separate outbox table (v1's
`sync_outbox_events` is deleted in the build wave), no resume token,
no `nextAttemptAt`.

The scheduler's own state (`running`, `online`, `timerDeadline`,
`timerHandle`) is in-memory and reconstructs on next launch from
NetInfo's first probe + the `AppState` initial event.

## 7. "Sync now" UX

### 7.1 Placement

A new **Sync** section in `app/(tabs)/settings.tsx`, separate from the
existing Profile sync widget on `app/profile.tsx` (the Profile widget
stays — it's a status surface, not a user-action surface).

Text-only mock:

```
Settings → Sync
┌─────────────────────────────────────────────┐
│ Sync status                                 │
│ Up to date                                  │
│ Last sync: 2 minutes ago                    │
│                                             │
│ [ Sync now ]                                │
│                                             │
│ Pending changes: 0                          │
│ Network: Wi-Fi · online                     │
│ Background sync: Available                  │
│                                             │
│ ─────────────────────────────────────────── │
│ Sync is enabled                       [ON]  │
│                                             │
│ [ Wipe local data ]   (dev only)            │
│ [ Wipe my remote data ]   (dev only)        │
└─────────────────────────────────────────────┘
```

### 7.2 Tap behaviour

- **Button disabled** if any of: not signed in, sync disabled,
  currently running a cycle, or `online === false`.
- **On tap:** call `requestSync()`. Because the button was enabled
  only when not-running and online, this either uses the existing
  scheduled deadline (if it's ≤ 1s away) or schedules a fresh
  deadline at `now + DEBOUNCE_INTERVAL`. In the worst case the user
  waits 1s before the cycle starts.
- **Feedback:** show a toast "Syncing…" immediately on tap (so the
  button feels responsive). When the next `running === false`
  transition lands, show "Sync complete" (success path) or "Sync
  failed: <message>" (cycle threw).

### 7.3 No "force" mode

In v1, "Sync now" needed `force: true` to bypass per-trigger
debounces. In v2 there is one debounce path and the button takes it.
No force mode; the worst-case wait is `DEBOUNCE_INTERVAL` (1s) and
the toast-on-tap covers the UX.

## 8. Sync status surface — what changes vs `profile-status.ts`

### 8.1 What v1 provides

`apps/mobile/src/sync/profile-status.ts:SyncProfileStatusSnapshot`
currently exposes: `isEnabled`, `kind`, `statusLabel`, `statusHint`,
`pendingCount`, `pendingCountCapped`, `lastSuccessfulSyncAt`,
`nextAttemptAt`, `errorMessage`, `retryHint`, `isOnline`.

Consumed by `app/profile.tsx` and tested in
`app/__tests__/sync-profile-status.test.ts` plus
`settings-profile-navigation.test.tsx`.

### 8.2 What v2 changes

1. **`pendingCount` source** changes from "rows in
   `sync_outbox_events`" to "count of rows where `local_dirty = 1`
   across the eight entity tables." Bounded by
   `SYNC_PROFILE_PENDING_SCAN_LIMIT` (currently 1000) for the cap
   semantics. `pendingCountCapped` is preserved.
2. **`nextAttemptAt` source** changes from the outbox's
   `next_attempt_at` to the scheduler's `timerDeadline`. The field
   is now "the next time a cycle is scheduled to start" — could be
   1s out (debounced) or 60s out (safety tick).
3. **New fields:**
   - `networkType: 'wifi' | 'cellular' | 'other' | 'none' | 'unknown'`
     — from NetInfo. Renders as "Wi-Fi · online" / "Offline".
   - `backgroundTaskStatus: 'available' | 'restricted' | 'denied' | 'unknown'`
     — from `BackgroundTask.getStatusAsync()`. Renders "Background
     sync: Available" or "Background sync: Disabled in iOS Settings".
   - `lastSyncDurationMs: number | null` — for honest power-user
     telemetry; populated by wrapping the cycle in a timer.
4. **`kind` enum extension** adds `'background_disabled'`
   (info-level, not blocking) when `backgroundTaskStatus ===
   'restricted'` and no higher-priority state is active. Suggests
   enabling Background App Refresh; not an error.
5. **`kind: 'retry_scheduled'` is dropped.** The v1 retry-scheduled
   state was driven by `outbox.calculateBackoffDelayMs`. v2 has no
   backoff state; a cycle that fails is indistinguishable in the
   status surface from a cycle that hasn't run yet. The user sees
   either "Up to date" (last cycle succeeded) or "Syncing…" (cycle
   in flight). Persistent failures show as a growing
   `lastSuccessfulSyncAt` staleness — the value the user actually
   cares about.
6. **`deliveryState.retryBlocked` path is dropped.** v1's
   "non-retryable backend response → block all sync" gate is gone
   in v2 because the only protocol-level non-retriable error is
   `FK_VIOLATION` (t2 §2.2), which is a structural bug fixed by app
   update; there is no class of normal errors we'd want to block
   on. `kind: 'action_required'` is reserved for the bootstrap-
   failed case (t2 §5 follow-up) and for `'background_disabled'`.

### 8.3 What stays the same

`deriveSyncProfileStatus` keeps its pure-function shape; new fields
slot into existing branches without rewriting decision logic. The
Profile widget and Settings widget can share one
`SyncProfileStatusSnapshot` source.

### 8.4 What gets deleted

`apps/mobile/src/sync/outbox.ts:getSyncDeliveryState` and its
`SyncDeliveryStateSnapshot` shape come out entirely when the outbox
table is dropped in the v2 build wave. The remaining "delivery"
information (last success, last error, pending count) moves to a
thinner `sync_runtime_state`-derived snapshot.

## 9. Battery / data tradeoffs

Honest quantification at the v2 default cadence.

### 9.1 Idle foreground (user has the app open but isn't editing)

- 1 cycle per `SAFETY_INTERVAL` = 60 cycles/hour.
- Per cycle, idle: 4 `sync_pull` RPCs (one empty page per layer),
  zero pushes. Each empty pull is ~300 bytes request + ~150 bytes
  response over TLS.
- ~60 cycles × ~2 KB ≈ **~120 KB/hour** of idle network traffic.
- See t2 §4.4's open follow-up about collapsing the 4 idle-layer
  probes into one — when that lands, idle drops to ~30 KB/hour.

### 9.2 Active editing (mid-workout)

- Edits arrive in bursts; the debounce coalesces them.
  Conservatively: 1 set logged ≈ 1 cycle (the 1s of typing collapses
  to one debounce window).
- A typical 1-hour workout has ~40 sets logged. **~40 cycles** in the
  hour.
- Per push cycle, a fresh set adds one row to `exercise_sets`
  (~200 bytes payload). The 4 pull probes still happen.
- ~40 cycles × (~2 KB pull + ~300 bytes push) ≈ **~100 KB/hour** of
  edit-driven traffic.

### 9.3 Background

- 1–4 `BGAppRefreshTask` firings per day for an actively-used app
  (Apple's typical figure; not a guarantee).
- Each firing is one cycle. Idle cost: ~2 KB. With dirty rows:
  ~2 KB + (payload).
- **Negligible** compared to foreground use.

### 9.4 Battery

The dominant cost is the radio waking for HTTPS requests, which
batches with the OS's general network activity (other apps' polling,
push, etc.) — iOS coalesces these. The marginal battery cost of our
~60 idle cycles/hour is small; the cellular-radio wake cost is the
real number and it amortises across the device's existing background
activity.

The 4-empty-pull idle waste (§9.1) is the only obviously-reducible
cost, tracked as a t2 follow-up. Until then, this is the v2 default.

### 9.5 Where this could go wrong

The numbers above assume "idle" means "no remote changes." A
two-device scenario where Device A is actively editing while Device
B is foregrounded but idle would have Device B's idle cycles
returning non-empty pull pages. With small payloads (typical
workout edits are <1 KB per row) the traffic stays within the same
order of magnitude. With pathological churn (one device looping
edits on hundreds of rows), Device B's pulls would amplify
accordingly — outside v2's scope (purpose #2 is explicitly low
priority).

## 11. t2 constraints emitted by t4

The consolidated index of every constraint t4's design places on t2.
These are scattered through the doc as the requirements t4 needs the
cycle to satisfy.

### 11.1 Cycle shape

1. **Cycle drains until empty.** A cycle continues issuing PULLs
   until the server has no new rows beyond the cursor (per-layer),
   and PUSHes until no local rows are dirty. Convergence to
   "nothing left on either side" is the cycle's job; the exact
   PULL/PUSH interleave is t2's choice (t2 §6.1). (§2.5)
2. **Cycle duration target.** Background cycles complete or cleanly
   abort with progress within **20 seconds** of wall-clock (10s
   margin under Apple's ~30s BG cap). Foreground cycles target
   ≤ 5 seconds for responsiveness (soft, not hard). (§4.3, §6.1)

### 11.2 Per-batch atomicity

3. **Batched independent commits.** A cycle internally operates in
   batches (push batches, pull pages) that commit independently on
   both server and client. No cycle-wide transaction; no
   all-or-nothing per cycle. Interruption mid-cycle (budget abort,
   suspend, network drop) leaves the database consistent through
   the last committed batch. The next cycle picks up from the next
   un-committed batch. (§4.3, §6.1)

### 11.3 Idempotency under interruption

4. **Push is idempotent.** Re-pushing the same row tuple
   `(owner_user_id, entity_type, id, client_updated_at_ms,
   payload)` twice yields the same final server state and clears
   the dirty bit either way. A retry of a previously-acked row
   succeeds (no error). (§6.1 #2, t2 §3.6)
5. **Pull is idempotent by cursor.** The cursor advances only when
   a page is fully applied to local SQLite. An interrupted apply
   leaves the cursor at its pre-page value; the next cycle
   re-fetches the same page. (§6.1 #3, t2 §4.7)

### 11.4 Surviving state

6. **Only dirty bits, the four per-layer pull cursors, and
   `last_emitted_ms` survive suspension.** Per t2 §9, these are
   the local SQLite surfaces. Any additional state t2 introduces
   must be either (a) reconstructible from these on the next cycle,
   or (b) persisted in SQLite with equivalent durability. No
   in-memory queue, no separate outbox persistence, no resume
   token, no `nextAttemptAt`. (§6.2)

### 11.5 Out of t4's scope but t2 owns

- The exact JSON shapes of `sync_push` and `sync_pull` requests
  and responses (t2 §3, §4).
- The PULL/PUSH ordering within a cycle (t2 §6.1).
- Push batching policy — topological order, FK closure, per-batch
  caps (t2 §3.4).
- The dirty-bit lifecycle including the push-in-flight race (t2 §7).
- Tombstone semantics (`deleted_at` only, no separate flag — t1 §1.1,
  t2 §2.1).
- The four per-layer pull-cursor semantics (t2 §4.4, §9.2).
- The clock-monotonicity guard (t2 §8).
