# t1 — Sync v2: Server schema & drift control

> Design doc only. Plan ref: `docs/plans/sync-v2/plan.md` § "t1: Server schema & drift control".
> Brainstorm: `docs/plans/sync-v2/brainstorm.md`.

## 0. Scope and shape of this doc

This doc specifies, in order:

1. The eight new `app_public.<entity>` tables — column-by-column mapping from the client
   Drizzle schemas to Postgres, including defaults and indexes.
2. The `extras jsonb` column convention.
3. The two new local columns added to every client entity table (`local_dirty`,
   `local_updated_at_ms`) and the local-schema additions for entities that currently
   lack `deletedAt`.
4. Deferrable foreign-key declarations and what `INITIALLY DEFERRED` buys the push RPC.
5. RLS policy text for every table.
6. The drift-detection mechanism — one approach, fully specified, with command, failure
   output, fix flow, and quality-gate wiring.
7. The agent-reminder mechanism — which file to edit and the exact wording.
8. A worked example end-to-end: developer adds `notes text` to `exercise_sets`.

It does NOT write the migration, the drift-check code, or edit `AGENTS.md`. Those land
in the t1 build wave that consumes this doc.

## 1. Ground rules

These are the invariants every section below honours.

- **Server schema mirrors client.** One `app_public.<entity>` table per client Drizzle
  table for the eight user-owned entities. No projection function. No event log. No
  per-row PL/pgSQL dispatch.
- **PK shape:** every server table uses composite `(owner_user_id, id)`. The current
  `20260514120000_user_scoped_pk_redesign.sql` already does this, but we drop those
  tables (cascade) and recreate from scratch — we keep the *idea* of the composite PK
  and discard the projection plumbing around it. Column-ordering switches to
  `(owner_user_id, id)` for clarity ("owner first") even though the older migration
  used `(id, owner_user_id)`; PK columns are unordered in semantics, the change is
  cosmetic and makes the canonical pull query (`where owner_user_id = … order by
  server_received_at`) literally use the leading column.
- **Composite FKs only.** Every cross-entity FK references the parent's composite PK
  `(owner_user_id, <parent_id>)`. There are no cross-owner references by construction.
- **All FKs are `DEFERRABLE INITIALLY DEFERRED`.** Push transactions can write a child
  before its parent without ordering hazards; constraints are checked at COMMIT.
- **Timestamps are `bigint` (epoch milliseconds), not `timestamptz`.** Mirrors the
  client. The only `timestamptz` on these tables is `server_received_at` (server-set,
  used as the pull cursor — see §4).
- **No `origin_scope_id` / `origin_source_id`.** v1's catalog-distinction columns are
  dropped; every row is per-user from day one.
- **The `set_type` CHECK constraint on `exercise_sets` is dropped server-side.**
  Validating enum text in two places is exactly the drift hazard we are designing
  away. The client validates; the server stores. (Same applies to `sessions.status`
  and `exercise_muscle_mappings.role`.) The server only enforces what's necessary for
  *storage* integrity: FKs, non-null, PK uniqueness. Application-level invariants
  live on the client.
- **`extras jsonb not null default '{}'::jsonb`** on every entity table. This is the
  forward-compat hatch — see §3.
- **RLS:** `owner_user_id = auth.uid()` for select/insert/update/delete. Identical
  shape for every table. See §6.
- **No `deleted_at` for the join tables** (`exercise_muscle_mappings`,
  `session_exercise_tags`). Membership is the data; "delete" is a row delete.
- **Hard cut from v1.** The migration drops every v1 server object
  (`sync_apply_projection_event`, `sync_events_ingest`, `sync_events_ingest_impl`,
  `sync_device_ingest_state`, `sync_ingested_events`, all eight legacy entity tables,
  the in-flight `user_scoped_pk_redesign` shape) and creates the v2 shape clean. Per
  plan.md, the hosted DB is wiped post-merge.

## 2. Per-entity mapping tables

Notation:

- "Client type" is the Drizzle column type as it appears in `apps/mobile/src/data/schema/<entity>.ts`.
- "Server type" is the Postgres type for the corresponding `app_public.<entity>` column.
- "Null" — `NO` means `not null`; `YES` means nullable.
- "Default" — server-side default. Empty cell means no default.
- "Indexed" — yes if a standalone btree index is created on that column. PK columns
  are not listed as "indexed" (the PK already covers them). Composite unique
  indexes / FK-supporting indexes are called out under "Other constraints / indexes".
- "FK" — the composite FK target if any. All FKs are `DEFERRABLE INITIALLY DEFERRED`.

Every table also gets these **identical** columns added on the server (not in any
client schema):

| Server column | Type | Null | Default | Purpose |
| --- | --- | --- | --- | --- |
| `owner_user_id` | `uuid` | NO | `auth.uid()` | Row owner. Part of PK. References `auth.users(id) on delete cascade`. |
| `extras` | `jsonb` | NO | `'{}'::jsonb` | Forward-compat blob; see §3. |
| `client_updated_at_ms` | `bigint` | NO | — | Client wall-clock at write, used for LWW. CHECK `>= 0`. |
| `deleted` | `boolean` | NO | `false` | Tombstone flag. Stays `true` after a delete; LWW still applies. |
| `server_received_at` | `timestamptz` | NO | `now()` | Server-set on every insert/update; the pull cursor's monotonic axis. |

Every table also gets these **identical** server indexes:

- `<table>_owner_received_idx` on `(owner_user_id, server_received_at)` — supports the
  pull cursor query `where owner_user_id = $1 and server_received_at > $2 order by
  server_received_at, id`. The PK trailing-key tiebreak (`id`) is included for stable
  pagination at the millisecond boundary.

Every table also gets this **identical** trigger:

- `<table>_touch_server_received_at` — `BEFORE UPDATE` setting
  `NEW.server_received_at = now()`. Insert side is covered by the column default.

These five columns, the `<table>_owner_received_idx` index, and the `touch` trigger
are not repeated in each table's mapping table below; they are part of the implicit
contract for every entity.

  per session (out of scope here; closed in t2). The server side only needs to
  document that the partial-unique constraint exists.

## 3. The `extras jsonb` convention

### 3.1 Purpose

Add a non-null `extras jsonb default '{}'::jsonb` column to every entity table. It
catches client fields that haven't been promoted to typed server columns yet. The
goal is to let small client additions (a single boolean, a stat, a marker) ship
without a backend migration on every PR — at the cost of those fields not being
queryable from SQL until they're promoted.

### 3.2 Lifecycle

Three states a client column can be in:

1. **Typed promoted** — column exists on the server with its own type and indexes
   (the default state for the eight entities' baseline columns; the mapping tables
   in §2 are the snapshot).
2. **Extras-only** — column exists on the client only. Drift check is silenced by an
   allowlist entry (see §7). Server sees the field through the `extras` blob; SQL
   on it requires `extras ->> 'field_name'` with appropriate casting.
3. **Promoted out of extras** — a later migration adds the typed column, the
   sync push code starts writing it as a typed column, and the allowlist entry is
   removed. Existing rows have the field in `extras` only; reading code should
   prefer `coalesce(typed_col, (extras ->> 'field_name')::<type>)` until a
   backfill migration moves data.

### 3.3 Allowlist file

The drift checker (see §7) reads a single source-of-truth file:

```
apps/mobile/src/data/schema/sync-extras.json
```

Shape:

```jsonc
{
  // "<entity>" is the snake_case client/server table name.
  "exercise_sets": {
    "extras_only_columns": [
      // each entry: { "column": "<client-camel>", "wire_key": "<snake_case>", "added": "YYYY-MM-DD" }
    ]
  },
  "exemptions": {
    // Synthetic / sync-internal columns that don't need a server counterpart.
    // Specifically: local_dirty, local_updated_at_ms, and the cross-cutting muscle_groups
    // taxonomy reference. These are listed once at top-level, not per-entity, because
    // they apply uniformly.
    "local_only_columns": ["local_dirty", "local_updated_at_ms"],
    "untyped_text_references": [
      { "entity": "exercise_muscle_mappings", "column": "muscleGroupId", "rationale": "muscle_groups is client-only taxonomy; see designs/t1.md §2.6.1" }
    ]
  }
}
```

### 3.4 Wire shape with extras

Push payload for one entity row (full row, see t2 for the envelope):

```json
{
  "type": "exercise_sets",
  "id": "01HX…",
  "owner_user_id": "<auth.uid()>",   // server validates against jwt; client may omit
  "client_updated_at_ms": 1748000000000,
  "deleted": false,
  "fields": {
    "session_exercise_id": "01HX…",
    "order_index": 2,
    "weight_value": "60",
    "reps_value": "10",
    "set_type": "rir_1",
    "created_at": 1747999900000,
    "updated_at": 1748000000000,
    "deleted_at": null
  },
  "extras": {
    "rpe": 8.5
  }
}
```

Server-side handling (the t2 push RPC, sketched here so the contract is clear):

1. For each typed column under `fields`, write `column = $value`.
2. Write the `extras` object verbatim into the `extras` column. Server does NOT
   merge with the existing extras — the client always sends the full extras blob.
   This is a deliberate constraint that makes "delete a field from extras" possible
   in a future migration without server-side merge logic.
3. The drift check (§7) guarantees that any field-named key in `extras` is registered
   as extras-only for that entity. The server does not enforce this; the gate fails
   the PR upstream.

### 3.5 Why this beats two alternatives

- **Opaque-blob server (the brainstorm's §3.1)** loses SQL queryability. Stats / MCP
  / groups don't get easier later.
- **No `extras` column, only typed columns** means every client addition needs a
  paired server migration. Friction is too high; we will skip migrations and ship
  drift.

The `extras` column is the compromise: small adds are quick (allowlist entry only),
queries are still SQL (`extras ->> 'rpe'` works in stats), and promotion later is a
single migration with an optional backfill.

## 4. Local schema additions

### 4.1 The two new sync-bookkeeping columns

Every one of the eight client entity tables gets two new columns. They are
**local-only** — they are listed under `exemptions.local_only_columns` in
`sync-extras.json` so the drift checker doesn't flag them.

| Column | Type | Default | Purpose |
| --- | --- | --- | --- |
| `local_dirty` | `integer` (0/1 boolean, SQLite convention) | `1` for new rows; `0` after a successful push acks the row | Set to 1 by every repo write path. Cleared to 0 by the push RPC's success callback for the rows the server confirmed. The clear path runs *only* if `local_updated_at_ms` hasn't moved since the push started; if it has, the row is still dirty and a follow-up push will run. |
| `local_updated_at_ms` | `integer` (epoch ms, bigint range) | `unixepoch() * 1000` at insert; reassigned via `max(Date.now(), last_emitted_for_row + 1)` at every write | The LWW clock for this row, monotonically non-decreasing. Distinct from `updated_at`: `updated_at` is the domain-meaningful "the user changed this" timestamp, `local_updated_at_ms` is the sync-engine clock used to detect "I have a fresh write the server hasn't seen". For most writes they're equal; for clock-skew correction they diverge. |

These columns are not the only state needed to drive the push (the sync runtime
also persists a per-batch in-flight set), but they are the *durable* state that
survives app restart.

> The monotonic-clock formula `max(Date.now(), last_emitted_for_row + 1)` lives in
> the sync runtime (t2 design closes the implementation detail). For the schema, all
> that matters is: the column is `bigint`-typed, monotone in practice, and the
> client never persists a value lower than the previous one.

### 4.2 New `deleted_at` columns

## 5. Deferrable foreign keys

### 5.1 What `INITIALLY DEFERRED` means in practice

Postgres checks every FK at the end of the statement by default. With
`DEFERRABLE INITIALLY DEFERRED`, the check is postponed to COMMIT time. Inside one
transaction you can:

- Insert an `exercise_sets` row that references a `session_exercises.id` whose own
  insert is the very next statement in the same transaction.
- Update a `session_exercises.session_id` and the matching `sessions.id` in either
  order without an intermediate violation.

At COMMIT all FKs are re-checked. If any are unsatisfied the entire transaction
rolls back. From the client's view, the push RPC is atomic per batch — either every
row landed or none did.

### 5.2 The four deferrable FKs

| Constraint | From | To | On delete | Notes |
| --- | --- | --- | --- | --- |
| `sessions_gym_fk` | `sessions(owner_user_id, gym_id)` | `gyms(owner_user_id, id)` | `set null` | Gym may be deleted while a historical session refers to it. |
| `session_exercises_session_fk` | `session_exercises(owner_user_id, session_id)` | `sessions(owner_user_id, id)` | `cascade` | Deleting a session removes its exercises (server side; client soft-deletes). |
| `session_exercises_exercise_definition_fk` | `session_exercises(owner_user_id, exercise_definition_id)` | `exercise_definitions(owner_user_id, id)` | `no action` | Soft-delete on `exercise_definitions` keeps the row addressable so session history reads still resolve. The FK is required (not stripped per v1's relax migration) because deferral solves the original ordering problem. |
| `exercise_sets_session_exercise_fk` | `exercise_sets(owner_user_id, session_exercise_id)` | `session_exercises(owner_user_id, id)` | `cascade` | Same rationale as `session_exercises_session_fk`. |
| `exercise_muscle_mappings_exercise_definition_fk` | `exercise_muscle_mappings(owner_user_id, exercise_definition_id)` | `exercise_definitions(owner_user_id, id)` | `cascade` | Mapping membership is meaningless without its definition. |
| `exercise_tag_definitions_exercise_definition_fk` | `exercise_tag_definitions(owner_user_id, exercise_definition_id)` | `exercise_definitions(owner_user_id, id)` | `cascade` | Same. |
| `session_exercise_tags_session_exercise_fk` | `session_exercise_tags(owner_user_id, session_exercise_id)` | `session_exercises(owner_user_id, id)` | `cascade` | Same. |
| `session_exercise_tags_exercise_tag_definition_fk` | `session_exercise_tags(owner_user_id, exercise_tag_definition_id)` | `exercise_tag_definitions(owner_user_id, id)` | `cascade` | Same. |

That's eight named FKs (counting the `sessions.gym_id` FK plus seven child→parent).
All declared `DEFERRABLE INITIALLY DEFERRED`. Migration template:

```sql
constraint <name>
  foreign key (owner_user_id, <child_col>)
  references app_public.<parent>(owner_user_id, id)
  on delete <action>
  deferrable initially deferred
```

### 5.3 What happens when a child arrives with no parent

If the client pushes an `exercise_sets` row whose parent `session_exercises` is also
in the same batch: deferral covers it.

If the client pushes an `exercise_sets` row whose parent is NOT in the same batch
(server doesn't have it yet — e.g., a partial batch from an offline device that
restarts mid-push): the COMMIT fails. The push RPC returns an error and the whole
batch is rejected. The client retries with the missing parent included; the dirty
bits on the failing batch stay set.

Note for t2: this is why the push protocol must include the full closure of any
dirty descendant relative to a clean ancestor — or accept that orphan child writes
will fail the batch. This design doc records the constraint; t2 decides the
strategy.

## 6. RLS policies

### 6.1 Universal shape

Identical for all eight tables. Substitute `<table>` per entity.

```sql
alter table app_public.<table> enable row level security;

drop policy if exists <table>_owner_select on app_public.<table>;
create policy <table>_owner_select on app_public.<table>
  for select to authenticated
  using (owner_user_id = auth.uid());

drop policy if exists <table>_owner_insert on app_public.<table>;
create policy <table>_owner_insert on app_public.<table>
  for insert to authenticated
  with check (owner_user_id = auth.uid());

drop policy if exists <table>_owner_update on app_public.<table>;
create policy <table>_owner_update on app_public.<table>
  for update to authenticated
  using (owner_user_id = auth.uid())
  with check (owner_user_id = auth.uid());

drop policy if exists <table>_owner_delete on app_public.<table>;
create policy <table>_owner_delete on app_public.<table>
  for delete to authenticated
  using (owner_user_id = auth.uid());
```

### 6.2 Why no service-role policies are needed

The sync push/pull RPCs run as `security definer` functions (defined in t2) that
own the data writes; the `authenticated` policies above are still applied because
the RPC sets `search_path` but does not bypass RLS. Direct client access to the
tables uses the `authenticated` role and goes through the same policies. The
`service_role` grants exist for ops scripts only (wipe-remote, support tools); RLS
is bypassed for `service_role` by Postgres convention.

### 6.3 Immutability of `owner_user_id`

Same as v1: a `BEFORE UPDATE` trigger on every table that raises if
`new.owner_user_id <> old.owner_user_id`. The existing
`app_public.enforce_owner_user_id_immutable()` function from v1 can be reused
verbatim — the trigger names follow the convention `<table>_owner_user_id_immutable`.

## 7. Drift-detection mechanism

### 7.1 Choice

**A Node script that introspects the Drizzle schema modules at runtime and diffs
columns against parsed `CREATE TABLE` blocks from the v2 migration tree.** It
fails on any unaccounted-for client column.

Location: `apps/mobile/scripts/check-sync-schema-drift.ts`.

Command: `npm run check:sync-drift` (added to `apps/mobile/package.json`).

Wired into `scripts/quality-slow.sh backend` as a new step (justification in §7.5).

### 7.2 Why this approach (and not the alternatives)

Picking a single tool, not surveying alternatives, per the brief. The justification:

- **Drizzle introspection over hand-maintained allowlists** — the script can `require`
  the schema modules directly and walk `table[Symbol.for('drizzle:Columns')]`. We
  already use this Symbol elsewhere in the codebase (sync engine type derivation).
  Hand-maintaining a list of "what columns the client has" duplicates the schema
  and breaks the exact moment we need it to work.
- **Parse migration SQL, don't run a live database** — drift is a static property of
  the source tree, runnable in CI without spinning up Postgres. The script
  concatenates every `supabase/migrations/*.sql` newer than the v2 baseline marker,
  feeds the text through a minimal `CREATE TABLE app_public.<x> (…)` parser
  scoped to the eight entity tables, and resolves each column's name and type.
  Subsequent `ALTER TABLE … ADD COLUMN` statements are applied in order to build the
  final shape.
- **One source of truth for exemptions: `sync-extras.json`** (see §3.3) — the
  drift checker, the push RPC's payload validator, and the documentation all refer
  to one file. No state lives in three places.

### 7.3 Algorithm

```
for each of the 8 entity table modules in apps/mobile/src/data/schema/:
  let clientCols = drizzle introspection of the table's column metadata
  let serverCols = parse v2 migrations under supabase/migrations/v2/ → final shape for app_public.<entity>
  let extras = sync-extras.json → entity entry
  let exemptions = sync-extras.json → top-level exemptions

  for each clientCol in clientCols:
    if clientCol.name is in exemptions.local_only_columns:
      continue   // local_dirty, local_updated_at_ms
    if clientCol matches a typed serverCol (name + base type compatible):
      continue
    if clientCol is in entity's extras.extras_only_columns:
      continue
    if clientCol is in exemptions.untyped_text_references:
      continue
    FAIL: report "<entity>.<column>" has no server counterpart

  for each serverCol in serverCols:
    if serverCol.name is in {extras, client_updated_at_ms, server_received_at, deleted, owner_user_id}:
      continue
    if serverCol matches a clientCol (by name):
      continue
    WARN: server has a column the client doesn't know about (probably stale, but warn-only)
```

Type compatibility map (intentionally narrow):

| Drizzle SQLite type | Acceptable server type |
| --- | --- |
| `text` | `text` |
| `integer` with `mode: 'timestamp_ms'` | `bigint` |
| `integer` (default, no mode) | `integer` *or* `bigint` |
| `real` | `double precision` *or* `numeric` |

Nullability and defaults are not checked; the goal is "field exists in both
places", not full structural equality. A future iteration can tighten this.

### 7.4 Failure output

Exactly one of three exit codes:

- `0`: no drift, no warnings.
- `1`: drift detected. Stdout lists, per entity:
  ```
  ✗ exercise_sets: client column "notes" (text) has no server counterpart and is not
    registered as extras-only.

    To fix in under 5 minutes, pick one:

    (a) Promote to a typed server column. Add a migration that does:

        alter table app_public.exercise_sets
          add column notes text;

        Then re-run `npm run check:sync-drift`.

    (b) Keep client-only via extras. Append to apps/mobile/src/data/schema/sync-extras.json:

        "exercise_sets": {
          "extras_only_columns": [
            { "column": "notes", "wire_key": "notes", "added": "2026-05-23" }
          ]
        }

        Then re-run `npm run check:sync-drift`.
  ```
- `2`: server has columns absent on the client. Warn-only; exits 0 by default,
  exits 2 only when called with `--strict`. (Slow-gate posture uses `--strict`.)

### 7.5 Wiring into `scripts/quality-slow.sh backend`

Justification for slow-gate placement (not fast):

- The script imports Drizzle modules and parses SQL — first run cold takes
  ~3-5 seconds on a clean Node install. Tolerable in `quality-slow.sh` (which
  already runs full Maestro suites measured in minutes); a measurable but not
  large addition to `quality-fast.sh`. **However**, drift is a backend-shaped
  concern (a schema mismatch breaks sync, not the unit-tested client), and the
  backend slow gate is the natural home: any PR that asserts "`quality-slow.sh
  backend` passed" is also asserting "no schema drift between client and v2
  server migrations". This matches the existing `test-sync-api-contract.sh`
  position.
- The script is also exposed as `npm run check:sync-drift` in `apps/mobile`, so
  PRs touching schema files can run it directly without invoking the slow gate.

Concrete addition to `scripts/quality-slow.sh`'s `run_backend()`:

```bash
if [[ ! -f "${REPO_ROOT}/apps/mobile/scripts/check-sync-schema-drift.ts" ]]; then
  echo "[quality-slow] skipping backend sync-schema-drift: script not found"
else
  echo "[quality-slow] backend: sync-schema-drift"
  (cd "${REPO_ROOT}/apps/mobile" && npm run check:sync-drift -- --strict)
fi
```

> Editing `quality-slow.sh` is out of scope for this design doc; the wiring above
> is the spec the t1 build wave implements.

### 8.1 Which file to edit

### 8.2 Exact wording

```markdown
## Client schema drift rule (Sync v2)

Modifying any file under `apps/mobile/src/data/schema/` for the eight user-owned

`exercise_definitions`, `exercise_muscle_mappings`, `exercise_tag_definitions`,
```

## 9. Worked example — developer adds `notes text` to `exercise_sets`

### 9.1 Scenario

A developer wants to let users type a free-text note per set ("felt easy", "tight
shoulder"). They expect this to take one PR and not need backend involvement
immediately — they want the field visible in the UI today, queryable in stats
later.

### 9.2 Step-by-step

**1. Edit the client schema.**

`apps/mobile/src/data/schema/exercise-sets.ts`:

```ts
export const exerciseSets = sqliteTable(
  'exercise_sets',
  {
    // …existing columns…
    notes: text('notes'),
    // …
  },
  // …
);
```

**2. Run the drift check locally.**

```
cd apps/mobile && npm run check:sync-drift
```

Output:

```
✗ exercise_sets: client column "notes" (text) has no server counterpart and is not
  registered as extras-only.
  To fix in under 5 minutes, pick one:
  …
```

**3. Pick the right path.**

The developer asks: *will stats queries need this field in the next ~month?*

- If **no** (just UI + history): take path (b), extras-only. Faster, no backend PR.
- If **yes**: take path (a), promote.

For this worked example, the developer picks (b).

**4. Register the field as extras-only.**

`apps/mobile/src/data/schema/sync-extras.json`:

```jsonc
{
  "exercise_sets": {
    "extras_only_columns": [
      { "column": "notes", "wire_key": "notes", "added": "2026-05-23" }
    ]
  },
  "exemptions": {
    "local_only_columns": ["local_dirty", "local_updated_at_ms"],
    "untyped_text_references": [
      { "entity": "exercise_muscle_mappings", "column": "muscleGroupId", "rationale": "muscle_groups is client-only taxonomy; see designs/t1.md §2.6.1" }
    ]
  }
}
```

**5. Update the local Drizzle migration.**

Generate the SQLite migration with `drizzle-kit generate` per existing convention
(picks up the new column).

**6. Re-run the drift check.**

```
npm run check:sync-drift
```

Exit code 0, no output beyond a success line. The PR can proceed.

**7. Sync engine handles the new field automatically.**

The push code (t2) serialises every column on a dirty row. The fields that match
typed server columns go under `fields`; the fields registered as extras-only go
into `extras`. Wire payload for a set with a note:

```json
{
  "type": "exercise_sets",
  "id": "01HX…",
  "client_updated_at_ms": 1748000000000,
  "deleted": false,
  "fields": {
    "session_exercise_id": "01HX…",
    "order_index": 0,
    "weight_value": "60",
    "reps_value": "10",
    "set_type": "rir_1",
    "created_at": 1748000000000,
    "updated_at": 1748000000000,
    "deleted_at": null
  },
  "extras": {
    "notes": "felt easy"
  }
}
```

Server stores the row, `extras = '{"notes":"felt easy"}'::jsonb`. On the next pull
the client gets the same shape back and Drizzle reads `notes` from the joined
projection (the pull RPC merges `fields` and `extras` before returning).

**8. (Optional, weeks later) Promote `notes` to a typed column.**

When stats wants to query `notes`:

- New migration `supabase/migrations/<ts>_promote_exercise_sets_notes.sql`:

  ```sql
  alter table app_public.exercise_sets
    add column notes text;

  update app_public.exercise_sets
    set notes = extras ->> 'notes'
    where extras ? 'notes';

  -- Optionally strip extras key:
  update app_public.exercise_sets
    set extras = extras - 'notes'
    where extras ? 'notes';
  ```

- Remove the `extras_only_columns` entry from `sync-extras.json`.

- Push code now writes `notes` under `fields` (automatically, because the drift
  check passes once the column is typed; the push code's "which bucket?" decision
  is "is this column in the server schema?").

- Drift check passes (`notes` is matched on both sides).

### 9.3 What the developer never has to do

- Edit a 1500-line projection function.
- Wire a new outbox event type.
- Update three sync API contract documents.
- Coordinate the change across two PRs in two repos.

That's the win.

## 10. Migration outline (informative, not authoritative)

A reader of this doc could write the v2 migration in one file. The shape:

1. Drop v1 server objects:
   - `drop function if exists app_public.sync_apply_projection_event(uuid, text, text, text, bigint, jsonb);`
   - `drop function if exists app_public.sync_events_ingest(...)`, `…_impl(...)` (look up signatures from M13 migration).
   - `drop table if exists app_public.sync_device_ingest_state cascade;`
   - `drop table if exists app_public.sync_ingested_events cascade;`
   - `drop table if exists app_public.session_exercise_tags cascade;` (and the other 7, in child→parent order even though `cascade` makes the order irrelevant).
2. Create the eight v2 tables per §2 (parent→child order so even if a future reader removes `if exists` they don't trip on FKs; deferrable FKs make this aesthetic only).
3. Add the universal index `<table>_owner_received_idx` and the `<table>_touch_server_received_at` trigger to each.
4. Add the owner-immutability trigger to each.
5. Apply RLS policies per §6.
6. Grant `select, insert, update, delete` to `authenticated` and `service_role` per table.

The migration is a single SQL file; the build task that writes it owns the
authoritative version.
