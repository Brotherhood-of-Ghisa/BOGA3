## 1. Ground rules

These are the invariants every section below honours.

- **Server schema mirrors client.** One `app_public.<entity>` table per client Drizzle
  table for the eight user-owned entities. No projection function. No event log. No
  per-row PL/pgSQL dispatch.
- **PK shape:** every server table uses composite `(owner_user_id, id)`. The current
  `20260514120000_user_scoped_pk_redesign.sql` already does this, but we drop those
  tables (cascade) and recreate from scratch — we keep the *idea* of the composite PK
  and discard the projection plumbing around it. Column-ordering switches to
  `(owner_user_id, id)` for clarity ("owner first") even though the older migration
  used `(id, owner_user_id)`; PK columns are unordered in semantics, the change is
  cosmetic and makes the canonical pull query (`where owner_user_id = … order by
  server_received_at`) literally use the leading column.
- **Composite FKs only.** Every cross-entity FK references the parent's composite PK
  `(owner_user_id, <parent_id>)`. There are no cross-owner references by construction.
- **All FKs are `DEFERRABLE INITIALLY DEFERRED`.** Push transactions can write a child
  before its parent without ordering hazards; constraints are checked at COMMIT.
- **Timestamps are `bigint` (epoch milliseconds), not `timestamptz`.** Mirrors the
  client. The only `timestamptz` on these tables is `server_received_at` (server-set,
  used as the pull cursor — see §4).
- **No `origin_scope_id` / `origin_source_id`.** v1's catalog-distinction columns are
  dropped; every row is per-user from day one.
- **No server-side validation.** The client is the only writer in v2 (all writes go
  through the push RPC, which is a thin upsert; there are no server-side triggers
  that mutate data, no admin UI, no service-role data entry tools writing to these
  tables). Validation on the server therefore *only* exists to catch a client bug,
  which is the same thing the client itself would catch. The general rule: **if the
  server is not a write source, the server does not validate.** Concretely this
  means:
    - **No CHECK constraints on enum text** (`sessions.status`,
      `exercise_sets.set_type`, `exercise_muscle_mappings.role`). v1's
      `('draft', 'active', 'completed')` CHECK on `sessions.status` is dropped — the
      v1 client schema actually only emitted `('active', 'completed')`, so the
      mismatch itself was an example of the drift-hazard this design closes.
    - **No CHECK constraints on numeric ranges** (`order_index >= 0`,
      `weight > 0`, `duration_sec >= 0`, `*_at >= 0`, etc.). The client guarantees
      these; the server stores whatever it's handed.
    - **No CHECK constraints on text content** (`length(btrim(name)) > 0`).
    - **No NOT NULL beyond what the wire envelope requires.** Domain columns are
      nullable if the client ever writes NULL on a legitimate write path; the wire
      envelope columns (`owner_user_id`, `client_updated_at_ms`,
      `server_received_at`) and the PK columns remain NOT NULL because
      they're structural, not validations.
    - **No triggers that validate row content.** The only triggers that exist are
      structural: `server_received_at` touch on UPDATE (needed because the column
      is a server-set cursor axis, not validation), and `owner_user_id`
      immutability (RLS-adjacent — see §6.3 for why this is structural, not
      validation).
  What the server *does* enforce — and these are structural, not validation:
    - **Primary key uniqueness** (`(owner_user_id, id)`). PKs are structural
      identity; without them an upsert has no anchor.
    - **Foreign keys** (the eight composite FKs in §5). FKs are referential
      integrity, not content validation.
    - **Row-Level Security policies** (§6). These are an authorisation boundary,
      not validation.
  **No other uniqueness constraints.** Earlier drafts of this design proposed
  partial-unique indexes on `(session_id, order_index)` /
  `(session_exercise_id, order_index)` and pair-unique constraints on the join
  tables. They are dropped: each one was the server enforcing an invariant that
  only the client can violate, which is double validation by the same rule
  above. The corresponding indexes still exist as **non-unique btree indexes**
  for query performance (see each entity's §2 mapping), and the invariants are
  emitted as t2 constraints (see §11).
  This rule is reread before every CHECK / NOT NULL / UNIQUE / trigger written
  into the v2 migration. The default answer is "no".
- **RLS:** `owner_user_id = auth.uid()` for select/insert/update/delete. Identical
  shape for every table. See §6.
- **Tombstones preserve the full row, undelete is lossless.** A delete sets
  `deleted_at = Date.now()` but does not clear any other column; LWW compares
  `client_updated_at_ms` and a newer write with `deleted_at = null` reconstructs
  the row exactly as it was. There is **no separate `deleted boolean` column** —
  `deleted_at IS NOT NULL` is the only tombstone marker, present on every entity
  table (including the join tables). See §1.1.
- **Hard cut from v1.** The migration drops every v1 server object
  (`sync_apply_projection_event`, `sync_events_ingest`, `sync_events_ingest_impl`,
  `sync_device_ingest_state`, `sync_ingested_events`, all eight legacy entity tables,
  the in-flight `user_scoped_pk_redesign` shape) and creates the v2 shape clean. Per
  plan.md, the hosted DB is wiped post-merge.

## 1.1 LWW and undelete semantics

LWW (last-writer-wins) is the conflict-resolution rule for every entity column.
The key it sorts on is `client_updated_at_ms`. The rule applies uniformly to live
writes, deletes, and undeletes — there is no special-case path. **Deletion is
expressed by `deleted_at` going non-null; undeletion is `deleted_at` going back
to null.** There is no separate `deleted boolean` column on the server or on
the wire.

### 1.1.1 The rule

For any incoming write to row `(owner_user_id, id)` with payload
`(client_updated_at_ms = T_in, fields…)` against an existing row with
`client_updated_at_ms = T_db`:

- If `T_in <= T_db`: the write is a **no-op** at the row level. (The push RPC, t2,
  may still ack the row as accepted to clear the client's dirty bit, but no column
  changes.)
- If `T_in > T_db`: every column in the payload is written verbatim, including
  `deleted_at`. There is no per-column comparison; the unit of LWW is the row.

A consequence: `deleted_at` can flip from a non-null timestamp back to `null`
purely by LWW. That is the undelete case.

### 1.1.2 The two undelete scenarios

**Scenario A — server has a newer tombstone, client locally undeletes (older).**

Sequence:
1. Device 1 deletes row X at T=100. Push lands. Server stores `deleted_at = 100,
   client_updated_at_ms = 100, fields = <row contents at delete time>`.
2. Device 2, offline, undeletes (or just edits) row X with a stale local clock at
   T=90.
3. Device 2 reconnects, pushes.

LWW resolves: `T_in=90 < T_db=100`. Write is a no-op. Server keeps the tombstone.
The next pull tells Device 2 the row is deleted (`deleted_at` non-null, full payload
preserved). Device 2 reconciles its local row to deleted on receipt. **The
user-visible behaviour: the undelete loses.** This is consistent with the
brainstorm's acknowledged trade-off for the two-device case (`brainstorm.md`
§"Two devices in use") and is the cost of trusting client clocks for v2.

**Scenario B — client has a newer undelete, server has an older tombstone.**

Sequence:
1. Device 1 deletes row X at T=100. Push lands. Server: `deleted_at = 100, T_db =
   100`.
2. Device 2 (or Device 1 itself, after recovering the row from undo / restore-from-
   history / "create with the same id" — see below) writes the row at T=200 with
   `deleted_at = null` and a full payload.
3. Push lands.

LWW resolves: `T_in=200 > T_db=100`. Every column from the new payload overwrites
the stored row, including `deleted_at = null`. The row is fully restored. **This
is undelete by LWW.**

### 1.1.3 Why undelete is lossless: tombstones preserve the row

The choice this design makes: on delete, the server stores `deleted_at =
<delete_timestamp>` and **every other column at its current value, unchanged**.
Specifically:

- A delete push from the client sets `deleted_at = Date.now()` and
  `client_updated_at_ms = Date.now()`. The `fields` object in the push payload
  still contains the last-known values for every typed column (the client always
  serialises the full row — see t2 §2.1 for the wire shape).
- The server's LWW upsert writes the full row including the latest `deleted_at`
  value. It does **not** null-out non-PK columns on a delete.

This means an undelete (scenario B) simply lands a newer payload with
`deleted_at = null` and re-uses the row's identity. No revival ceremony, no
skeleton row, no special server logic. The push RPC code path for "undelete" is
identical to the push RPC code path for "update": one upsert with LWW.

> Why not a leaner "PK-skeleton tombstone" (delete clears non-PK columns)? Two
> reasons. First, it forces the server to know which columns are
> domain-meaningful vs envelope — that knowledge belongs on the client and
> tangling it into the push RPC violates §1's "server only stores" rule. Second,
> undelete via LWW (scenario B) would then have to reconstruct columns from the
> client's payload, which is what we're doing anyway — clearing on delete just
> throws the data away and re-fetches it on undelete, with a brief window where
> the tombstone is missing data that another reader (e.g. a stats query, a future
> audit feature) might want.

### 1.1.4 `deleted_at` is the single deletion signal

`deleted_at` is the **only** column the protocol uses to express deletion. It is
just another column under LWW: on delete the client sets `deleted_at =
Date.now()`; on undelete (scenario B) the client sets `deleted_at = null`. The
server stores it as written. Local readers filter `WHERE deleted_at IS NULL` to
hide tombstoned rows from the UI.

All eight entity tables — including the two join tables
(`exercise_muscle_mappings`, `session_exercise_tags`) — carry `deleted_at` so a
single LWW path works for every entity. (Earlier drafts had `deleted_at` only on
the six non-join entities and used a separate `deleted boolean` wire flag for
the join tables; both decisions were reversed during t2 reconciliation — see
§11.)

### 1.1.5 Reuse of `id` is the only practical undelete path

The client never resurrects a deleted row through normal UI — once deleted, the row
is hidden. Undelete in scenario B can occur via:
- A future "restore from history" UI that re-emits the original row id with a fresh
  `client_updated_at_ms`.
- A bug that re-uses an id after a local delete completes but before the server's
  ack reaches the device (rare, narrow window).
- A test fixture / dev tool that bypasses the UI.

All three converge on the same wire shape and the same LWW resolution. The design
doesn't need a special API; the existing push RPC handles undelete by construction.

### 1.1.6 What t2 will reference, not redefine

t2 owns the wire-level shape of the push payload, the batching, and the ack
protocol. The rules above are the *semantic* contract t2 implements; t2's wire
section will cite this subsection rather than restate it.

## 2. Per-entity mapping tables

Notation:

- "Client type" is the Drizzle column type as it appears in `apps/mobile/src/data/schema/<entity>.ts`.
- "Server type" is the Postgres type for the corresponding `app_public.<entity>` column.
- "Null" — `NO` means `not null`; `YES` means nullable.
- "Default" — server-side default. Empty cell means no default.
- "Indexed" — yes if a standalone btree index is created on that column. PK columns
  are not listed as "indexed" (the PK already covers them). Composite (multi-column)
  indexes and FK-supporting indexes are called out under "Other constraints / indexes".
- "FK" — the composite FK target if any. All FKs are `DEFERRABLE INITIALLY DEFERRED`.

Every table also gets these **identical** columns added on the server (not in any
client schema):

| Server column | Type | Null | Default | Purpose |
| --- | --- | --- | --- | --- |
| `owner_user_id` | `uuid` | NO | `auth.uid()` | Row owner. Part of PK. References `auth.users(id) on delete cascade`. |
| `client_updated_at_ms` | `bigint` | NO | — | Client wall-clock at write, used for LWW. No CHECK; per §1 the server does not validate. |
| `server_received_at` | `timestamptz` | NO | `now()` | Server-set on every insert/update; the pull cursor's monotonic axis. |

Deletion is expressed through the per-entity `deleted_at bigint` domain column
(see each entity's mapping below). There is **no separate `deleted boolean`
column** on the server and no `extras jsonb` column either — both were dropped
during t2 reconciliation (see §11).

Every table also gets these **identical** server indexes:

- `<table>_owner_received_idx` on `(owner_user_id, server_received_at)` — supports the
  pull cursor query `where owner_user_id = $1 and server_received_at > $2 order by
  server_received_at, id`. The PK trailing-key tiebreak (`id`) is included for stable
  pagination at the millisecond boundary.

Every table also gets this **identical** trigger:

- `<table>_touch_server_received_at` — `BEFORE UPDATE` setting
  `NEW.server_received_at = now()`. Insert side is covered by the column default.

These three columns, the `<table>_owner_received_idx` index, and the `touch`
trigger are not repeated in each table's mapping table below; they are part of
the implicit contract for every entity.

#### 2.6.1 muscle_groups is intentionally not synced

`muscle_groups` is a client-only seeded taxonomy with `isEditable = 0` and no
intended user mutation (`docs/specs/05-data-model.md` §"Static/system data"). It is
NOT one of the eight user-owned entities. The server has no `app_public.muscle_groups`
table. The `muscle_group_id` column on `exercise_muscle_mappings` is durable
metadata (a text token like `chest_sternal`) that the client resolves locally; the
server treats it as opaque text.

This is captured in §4.1's `exemptions.untyped_text_references` block (the
`muscleGroupId` entry) so the drift checker doesn't flag it as a missing
server counterpart.

## 3. (Removed) `extras jsonb` convention

**This section is intentionally empty.** Earlier drafts of this design used a
universal `extras jsonb` column on every entity table as a forward-compat hatch
for client-only columns, with an `extras_only_columns` allowlist in
`sync-extras.json` registering them. The mechanism is dropped during t2
reconciliation in favour of an unconditional **server-first deploy** rule for
every schema addition.

What replaces it:

- Every new entity column lands on the server first (with a migration), then on
  the client.
- The drift checker (§7) verifies that every typed client column has either a
  typed server counterpart or a `local_only` registration in
  `exemptions.local_only_columns`. No `extras_only` registration exists.
- The wire shape carries typed columns only; see t2 §2.1 for the envelope. There
  is no `extras` blob on push or pull.

For the dev-workflow rationale (server is deployed first, always), see the
"Server-first deploy" notes in §4.1 and the worked example in §9.

The section number `3` is preserved as a tombstone so cross-doc references to
later sections (§4 onwards) do not shift.

## 4. Local schema additions

### 4.1 Local-only sync-bookkeeping columns (mechanism only)

Sync v2 introduces local-only columns on the eight client entity tables to
support sync bookkeeping — these are set and cleared by the sync engine and
**never travel on the wire**. The specific columns (names, types, semantics)
are owned by **t2** (the sync protocol design), not by this document. t1's
contribution is the *mechanism* for tolerating them:

1. Local-only columns are registered in
   `apps/mobile/src/data/schema/sync-extras.json` under
   `exemptions.local_only_columns`.
2. The drift checker (§7) skips any client column listed there — no server
   counterpart is required.
3. The push code-path (t2) ignores any column registered as local-only when
   serialising a row to the wire.

That is the entire t1 contract. The full canonical list of which sync columns
exist, what they are named, what types they are, and how they are maintained,
lives in t2's design — for reference, t2 §9 pins it down as two columns
(`local_dirty`, `local_updated_at_ms`) on every entity table, plus singleton
state on `sync_runtime_state`.

**Server-first deploy is the unconditional rule for every schema addition.**
Every new entity column lands on the server first (via a migration deployed to
production), then on the client. There is no `extras_only` short-circuit — t2's
reconciliation dropped that mechanism. A client that ships a typed column ahead
of its server counterpart will fail the drift check (§7) at PR time. See §9 for
the worked example and §3 for the historical context.

## 5. Deferrable foreign keys

### 5.1 What `INITIALLY DEFERRED` means in practice

Postgres checks every FK at the end of the statement by default. With
`DEFERRABLE INITIALLY DEFERRED`, the check is postponed to COMMIT time. Inside one
transaction you can:

- Insert an `exercise_sets` row that references a `session_exercises.id` whose own
  insert is the very next statement in the same transaction.
- Update a `session_exercises.session_id` and the matching `sessions.id` in either
  order without an intermediate violation.

At COMMIT all FKs are re-checked. If any are unsatisfied the entire transaction
rolls back. From the client's view, the push RPC is atomic per batch — either every
row landed or none did.

### 5.2 The four deferrable FKs

| Constraint | From | To | On delete | Notes |
| --- | --- | --- | --- | --- |
| `sessions_gym_fk` | `sessions(owner_user_id, gym_id)` | `gyms(owner_user_id, id)` | `set null` | Gym may be deleted while a historical session refers to it. |
| `session_exercises_session_fk` | `session_exercises(owner_user_id, session_id)` | `sessions(owner_user_id, id)` | `cascade` | Deleting a session removes its exercises (server side; client soft-deletes). |
| `session_exercises_exercise_definition_fk` | `session_exercises(owner_user_id, exercise_definition_id)` | `exercise_definitions(owner_user_id, id)` | `no action` | Soft-delete on `exercise_definitions` keeps the row addressable so session history reads still resolve. The FK is required (not stripped per v1's relax migration) because deferral solves the original ordering problem. |
| `exercise_sets_session_exercise_fk` | `exercise_sets(owner_user_id, session_exercise_id)` | `session_exercises(owner_user_id, id)` | `cascade` | Same rationale as `session_exercises_session_fk`. |
| `exercise_muscle_mappings_exercise_definition_fk` | `exercise_muscle_mappings(owner_user_id, exercise_definition_id)` | `exercise_definitions(owner_user_id, id)` | `cascade` | Mapping membership is meaningless without its definition. |
| `exercise_tag_definitions_exercise_definition_fk` | `exercise_tag_definitions(owner_user_id, exercise_definition_id)` | `exercise_definitions(owner_user_id, id)` | `cascade` | Same. |
| `session_exercise_tags_session_exercise_fk` | `session_exercise_tags(owner_user_id, session_exercise_id)` | `session_exercises(owner_user_id, id)` | `cascade` | Same. |
| `session_exercise_tags_exercise_tag_definition_fk` | `session_exercise_tags(owner_user_id, exercise_tag_definition_id)` | `exercise_tag_definitions(owner_user_id, id)` | `cascade` | Same. |

That's eight named FKs (counting the `sessions.gym_id` FK plus seven child→parent).
All declared `DEFERRABLE INITIALLY DEFERRED`. Migration template:

```sql
constraint <name>
  foreign key (owner_user_id, <child_col>)
  references app_public.<parent>(owner_user_id, id)
  on delete <action>
  deferrable initially deferred
```

### 5.3 What happens when a child arrives with no parent

If the client pushes a child row whose parent is included in the same batch:
deferral covers it. Postgres allows the child INSERT to be in the transaction
before the parent INSERT; at COMMIT the FK check finds the parent and the
transaction succeeds. This is the whole point of `DEFERRABLE INITIALLY
DEFERRED`.

If the client pushes a child row whose parent is **neither in the same batch
nor already on the server**, the COMMIT fails. From the database's
perspective the FK is unsatisfied. The push RPC will roll the transaction
back and return an error.

**This is a t2 invariant violation, not a normal occurrence.** The sync
protocol is responsible for guaranteeing that every dirty child is pushed
alongside its dirty parent (or with a parent that the server already has).
There is no "retry with parent added" loop on the happy path; reaching this
code path means t2 batched incorrectly.

**t2 constraint:** the client must never push a child row whose parent is
neither in the same batch nor already present on the server. t2's batching
rules must guarantee this — concretely, when a child is dirty, t2 must
include any dirty ancestor in the same batch (transitively, up to a clean
ancestor or a server-resident row). The COMMIT-time failure described above
exists only as a defense-in-depth backstop against a t2 bug; this design
doc does not specify a retry strategy because there is nothing to retry —
t2 must not produce orphan-child batches in the first place.

## 6. RLS policies

### 6.1 Universal shape

Identical for all eight tables. Substitute `<table>` per entity.

```sql
alter table app_public.<table> enable row level security;

drop policy if exists <table>_owner_select on app_public.<table>;
create policy <table>_owner_select on app_public.<table>
  for select to authenticated
  using (owner_user_id = auth.uid());

drop policy if exists <table>_owner_insert on app_public.<table>;
create policy <table>_owner_insert on app_public.<table>
  for insert to authenticated
  with check (owner_user_id = auth.uid());

drop policy if exists <table>_owner_update on app_public.<table>;
create policy <table>_owner_update on app_public.<table>
  for update to authenticated
  using (owner_user_id = auth.uid())
  with check (owner_user_id = auth.uid());

drop policy if exists <table>_owner_delete on app_public.<table>;
create policy <table>_owner_delete on app_public.<table>
  for delete to authenticated
  using (owner_user_id = auth.uid());
```

### 6.2 Why no service-role policies are needed

The sync push/pull RPCs run as `security definer` functions (defined in t2) that
own the data writes; the `authenticated` policies above are still applied because
the RPC sets `search_path` but does not bypass RLS. Direct client access to the
tables uses the `authenticated` role and goes through the same policies. The
`service_role` grants exist for ops scripts only (wipe-remote, support tools); RLS
is bypassed for `service_role` by Postgres convention.

### 6.3 Immutability of `owner_user_id`

Same as v1: a `BEFORE UPDATE` trigger on every table that raises if
`new.owner_user_id <> old.owner_user_id`. The existing
`app_public.enforce_owner_user_id_immutable()` function from v1 can be reused
verbatim — the trigger names follow the convention `<table>_owner_user_id_immutable`.

This is the **only** trigger that touches row content, and it lives in §6 (RLS)
deliberately. It is an authorisation boundary, not validation: it prevents a row
from being silently re-homed to another user, which the RLS `with check` clauses
already block (the `with check` says `owner_user_id = auth.uid()`, so an update
attempting to set it to another user's id is denied; the trigger is the
belt-and-braces guarantee against future writes through SECURITY DEFINER
functions that might forget to re-evaluate `with check`). Treat it the same way
as RLS itself: structural, not validation.

## 7. Drift-detection mechanism

### 7.1 Choice

**DB-to-DB introspection on both sides.** The drift checker spins up two
ephemeral databases — a local Postgres for the server side and an in-memory
SQLite for the client side — applies every migration to each, then
introspects the live system catalogs (`information_schema` / `pg_catalog` on
Postgres, `pragma table_info` / `pragma index_list` / `pragma foreign_key_list`
on SQLite) and diffs the results. Drizzle's TypeScript metadata
(`getTableConfig`) is consulted only as a tie-breaker for things the SQLite
catalog can't disambiguate (see §7.6).

The check runs the migrations the same way production will:

- **Server side:** `supabase db reset --local`, which drops the local
  Postgres and re-applies the full migration tree under
  `supabase/migrations/` from scratch. What ends up in `information_schema`
  is what production will end up with.
- **Client side:** `drizzle-kit export` emits the SQL DDL for the current
  TypeScript schema; the checker executes that DDL against an in-memory
  `better-sqlite3` database. What ends up in `pragma table_info` is what a
  fresh app install will end up with after `drizzle-kit push` has been
  applied.

This eliminates a whole class of brittleness that a static parser introduces
on either side — `DEFAULT` expressions, generated columns, trigger-set
values, type coercions on `ALTER TYPE`, CHECK / NOT NULL set via subsequent
migrations, and Drizzle-side quirks like `text({ enum: [...] })` not
surfacing the enum values in `getTableConfig` — all materialise correctly
because the respective database engine itself ran the SQL.

Location: `apps/mobile/scripts/check-sync-schema-drift.ts`.

Command: `npm run check:sync-drift` (added to `apps/mobile/package.json`).

Wired into `scripts/quality-slow.sh backend` as a new step (justification in §7.5).

### 7.2 Why a live database (and not a SQL parser)

The earlier draft of this section proposed parsing `CREATE TABLE` blocks out of the
migration files. That is brittle in ways that matter:

- **`ALTER TABLE … ALTER COLUMN TYPE …`** with `USING` clauses — the static parser
  has to model Postgres's type coercion to know the result.
- **`DEFAULT now()` / `DEFAULT gen_random_uuid()` / `DEFAULT auth.uid()`** — the
  parser has to recognise these as different from a literal default to avoid
  reporting a false positive on the client (which has no equivalent).
- **Triggers that set column values** — `server_received_at` is touched by the
  `<table>_touch_server_received_at` trigger; an SQL parser can't know the column
  is server-managed without modelling trigger effects.
- **GRANT / RLS** — the static parser would have to track grants and RLS policy
  scopes across migrations to check anything beyond column names.
- **Two migrations that disagree** — if migration A creates `notes text not null
  default ''` and migration B does `alter table … alter column notes drop not
  null`, only running both in order produces the right answer.

A live database evaluates every one of those correctly because it *is* Postgres.
The cost is a one-time cold-start of a local container; see §7.5.

The same argument applies to the client side: SQLite's `pragma table_info`
is the source of truth for "what columns does the app actually have after
migrations". Drizzle's TypeScript metadata is a useful supplement (it tells
us which column is a `timestamp_ms` integer vs a plain integer, which the
SQLite catalog doesn't preserve) but it has known gaps documented in §7.6;
the SQLite catalog is the primary signal.

The drift checker is, in this revision, a thin tool around live introspection
on both sides. It does not try to be cleverer than the databases.

### 7.3 Algorithm

```
1. Server side — ensure Supabase CLI is installed and a local Postgres is
   reachable. (`supabase status` returns a running stack, or `supabase start`
   is invoked.)

2. Server side — run `supabase db reset --local --yes` to drop + recreate the
   local Postgres and apply every migration under `supabase/migrations/` in
   order. Connect as a role that can see all schemas (the CLI exposes a
   service-role URI for local use).

3. Client side — materialise the Drizzle schema into an ephemeral SQLite DB:

   a. Run `drizzle-kit export --config=apps/mobile/drizzle.config.ts` to emit
      the SQL DDL for the current TypeScript schema. (drizzle-kit's `export`
      command produces `CREATE TABLE` statements for every table defined in
      the schema; verified against the drizzle-kit 0.31.x docs.)
   b. Open a fresh in-memory `better-sqlite3` database (`new Database(':memory:')`)
      and execute the DDL.
   c. The result: a SQLite database containing exactly what a freshly-installed
      app's Drizzle migration history would produce.

4. For each of the 8 entity table names:

   a. Introspect the live server table:

      select
        column_name,
        data_type,
        udt_name,                -- distinguishes bigint vs integer
        is_nullable,
        column_default,
        ordinal_position
      from information_schema.columns
      where table_schema = 'app_public'
        and table_name   = $entity
      order by ordinal_position;

      Plus parallel queries for:
        - indexes (`pg_indexes`, scoped to the table)
        - foreign keys (`information_schema.referential_constraints` joined to
          `key_column_usage`)
        - RLS policies (`pg_policies`)
        - check constraints (`information_schema.check_constraints` joined to
          `constraint_column_usage`) — expected empty per §1; any present is a bug
        - triggers (`information_schema.triggers`) — expected only the two
          structural triggers from §1

   b. Introspect the client SQLite table:

      pragma table_info('<entity>');     -- columns, types, nullability, defaults
      pragma index_list('<entity>');     -- indexes, unique flag
      pragma foreign_key_list('<entity>'); -- FK targets

      This is the *primary* signal for "what columns the client actually has".
      Drizzle's TypeScript metadata is consulted as a tie-breaker only — see
      §7.6.

   c. Load `sync-extras.json` for the top-level `exemptions`. (There are no
      per-entity blocks after the t2 reconciliation — the `extras_only_columns`
      mechanism is dead; see §3.)

   d. For each clientCol in clientCols:
        if clientCol is in exemptions.local_only_columns: continue
        if clientCol matches a typed serverCol (snake_case name + compatible type): continue
        if clientCol is in exemptions.untyped_text_references: continue
        FAIL: "<entity>.<column>" has no server counterpart.

   e. For each serverCol not in {client_updated_at_ms, server_received_at,
      owner_user_id} (the wire-envelope columns from §2):
        if serverCol matches a clientCol (snake/camel mapping): continue
        WARN (or FAIL under --strict): server has a column the client doesn't know
        about — usually a stale migration that should be reverted.

   f. Sanity checks on the universal contract:
        - `<entity>_owner_received_idx` index present
        - `<entity>_touch_server_received_at` trigger present
        - `<entity>_owner_user_id_immutable` trigger present
        - RLS enabled with the four policies named per §6.1
        - No CHECK constraints (per §1; presence is a regression)
        - No `extras` column on the server (per §3; presence is a regression)
        - No `deleted` boolean column on the server (per §1; presence is a regression)
      Any failure here exits 1 with a precise message.

5. Tear down: the local Postgres is left running for the next call (no shutdown).
   CI starts fresh per run; local dev preserves the container.
```

In addition to the per-entity walk above, the checker runs a **topological FK
order assertion** — see §7.7.

Type compatibility map (intentionally narrow):

| Drizzle SQLite type | Acceptable `udt_name` in Postgres |
| --- | --- |
| `text` | `text` |
| `integer` with `mode: 'timestamp_ms'` | `int8` (bigint) |
| `integer` (default, no mode) | `int4` (integer) *or* `int8` (bigint) |
| `real` | `float8` (double precision) *or* `numeric` |

Nullability and defaults are not part of the column-match check — the goal is
"field exists in both places", not full structural equality. A future iteration
can tighten this. The structural-trigger / RLS / no-CHECK sanity checks (step 4f)
ARE strict, because they assert §1's ground rules.

### 7.4 Failure output

Exactly one of three exit codes:

- `0`: no drift, no warnings.
- `1`: drift detected (client column without a server counterpart, or a §1
  ground-rule regression like a CHECK constraint appearing). Stdout lists, per
  entity:
  ```
  ✗ exercise_sets: client column "notes" (text) has no server counterpart.

    The local Postgres applied every migration and ended up with these columns
    on app_public.exercise_sets:
      id, session_exercise_id, order_index, weight_value, reps_value, set_type,
      created_at, updated_at, deleted_at, owner_user_id,
      client_updated_at_ms, server_received_at

    To fix, add a server migration under supabase/migrations/ that promotes the
    column to typed, then deploy it:

      alter table app_public.exercise_sets
        add column notes text;

    Server-first deploy is unconditional (per designs/t1.md §4.1 and §9). Once
    deployed, re-run `npm run check:sync-drift` — the local DB reset will pick
    up the new migration and the check will pass.
  ```
- `2`: server has columns absent on the client. Warn-only by default; exits 2
  only when called with `--strict`. (Slow-gate posture uses `--strict`.)

The fix flow is a single path: write the server migration, deploy it, then the
client column ships. There is no `extras_only` short-circuit — that mechanism
was removed in the t2 reconciliation (see §3).

### 7.5 Wiring into `scripts/quality-slow.sh backend`

Justification for slow-gate placement (and the cost is now more obviously a
slow-gate cost):

- **Cold-start cost.** First invocation in CI starts a Postgres container via
  `supabase start` (~15-30 seconds on warm hardware, longer if the image isn't
  cached) and runs `supabase db reset --local --yes` (~5-15 seconds for the
  current migration set). The introspection queries themselves are fast (sub-
  second). Total: ~20-45 seconds on a cold runner, ~10 seconds on a warm local
  dev box. This is squarely in slow-gate territory.
- **Backend gate is the natural home.** A schema mismatch breaks sync, not the
  unit-tested client. Any PR that asserts "`quality-slow.sh backend` passed" is
  also asserting "no schema drift between client and v2 server migrations".
  Matches the existing `test-sync-api-contract.sh` position, which already pays
  the supabase-stack cost.
- **The local-dev path is faster.** If the developer already has the Supabase
  stack running from `supabase start`, `npm run check:sync-drift` skips the
  start step and just runs `supabase db reset --local --yes` + introspection
  (~5-15 seconds). PRs touching schema files can invoke this directly without
  triggering the slow gate.
- **Reusing the existing CLI invocation pattern.** `supabase/scripts/reset-local.sh`
  already wraps `run_supabase db reset --local --yes`; the drift checker reuses
  the same wrapper so credentials, ports, and env-var handling don't diverge.

Concrete addition to `scripts/quality-slow.sh`'s `run_backend()`:

```bash
if [[ ! -f "${REPO_ROOT}/apps/mobile/scripts/check-sync-schema-drift.ts" ]]; then
  echo "[quality-slow] skipping backend sync-schema-drift: script not found"
else
  echo "[quality-slow] backend: sync-schema-drift (boots local supabase + introspects)"
  (cd "${REPO_ROOT}/apps/mobile" && npm run check:sync-drift -- --strict)
fi
```

The `npm run check:sync-drift` script itself ensures `supabase status` reports
running before continuing; if the stack is down it invokes `supabase start` (or
fails with a useful message in CI environments that should have started the
stack already).

> Editing `quality-slow.sh` is out of scope for this design doc; the wiring above
> is the spec the t1 build wave implements.

### 7.6 Drizzle metadata extraction and confidence claim

A previous draft of this design described the client side as "diff against
the Drizzle client schema metadata." That phrasing under-promises and
over-relies on Drizzle's TypeScript reflection in ways that are worth
nailing down.

**The API used.** When Drizzle metadata is consulted, the entry point is
`getTableConfig(table)` from `drizzle-orm/sqlite-core`. Per the upstream
documentation, this returns
`{ columns, indexes, foreignKeys, checks, primaryKeys, name, schema }` for
the table. (Verified against drizzle-orm docs via Context7,
drizzle-kit 0.31.x.)

**What `getTableConfig` exposes reliably:**

- Column names (camelCase as declared in TypeScript).
- The wire name passed as the first arg of the column builder (e.g.
  `text('order_index')` → `order_index`). The drift checker uses this for
  snake-case mapping.
- Base column type (`SQLiteText`, `SQLiteInteger`, `SQLiteReal`, etc.).
- Nullability (`notNull` builder applied or not).
- Foreign key target tables and column tuples.
- Index declarations (column tuples, unique flag).
- Primary key column tuples.

**What `getTableConfig` exposes unreliably or not at all:**

- **Precise SQL `DEFAULT` expressions.** `default(sql\`unixepoch() * 1000\`)`
  is stored as an opaque `SQL` chunk; the textual form requires evaluating
  the chunk against a dialect, which the drift checker does not do.
- **Enum values from `text({ enum: [...] })`.** The Drizzle SQLite text
  column type tracks the TypeScript enum in its type parameter but does
  not always surface the array of literals at runtime via
  `getTableConfig`; we have observed cases where the field is undefined on
  released builds.
- **The `mode` discriminator** for integer columns (`'timestamp_ms'`,
  `'boolean'`, plain). It is on the column builder result, accessible via
  `column.config.mode` rather than the top-level `getTableConfig` shape;
  the drift checker reads this directly off the column to distinguish
  `bigint`-shaped client columns from plain integers.
- **Generated / computed columns** if introduced via raw SQL.
- **Column ordering after `ALTER TABLE`** on the SQLite side; Drizzle
  reflects the TypeScript declaration order, not the migration-applied
  order.

**The fallback for unreliable items: DB-to-DB introspection.** The drift
checker materialises the Drizzle schema into an ephemeral in-memory SQLite
database (per §7.3 step 3) and queries `pragma table_info`,
`pragma index_list`, and `pragma foreign_key_list` against that database.
Those PRAGMAs give us, on the client side, the same authoritative answers
that `information_schema` gives us on the server side. Where the two
sources of client-side truth disagree (Drizzle metadata vs SQLite catalog),
the SQLite catalog wins; Drizzle metadata is consulted only for the items
the SQLite catalog can't express (e.g., distinguishing `integer mode:
'timestamp_ms'` from a plain integer for the type-compatibility table in
§7.3).

**Tooling note.** `drizzle-kit export` (verified present in 0.31.x) emits
`CREATE TABLE` SQL for the current schema; piping that into a
`better-sqlite3` in-memory database is the materialisation step. If a
future drizzle-kit version changes the `export` interface, the fallback
is to invoke `drizzle-kit push` against a throwaway SQLite file path; both
produce the same DDL result.

**Confidence claim.** With DB-to-DB introspection on both sides:

- **High confidence** that the checker detects: client columns that
  don't exist on the server, server columns that don't exist on the
  client, name-mapping mismatches, missing FKs, missing or extra
  indexes, missing structural triggers, missing RLS policies,
  unexpected CHECK constraints, and the universal-contract sanity
  checks of §7.3 step 4f.
- **Medium confidence** on type compatibility, because the type-
  compatibility map in §7.3 is intentionally narrow (text↔text,
  integer↔int4/int8, real↔float8/numeric). A column whose Drizzle type
  is exotic (e.g., a custom `customType` wrapper) may not map cleanly;
  the checker errs on the side of FAIL and the developer adds an
  explicit mapping when they introduce such a column.
- **Low confidence (deliberately not enforced)** on:
  - Default-expression equality. Server `default now()` vs client
    `default unixepoch() * 1000` are not compared.
  - Enum literal set equality. The server stores arbitrary text per
    §1, so the enum is a client-only contract anyway.
  - Index ordering / btree options. The checker asserts presence and
    column tuple, not opclass or sort direction.
- **The remaining gap.** The checker does not detect: data-level
  inconsistencies (rows that violate the t2 constraints emitted by §11
  — that is t2's tests' job), trigger *body* changes (it asserts
  presence by name), or grant changes outside RLS policies. None of
  these are schema drift in the sense this section is targeting; they
  are operational concerns the slow gate's other steps cover.

In short: the part of the check that runs against Drizzle metadata is
narrow and well-bounded, and the rest of the check runs against a live
SQLite database, which is the same property §7.2 gives us on the
server side.

### 7.7 Topological FK order assertion

t2's push batch builder (t2 §3.4.1) hardcodes a per-table topological order
that the eight entity tables are walked in when collecting dirty rows. The
order is:

```
gyms, exercise_definitions, exercise_tag_definitions,
sessions, exercise_muscle_mappings,
session_exercises,
exercise_sets, session_exercise_tags
```

This list lives at a known client-side location (working name:
`apps/mobile/src/sync/topo-order.ts`; the build wave pins the exact path).

The drift checker, after introspecting the live FK graph in §7.3, asserts the
hardcoded list is a valid topological sort:

```
1. Read TOPO_ORDER from apps/mobile/src/sync/topo-order.ts (as a TypeScript
   const array, parsed via tsx import or static parse — the checker already
   loads TS files for the Drizzle schema, so this is reusing existing
   infrastructure).
2. Read all FK edges from the live introspected schema (the same query that
   feeds the per-entity walk in §7.3 step 4a's foreign-keys subquery).
3. For every FK edge T_child → T_parent:
     if index_of(T_parent, TOPO_ORDER) is undefined: FAIL ("T_parent missing
       from topo order list; add it before its child tables.")
     if index_of(T_child, TOPO_ORDER) is undefined: FAIL ("T_child missing
       from topo order list; add it after its parents.")
     if index_of(T_parent) >= index_of(T_child): FAIL ("topo order violation:
       T_child precedes its parent T_parent; reorder the list.")
4. Verify TOPO_ORDER lists exactly the eight entity tables (no extras, no
   missing). Mismatch is a FAIL.
```

The assertion runs once per drift check (not per entity). Cost is negligible.

**Why this check exists.** The no-orphan-child guarantee on push (t2 §3.4.3)
depends on the topological walk being correct. If a future migration adds a
new entity table or a new FK without updating `topo-order.ts`, the push could
silently start producing batches where a dirty child precedes its dirty parent
in the same batch — deferral handles same-batch ordering, but a child that
ends up in an *earlier* batch than its parent (because the topo list claims
the child layer is earlier) would COMMIT against an absent parent and
`FK_VIOLATION` the whole batch. Catching this at PR time is much cheaper than
catching it as a production sync failure.

**Failure output (exit 1).**

```
✗ topo-order: violation detected in apps/mobile/src/sync/topo-order.ts.
  FK edge session_exercises → sessions but topo lists session_exercises
  before sessions.
  Reorder TOPO_ORDER so every parent precedes its children.
```

### 8.1 Which file to edit

The t1 build wave adds a new subsection at the end of `docs/specs/05-data-model.md`

### 8.2 Exact wording

```markdown
## Client schema drift rule (Sync v2)

Modifying any file under `apps/mobile/src/data/schema/` for the eight user-owned
entities (`gyms`, `sessions`, `session_exercises`, `exercise_sets`,
`exercise_definitions`, `exercise_muscle_mappings`, `exercise_tag_definitions`,
`session_exercise_tags`) to add a domain column requires a paired server
migration under `supabase/migrations/` that adds the matching
`app_public.<entity>` column with a compatible Postgres type, **and the server
migration must be deployed to production before the client change ships**.

Why "server first": a client that depends on a typed server column not yet
deployed will fail to round-trip that column; the server has nowhere typed to
store it. Server-ahead-of-client is always safe (the column sits unwritten until
the client catches up). The earlier `extras_only_columns` short-circuit was
dropped during the t2 reconciliation; see
`docs/plans/sync-v2/designs/t1.md` §3 and §4.1 for the full rationale.

The drift checker (`apps/mobile/scripts/check-sync-schema-drift.ts`, invoked via
`npm run check:sync-drift` and gated by `./scripts/quality-slow.sh backend`)
enforces the rule by booting a local Postgres, applying every migration, and
introspecting the live schema. PRs failing the gate cannot merge. The checker
also asserts the hardcoded topological table order in
`apps/mobile/src/sync/topo-order.ts` against the live FK graph (see
`designs/t1.md` §7.7) — adding a new entity table or FK without updating that
list also fails the gate.

This rule does NOT apply to: `muscle_groups` (client-only taxonomy),
`smoke_records`, `sync_outbox_events`, `sync_delivery_state`,
`sync_runtime_state` (test/runtime scaffolding), or the two local-only
sync-bookkeeping columns (`local_dirty`, `local_updated_at_ms`) on each entity
table. All of these are listed under `exemptions` in `sync-extras.json`.

If your client change adds a value to an existing column (e.g., a new enum literal),
the rule does not apply because the column already exists on both sides; the client
is free to validate the enum and the server stores arbitrary text per the v2
no-server-validation policy in `docs/plans/sync-v2/designs/t1.md` §1.
```

### 8.3 Where the AGENTS.md side lives

The rule lives in the spec; AGENTS.md doesn't need a direct edit because

## 9. Worked example — developer adds `notes text` to `exercise_sets`

### 9.1 Scenario

A developer wants to let users type a free-text note per set ("felt easy",
"tight shoulder"). They want the field on the UI and queryable in stats.

The server-first deploy rule (§4.1) is unconditional: the server migration
lands first, then the client. Two PRs, in that order. (Earlier drafts of this
doc described an alternative `extras_only` path; that mechanism was dropped
during the t2 reconciliation — see §3.)

### 9.2 Step 1 — Server PR (lands first)

New migration `supabase/migrations/<ts>_add_exercise_sets_notes.sql`:

```sql
alter table app_public.exercise_sets
  add column notes text;
```

Merge and deploy to production. Confirm healthy. The production DB now has
the column; older clients in the wild that don't yet write it simply leave
the column NULL for their rows.

### 9.3 Step 2 — Client PR (lands second)

After the server is deployed:

**1. Edit the client schema.**

`apps/mobile/src/data/schema/exercise-sets.ts`:

```ts
export const exerciseSets = sqliteTable(
  'exercise_sets',
  {
    // …existing columns…
    notes: text('notes'),
    // …
  },
  // …
);
```

**2. Generate the local Drizzle migration** with `drizzle-kit generate` per
existing convention.

**3. Run the drift check locally.**

```
cd apps/mobile && npm run check:sync-drift
```

The check boots local Supabase (if not running), resets the local DB to apply
all migrations (including the one from Step 1), introspects
`app_public.exercise_sets`, and compares against the client schema. It passes
because both sides now have the column.

**4. Update read code** that wants to query `notes`.

**5. Ship.**

After both PRs are deployed, the push code routes `notes` under `fields`
automatically — the push serialiser closes over the typed server-column list
and emits every typed column on the dirty row.

Wire payload for a set with a note (per t2 §2.1):

```json
{
  "type": "exercise_sets",
  "id": "01HX…",
  "client_updated_at_ms": 1748000000000,
  "fields": {
    "session_exercise_id": "01HX…",
    "order_index": 0,
    "weight_value": "60",
    "reps_value": "10",
    "set_type": "rir_1",
    "notes": "felt easy",
    "created_at": 1748000000000,
    "updated_at": 1748000000000,
    "deleted_at": null
  }
}
```

### 9.4 What the developer never has to do

- Edit a 1500-line projection function.
- Wire a new outbox event type.
- Update three sync API contract documents.
- Coordinate one change across multiple service repos.

That's the win.

## 10. Migration outline (informative, not authoritative)

A reader of this doc could write the v2 migration in one file. The shape:

1. Drop v1 server objects:
   - `drop function if exists app_public.sync_apply_projection_event(uuid, text, text, text, bigint, jsonb);`
   - `drop function if exists app_public.sync_events_ingest(...)`, `…_impl(...)` (look up signatures from M13 migration).
   - `drop table if exists app_public.sync_device_ingest_state cascade;`
   - `drop table if exists app_public.sync_ingested_events cascade;`
   - `drop table if exists app_public.session_exercise_tags cascade;` (and the other 7, in child→parent order even though `cascade` makes the order irrelevant).
2. Create the eight v2 tables per §2 (parent→child order so even if a future reader removes `if exists` they don't trip on FKs; deferrable FKs make this aesthetic only).
3. Add the universal index `<table>_owner_received_idx` and the `<table>_touch_server_received_at` trigger to each.
4. Add the owner-immutability trigger to each.
5. Apply RLS policies per §6.
6. Grant `select, insert, update, delete` to `authenticated` and `service_role` per table.

The migration is a single SQL file; the build task that writes it owns the
authoritative version.

## 11. t2 constraints emitted by t1

This is the consolidated index of every constraint t1's design places on the
t2 (sync protocol) design. When t2 is (re)written, the agent should treat
each of these as a hard input. They are scattered through the doc as
"**t2 constraint:**" call-outs; this section is the single index.

### 11.1 Uniqueness invariants (client-enforced because the server doesn't)

Per the no-server-validation rule of §1, the server enforces only PK and
FK uniqueness. The remaining uniqueness invariants are the client's job:

1. **Active `session_exercises` ordering.** The client must never push a
   state with two active rows (`deleted_at IS NULL`) sharing
   `(session_id, order_index)`. Reorder writes are serialised through a
   single push batch per session. (§2.3)
2. **Active `exercise_sets` ordering.** Same rule for two active
   `exercise_sets` rows sharing `(session_exercise_id, order_index)`.
   (§2.4)
3. **`exercise_muscle_mappings` pair uniqueness.** No two rows for the
   same owner sharing `(exercise_definition_id, muscle_group_id)`.
   (§2.6)
4. **`exercise_tag_definitions` normalized-name uniqueness.** No two
   active rows for the same owner sharing
   `(exercise_definition_id, normalized_name)`. (§2.7)
5. **`session_exercise_tags` pair uniqueness.** No two rows for the
   same owner sharing
   `(session_exercise_id, exercise_tag_definition_id)`. (§2.8)

The corresponding server-side indexes still exist as non-unique btree
indexes for query performance, but the database does not reject a
duplicate. A duplicate landing on the server is a t2 bug.

### 11.2 Foreign-key closure on push batches

6. **No orphan-child pushes.** The client must never push a child row
   whose parent is neither in the same batch nor already present on the
   server. t2's batching rules must guarantee this — concretely, when a
   child row is dirty, t2 must include any dirty ancestor in the same
   batch (transitively, up to a clean ancestor or a server-resident
   row). Reaching the COMMIT-time FK failure that deferral can't paper
   over is a t2 bug, not a normal retry path. (§5.3)

### 11.3 Local-only column registration

7. **Local-only sync-bookkeeping columns registered in
   `sync-extras.json`.** Any local-only column t2 introduces on a
   client entity table (sync bookkeeping that never travels on the
   wire) must be registered in
   `apps/mobile/src/data/schema/sync-extras.json` under
   `exemptions.local_only_columns`. The drift checker flags any client
   column that has neither a server counterpart nor a `local_only`
   registration. t2 also owns the responsibility of ensuring the push
   serialiser does not emit these columns on the wire. (§4.1)

### 11.4 (Removed) Extras blob handling on the wire

Items 8 and 9 in earlier drafts of this index covered `extras` blob handling
on the wire (full-blob replace; per-key registration). Both are **dead** after
the t2 reconciliation: there is no `extras` blob on the wire and no
`extras` column on the server (see §3). The item numbers `8` and `9` are
reserved as a tombstone to keep subsequent item numbers stable.

### 11.5 LWW semantics that t2 implements

10. **Row-level LWW on `client_updated_at_ms`.** The unit of LWW is
    the row, not the column. On `T_in > T_db` every column in the
    payload is written verbatim (including `deleted_at`); on `T_in
    <= T_db` the write is a no-op at the row level. (§1.1.1)
11. **Tombstones preserve the full row.** A delete push sets
    `deleted_at = Date.now()` and a fresh `client_updated_at_ms`, but
    the `fields` blob still carries the row's last-known values for
    every other column. The server upsert writes the full row
    including those fields. This makes undelete via LWW lossless.
    The protocol expresses this without any top-level `deleted` flag
    — the constraint is satisfied because every push carries the
    full row in `fields` (including `deleted_at`). (§1.1.3)
12. **`deleted_at` is the only column the protocol uses to express
    deletion.** It is just another column under LWW. On delete the
    client sets `deleted_at = Date.now()`. On undelete it sets
    `deleted_at = null`. The server stores it as written; there is
    no separate `deleted boolean` column on the server or on the
    wire. (§1.1.4)

### 11.6 Out of t1's scope but t2 must own

- The exact JSON shapes of `/sync/push` and `/sync/pull` requests and
  responses.
- Push batching strategy (which rows go in which batch; transitive
  parent inclusion; back-pressure on batch size).
- The dirty-bit lifecycle (when set, when cleared, how the clear
  interacts with concurrent writes).
- The local-only sync-column inventory itself (names, types, defaults,
  semantics). t1 only specifies the registration mechanism.
- Tombstone-purge policy (whether and when server tombstones are
  garbage-collected; t1 stores them indefinitely).
