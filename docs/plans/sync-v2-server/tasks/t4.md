**Type:** build

**Problem:** v2's pull side, like push, replaces the M13 event-log
projection with a thin PostgREST RPC. `sync_pull` accepts a per-layer
cursor envelope, returns up to `limit` typed rows from tables in that
topological layer, and emits a `next_cursor` plus `has_more` flag the
client uses to drain layer-by-layer. The wire shape and cursor semantics
are pinned in `docs/plans/sync-v2/designs/t2.md` §4; this task implements
the RPC body and the contract tests that exercise pagination, RLS, and
the layered drain.

**Inputs:**

- t1 (this plan) merged. The eight v2 entity tables exist with
  `(owner_user_id, server_received_at)` indexes per t1 §2's universal
  index spec.
- `docs/plans/sync-v2/designs/t2.md` §1 (PostgREST endpoint pattern,
  bigint integer serialization), §2.1 (Entity envelope — same shape
  emitted by pull), §2.2 (error envelope — `AUTH_REQUIRED` and
  `INTERNAL`; `FK_VIOLATION` does not apply to pull), §4.1 (request
  shape: `layer 0..3`, opaque `cursor`, `limit 1..200`), §4.2 (response
  shape: `{entities, next_cursor, has_more}`), §4.3 (the
  `ORDER BY server_received_at, owner_user_id, type, id` and the WHERE
  clause that uses row-value `>` against the cursor, plus the
  documented same-millisecond cursor-race limitation), §4.4 (layer-by-
  layer drain pseudocode the client follows — informational here, the
  RPC just serves one layer at a time), §4.7 (cursor lifecycle table).
- `docs/plans/sync-v2/designs/t1.md` §2 (`<table>_owner_received_idx`

  see their own rows).

**Outcomes:**

- Migration ships `app_public.sync_pull(payload jsonb)` marked
  `language plpgsql security invoker`, granted `execute … to
  authenticated`. Exposed at `POST /rest/v1/rpc/sync_pull`.
- Request validation (structural):
  - `payload->>'layer'` is an integer in `0..3`; otherwise
    `INTERNAL`-shaped error.
  - `payload->>'limit'` is an integer in `1..200` (default 200 if
    absent).
  - `payload->'cursor'` is either JSON null / absent OR a JSON object
    with all four of `server_received_at` (ISO 8601 with ms),
    `owner_user_id` (UUID), `type` (string, one of the eight entity
    names), `id` (string).
  - `auth.uid()` is non-null; otherwise `AUTH_REQUIRED`.
- Query body:
  - Resolve the requested layer to its set of entity types per t2 §4.1
    mapping (Layer 0: `gyms`, `exercise_definitions`,
    `exercise_tag_definitions`; Layer 1: `sessions`,
    `exercise_muscle_mappings`; Layer 2: `session_exercises`; Layer 3:
    `exercise_sets`, `session_exercise_tags`). Implementation may be a
    `case` expression mapping layer → `text[]`, hardcoded in the
    function body.
  - UNION ALL the per-entity selects (one per entity in the layer), each
    projecting the columns into the wire envelope shape: `type` as a
    constant string, `id`, `client_updated_at_ms`, and `fields` as a
    `jsonb_build_object(...)` of every typed column listed in t1 §2 for
    that entity (including `deleted_at`).
  - Apply `WHERE owner_user_id = auth.uid()` (RLS also enforces this;
    the explicit predicate keeps the query planner on the universal
    `(owner_user_id, server_received_at)` index per t1 §2).
  - Apply the cursor predicate per t2 §4.3 — row-value `>` on
    `(server_received_at, owner_user_id, type, id)` when cursor is
    non-null; no predicate when null (snapshot).
  - `ORDER BY server_received_at ASC, owner_user_id ASC, type ASC, id
    ASC` and `LIMIT :limit + 1` (the `+1` lets the RPC compute
    `has_more` cheaply).
- Response: a `jsonb` object
  ```json
  {
    "entities": [ /* up to `limit` envelopes */ ],
    "next_cursor": { server_received_at, owner_user_id, type, id },
    "has_more": <boolean>
  }
  ```
  per t2 §4.2:
  - When the query returned `limit + 1` rows, drop the last row,
    `has_more = true`, `next_cursor` echoes the last *kept* row's sort
    key.
  - When the query returned `<= limit` rows, `has_more = false`,
    `next_cursor` echoes the last returned row's sort key — or, on an
    empty page, echoes the input `cursor` unchanged per t2 §4.2.
- Same-transaction `server_received_at` consistency: not relevant for
  pull (it's a read), but document in code comments that the cursor's
  `server_received_at` field is the upstream `now()` value the push RPC
  stamped, and the v2-acknowledged concurrent-commit race from t2 §4.3
  is not solved here (a follow-up may switch to
  `pg_xact_commit_timestamp(xmin)` per that section).
- Contract tests under `supabase/tests/sync-pull-contract.sh` cover:
  - **Snapshot pull (cursor=null)** — seed three rows for user A across
    Layer 0 entities (one `gyms`, two `exercise_definitions`); POST
    `{layer: 0, cursor: null, limit: 10}` as A; response contains all
    three in `(server_received_at, owner_user_id, type, id)` order;
    `has_more = false`; `next_cursor` matches the last row's key.
  - **Paginated drain** — seed five rows in Layer 0; pull with
    `limit: 2`; first response has 2 rows + `has_more: true`; second
    pull with the returned `next_cursor` has 2 rows + `has_more: true`;
    third pull has 1 row + `has_more: false`; union of all three pages
    equals the seeded set exactly once each, in cursor order.
  - **Layer→type mapping integrity (all four layers, all eight
    entities).** Seed at least one row of *every* entity type for user A
    (one each of `gyms`, `exercise_definitions`,
    `exercise_tag_definitions`, `sessions`, `exercise_muscle_mappings`,
    `session_exercises`, `exercise_sets`, `session_exercise_tags`). Pull
    each layer in turn (0, 1, 2, 3) with `cursor: null, limit: 100`.
    Assert that the set of `type` values in each response equals
    *exactly* the t2 §4.4 mapping: `{gyms, exercise_definitions,
    exercise_tag_definitions}` for layer 0, `{sessions,
    exercise_muscle_mappings}` for layer 1, `{session_exercises}` for
    layer 2, `{exercise_sets, session_exercise_tags}` for layer 3.
    Assert further that the union of the four type-sets equals all
    eight entity types and the four sets are pairwise disjoint (i.e.
    every entity is reachable via *exactly* one layer). This is the
    structural guarantee a client drain depends on; if any entity is
    misclassified the client will FK-fail on insert.
  - **RLS isolation** — seed rows for user B; pull as user A; A's
    responses contain none of B's rows even though the cursor query
    would otherwise sort them in.
  - **Tombstones included** — soft-delete a row (via direct
    service-role UPDATE setting `deleted_at`); pull returns the row
    with `fields.deleted_at` non-null. (Tombstones are pulled like any
    other row per t1 §1.1; clients filter `deleted_at IS NULL`
    locally.)
  - **Empty page after drain** — pull a layer to `has_more: false`;
    re-pull with the same `next_cursor`; response is
    `{entities: [], next_cursor: <echoes input>, has_more: false}`
    per t2 §4.2.
  - **Same-millisecond tiebreak** — directly INSERT two rows for the
    same owner in the same transaction (so they share
    `server_received_at`); pull with `limit: 1` returns the
    lexicographically-earlier `(type, id)` first; second pull with the
    emitted `next_cursor` returns the second row.
  - **Limit bounds** — `limit: 0` rejected; `limit: 201` rejected;
    `limit: 200` accepted.
  - **Layer bounds** — `layer: -1` rejected; `layer: 4` rejected;
    `layer: 0..3` each accepted.
  - **AUTH_REQUIRED** — pull without a JWT returns the
    `AUTH_REQUIRED` envelope.
- `./scripts/quality-slow.sh backend` invokes the new test script via
  the same wrapper pattern as t3's `test-sync-push-contract.sh`.

- Solving the concurrent-commit cursor-race documented in t2 §4.3 —
  the future fix (`pg_xact_commit_timestamp` or

  ships with the documented limitation.

  plan 2.

- The "any layer dirty?" probe optimisation flagged as a follow-up in
  t2 §4.4 — that's a post-v2 polish.
