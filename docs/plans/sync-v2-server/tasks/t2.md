# t2: drift checker tooling + spec edit

**Type:** build

**Problem:** Without an automated check, a future developer adds a Drizzle
column to one of the eight entity schemas and ships it before the matching
server migration lands — the column has nowhere typed to live on the
server, sync silently drops it, and the bug is invisible until the next
hand-debug session. t1 §7 specifies a DB-to-DB introspection checker that
boots a local Supabase, applies every migration, materialises the Drizzle
schema into in-memory SQLite, and asserts column-by-column parity with the
exemptions in `apps/mobile/src/data/schema/sync-extras.json`. It also
asserts the structural triggers, RLS policies, no-CHECK rule, and the
topological FK order in `apps/mobile/src/sync/topo-order.ts` per t1 §7.7.
This task ships the checker plus the spec edit that documents the rule for
future agents.

**Outcomes:**

- `apps/mobile/scripts/check-sync-schema-drift.ts` implements the t1 §7.3
  algorithm:
  1. Calls `supabase/scripts/reset-local.sh` (or equivalent) to reset the
     local DB and apply every migration.
  2. Runs `drizzle-kit export --config=apps/mobile/drizzle.config.ts`,
     captures the SQL, executes it against a fresh in-memory
     `better-sqlite3` database.
  3. Introspects both sides per t1 §7.3 step 4a–b (Postgres via
     `information_schema.columns`, `pg_indexes`,
     `information_schema.referential_constraints`,
     `information_schema.key_column_usage`, `pg_policies`,
     `information_schema.check_constraints`, `information_schema.triggers`;
     SQLite via `pragma table_info`, `pragma index_list`,
     `pragma foreign_key_list`).
  4. Loads `apps/mobile/src/data/schema/sync-extras.json` for
     `exemptions.local_only_columns` and
     `exemptions.untyped_text_references`.
  5. For each entity in the **derived** entity list (every
     `app_public.*` table with an `owner_user_id` column — t1 §7.7's
     deliberate choice over hardcoding 8): walks every client column
     and asserts a typed server counterpart exists, walks every server
     column (minus the wire-envelope universals) and asserts a client
     counterpart, runs the §7.3 step 4f sanity checks (universal index,
     two structural triggers present, RLS policies present and bodies
     hashed against a fixture file at
     `apps/mobile/scripts/check-sync-schema-drift.fixtures.json`, no
     CHECK constraints, no `extras` column, no `deleted` boolean).
  6. Runs the t1 §7.7 topological FK order assertion against
     `TOPO_LAYERS` exported from `apps/mobile/src/sync/topo-order.ts`.
     (This file is created in this PR — see below.)
  7. Type compatibility map from t1 §7.3 (text↔text, integer mode:
     'timestamp_ms'↔int8, integer↔int4/int8, real↔float8/numeric).
  8. Exit code 0 (no drift), 1 (drift / regression), 2 (server-only
     column, warn-only by default; FAIL under `--strict`).
- The failure output for missing-server-counterpart matches the example
  in t1 §7.4 — includes the offending column, the full set of server
  columns, and the literal `alter table app_public.<entity> add column …`
  fix command.
- `apps/mobile/package.json` exposes the script as `npm run
  check:sync-drift` (forwards `--` args to the TS script). Local invocation
  works against a developer-started Supabase; CI invocation runs
  `supabase start` first if needed.
- `apps/mobile/scripts/check-sync-schema-drift.fixtures.json` ships with
  the SHA-256 hashes (whitespace-collapsed, comments-stripped) of the
  v2 `enforce_owner_user_id_immutable` function body and the four RLS
  policy expressions per entity table — checker compares actual against
  fixture per t1 §7.3 step 4f.
- `apps/mobile/src/sync/topo-order.ts` exports `TOPO_LAYERS` exactly as
  t1 §7.7 prescribes (Layer 0: `gyms`, `exercise_definitions`,
  `exercise_tag_definitions`; Layer 1: `sessions`,
  `exercise_muscle_mappings`; Layer 2: `session_exercises`; Layer 3:
  `exercise_sets`, `session_exercise_tags`). The drift checker imports
  this file; no other consumers in this PR (the sync engine that uses it
  ships in plan 2).
- `apps/mobile/src/data/schema/sync-extras.json` exists with the t1 §4.1
  shape: `exemptions.local_only_columns: ["local_dirty",
  "local_updated_at_ms"]` and `exemptions.untyped_text_references` with
  the one entry for `exercise_muscle_mappings.muscleGroupId` per t1
  §2.6.1. (The local-only columns themselves don't yet exist on the
  client — they ship in plan 2 — but the registration is harmless: the
  checker treats a registered name that doesn't appear anywhere as a
  no-op.)
- `scripts/quality-slow.sh` `run_backend()` is updated (t1 added the
  skip-path; this task converts the skip into a live invocation):
  `npm run check:sync-drift -- --strict`. Wired exactly as t1 §7.5
  shows.
- `docs/specs/05-data-model.md` gains a new "Client schema drift rule
  (Sync v2)" subsection with the **verbatim wording from t1 §8.2** —
  modifications to client schema files require a paired server
  migration deployed first, the drift checker enforces the rule, lists
  the exemptions, and excludes `muscle_groups` and runtime-scaffolding
  tables. Do not paraphrase; copy the t1 §8.2 markdown block.
- Self-test: in this PR, running `./scripts/quality-slow.sh backend`
  passes. Manually adding a `notes text` column to
  `apps/mobile/src/data/schema/exercise-sets.ts` (then reverting) shows
  the gate exits non-zero with a t1 §7.4-shaped message before the
  revert.
