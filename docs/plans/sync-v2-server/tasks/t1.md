# t1: clean-room migration — drop v1, create v2 schema

**Type:** build

**Problem:** v1's sync server surface — the M13 projection function, the
event-log tables, and the legacy entity tables from the user-scoped PK
redesign — is broken and must be replaced wholesale. v2's design
(`docs/plans/sync-v2/designs/t1.md`) commits to a typed mirror of the
client Drizzle schemas with composite `(owner_user_id, id)` PKs, deferrable
FKs, RLS, and zero CHECK constraints. Per the user-supplied constraints
this plan inherits, v1 objects are dropped — not migrated, not coexisting
— in the same migration that creates v2. The hosted DB is wiped after
merge; there is no data preservation requirement.

**Inputs:**

- `docs/plans/sync-v2/designs/t1.md` — authoritative for the v2 schema
  shape. Especially §1 (ground rules), §1.1 (LWW + undelete), §2 (per-entity
  mapping tables for all eight entities), §4.2 (the join tables also carry
  `deleted_at`), §5 (deferrable FKs — eight of them), §6 (RLS and the
  owner-immutability trigger), §10 (informative migration outline).
- Current state to survey for what to drop:
  - `supabase/migrations/20260306170000_m13_sync_events_ingest_projection.sql`
    — defines `sync_apply_projection_event`, `sync_events_ingest`,
    `sync_events_ingest_impl`, `sync_ingest_failure`,
    `sync_device_ingest_state`, `sync_ingested_events`.
  - `supabase/migrations/20260307120000_m14_sync_set_type_projection_patch.sql`
    — patches the M13 projection (drops the function and re-creates).
  - `supabase/migrations/20260505213500_relax_session_exercise_definition_fk.sql`
    — relaxes one of the eight FKs.
  - `supabase/migrations/20260514120000_user_scoped_pk_redesign.sql` —
    creates the current eight entity tables with their composite PKs,
    triggers, and policies. Survey this for the complete list of v1
    objects to drop.
  - `supabase/migrations/20260515190609_idempotent_missing_sync_deletes.sql`
    — touches the projection-event helpers.
  - `supabase/migrations/20260523120000_m15_gym_coordinates.sql` — adds
    `latitude` / `longitude` columns to v1 `gyms`. Survey for any v1

  - `supabase/hosted-clear-sync-state.sql`,

**Outcomes:**

- One new migration file under `supabase/migrations/<timestamp>_sync_v2_clean_room.sql`
  (timestamp later than the most recent existing migration). The file:
  1. **Drops every v1 sync server object** in dependency-safe order
     (functions and dependent triggers first, then tables with `cascade`):
     `app_public.sync_events_ingest`, `app_public.sync_events_ingest_impl`,
     `app_public.sync_apply_projection_event`,
     `app_public.sync_ingest_failure`,
     `app_public.sync_device_ingest_state`,
     `app_public.sync_ingested_events`, and the eight legacy entity tables
     (`app_public.session_exercise_tags`, `exercise_tag_definitions`,
     `exercise_muscle_mappings`, `exercise_sets`, `session_exercises`,
     `sessions`, `exercise_definitions`, `gyms`). Use `drop … if exists …
     cascade` so the migration is idempotent on a fresh DB that doesn't
     have v1.
  2. **Creates the eight v2 `app_public.<entity>` tables** with the
     columns, types, defaults, indexes, and composite PKs in t1 §2.1–§2.8.
     Includes the universal `owner_user_id` (NOT NULL, DEFAULT
     `auth.uid()`, references `auth.users(id) on delete cascade`),
     `client_updated_at_ms` (`bigint NOT NULL`), `server_received_at`
     (`timestamptz NOT NULL DEFAULT now()`), and `deleted_at` (`bigint
     NULL` per entity — present on all eight including the two join
     tables per t1 §4.2).
  3. **Declares the eight FKs** per t1 §5.2 as composite
     `(owner_user_id, <child_col>) → (owner_user_id, id)`,
     `DEFERRABLE INITIALLY DEFERRED`, with the `on delete` actions in
     t1 §5.2's table.
  4. **Adds the universal indexes**: `<table>_owner_received_idx` on
     `(owner_user_id, server_received_at)` for every entity, plus the
     per-entity btree indexes listed in t1 §2 (including the partial
     `WHERE deleted_at IS NULL` indexes for `session_exercises`,
     `exercise_sets`, `exercise_tag_definitions`).
  5. **Installs the two universal triggers** on every entity:
     `<table>_touch_server_received_at` (BEFORE UPDATE WHEN NEW IS
     DISTINCT FROM OLD, setting `NEW.server_received_at = now()`, per
     t1 §2's universal-trigger paragraph), and `<table>_owner_user_id_immutable`
     (BEFORE UPDATE, calling `app_public.enforce_owner_user_id_immutable()`).
     The function body matches t1 §6.3 verbatim — uses `IS DISTINCT FROM`
     and refuses when `auth.uid() IS NULL`.
  6. **Enables RLS and creates the four policies** per t1 §6.1 on every
     entity table — `<table>_owner_select`, `_owner_insert`,
     `_owner_update`, `_owner_delete`, all keyed on `owner_user_id =
     auth.uid()`. Policies use `to authenticated`.
  7. **Grants** `select, insert, update, delete` to `authenticated` and
     `service_role` on every entity table per t1 §10 step 6.
  8. The migration has no `down` / rollback path — the hard-cut model in
     t1 §1 makes the down side undefined. If Supabase migration framework
     requires a down section, supply a `RAISE EXCEPTION 'sync-v2 clean-room
     migration is irreversible';` body.
- `supabase db reset --local --yes` against the migration tree (after this
  PR's migration is added) completes without error, leaves zero v1
  objects in `pg_proc` / `pg_class`, and leaves all eight v2 tables
  present with the columns, indexes, FKs, RLS policies, and triggers
  enumerated above.
- A new SQL smoke test under `supabase/tests/sync-v2-schema-smoke.sql`
  (or shell wrapper under `supabase/tests/`) asserts, against the freshly
  reset DB:
  - All eight v2 tables exist in `app_public`.
  - Every v1 object name above is absent from `information_schema`.
  - Each table has RLS enabled and the four named policies.
  - Each table has the two universal triggers.
  - The eight FKs are present with `confdeltype` matching t1 §5.2 and
    `condeferrable = true`, `condeferred = true`.
  - No CHECK constraints exist on any of the eight entity tables.
- `./scripts/quality-slow.sh backend` is updated in this same PR with the
  drift-checker wiring described in t1 §7.5 (the conditional block that
  skips when `apps/mobile/scripts/check-sync-schema-drift.ts` is missing —
  the script doesn't exist yet, so this PR adds the **skip path only**; the
  actual `npm run check:sync-drift` invocation goes live once t2 lands its
  script). Confirms the wiring is present so t2 doesn't have to re-touch
  the slow gate.

**Output artifact:**

- `supabase/migrations/<timestamp>_sync_v2_clean_room.sql` — single SQL

- `supabase/tests/sync-v2-schema-smoke.sql` (or `.sh` wrapper named
