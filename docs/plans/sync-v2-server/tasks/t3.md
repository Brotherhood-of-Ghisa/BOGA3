# t3: `sync_push` RPC

**Type:** build

**Problem:** v2 replaces the M13 projection-function entry point with a thin
PostgREST RPC, `sync_push`, that takes a batch of typed-row payloads, runs
one transaction with deferrable FKs, upserts each row under per-row LWW,
and returns a single batch ack. The full wire shape and semantics are
pinned in `docs/plans/sync-v2/designs/t2.md` §3 and the schema-level
constraints in `docs/plans/sync-v2/designs/t1.md`; this task writes the
RPC body and the focused unit/contract tests that exercise it.

**Inputs:**

- t1 (this plan) merged. The eight v2 entity tables, deferrable FKs, RLS,
  and `enforce_owner_user_id_immutable` exist on the server.
- `docs/plans/sync-v2/designs/t2.md` §1 (endpoint spec, PostgREST bigint
  serialization), §2.1 (Entity envelope — `type`, `id`,
  `client_updated_at_ms`, `fields`), §2.2 (`FK_VIOLATION`, `AUTH_REQUIRED`,
  `INTERNAL` error envelope), §3.1 (request shape, `entities.length
  1..200`), §3.2 (single transaction with `SET CONSTRAINTS ALL
  DEFERRED`), §3.3 (what goes in `fields`), §3.5 (success response
  `{ok: true, server_received_at}` — single `now()` per transaction),
  §3.6 (LWW makes retry idempotent), §3.7 (200-row batch cap).
- `docs/plans/sync-v2/designs/t1.md` §1 (no server validation, future-clock
  clamp to `now_ms() + 5 minutes`), §1.1.1 (the LWW rule:
  `incoming.client_updated_at_ms > stored.client_updated_at_ms`
  ⇒ overwrite every column in `fields`; equal or less ⇒ no-op), §2
  (per-entity typed column lists for the upsert column projection), §5
  (deferrable FK declarations are the structural anchor for the
  no-orphan-child guarantee).
- Current state to survey:
  - `supabase/tests/session-sync-api-contract.sh` and
    `supabase/tests/sync-events-ingest-contract.sh` — existing curl-based
    contract test patterns; t3's tests follow the same shape.
  - `supabase/scripts/auth-fixture-constants.sh` — fixture user UUIDs /
    JWTs for the multi-user RLS tests.

**Outcomes:**

- The migration ships the function `app_public.sync_push(payload jsonb)`
  (or a SQL/PLPGSQL body — the implementor picks; the wire-level
  behaviour is what counts) marked `language plpgsql security invoker`
  (so RLS evaluates as the caller per t1 §6.2, NOT `security definer`).
  Granted `execute … to authenticated`. Exposed at `POST
  /rest/v1/rpc/sync_push`.
- Request validation (structural only per t1 §1, t2 §2.2):
  - `payload->'entities'` is a JSON array; otherwise raise
    `INTERNAL`-shaped error.
  - Length is `1..200`; otherwise raise `INTERNAL`-shaped error.
  - `auth.uid()` is non-null; otherwise raise `AUTH_REQUIRED`-shaped
    error.
  - No further validation — types, ranges, enums, lengths are NOT
    checked. The drift checker (t2) catches schema-level malformation
    at PR time.
- Transaction body:
  1. `SET CONSTRAINTS ALL DEFERRED;`
  2. For each entity in the payload, dispatch on `type` to the
     correct `app_public.<entity>` upsert. The upsert clamps
     `incoming.client_updated_at_ms` to `least(incoming, (extract(epoch
     from now()) * 1000)::bigint + 5*60*1000)` per t1 §1's future-clock
     clamp, and uses the LWW `ON CONFLICT … DO UPDATE … WHERE
     incoming.client_updated_at_ms > app_public.<entity>.client_updated_at_ms`
     predicate per t1 §1.1.1. Every typed column from `fields` is written
     verbatim on overwrite, including `deleted_at` (per t1 §1.1 — there
     is no special delete path).
  3. The implicit COMMIT at function return re-checks all FKs. Failure
     surfaces as the `FK_VIOLATION` error envelope per t2 §2.2 — the
     entire transaction rolls back; no partial state lands.
- Response: `jsonb` `{"ok": true, "server_received_at": "<iso8601 with
  ms>"}` per t2 §3.5. `server_received_at` is the transaction's `now()`,
  consistent across all rows in the batch.
- PostgREST integer serialization: the function returns bigint epoch-ms
  values as JSON integers (not strings). If a Supabase role config
  change is needed (`db-aggregates-enabled=true` or similar per t2 §1),
  ship it in the same migration; otherwise verify the default
  serialization is already integers and document in the PR body.
- Contract tests under `supabase/tests/sync-push-contract.sh` (and any
  helpers consistent with the existing `session-sync-api-contract.sh`
  pattern) cover:
  - **Happy path single-row** — POST one `gyms` entity for user A,
    response is `{ok: true, server_received_at: <iso>}`, row visible to A
    via a direct `select` (RLS as A), not visible to B (RLS as B).
  - **Happy path multi-layer batch** — POST a batch including
    `exercise_definitions` (Layer 0) → `session_exercises` (Layer 2)
    → `exercise_sets` (Layer 3) **out of topological order in the array**;
    transaction succeeds because of deferrable FKs.
  - **LWW newer wins** — POST a `gyms` row at T=100, then again at
    T=200 with a different `name`; the second POST overwrites every
    column.
  - **LWW older no-op** — POST a `gyms` row at T=200, then again at
    T=100 with a different `name`; the second POST returns `ok: true`
    but the stored row is unchanged.
  - **LWW retry idempotent** — POST identical payload twice in quick
    succession; both return `ok: true`; the stored row matches the
    payload.
  - **Soft-delete via `deleted_at`** — POST a row with `deleted_at` set
    and a newer `client_updated_at_ms`; verify the row's `deleted_at` is
    non-null after the call.
  - **Undelete via LWW** — POST a row with `deleted_at = null` and a
    `client_updated_at_ms` strictly greater than the prior tombstone's;
    verify `deleted_at` is null after the call (t1 §1.1.2 scenario B).
  - **Future-clock clamp** — POST a row with `client_updated_at_ms`
    set to `now()+1day`; the stored `client_updated_at_ms` is
    `<= now()+5min`. (Query the stored value via a service-role select
    to confirm clamping.)
  - **FK closure: parent in same batch** — POST a `sessions` row plus
    its `session_exercises` child in one batch (child-before-parent in
    the array order); transaction commits.
  - **FK closure: missing parent** — POST a `session_exercises` row
    whose parent `sessions` row is neither in the batch nor on the
    server; transaction rolls back with the `FK_VIOLATION` envelope per
    t2 §2.2; verify no rows landed.
  - **RLS: cross-owner injection** — POST a payload where the JWT is
    user A but the `fields` would imply user B (in practice, the RPC
    derives `owner_user_id` from `auth.uid()`, so this should fail
    cleanly or land under A — the test pins which). Per t1 §6.1, RLS's
    `with check (owner_user_id = auth.uid())` policy + the immutability
    trigger blocks the cross-owner write path.
  - **Batch size bounds** — empty array rejected; 201-row batch
    rejected; 200-row batch accepted.
  - **AUTH_REQUIRED** — POST without a JWT returns the `AUTH_REQUIRED`
    envelope.
- `./scripts/quality-slow.sh backend` invokes the new test script
  (wire it into the same `run_backend()` block that already runs
  `test-sync-api-contract.sh`; this PR adds a parallel
  `test-sync-push-contract.sh` invocation if a wrapper script is
  needed for parity with the existing pattern).
