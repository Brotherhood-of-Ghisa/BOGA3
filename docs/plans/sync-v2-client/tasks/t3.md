# t3: manual-wipe procedure documentation

**Type:** build (docs-only)

**Problem:** The v2 client assumes a clean local SQLite (no v1
`sync_outbox_events` / `sync_delivery_state` tables; no entity rows
with stale `local_dirty = 0` semantics; no `sync_runtime_state` row in
the v1 shape). Rather than ship an auto-wipe code path with a version
marker — which adds first-boot complexity, has subtle ordering
constraints relative to the Drizzle migrator, and was the source of
two open design questions (d1: storage; d2: migration approach
coupling) — plan 2 ships the v2 build assuming a one-time **manual**
wipe by every dev, TestFlight tester, and (eventually) production
user. This task is a short documentation page covering the wipe
procedure on iOS simulator, Android emulator, and physical devices,
plus the TestFlight-specific guidance ("delete the v1 build first;
do not update in place"). No code, no marker, no app-side logic.

**Inputs:**

- The plan's `## Goal` paragraph (clean-slate assumption).
- The plan's `## Outcomes` block — specifically the manual-wipe
  outcome that replaces the prior version-marker outcome.
- `apps/mobile/README.md` (or the equivalent dev-onboarding doc) —
  for a cross-link from project docs into the wipe procedure.
- Existing TestFlight rollout notes if any exist under
  `docs/plans/sync-v2-client/` or `apps/mobile/` (none currently;
  this task creates the first one for plan 2).

**Outcomes:**

- A new doc at `docs/plans/sync-v2-client/manual-wipe.md` exists,
  short (well under 200 lines), with these sections:
  - **Why this is needed** — one paragraph explaining the v1→v2
    schema break and why an automated migration is intentionally
    not shipped.
  - **iOS Simulator** — two paths:
    (a) Xcode → Device → Erase All Content and Settings (nukes
        the whole simulator state); or
    (b) Delete the app from the simulator's home screen (long-press
        → remove app), which removes the app sandbox including the
        SQLite file.
  - **Android Emulator** — two paths:
    (a) Settings → Apps → BOGA3 → Storage → Clear Storage; or
    (b) AVD Manager → wipe data on the emulator (nuclear option).
  - **Physical device (iOS and Android)** — uninstall and
    reinstall. Both platforms delete the app sandbox (including
    the SQLite file) on uninstall, so a fresh install starts
    clean.
  - **TestFlight** — explicit guidance for testers receiving the
    v2 build: **delete the existing v1 build from the device
    first, then install the v2 build from TestFlight. Do NOT
    update in place** — updating in place preserves the app
    sandbox and the v2 build will boot against a v1-shaped local
    DB with undefined behaviour.
  - **What "undefined behaviour" means** — one sentence: rows may
    fail to sync (dirty bits not set after the t2 migration),
    cursor state may be missing, and the cycle may push entities
    the server has never seen and pull nothing back. Diagnosis
    almost always lands on "the local DB was not wiped" — wipe
    and re-bootstrap fixes it.
- A cross-link from `apps/mobile/README.md` (or the closest
  equivalent dev-onboarding section — surface this in the PR body
  if a different file is chosen) to the manual-wipe doc, in a
  section titled e.g. **"Upgrading from v1 sync (one-time wipe)"**.
- No code changes anywhere. No app-side marker. No first-boot
  detection logic. No new dependencies. The `apps/mobile/`
  TypeScript surface is untouched by this task.
- `npm run lint && npm run typecheck && npm run test` from
  `apps/mobile/` still passes (it was already passing; this task
  is doc-only so the gate is verifying no accidental code edits
  slipped in).

**Output artifact:**

- New file `docs/plans/sync-v2-client/manual-wipe.md`.
- Edit to `apps/mobile/README.md` (or the closest equivalent —
  the t3 builder picks the most discoverable location and surfaces
  the choice in the PR body's `## Deviations from card` if it
  isn't `apps/mobile/README.md`).

**Out of scope:**

- ANY app-side code (no marker module; no boot-time check; no
  wipe function; no migration; no test). The whole point of this
  task is to NOT ship code.
- The dev-only `wipe-local` button (t9 owns) — that uses the
  existing `resetLocalAppData()` from `apps/mobile/src/data/bootstrap.ts`
  for dev iteration and is unrelated to the v1→v2 manual wipe
  this doc covers.
- Server-side wipe affordances (t9's `wipe-remote-for-me`).
- Tests in CI for the manual-wipe procedure — it's a runbook
  step performed by humans on the device side, not a code path.
  tFINAL explicitly notes this in its `Out of scope`.
- Renaming or moving the doc later — if plan 3 (`sync-v2-launch`)
  needs a more discoverable home (e.g. a top-level `docs/runbooks/`
  directory), it can move the file then.
