# t3: version-marker + wipe-on-v2-boot

**Type:** build

**Problem:** Devices upgrading from a v1 build of `apps/mobile` carry
the v1 `sync_outbox_events` / `sync_delivery_state` tables in their
on-device SQLite, plus entity rows with `local_dirty` / `local_updated_at_ms`
unset, plus a `sync_runtime_state` row in the v1 shape. Forward
migrations alone cannot recover from this safely: the v2 cycle (t6)
assumes the dirty bit is the source of truth for "this row needs a
push," but every existing row on a v1-upgraded device has
`local_dirty = 0` after the t2 migration adds the column with default
0, which would mean "everything is in sync with the server" — which
is wrong (the v1 row was sync'd via the outbox, but v2's server is a
clean-room redeploy with no rows). The cleanest fix is to wipe the
local DB on the first v2 boot of any device that had a v1 install,
gated by a one-shot version marker so the wipe never re-runs.
`apps/mobile/src/data/bootstrap.ts` already exposes
`resetLocalAppData()` (`deleteDatabaseAsync` + recreate); t3 wires it
behind the marker.

**Inputs:**

- t2 merged (v2 entity schemas + reshaped `sync_runtime_state` are
  in place; Drizzle migrator will create the v2 shape when the wipe
  recreates the DB).
- `designs/d1.md ## Decision` — binding contract for the
  version-marker storage mechanism.

> Updated from d1: see `designs/d1.md ## Decision` for the
> version-marker storage contract this task implements.

- `apps/mobile/src/data/bootstrap.ts` — current `resetLocalAppData()`
  + `bootstrapLocalDataLayer()` shape. The wipe path closes the
  current handle, calls `deleteDatabaseAsync(LOCAL_DATABASE_NAME)`,
  then re-opens and runs migrations.
- `apps/mobile/app/_layout.tsx` — the call site that ultimately
  invokes `bootstrapLocalDataLayer()`. t3 inserts a marker-check
  step before this call.
- `apps/mobile/src/data/dev-reset.ts` — informs the "wipe semantics"
  (which tables / which markers).

**Outcomes:**

- A new module under `apps/mobile/src/data/` (exact filename per
  the d1 storage decision — e.g. `v2-boot-marker.ts`) exposes:
  - `hasV2BootCompleted(): Promise<boolean>` — reads the marker
    from the storage chosen by d1.
  - `markV2BootCompleted(): Promise<void>` — writes the marker
    idempotently. Re-calls are no-ops.
  - `clearV2BootMarker(): Promise<void>` — explicit dev-mode
    helper (called by t9's `wipe-local`).
- A new bootstrap step in `apps/mobile/src/data/bootstrap.ts` (or a
  wrapper used by `app/_layout.tsx`) implements the gate:
  ```
  if (!(await hasV2BootCompleted())) {
    await resetLocalAppData();        // delete + recreate via Drizzle migrator
    await markV2BootCompleted();
  }
  await bootstrapLocalDataLayer();
  ```
  The exact placement (inside `bootstrapLocalDataLayer` or one level
  up in `_layout.tsx`) follows d1's recommendation — the marker
  storage chosen by d1 may impose ordering constraints (e.g. if the
  marker reads synchronously the gate can be inline; if async it
  must await before the Drizzle migrator runs).
- The wipe is **idempotent on a fresh install**: a device with no
  prior v1 state still sets the marker, just without an actual
  wipe operation costing anything observable (Drizzle migrator
  runs from idx 0 once, on whatever shape Drizzle finds — empty
  on a fresh install, v1 on an upgrade after the delete).
- The marker is wipeable via `clearV2BootMarker()` from t9's
  dev-only `wipe-local` flow.
- A Jest test exercises the gate in four scenarios:
  1. **Fresh install** (no DB file, no marker): wipe is a no-op
     (no file to delete); marker is set; `bootstrapLocalDataLayer`
     produces the v2 shape.
  2. **v1 upgrade** (DB file with v1 tables, no marker): wipe
     deletes the file; marker is set; `bootstrapLocalDataLayer`
     produces the v2 shape; no v1 tables exist after the wipe.
  3. **Second launch after wipe** (DB file with v2 tables, marker
     set): wipe does NOT run; `bootstrapLocalDataLayer` is the only
     call.
  4. **Dev-mode clear** (DB file with v2 tables, marker set, then
     `clearV2BootMarker()`): a subsequent boot re-wipes and re-sets
     the marker.
- `npm run lint && npm run typecheck && npm run test` passes from
  `apps/mobile/`.
- The PR body asserts `test:e2e:ios:smoke` and `test:e2e:ios:data-smoke`
  pass locally (per the `## Orchestration` rule binding UI-touching
  tasks).

**Output artifact:**

- New file `apps/mobile/src/data/<v2-boot-marker-filename>.ts`
  (exact name per d1).
- Edit to `apps/mobile/src/data/bootstrap.ts` and/or

- New Jest test under `apps/mobile/app/__tests__/v2-boot-marker.test.ts`
  (or similar) covering the four scenarios above.
- If d1's decision adds a dependency or a plugin entry to
