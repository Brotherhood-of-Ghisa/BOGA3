# d2: decide sync_runtime_state migration approach

**Type:** design

**Problem:** The plan's t2 must reshape the existing `sync_runtime_state`
Drizzle table:

- **Drop** v1 columns: `is_enabled`, `bootstrap_user_id`,
  `last_bootstrap_error`, `last_bootstrap_attempt_at`,
  `seeds_applied_at`.
- **Add** v2 columns per t2 §9: `pull_cursor` (JSON / text, default
  `'{}'`), `last_emitted_ms` (integer default `0`),
  `applied_seed_migration_app_version` (integer default `0`).
- **Keep** `bootstrap_completed_at` (still a useful UX surface per t2 §5).

SQLite (and hence Drizzle's SQLite migrator) does not support
`ALTER TABLE DROP COLUMN` in older versions; the canonical recipe is
the "12-step procedure" (PRAGMA foreign_keys=OFF; CREATE TABLE
`__new_*`; INSERT … SELECT …; DROP TABLE; RENAME; PRAGMA
foreign_keys=ON) — see `apps/mobile/src/data/migrations/index.ts`
m0004 / m0013 for two existing examples. But this plan also ships
t3's "wipe the local DB on first v2 boot," which makes the migration
itself partly redundant — every device that ships plan 2 either (a)
has v1 state, in which case t3 wipes the DB before any migrator
runs, or (b) is a fresh install, in which case there is no v1
`sync_runtime_state` to migrate.

There are at least three real options:

1. **Full additive + 12-step table swap in the same Drizzle migration**
   (the safe-conservative path).
2. **Drop-and-recreate `sync_runtime_state` directly** (assumes the
   t3 wipe always runs first on devices with v1 data — coupling
   migration safety to t3 correctness).
3. **Additive only: add new columns, leave v1 columns dead** (lowest
   churn, but `seedsAppliedAt` stays as a dead column on the table
   forever and the drift checker / spec needs an exemption).

Each option has materially different blast radius, reversibility,
test surface, and coupling with t3. Two reasonable builders would
ship different migrations from the t2 card alone.

**Inputs:**

- The plan `## Outcomes` block — specifically the
  `sync_runtime_state` rewrite outcome.
- `apps/mobile/src/data/schema/sync-runtime-state.ts` — current
  v1 schema.
- `apps/mobile/src/data/migrations/index.ts` — the existing migration
  tree. Particularly m0004 (the `sessions` table swap, a worked
  12-step procedure example) and m0013 (the `gyms` table swap).
- `apps/mobile/src/data/exercise-catalog-seeds.ts` — consumer of
  `seedsAppliedAt`; will migrate to
  `applied_seed_migration_app_version` per d2 + t2.
- `docs/plans/sync-v2/designs/t2.md` §8.3, §9.2, §9.3 — the
  v2-canonical column shape (binding contract).
- `docs/plans/sync-v2-client/tasks/t3.md` — t3's wipe semantics
  (decision in d1).
- Downstream task blocked: t2 (and indirectly t3, which depends on t2).

**Constraints:**

- The decision MUST work for both (a) devices upgrading from a v1
  build with existing `sync_runtime_state` rows and (b) fresh
  installs with no `sync_runtime_state` row.
- The decision must compose with t3's wipe — explicitly state the
  ordering: does the migration run BEFORE the wipe (always), AFTER
  the wipe (always), or both depending on the version-marker state?
- Adding columns is cheap; dropping columns is what makes this
  non-trivial.
- Drizzle's `migrate()` (`drizzle-orm/expo-sqlite/migrator`) runs all
  pending migrations sequentially on `bootstrapLocalDataLayer()`
  (`apps/mobile/src/data/bootstrap.ts`).
- `pull_cursor` storing JSON: t2 §9.2 says "jsonb (or text holding
  JSON)." SQLite doesn't have `jsonb`; the column is `text`. Drizzle
  has a `mode: 'json'` text helper that auto-serialises — the design
  should pick which (text + manual JSON.parse vs Drizzle's `mode: 'json'`).

**Decision authority:** designer recommends; the human decides if the
"trust t3's wipe" coupling is acceptable.

**Outcomes:** `docs/plans/sync-v2-client/designs/d2.md` exists,
follows the design protocol structure (Question, Constraints, Options
considered with ≥ 2 options + tradeoffs, exactly one Decision,
Downstream impact section listing t2). Downstream card `tasks/t2.md`
is updated with a pointer marker (`> Updated from d2: see
designs/d2.md ## Decision`) and its `Inputs` field cites the design
doc. Doc < 200 lines.

**Output artifact:** `docs/plans/sync-v2-client/designs/d2.md` and a
pointer-only edit to `docs/plans/sync-v2-client/tasks/t2.md`.

**Out of scope:**

- The Drizzle migration SQL itself (t2 owns the build, per d2's
  decision).
- The entity-table local-column additions (`local_dirty`,
  `local_updated_at_ms`, `deleted_at` on the five schemas) — those
  are simple `ALTER TABLE ADD COLUMN` and don't need a design.
- The `pull_cursor` JSON shape — that's pinned by t2 §9.2 already.
