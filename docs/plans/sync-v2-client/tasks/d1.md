# d1: decide version-marker storage

**Type:** design

**Problem:** t3 wipes the local SQLite database on first v2 boot to
drop the v1 tables (`sync_outbox_events`, `sync_delivery_state`) and
let the Drizzle migrator recreate the v2 shape clean. The wipe is
gated by a one-shot marker that must:

- Survive the wipe itself (otherwise the wipe re-runs every launch).
- Survive an app restart and a process kill.
- Be readable synchronously enough that the gate fires before the
  rest of app init touches the DB.
- Be writable atomically (a half-written marker that thinks the wipe
  already ran would leave a v1 DB with a "completed" flag).
- Be wipeable by the dev-only `wipe-local` affordance in t9 so a dev
  can replay the wipe on demand.

The candidates carry different tradeoffs (iOS keychain persistence
across uninstall in SecureStore; AsyncStorage availability under
Expo SDK 54; a tiny Drizzle "marker" table outside the wiped scope
conflicting with the wipe semantics; an Expo `FileSystem` JSON file).
The decision picks ONE storage; t3 implements against it. This is a
real design question — two reasonable builders would ship materially
different markers given only the t3 problem statement.

**Inputs:**

- The stub `docs/plans/sync-v2-client/plan.md ## Outcomes` (the
  "one-shot version-marker" outcome).
- `apps/mobile/src/data/bootstrap.ts` — current `LOCAL_DATABASE_NAME
  = 'scaffolding-local.db'` and `deleteDatabaseAsync` /
  `bootstrapLocalDataLayer` shape.
- `apps/mobile/package.json` — `expo-secure-store@~15.0.8` is already
  installed; `@react-native-async-storage/async-storage` is NOT
  installed (would need to be added).
- `apps/mobile/src/auth/storage.ts` — existing `expo-secure-store`
  usage in the codebase (reference for keychain-suite naming
  conventions).
- `apps/mobile/app.config.ts` — already has `"expo-secure-store"` in
  the plugins array.
- Downstream task blocked: t3.

**Constraints:**

- Must work in iOS dev-client builds (`com.phano.boga3.dev`) and
  TestFlight / App Store builds without prebuild churn beyond what
  Expo SDK 54 plugins already provide.
- Must NOT add a Drizzle table inside the wiped scope (any table the
  Drizzle migrator manages is wiped by t3's `deleteDatabaseAsync`).
- Must be readable BEFORE the Drizzle migrator runs (the migrator is
  what creates the v2 schema; if the marker lives in the DB we
  cannot read it before recreate without ordering gymnastics).
- iOS keychain quirks matter: SecureStore on iOS persists across
  uninstall (keychain survives), which would make a "fresh install
  after uninstall" never re-wipe. Discuss whether that's a feature
  or a bug.
- Wipe affordance (t9) must be able to clear the marker so a dev can
  re-run the wipe path on demand.
- Decision must compose with the dev-only seed reset path in
  `apps/mobile/src/data/dev-reset.ts` (which currently calls
  `__clearSeedsAppliedMarkerForReset`).

**Decision authority:** designer recommends; the human decides if
the iOS-uninstall-persistence tradeoff goes the unexpected way.

**Outcomes:** `docs/plans/sync-v2-client/designs/d1.md` exists,
follows the design protocol structure (Question, Constraints, Options
considered with ≥ 2 options + tradeoffs, exactly one Decision,
Downstream impact section listing t3). Downstream card `tasks/t3.md`
is updated with a pointer marker (`> Updated from d1: see
designs/d1.md ## Decision`) and its `Inputs` field cites the design
doc. Doc < 200 lines.

**Output artifact:** `docs/plans/sync-v2-client/designs/d1.md` and a
pointer-only edit to `docs/plans/sync-v2-client/tasks/t3.md`.

**Out of scope:**

- The wipe mechanism itself (t3 owns).
- The dev-only wipe-local affordance (t9 owns).
- The exact key string / value format of the marker — that is t3's
  call once the storage is picked.
