# t7: scheduler implementation per t4 §2

**Type:** build

**Problem:** The scheduler implements the t4 §2 four-state machine
(OFFLINE / LONG_TIMEOUT / SHORT_TIMEOUT / RUNNING) that decides
WHEN to call `runSyncCycle()` (t6). The contract is a literal
encoding of the two transition tables in t4 §2.2 (external inputs:
`request sync` / `go online` / `go offline`) and t4 §2.3 (internal
events: `timer fires` / `cycle ends`). NetInfo is the sole
authority on online state per t4 §3.4. `AppState` `background →
active` emits `requestSync` per t4 §2.5. There is no manual "Sync
now" affordance in v2 per t4 §2.5 (that's a plan 3 surface).

**Inputs:**

- t6 merged (`runSyncCycle` available).
- `docs/plans/sync-v2/designs/t4.md` §2 (the four states + the two
  transition tables), §3 (NetInfo wiring), §5.1 (logging requirements),
  §6.1 (the per-cell test list this task must implement).
- `@react-native-community/netinfo` — added as a dependency by t8
  (BG-task task adds all the deps together via `expo install`).
  **Order constraint:** t7's PR can add the dependency itself
  (`expo install @react-native-community/netinfo`) without
  conflicting with t8, since t8 adds three OTHER deps
  (`expo-background-task`, `expo-task-manager`, `expo-network`).
- `apps/mobile/app/_layout.tsx` — t7 adds a sync-init call back
  in (t1 removed the v1 wiring); the call is the scheduler boot
  (e.g. `startSyncScheduler()`).
- `apps/mobile/src/sync/topo-order.ts` — unused by t7 directly
  (the scheduler doesn't touch layer ordering) but imported by
  t6's cycle.

**Outcomes:**

- A new file `apps/mobile/src/sync/scheduler.ts` exports:
  - `startSyncScheduler(): void` — wires the NetInfo listener,
    the `AppState` listener, and arms the initial state
    (OFFLINE per t4 §2.1).
  - `stopSyncScheduler(): void` — tears down listeners; cancels
    any armed timer. Used by Jest setup / teardown.
  - `requestSync(): void` — the single external entry point per
    t4 §2.5. No `reason` parameter. Called by repo writes (the
    repos can call it via a sync-layer adapter that t7 exposes
    or via direct import — pick the simplest), by the AppState
    listener on `background → active`, and by the cold-launch
    bootstrap.
  - `__getSchedulerStateForTests(): State` — module-internal
    test inspector returning the current state and the next
    timer deadline (if armed).
- The four states (`OFFLINE` / `LONG_TIMEOUT` / `SHORT_TIMEOUT` /
  `RUNNING`) are literal in the code — discriminated-union types
  or an enum — not buried in a `deadline: number | null` field
  (per t4's "make the four states (OFFLINE, LONG_TIMEOUT,
  SHORT_TIMEOUT, RUNNING) and the transition tables literal in
  the code — not buried in conditional branches" in t4 §8).
- `SHORT_INTERVAL = 1000` and `LONG_INTERVAL = 60_000` per t4 §2.6.
  No `isDevMode()` / `__DEV__` override on these constants per t4
  §2.6 ("Dev reflects production for both intervals").
- The external-input transition table (t4 §2.2):
  - OFFLINE × request sync → no-op.
  - OFFLINE × go online → SHORT_TIMEOUT, arm SHORT_INTERVAL.
  - OFFLINE × go offline → no-op.
  - LONG_TIMEOUT × request sync → cancel timer, →
    SHORT_TIMEOUT, arm SHORT_INTERVAL.
  - LONG_TIMEOUT × go online → no-op (already online).
  - LONG_TIMEOUT × go offline → cancel timer, → OFFLINE.
  - SHORT_TIMEOUT × request sync → **no-op (DO NOT restart the
    timer — heart of the design per t4 §2.2 final paragraph)**.
  - SHORT_TIMEOUT × go online → no-op.
  - SHORT_TIMEOUT × go offline → cancel timer, → OFFLINE.
  - RUNNING × * → no-op for all external inputs.
- The internal-transition table (t4 §2.3):
  - LONG_TIMEOUT / SHORT_TIMEOUT × timer fires → RUNNING, start
    cycle.
  - RUNNING × cycle ends (online) → LONG_TIMEOUT, arm
    LONG_INTERVAL.
  - RUNNING × cycle ends (offline) → OFFLINE.
  - The "currently online" check at cycle-end consults the
    latest NetInfo projection per t4 §2.3 final paragraph.
- NetInfo wiring per t4 §3.2:
  - One `addEventListener` subscription projects
    `isInternetReachable === true` to a boolean.
  - On projection change only, emits `go online` / `go offline`.
  - Initial projection is `false` (start in OFFLINE per t4 §3.3
    last row — wait for the first definitive `true`).
- `AppState` listener per t4 §1.1 / §2.5:
  - On `background → active` → `requestSync()`.
  - On `inactive` transitions → no-op.
  - On `active → background` → no-op.
- Cold-launch trigger per t4 §2.5: a single `requestSync()` call
  at the end of `_layout.tsx`'s bootstrap sequence; if still
  OFFLINE the call is a no-op (the first NetInfo `true` will
  take the scheduler to SHORT_TIMEOUT instead).
- Structured logging per t4 §5.1: every transition logs the
  triggering event, the from-state, the to-state, the newly
  armed timer deadline (if any), and a millisecond timestamp.
  Use the existing `logEvent` helper at
  `apps/mobile/src/logging/logEvent.ts`.
- Jest tests cover, **one per cell of the §2.2 / §2.3 transition
  tables** (≥ 20 cells per t4 §6.1):
  - Every external-input cell × every state.
  - Every internal-event cell × every state.
  - Plus the integration tests t4 §6.1 calls out: foreground edge
    via the `AppState` listener; NetInfo projection emits inputs
    only on change; logging covers every transition exercised.
- `apps/mobile/app/_layout.tsx` calls `startSyncScheduler()` in
  the bootstrap `useEffect`. Teardown calls `stopSyncScheduler()`.
- `npm run lint && npm run typecheck && npm run test` passes from
  `apps/mobile/`.
- `test:e2e:ios:smoke` and `test:e2e:ios:data-smoke` pass locally;
  the PR body's Standard checklist asserts both with the built
  git sha.
