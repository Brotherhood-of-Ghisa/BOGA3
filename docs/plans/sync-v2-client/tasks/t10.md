# t10: drop inlined SQL from migrations/index.ts

**Type:** build

**Problem:** Today
`apps/mobile/src/data/migrations/index.ts` carries a hand-copied SQL
string under `migrations.m0000` that duplicates the authoritative
output of `drizzle-kit generate` at
`apps/mobile/drizzle/0000_living_bucky.sql`. Every regeneration forces
a manual copy-paste between the two; silent drift between them is
undetected by `npm run check:sync-drift` (the drift checker compares
client Drizzle schemas vs the server, not the runtime migration bundle
vs `drizzle/*.sql`). This was flagged during t2 review — see plan-2
Deviations log entry for t2. t10 removes the duplication so the
runtime bundle is a thin wrapper around the generated SQL file, and a
single `npm --prefix apps/mobile run db:generate` invocation
regenerates both.

**Inputs:**

- t2 merged (the v2-canonical baseline SQL now lives at
  `apps/mobile/drizzle/0000_living_bucky.sql`).
- `apps/mobile/src/data/migrations/index.ts` — the current bundle
  carrying inlined SQL; to be rewritten as a thin wrapper.
- `apps/mobile/drizzle/0000_living_bucky.sql` — the authoritative
  baseline SQL produced by `drizzle-kit generate`.
- `apps/mobile/drizzle/meta/_journal.json` — Drizzle's per-config
  journal; the t10 bundle uses this (or a re-derived equivalent) as
  the `journal.entries` source.
- `apps/mobile/drizzle.config.ts` — Drizzle Kit config; Option C
  changes this file. Option B leaves it alone.
- `apps/mobile/package.json` — the `db:generate` script entry; both
  options touch this so a single command regenerates the bundle.
- `apps/mobile/app/__tests__/domain-schema-migrations.test.ts` — the
  squash-invariant test that asserts a single `m0000` entry. The
  assertion shape stays the same; only the import path may shift if
  Option C lands.
- Drizzle Kit version pinned in `package.json` (currently
  `drizzle-kit ^0.31.5`, `drizzle-orm ^0.44.6`) — Option C needs the
  installed Drizzle Kit to support the Expo `driver: 'expo'` bundle
  output.

**Outcomes:**

- `apps/mobile/src/data/migrations/index.ts` contains **zero** raw
  DDL tokens — no `CREATE TABLE`, `ALTER TABLE`,
  `CREATE INDEX`, `CREATE UNIQUE INDEX`, or `statement-breakpoint`
  string literals — and is a thin wrapper around a generated bundle
  file. The exported symbol name (`localRuntimeMigrations`) and its
  runtime shape (`{ journal: { entries: [...] }, migrations: { m0000:
  <string> } }`) match `apps/mobile/node_modules/drizzle-orm/expo-sqlite/migrator.d.ts`'s
  `MigrationConfig` exactly, so existing call sites of
  `migrate(db, localRuntimeMigrations)` keep compiling.
- A single command — `npm --prefix apps/mobile run db:generate` —
  regenerates **both** `apps/mobile/drizzle/0000_<name>.sql` AND the
  bundle file (or, under Option C, both in one Drizzle Kit pass). No
  second manual copy step is required and no comment in the codebase
  instructs anyone to copy SQL by hand.
- The bundle file IS committed (Drizzle's Expo pattern ships it in
  the JS bundle; the script-based Option B bundle is likewise
  committed so cold-start doesn't depend on regeneration).
- `apps/mobile/app/__tests__/domain-schema-migrations.test.ts` still
  passes verbatim, including the squash-invariant
  `expect(Object.keys(localRuntimeMigrations.migrations)).toEqual(['m0000'])`
  and the journal-entry count check. The test's import path may
  change ONLY if Option C lands and the bundle lives at a different
  module path; the assertion shape is untouched.
- `./scripts/quality-fast.sh frontend && (cd apps/mobile && npm run
  check:sync-drift -- --strict)` exits zero (drift checker matters —
  if the bundle has different SQL than the schema, drift will flag
  it).
- Cold-start sim-smoke succeeds — `npm --prefix apps/mobile run
  test:e2e:ios:smoke` plus `npm --prefix apps/mobile run
  test:e2e:ios:data-smoke` pass — proving the migrator runs on first
  boot and the schema lands correctly.

**Out of scope (do NOT touch):**

- Any schema change. Adding / removing columns is t2's territory and
  is done.
- Renaming `localRuntimeMigrations`, `migrations/index.ts`, or any
  call site of `migrate(db, ...)`.
- Server-side / Supabase migrations.
- Drift-checker logic. If t10 surfaces a real drift between the
  generated SQL bundle and the Drizzle TS schemas, that is by design
  — fix the schemas (or re-run `db:generate`) rather than weakening
  the checker.

**Approach — three options, try in order:**

1. **Option C (preferred):** Drizzle's stock Expo bundle output —
   set `driver: 'expo'` in `drizzle.config.ts` so `drizzle-kit generate`
   emits a `drizzle/migrations.js` that imports `_journal.json` and
   the `.sql` file. The runtime `migrations/index.ts` becomes a
   one-line re-export. Verify the installed `drizzle-kit ^0.31.5`
   supports the Expo bundle output; if it does, this is the cleanest
   path. Note: the Drizzle Expo pattern requires Metro to load `.sql`
   as source (via `metro.config.js` adding `sql` to `sourceExts`) and
   Babel to inline-import `.sql` files (via `babel-plugin-inline-import`
   in a project-level `babel.config.js`). Both files do not exist
   today — adding them counts as in-scope for Option C since they
   are documented prerequisites of Drizzle's Expo recipe. Jest must
   also resolve `.sql` imports (either via the same Babel plugin or a
   small Jest moduleNameMapper / transform).

2. **Option B (fallback):** Custom bundle script. Add
   `apps/mobile/scripts/bundle-migrations.ts` that reads
   `apps/mobile/drizzle/meta/_journal.json` plus every
   `apps/mobile/drizzle/*.sql` file in journal order and emits
   `apps/mobile/drizzle/migrations.generated.ts` exporting
   `{ journal, migrations }`. Extend `db:generate` to
   `drizzle-kit generate && tsx scripts/bundle-migrations.ts`.
   Rewrite `migrations/index.ts` as a thin wrapper importing from
   `../../drizzle/migrations.generated`. No Metro / Babel / Jest
   surgery required — `.ts` imports already work everywhere.

3. **Option A (last resort):** Metro asset import for `.sql` without
   Drizzle's blessing. Do NOT pick this unless C and B both fail; it
   is the most invasive and least documented.

If all three options fail, post a `[builder]` comment on the PR (open
or draft) describing the obstacle and surface to the coordinator. Do
NOT silently revert to the inlined approach — t10's whole point is
removing the duplication.

**Quality gate:**

- `./scripts/quality-fast.sh frontend && (cd apps/mobile && npm run check:sync-drift -- --strict)`
- `npm --prefix apps/mobile run test:e2e:ios:smoke`
- `npm --prefix apps/mobile run test:e2e:ios:data-smoke`

**Output artifact:** the rewritten
`apps/mobile/src/data/migrations/index.ts` (zero raw DDL tokens, thin
wrapper) plus the generated bundle file it imports from (`drizzle/migrations.js`
under Option C or `drizzle/migrations.generated.ts` under Option B).
The single `db:generate` command produces both deterministically.
