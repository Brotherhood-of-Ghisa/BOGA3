# t8: BG-task registration per t4 §4

**Type:** build

**Problem:** Sync needs to run while the app is backgrounded so
users see fresh data when they return — but the foreground
scheduler stops ticking when iOS suspends the JS thread. Per t4 §4,
`expo-background-task` wrapping iOS `BGAppRefreshTask` is the chosen
mechanism. The task body runs **one cycle directly** outside the
four-state machine per t4 §4.4. Registration happens once at app
init. The Expo plugin block in `app.config.ts` lets Expo write the
required `UIBackgroundModes` and `BGTaskSchedulerPermittedIdentifiers`
entries into the generated `Info.plist` during prebuild per t4 §4.5.

**Inputs:**

- t7 merged (the foreground scheduler is wired so the BG task can
  share `runSyncCycle` cleanly).
- `docs/plans/sync-v2/designs/t4.md` §4 (BG-task design), §4.4 (the
  task body — pre-flight network check then one cycle), §4.5
  (config / plugin block), §6.1 ("Background-task wrapper:" test
  list).
- `apps/mobile/app.config.ts` — current plugins array. Already has
  `"expo-secure-store"`, `"expo-router"`, `"expo-splash-screen"`,
  `"expo-location"`. t8 appends `"expo-background-task"`.
- `apps/mobile/package.json` — t8 adds `expo-background-task`,
  `expo-task-manager`, and `expo-network` via `expo install`
  (t7 added `@react-native-community/netinfo`).
- `apps/mobile/src/sync/cycle.ts` — `runSyncCycle` from t6, called
  by the BG-task body.

**Outcomes:**

- A new file `apps/mobile/src/sync/background-task.ts` exports:
  - `registerBackgroundSyncTask(): Promise<void>` — called once at
    app init from `_layout.tsx`. Calls
    `TaskManager.defineTask(BACKGROUND_SYNC_TASK_IDENTIFIER, ...)`
    at module-load time (per t4 §4.4 the task definition lives at
    module-load, not inside the registration call). Then calls
    `BackgroundTask.registerTaskAsync(BACKGROUND_SYNC_TASK_IDENTIFIER,
    { minimumInterval: 15 })`.
  - `BACKGROUND_SYNC_TASK_IDENTIFIER` constant — the task
    identifier string. The same constant referenced in `defineTask`
    AND `registerTaskAsync` AND the `Info.plist`'s
    `BGTaskSchedulerPermittedIdentifiers` array (the plugin writes
    this — t4 §4.5 final paragraph notes identifier mismatch as
    the #1 failure mode for this API). A test asserts the
    constant matches.
- The task body per t4 §4.4:
  1. **Pre-flight network check.** Call
     `Network.getNetworkStateAsync()` from `expo-network` (NOT
     NetInfo — NetInfo's long-lived listener isn't running in
     the short-lived BG context). If `isInternetReachable !== true`,
     return `BackgroundTaskResult.Success` without starting a
     cycle (per t4 §4.4 step 1 — Success preserves Apple's
     scheduling preference).
  2. **Run one cycle directly.** `await runSyncCycle()`. Returns
     `BackgroundTaskResult.Success` on resolve;
     `BackgroundTaskResult.Failed` on throw.
- The BG-task body does NOT consult or mutate the foreground
  scheduler's state (per t4 §4.4 final paragraph). It runs in a
  stateless context.
- `apps/mobile/app.config.ts` plugins array gains
  `"expo-background-task"` (no options needed per t4 §4.5). The
  PR body asserts the post-prebuild `Info.plist` carries
  `UIBackgroundModes: ["processing"]` (or the union of any
  existing values + `"processing"`) and the task identifier in
  `BGTaskSchedulerPermittedIdentifiers`.
- `apps/mobile/app/_layout.tsx` calls
  `registerBackgroundSyncTask()` once in the bootstrap
  `useEffect`, alongside `startSyncScheduler()` from t7.
- Jest tests cover, per t4 §6.1 "Background-task wrapper":
  - Network online: task body awaits the cycle stub and returns
    `Success`.
  - Network offline (pre-flight fails): no cycle runs; returns
    `Success`.
  - Cycle throws: returns `Failed`.
  - The constant identifier match across `defineTask`,
    `registerTaskAsync`, and the literal expected to appear in
    the `Info.plist` (assert via reading
    `apps/mobile/app.config.ts` to verify the constant doesn't
    drift — defence against the t4 §4.5 failure mode).
  - Foreground-scheduler isolation: the BG-task body does NOT
    call `requestSync()` or read scheduler state (via a spy on
    those exports).
- `npm run lint && npm run typecheck && npm run test` passes from
  `apps/mobile/`.
- PR body asserts `test:e2e:ios:smoke` and `test:e2e:ios:data-smoke`
  pass locally; the builder also runs a manual prebuild against
  the dev profile (`expo prebuild --platform ios --no-install`) to
  verify the `Info.plist` mutations land correctly. Manual-verify
  result is noted in the PR body.
