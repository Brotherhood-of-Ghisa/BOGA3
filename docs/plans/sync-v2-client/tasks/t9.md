# t9: dev-only wipe affordances behind isDevMode()

**Type:** build

**Problem:** Developers debugging sync issues (especially during the
TestFlight rollout of plan 2) need fast escape hatches:

- **wipe-local** — drop the local SQLite DB and re-bootstrap. Useful
  for testing the cycle against a clean local DB without having to
  reinstall the app (the manual-wipe procedure documented in t3 is
  the user-facing escape hatch; this is the dev-facing in-app
  equivalent for fast iteration).
- **wipe-remote-for-me** — delete every row owned by the current
  user from the server. Useful for testing the full bootstrap
  flow against an empty server side.

These MUST be gated by `isDevMode()` per
`apps/mobile/src/utils/isDevMode.ts` (NOT `__DEV__` per the user's
memory `feedback_no_dev_global.md` — `__DEV__` is false in the
`com.phano.boga3.dev` TestFlight build, which is exactly the build
that needs these affordances). The Settings screen at
`apps/mobile/app/(tabs)/settings.tsx` already has an `isDevMode()`
dev block (line 83+) — t9 extends it.

**Inputs:**

- t6 merged (the cycle exists so a post-wipe re-bootstrap can pull
  fresh).
- `apps/mobile/src/utils/isDevMode.ts` — the gate.
- `apps/mobile/app/(tabs)/settings.tsx` — existing dev block.
- `apps/mobile/src/data/dev-reset.ts` — existing wipe helper
  (`resetLocalDataAndReseed`). t9 may re-export it via a new
  helper or call it directly; either is fine.
- `apps/mobile/src/data/bootstrap.ts` — `resetLocalAppData()` for
  the local wipe.
- The Supabase service-role client — t9's `wipe-remote-for-me`
  needs a service-role helper RPC. Check whether one exists under
  `supabase/migrations/`; if not, t9 adds it (e.g.
  `app_public.dev_wipe_my_data()` granted to `authenticated`, with
  a guard against running in production). Coordinate with the
  reviewer if a fresh server-side migration is needed — that
  raises the t9 quality gate from frontend to
  `./scripts/quality-slow.sh backend` as well.

**Outcomes:**

- A new module `apps/mobile/src/sync/dev-affordances.ts` exports:
  - `wipeLocalAndReBootstrap(): Promise<void>` — calls
    `resetLocalAppData()` (from `apps/mobile/src/data/bootstrap.ts`)
    then `bootstrapLocalDataLayer()` again. Throws synchronously if
    `isDevMode() !== true`.
  - `wipeRemoteForCurrentUser(): Promise<{ rowsDeleted: number }>` —
    POSTs to the service-role helper RPC. Throws synchronously if
    `isDevMode() !== true`. Returns the RPC's row count.
- The Settings dev block at
  `apps/mobile/app/(tabs)/settings.tsx` (inside the existing
  `isDevMode()` gate) gains two new buttons:
  - **"Wipe local & re-bootstrap"** — calls
    `wipeLocalAndReBootstrap`. After the wipe, the cycle (t6 via
    the t7 scheduler) re-pulls from the server. No version-marker
    state to reset (there is no marker — see t3).
  - **"Wipe remote (my data)"** — calls
    `wipeRemoteForCurrentUser`. Confirmation modal first ("This
    deletes EVERY row on the server owned by your account. Are
    you sure?"). On confirm, the RPC runs; then the local DB is
    also wiped via `wipeLocalAndReBootstrap` (so the local state
    doesn't immediately re-push the just-deleted rows).
- Both buttons are NOT visible / interactable when `isDevMode()`
  is false. An ESLint guard on `__DEV__` literals must continue
  to pass — t9 does not introduce a bare `__DEV__` check.
- If t9 ships a server-side migration for
  `app_public.dev_wipe_my_data()`, the migration:
  - Lives under `supabase/migrations/` with a timestamp newer
    than plan 1's clean-room migration.
  - Is `security definer` and `set search_path = app_public`
    (standard helper-RPC shape in this repo).
  - Asserts a non-prod environment via `current_setting('app.env',
    true) IN ('local', 'staging', 'dev')` — if the setting is
    `'production'` or unset, the RPC raises a structured error.
  - Deletes from every entity table where `owner_user_id =
    auth.uid()` in a single transaction.
  - The migration's quality gate is `./scripts/quality-slow.sh
    backend` (the backend slow gate), declared in the PR body.
- Jest tests cover:
  - `wipeLocalAndReBootstrap` throws when `isDevMode()` returns
    false; succeeds when true.
  - `wipeRemoteForCurrentUser` throws when `isDevMode()` returns
    false; succeeds when true (stubbed RPC).
  - Settings screen renders neither button when `isDevMode()`
    returns false; renders both when true.
  - The two-step "remote wipe also wipes local" flow.
- `npm run lint && npm run typecheck && npm run test` passes from
  `apps/mobile/`.
- PR body asserts `test:e2e:ios:smoke` and `test:e2e:ios:data-smoke`
  pass locally.

**Output artifact:**

- New file `apps/mobile/src/sync/dev-affordances.ts`.
- Edit to `apps/mobile/app/(tabs)/settings.tsx`.
- New Jest test files (split across button-renders, helper logic,
  and the `isDevMode` gate).
- *Optional* new file under `supabase/migrations/<timestamp>_dev_wipe_my_data.sql`
  if the service-role helper RPC doesn't already exist. If t9
  ships this, also wire a contract test under
  `supabase/tests/` per existing repo convention.

**Out of scope:**

- The non-dev sync surface (deferred to plan 3 per t4 §5.2).
- A confirmation flow other than the modal above.
- Wiping `muscle_groups` (taxonomy table — local-only, untouched
  by sync; the wipe affordance doesn't need to clear it because
  the seed loader will re-create it).
- Adjusting `isDevMode()`'s implementation.
