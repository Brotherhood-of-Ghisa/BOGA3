# t4: nowMonotonic clock helper

**Type:** build

**Problem:** Per t2 §8 the client needs a single monotonic clock
helper `nowMonotonic(): number` that returns
`Math.max(Date.now(), last_emitted_ms + 1)`, persists the result
**synchronously in the same SQLite transaction as any entity write
that consumed the value** (t2 §8.3), and serves as the LWW key on
every dirtying write (t2 §7.2). Fire-and-forget persistence is
explicitly unsafe per §8.3 — a crash between row commit and counter
persist would leave the counter behind and the next launch would
emit a stale timestamp ≤ already-persisted entity timestamps, which
the server's LWW would silently reject. The helper is the **single
source of monotonic time** across the app per §8.2; t4 ships it
under `apps/mobile/src/data/clock.ts` so the conspicuous filename
matches §8.2's recommendation.

**Inputs:**

- t2 merged (`sync_runtime_state.last_emitted_ms` column exists,
  default `0`).
- `docs/plans/sync-v2/designs/t2.md` §8 — the contract (formula,
  persistence rule, cache strategy).
- `apps/mobile/src/data/bootstrap.ts` — the `LocalDatabase` type
  used in repo write transactions.
- `apps/mobile/src/data/schema/sync-runtime-state.ts` — the column
  `nowMonotonic()` reads / writes.

**Outcomes:**

- A new file `apps/mobile/src/data/clock.ts` exports:
  - `nowMonotonic(tx: Transaction): number` — the canonical helper.
    Takes a Drizzle SQLite transaction handle so the SELECT /
    UPDATE on `sync_runtime_state.last_emitted_ms` happens inside
    the caller's transaction, satisfying t2 §8.3. Returns the
    computed integer.
  - `__resetClockForTests(): void` — module-internal cache reset for
    Jest setup. Production code must not call this.
- The helper algorithm:
  1. Read the module-scoped cached `cachedLastEmitted` (initialised
     `null`).
  2. If cache is `null`, SELECT `last_emitted_ms` from
     `sync_runtime_state` inside the caller's tx; populate cache.
  3. Compute `next = Math.max(Date.now(), cached + 1)`.
  4. UPDATE `sync_runtime_state.last_emitted_ms = next` inside the
     same tx.
  5. Update the cache to `next`.
  6. Return `next`.
- `sync_runtime_state` row existence: if no row exists, the helper
  creates one with id = the singleton id used by t2 (the same
  `PRIMARY_RUNTIME_STATE_ID` constant the v1 `runtime.ts` used —
  preserved by t2 in the renamed module / file path). Insert is
  inside the caller's tx.
- Jest unit tests cover:
  - Cold start: first call reads the persisted `last_emitted_ms`
    correctly (round-trip a fixture row written before the helper
    is imported).
  - Cache hit: a second call within the same process does NOT
    SELECT (verify via a spy on the DB).
  - `Date.now()` ahead of cache: `next = Date.now()`.
  - Cache ahead of `Date.now()` (clock skew): `next = cached + 1`.
  - Strict monotonicity: 10000 sequential calls produce a strictly
    increasing sequence even when `Date.now()` is mocked to a
    constant.
  - Persistence: after 10 calls inside one transaction, a fresh
    SELECT on `sync_runtime_state.last_emitted_ms` reads the
    10th-emitted value (no fire-and-forget loss).
  - Persistence-across-restart: `__resetClockForTests()` simulates
    a cold start; the next call reads the persisted value, not 0.
- The helper accepts ONLY a transaction argument — there is no
  fire-and-forget overload. Callers without a tx must wrap their
  write in `database.transaction(...)`.
- `npm run lint && npm run typecheck && npm run test` passes from
  `apps/mobile/`.

**Output artifact:**

- New file `apps/mobile/src/data/clock.ts` exporting `nowMonotonic`
  and `__resetClockForTests`.
- New Jest test `apps/mobile/app/__tests__/clock.test.ts` covering
  the cases enumerated above.

**Out of scope:**

- Wiring `nowMonotonic` into entity repo write paths (t5a / t5b).
- The cycle's serialisation logic that consumes
  `local_updated_at_ms` (t6).
- Any helper that doesn't write to `sync_runtime_state.last_emitted_ms`
  (e.g. UI timestamps, log timestamps — those keep using
  `Date.now()`).
