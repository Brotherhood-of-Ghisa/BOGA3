# t1: delete v1 sync code paths

**Type:** build

**Problem:** The v1 client sync stack is the largest source of
incidental complexity in `apps/mobile/`: ~3000 lines under
`apps/mobile/src/sync/` plus `enqueueSyncEvent` / `enqueueSyncEventsTx`
call sites in every repo, plus the `sync_outbox_events` and
`sync_delivery_state` Drizzle tables, plus the wiring in
`apps/mobile/app/_layout.tsx`, plus a ~200-line "Backup sync" UI card
on `apps/mobile/app/profile.tsx` that imports
`loadSyncProfileStatus`, `setSyncEnabled`, and
`SyncProfileStatusSnapshot` from `@/src/sync`. The v2 design replaces
the entire event-log + outbox + projection model with two RPCs
(push, pull) and two local-only columns (`local_dirty`,
`local_updated_at_ms`); none of the v1 surface survives. Removing it
cleanly BEFORE adding the v2 shape keeps every downstream PR's diff
small and prevents the v1 + v2 codepaths from briefly coexisting in
a confusing half-state. Critically, if t1 ships without stripping
the `profile.tsx` sync card, deleting the `@/src/sync/*` exports
leaves `profile.tsx` with broken imports and the app no longer
compiles — directly violating the user's "app must ALWAYS be in
working state" directive. So `profile.tsx` is in scope for this PR.
This task is deletion-heavy (~3000 lines from `apps/mobile/src/sync/`
+ ~200 lines from `profile.tsx` + the eight v1 sync tests). Net
additions remain tiny (a minimal re-export shim if needed and the
Drizzle migration dropping the two v1 tables). Even with the
expanded scope, the PR sits well under the ~2000-line size budget
because almost the entire diff is deletion.
