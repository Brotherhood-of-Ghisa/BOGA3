# t11: post-sign-in sync authentication — session lost between sign-in and cycle

**Type:** build

**Problem:** After a *successful* sign-in (`onAuthStateChange` fires `SIGNED_IN`
with a valid JWT), the sync cycle's Supabase client reports `getSession() → null`
and the server returns `AUTH_REQUIRED` on every cycle. The shared
"route-to-sign-in" selector (`apps/mobile/src/sync/use-auth-required-redirect.ts`,
`selectShouldRouteToSignIn` → returns `authRequiredSignal` when a session is
present) then faithfully bounces the just-signed-in user back to `/sign-in`,
producing a redirect loop (and, with the dev Maestro harness, a React "maximum
update depth" loop). The guard logic is correct; its **premise is false** — the
cycle raises a spurious `AUTH_REQUIRED` because the session it should see is gone.
There is a SINGLE memoized Supabase client (`apps/mobile/src/auth/supabase.ts`
`getSupabaseMobileClient()`, `persistSession: true`, `autoRefreshToken: true`,
custom storage via `getAuthStorageAdapter()`), so this is NOT a two-client
instance mismatch — it is a session persistence / refresh / timing defect.

This was surfaced while making the sync-gate Maestro flow pass; it reproduces on
the already-merged `apps/mobile/.maestro/flows/launch-requires-sign-in.yaml`, and
that whole lane (`npm run test:e2e:ios:auth-profile`) is **excluded from CI's
mandated `test:e2e:ios:gates`**, so login→sync was never verified end-to-end —
which is how it stayed hidden.

**Inputs:**
- Depends on t1 (merged): the auth guard, sign-in route, the AUTH_REQUIRED signal
  (`apps/mobile/src/sync/auth-required-signal.ts`), and the shared selector
  (`use-auth-required-redirect.ts`). Branch from latest `origin/main`.
- Composes with the merged scheduler/cycle (plan 2):
  `apps/mobile/src/sync/cycle.ts` (`runSyncCycle`, `classifyRpcResult` →
  `AUTH_REQUIRED`), `apps/mobile/src/sync/scheduler.ts`. Any server RPC uses
  `client.schema('app_public').rpc(...)`.
- Auth client + storage: `apps/mobile/src/auth/supabase.ts`,
  `apps/mobile/src/auth/storage.ts` (`getAuthStorageAdapter`),
  `apps/mobile/src/auth/service.ts`, `apps/mobile/src/auth/provider.tsx`.
- Maestro lanes: `apps/mobile/scripts/maestro-ios-*.sh`, the `test:e2e:ios:auth-profile`
  lane and its local-Supabase fixture (`.env.local` baseline), plus the infra
  lane conventions (`test:sync:infra`, branch-provisioned —
  `SUPABASE_BRANCH_URL` / `SUPABASE_BRANCH_ANON_KEY`).

**First job — determine severity (real vs. fixture):** before fixing, establish
whether the session loss reproduces against the **real (branch-provisioned)
Supabase backend** (⇒ launch-blocking: login→sync→restore is broken for every
user) or **only against the local auth-profile fixture** (⇒ a test-infra defect:
e.g. `autoRefreshToken` failing against the fixture's token/clock and clearing
the session). Run the branch-provisioned infra lane if the env is available;
otherwise isolate the root cause by instrumentation + static analysis (storage
adapter read/write in the sim, auto-refresh behaviour, sign-in→cycle ordering)
and state the conclusion explicitly. Record the finding in the PR.

**Outcomes:**
- Root cause identified and stated (real-backend vs. fixture-only), with evidence.
- After a successful sign-in, the sync cycle's client sees the session: a normal
  cycle authenticates (no spurious `AUTH_REQUIRED`), and a validly-signed-in user
  is NOT routed back to sign-in. The redirect/`maximum update depth` loop is gone.
- A genuinely-expired/absent session still correctly yields `AUTH_REQUIRED` →
  route to sign-in (don't fix the loop by suppressing the real signal).
- `apps/mobile/.maestro/flows/launch-requires-sign-in.yaml` (t1's flow) passes
  green in the auth-profile lane (sign-in → reaches a data/gate surface, no bounce
  loop).
- The login→sync path is covered by an **enforced** gate going forward (wire the
  auth-profile sign-in→cycle assertion into a run that actually gates, or hand
  tFINAL a clear contract to assert it against real Supabase) so this class of
  defect cannot silently hide again. State which.
- Jest covers the session-visibility fix at the unit level (the cycle/auth path
  sees the signed-in session; a real missing session still classifies
  `AUTH_REQUIRED`).
- Quality gate `./scripts/quality-fast.sh frontend` passes; the auth-profile
  Maestro lane passes for t1's flow (`npm run test:e2e:ios:gates` for the
  mandated lanes too).
