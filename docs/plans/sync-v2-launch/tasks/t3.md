# t3: bootstrapper integration — seeder runs only after the first cycle

**Type:** build

**Problem:** Today the exercise-catalog seeder runs unconditionally at boot:
`apps/mobile/src/data/bootstrap.ts` calls `seedSystemExerciseCatalog(database)`
inside `prepareLocalDataLayer`, before any sync. That re-creates seed rows on a
reinstall even when the server already holds the user's (possibly deleted or
renamed) catalog — defeating device recovery. The launch contract requires the
seeder to run only AFTER the first full pull drains and ONLY when that pull
returned zero rows, then set `bootstrap_completed_at` last. This is the
single load-bearing reorder of the launch.

**Inputs:**
- tPROG merged: the sync/bootstrap progress-reporting contract (binding —
  cite, do not restate): `designs/tPROG.md ## Decision` fixes the phase
  representation, the intra-phase progress representation, and where each is
  sourced. The bootstrapper PRODUCES phase + intra-phase progress per that
  contract (per-layer pull, seed).

- t1 merged: the auth-gated entry point. The bootstrapper runs on the first

**Outcomes:**
- The entity-catalog seeder no longer runs unconditionally at boot; it runs only
  inside the bootstrapper, gated on the first full pull returning zero rows.
  (Maps to plan outcome: seeder runs only after `bootstrap_completed_at` is set
  / per the bootstrapper.)
- The bootstrapper: runs when `bootstrap_completed_at IS NULL`; invokes the
  first full pull; seeds via the normal repo path iff `rowsPulled == 0`
  (tombstones count toward `rowsPulled`); sets `bootstrap_completed_at` LAST.
- A crash between the pull and the flag-set leaves `bootstrap_completed_at`
  null so the next sign-in re-attempts cleanly (the flag is the last write).
- The seeder's writes go through repo paths that flip `local_dirty` naturally;
  the bootstrapper never writes `local_dirty` / `local_updated_at_ms` directly.
- On a reinstall where the server holds rows or tombstones for the user
  (`rowsPulled > 0`), the seeder is a no-op and the pulled state stands.
- The bootstrapper emits **phase** (per-layer pull, seed, …) and **intra-phase
  progress** per `designs/tPROG.md ## Decision`, sourced as that contract decides
  (e.g. derived from existing log events / `sync_runtime_state` / local row
  counts, or via progress signals the contract threads through the pull leg). If
  tPROG assigns ownership of cycle instrumentation to t3, t3 adds exactly that
  instrumentation; if it assigns it elsewhere, t3 only reads/forwards the
  contract's shape. (Maps to plan outcome: Sync gate surfaces phase + advancing
  activity — the bootstrapper is the producer.)
- Jest covers: bootstrapper skips when flag already set; seeds when
  `rowsPulled == 0`; no-ops when `rowsPulled > 0` including the tombstone-only
  case; sets the flag last; re-attempts after a simulated mid-bootstrap crash;
  emits the phase + intra-phase progress shape per `designs/tPROG.md ##
  Decision` (asserted against the contract's typed output, not a UI render).
- Quality gate `./scripts/quality-fast.sh frontend` passes.

**Output artifact:**
- A bootstrapper entry point (e.g. `apps/mobile/src/sync/bootstrapper.ts`

- `apps/mobile/src/data/bootstrap.ts` no longer seeds the entity catalog

- The phase + intra-phase progress emission per `designs/tPROG.md ## Decision`,

**Size note (~2000-added-line budget):** the bootstrapper reorder is the bulk;
the added progress-emission scope is small (emit phase/progress values from
points the bootstrapper + pull leg already traverse, sourced as tPROG decides) —
the heavy "which representation" reasoning lives in tPROG, not here. Expected to
stay well under the budget. Re-check after tPROG lands: if the contract assigns
t3 substantial new cycle instrumentation (e.g. progress callbacks threaded
through every layer of `cycle.ts` plus new tests), split into **t3a**
(bootstrapper reorder + flag-last) and **t3b** (cycle/bootstrapper progress
instrumentation) so each PR stays ≤ ~2000 added lines.
