# t3: bootstrapper integration — seeder runs only after the first cycle

**Type:** build

**Problem:** Today the exercise-catalog seeder runs unconditionally at boot:
`apps/mobile/src/data/bootstrap.ts` calls `seedSystemExerciseCatalog(database)`
inside `prepareLocalDataLayer`, before any sync. That re-creates seed rows on a
reinstall even when the server already holds the user's (possibly deleted or
renamed) catalog — defeating device recovery. The launch contract requires the
seeder to run only AFTER the first full pull drains and ONLY when that pull
returned zero rows, then set `bootstrap_completed_at` last. This is the
single load-bearing reorder of the launch.

**Inputs:**
- t1 merged: the auth-gated entry point. The bootstrapper runs on the first
  successful sign-in for an account (it observes `bootstrap_completed_at IS
  NULL`), so it sits downstream of the auth guard.
- Binding design (authoritative — cite, do not restate):
  `docs/plans/sync-v2/designs/t3.md` §1 (the bootstrapper:
  `rowsPulled := runFirstFullPull(); if rowsPulled == 0: runSeeder();
  bootstrap_completed_at := now()` — set LAST), §3 (the single `rowsPulled == 0`
  gate; tombstones count), §1.4 (seeder writes via the normal repo path; never
  touches `local_dirty` / `local_updated_at_ms` directly), §8.2 #4 (set the flag
  last so a crash re-attempts cleanly), §8.2 #5 (seeder runs in a single Drizzle
  transaction).
- Existing seeder + marker helpers (already as-built):
  `apps/mobile/src/data/exercise-catalog-seeds.ts` —
  `seedSystemExerciseCatalog(database, now?)`, `SEED_CATALOG_BUNDLE_VERSION = 1`,
  and the `appliedSeedMigrationAppVersion` read/write marker helpers.
- The cycle and its first-full-pull: `apps/mobile/src/sync/cycle.ts`
  (`runSyncCycle`, the per-layer pull leg that drains to `has_more = false`).
  The corrected topological layer partition is inherited from plan 1/plan 2
  (`apps/mobile/src/sync/topo-order.ts`) — do NOT redefine the layers.
- `sync_runtime_state.bootstrap_completed_at` +
  `applied_seed_migration_app_version` columns + `PRIMARY_RUNTIME_STATE_ID`
  already exist (`apps/mobile/src/data/schema/sync-runtime-state.ts`,
  `apps/mobile/src/data/clock.ts`).
- All server-touching calls use `client.schema('app_public').rpc(...)` (the
  cycle already does; any new RPC must match).
- `muscle_groups` stays client-only and seeds independently of the bootstrapper
  (design t3.md §4); t6 owns the idempotent extension of that path. t3 keeps
  `muscle_groups` seeding decoupled from the entity bootstrapper.

**Outcomes:**
- The entity-catalog seeder no longer runs unconditionally at boot; it runs only
  inside the bootstrapper, gated on the first full pull returning zero rows.
  (Maps to plan outcome: seeder runs only after `bootstrap_completed_at` is set
  / per the bootstrapper.)
- The bootstrapper: runs when `bootstrap_completed_at IS NULL`; invokes the
  first full pull; seeds via the normal repo path iff `rowsPulled == 0`
  (tombstones count toward `rowsPulled`); sets `bootstrap_completed_at` LAST.
- A crash between the pull and the flag-set leaves `bootstrap_completed_at`
  null so the next sign-in re-attempts cleanly (the flag is the last write).
- The seeder's writes go through repo paths that flip `local_dirty` naturally;
  the bootstrapper never writes `local_dirty` / `local_updated_at_ms` directly.
- On a reinstall where the server holds rows or tombstones for the user
  (`rowsPulled > 0`), the seeder is a no-op and the pulled state stands.
- Jest covers: bootstrapper skips when flag already set; seeds when
  `rowsPulled == 0`; no-ops when `rowsPulled > 0` including the tombstone-only
  case; sets the flag last; re-attempts after a simulated mid-bootstrap crash.
- Quality gate `./scripts/quality-fast.sh frontend` passes.

**Output artifact:**
- A bootstrapper entry point (e.g. `apps/mobile/src/sync/bootstrapper.ts`
  exporting `runBootstrapper()` or similar) wired into the boot/sign-in path.
- `apps/mobile/src/data/bootstrap.ts` no longer seeds the entity catalog
  unconditionally at boot (the `seedSystemExerciseCatalog` call is relocated
  behind the bootstrapper's `rowsPulled == 0` branch). `muscle_groups` seeding
  stays at boot.
- Jest specs under `apps/mobile/app/__tests__/`.

**Out of scope:**
- The `sys_*` → `seed_*` bundle slug rename (t4).
- The bundle-migration runtime loop (t5).
- The `muscle_groups` idempotent-extension change (t6).
- The full-screen gate UI (t2) — t3 only sets the flag the gate reads.
- The sign-out / account-switch wipe (t8).
