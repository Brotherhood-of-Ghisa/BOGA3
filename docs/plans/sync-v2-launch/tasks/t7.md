# t7: soft-delete everywhere ‚Äî convert hard-delete repo paths

**Type:** build

**Problem:** Deletion must survive reinstall (device recovery): a deleted row
becomes a tombstone (`deleted_at` non-null) that pushes to the server and pulls
back on the next bootstrap, so the seeder never resurrects it. The schema is
already there ‚Äî every one of the 8 entities carries `deleted_at` + `deletedAtIdx`
and the cycle serialises `deleted_at` on the wire. What remains is that some
write paths still issue a hard SQL `DELETE` instead of a soft-delete, and some
readers do not filter tombstones. The launch contract: every user-facing
hard-delete path on the 8 entities becomes a soft-delete (set
`deleted_at = Date.now()` + flip the dirty bit via the normal repo path), and
local readers filter `WHERE deleted_at IS NULL`.

**Inputs:**
- No build dependency on t3/t4/t5/t6 (independent surface). Branch from latest
  `origin/main`.
- `deleted_at` columns + `deletedAtIdx` already exist on all 8 entity schemas
  (`apps/mobile/src/data/schema/*.ts`) and the cycle emits `deleted_at`
  (`apps/mobile/src/sync/cycle.ts`) ‚Äî t7 adds NO schema column.
- Known remaining hard delete to convert: `removeTagAssignment`
  (`apps/mobile/src/data/exercise-tags.ts` ‚Äî `db.delete(sessionExerciseTags)`).
- Other candidate hard-delete sites across the 8 entities to evaluate and
  convert where they represent a user delete:
  - `apps/mobile/src/data/exercise-catalog.ts` (`tx.delete(exerciseMuscleMappings)`).
  - `apps/mobile/src/data/session-drafts.ts` (the session-rebuild cascade that
    `tx.delete(...)`s `session_exercise_tags`, `exercise_sets`,
    `session_exercises` before re-inserting). This is a rebuild, not a simple
    delete ‚Äî converting it to soft-delete-then-reconcile is the size/complexity
    risk in this task; see the split note below.
  - Grep `db.delete(` / `.delete(` across the 8 entities for any others.
- Exempt (NOT user deletes ‚Äî leave as hard wipes): `apps/mobile/src/data/dev-reset.ts`
  (full local-table wipe), `apps/mobile/src/maestro/**` fixtures, and any
  in-memory `Set/Map.delete` / listener teardown.
- Reader filtering: local read queries on the 8 entities must exclude tombstones
  (`isNull(<table>.deletedAt)`) unless they explicitly need deleted rows (the tag
  store already takes an `includeDeleted` flag ‚Äî match that pattern where a
  reader legitimately needs both).
- Binding design context: `docs/plans/sync-v2/designs/t3.md` ¬ß3.1 / ¬ß5.2
  (tombstones count toward `rowsPulled`; deletion survives reinstall),
  `docs/plans/sync-v2/designs/t1.md` (deletion is `deleted_at != null`, no
  `deleted boolean`).

**Outcomes:**
- Every user-facing delete path on the 8 entities sets `deleted_at = Date.now()`
  and flips the dirty bit through the normal repo path; no user-facing hard
  `DELETE` against the 8 entities remains in `apps/mobile/src/` outside the
  exempt sites. (Maps to plan outcome: every prior hard-DELETE path is now a
  soft-delete; grep test confirms no `db.delete(<entity>)` for the 8 entities
  except the dev wipe affordance.)
- `removeTagAssignment` soft-deletes the `session_exercise_tags` row instead of
  hard-deleting it.
- Local readers on the 8 entities filter `WHERE deleted_at IS NULL` (except
  readers that explicitly opt into deleted rows).
- A grep-style automated test confirms no disallowed `db.delete` / `.delete(`
  against the 8 entity tables remains outside the exempt sites.
- Jest covers each converted path: the row is tombstoned (not removed), is
  dirty after the operation, and is hidden from the default reader; an
  `includeDeleted` reader (where one exists) still sees it.
- Quality gate `./scripts/quality-fast.sh frontend` passes; if any UI surface
  changes, `npm run test:e2e:ios:gates` passes.

**Output artifact:**
- Converted repo paths across `apps/mobile/src/data/` (at minimum
  `exercise-tags.ts`, plus every other user-delete site the grep surfaces) and
  the reader-filter updates.
- A grep/guard Jest test (e.g. `apps/mobile/app/__tests__/soft-delete-guard.test.ts`)
  asserting no disallowed hard-delete against the 8 entities.
- Per-path Jest specs under `apps/mobile/app/__tests__/`.

**Out of scope:**
- Any schema migration (columns + index already exist).
- The dev-reset full-table wipe and Maestro fixtures (exempt).
- The dev wipe affordances (t10) and the sign-out wipe (t8), which clear local
  tables wholesale and are not per-row user deletes.

**Split note (size discipline):** the converted paths + reader filters + tests
are expected to land well under the ~2000-added-line budget in one PR. The
session-rebuild cascade in `session-drafts.ts` is the one path whose
soft-delete-then-reconcile conversion is non-trivial. If, while building, the
total approaches the budget OR the session-drafts conversion turns out to be a
materially separate piece of work, STOP and surface a split ‚Äî e.g. split into
"t7a: tag + mapping + simple delete paths" and "t7b: session-rebuild cascade
conversion" ‚Äî rather than shipping a mega-PR. Both halves would feed tFINAL.
