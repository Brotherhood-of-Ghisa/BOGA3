# t10: dev-gated wipe affordances — confirm correct against launch state

**Type:** build

**Problem:** Plan 2 already shipped the dev-only wipe affordances:
`apps/mobile/src/sync/dev-affordances.ts` exports `wipeLocalAndReBootstrap()` +
`wipeRemoteForCurrentUser()` (both throw synchronously when `!isDevMode()`),
both buttons already live in the `isDevMode()` block of
`apps/mobile/app/(tabs)/settings.tsx`, and the `app_public.dev_wipe_my_data()`
RPC is migrated + contract-tested. So this is NOT net-new — it is a verification
delta: confirm the affordances still behave correctly against the launch state
(the bootstrapper reorder of t3, the soft-delete world of t7, the sign-out wipe
of t8) and ensure the Settings entry-point is wired and gated. The launch
outcome ("wipe-local + wipe-remote-for-me work behind the `isDevMode()` gate")
must hold on the integration branch with t3/t7/t8 merged.

**Inputs:**
- t3 merged: `wipeLocalAndReBootstrap()` ends by re-bootstrapping; with the
  bootstrapper reorder, a local wipe must leave the device in a state where the
  next cycle/bootstrapper re-pulls correctly. Confirm the affordance composes
  with the reordered bootstrap (it should: wipe-local re-enters the same boot
  path). (See t3 `## Output artifact`.)
- As-built affordances (cite, do not duplicate):
  `apps/mobile/src/sync/dev-affordances.ts` (`wipeLocalAndReBootstrap`,
  `wipeRemoteForCurrentUser`, `assertDevModeSync`, the
  `client.schema('app_public').rpc('dev_wipe_my_data')` call) and the
  `isDevMode()`-gated buttons in `apps/mobile/app/(tabs)/settings.tsx`
  (`settings-dev-wipe-local-button`, `settings-dev-wipe-remote-button`).
- `isDevMode()` (`apps/mobile/src/utils/isDevMode.ts`) — the gate must keep the
  affordances live in the `com.phano.boga3.dev` TestFlight build and inert in
  release. Never `__DEV__`.
- CONSTRAINT: the dev-wipe RPC is non-prod-gated AND a server delete; it is
  distinct from t8's production LOCAL sign-out wipe. Do not conflate them.

**Outcomes:**
- `wipe-local` (wipe local + re-bootstrap) and `wipe-remote-for-me` are reachable
  from Settings ONLY when `isDevMode()` is true, and both throw synchronously
  before any I/O when `isDevMode()` is false. (Maps to plan outcome AND the
  final-test-card cross-cutting outcome #5: wipe-local + wipe-remote-for-me work
  behind the `isDevMode()` gate.)
- Both affordances behave correctly against the launch integration branch:
  after `wipe-local`, the device re-bootstraps and a cycle re-pulls server state;
  after `wipe-remote-for-me`, the server rows are deleted and local is wiped so
  they are not re-pushed.
- The Settings dev-tools entry-point is wired and gated; the buttons carry
  stable testIDs.
- Jest covers: both affordances throw synchronously when `!isDevMode()`;
  `wipeRemoteForCurrentUser` dispatches via `client.schema('app_public').rpc`;
  the Settings buttons render only under `isDevMode()`.
- UI gate: `npm run test:e2e:ios:gates` passes; a Maestro flow (dev build)
  exercises wipe-local and asserts the success feedback and re-bootstrap.

**Output artifact:**
- Any fixes needed to `apps/mobile/src/sync/dev-affordances.ts` and/or the
  `isDevMode()` card in `apps/mobile/app/(tabs)/settings.tsx` so the affordances
  are correct against the launch state (no net-new mechanism expected; if the
  affordances already pass, the deliverable is the confirming tests + any wiring
  fix).
- Jest / Maestro coverage proving the gated behaviour and the wipe round-trips.

**Out of scope:**
- The production sign-out / account-switch local wipe (t8) — different gate,
  different mechanism, no server delete.
- Net-new wipe features beyond confirming/wiring the existing two.
- Changing the `dev_wipe_my_data` server RPC.
