# t2: sync-gate full-screen block until first cycle drains

**Type:** build

**Problem:** After sign-in, the app must not expose data screens until the
first sync cycle has drained and the device holds the user's restored data.
The gate is keyed on `sync_runtime_state.bootstrap_completed_at`: while it is
null, the app shows a full-screen "Setting up your data…" block; once it is
non-null the block dismisses and the normal routes render. On a cycle error the
block shows the error message plus a single Retry CTA — but a Retry must not
re-fire a cycle when the latest outcome is `AUTH_REQUIRED` (a retry would just
re-hit the same envelope; the user needs to sign in first). This is the
device-recovery waiting room a returning user sees.

  representation, the intra-phase activity/progress representation, the
  network-unreachable signal, and where each is sourced. The gate RENDERS this
  contract.

**Outcomes:**
- While `bootstrap_completed_at` is null for the signed-in user, a full-screen
  "Setting up your data…" block renders and no data screen is reachable. (Maps
  to plan outcome: app usage blocked until `bootstrap_completed_at` non-null.)
- The block dismisses and the normal routes render within the foreground window
  once `bootstrap_completed_at` is set.
- While the block is up, it renders the **current phase** label of the
  first-sync/bootstrap (per `designs/tPROG.md ## Decision`). (Maps to plan
  outcome: Sync gate surfaces the current phase.)
- While the block is up and work is happening, it renders an **intra-phase
  activity/progress indicator** that visibly advances (the representation —
  percentage / item-row count / page-cycle counter / phase+heartbeat — is fixed
  by `designs/tPROG.md ## Decision`; t2 renders whatever shape that contract
  defines, sourced from the t9 accessor). (Maps to plan outcome: Sync gate
  surfaces an advancing activity signal.)
- When the device is offline (the network-unreachable signal from the contract),
  the block shows the **offline message** rather than an indefinite spinner, so a
  non-advancing gate is self-explanatory. (Maps to plan outcome: Sync gate
  surfaces a network-unreachable/offline message.)
- On a cycle error (non-`AUTH_REQUIRED`), the block shows the error message and
  a single Retry button; Retry fires one cycle via `requestSync()`.
- When the latest outcome is `AUTH_REQUIRED`, the gate routes to the sign-in
  screen (t1) and does NOT render or fire Retry.
- A boot-time scheduler wiring failure is not swallowed into a gate error state
  (it surfaces as the crash it is).
- Jest covers: gate renders block when flag null; dismisses when flag set;
  shows error + Retry on cycle error; Retry calls `requestSync()` exactly once;
  `AUTH_REQUIRED` routes to sign-in and suppresses Retry; renders the current
  phase label from a stubbed progress source; renders the intra-phase activity
  indicator and that it advances when the stubbed signal advances; renders the
  offline message when the stubbed network-unreachable signal is set.
- UI gate: `npm run test:e2e:ios:gates` passes; a Maestro flow asserts: fresh
  install → sign-in → block visible (with a phase label and an activity/progress
  indicator) → block dismisses after the first cycle completes within 1 minute of
  foreground. (The offline-message path is asserted at Jest level here and, where
  reachability-toggling is feasible, end-to-end by tFINAL.)

**UX Contract:**
- Flow — First-sync block:
  - Trigger: signed-in user whose `bootstrap_completed_at` is null.
  - Steps: render full-screen "Setting up your data…" with the current **phase
    label** and an **intra-phase activity/progress indicator** that advances as
    work proceeds → first cycle drains → flag set → block dismisses → normal
    routes render.
  - Success outcome: data restored and reachable within 1 min of foreground;
    while it loads, the phase + advancing activity indicator make it visible that
    work is happening.
  - Failure/edge: cycle error → error message + single Retry; `AUTH_REQUIRED`
    → route to sign-in (no Retry); **offline** (network-unreachable signal) →
    show the offline message instead of an indefinite spinner.
- Notes: single full-screen state, single primary action (Retry); the phase
  label, the activity/progress indicator, and the offline message are display-only
  (no extra primary actions); reuse `@/components/ui` primitives; render the
  phase / progress / offline shape exactly as `designs/tPROG.md ## Decision`
  defines it (do not invent a parallel representation); no new interaction
  pattern (if one is introduced, update `docs/specs/ui/ux-rules.md` in the same
  PR).
