# t2: sync-gate full-screen block until first cycle drains

**Type:** build

**Problem:** After sign-in, the app must not expose data screens until the
first sync cycle has drained and the device holds the user's restored data.
The gate is keyed on `sync_runtime_state.bootstrap_completed_at`: while it is
null, the app shows a full-screen "Setting up your data…" block; once it is
non-null the block dismisses and the normal routes render. On a cycle error the
block shows the error message plus a single Retry CTA — but a Retry must not
re-fire a cycle when the latest outcome is `AUTH_REQUIRED` (a retry would just
re-hit the same envelope; the user needs to sign in first). This is the
device-recovery waiting room a returning user sees.

**Inputs:**
- t1 merged: the route-layer auth guard and the sign-in route. The sync gate
  sits AFTER the auth guard — it only renders for a signed-in user. Consume
  t1's `AUTH_REQUIRED` → route-to-sign-in hook so an `AUTH_REQUIRED` outcome at
  the gate routes to sign-in instead of showing Retry. (See t1
  `## Output artifact`.)
- `sync_runtime_state` schema (already exists):
  `apps/mobile/src/data/schema/sync-runtime-state.ts` —
  `bootstrapCompletedAt` (nullable `timestamp_ms`), singleton row id
  `PRIMARY_RUNTIME_STATE_ID = 'primary'` (`apps/mobile/src/data/clock.ts`).
- The scheduler entry point `requestSync()` takes NO arguments
  (`apps/mobile/src/sync/scheduler.ts`). The gate triggers a cycle via the
  parameterless `requestSync()`.
- Last-cycle error signal: the scheduler emits structured log events
  (`sync_scheduler_cycle_error`, etc.) via `apps/mobile/src/logging/logEvent.ts`;
  the cycle classifies errors as `AUTH_REQUIRED` / `FK_VIOLATION` / `INTERNAL`
  (`apps/mobile/src/sync/cycle.ts`). The gate needs a read path for "did the
  last cycle error, and with what code." (t9 builds the durable scheduler-state
  read API; t2 may read the minimal error signal it needs from the same
  source or a small accessor — keep it scoped to the gate's needs.)
- A scheduler wiring failure at boot CRASHES (re-throw from
  `startSyncScheduler`); the gate must treat that as a crash, not a recoverable
  gate error state.
- UX inputs (binding): `docs/specs/08-ux-delivery-standard.md`,
  `docs/specs/ui/ux-rules.md`, `docs/specs/ui/navigation-contract.md`,
  `docs/specs/ui/screen-map.md`, `docs/specs/ui/components-catalog.md`.

**Outcomes:**
- While `bootstrap_completed_at` is null for the signed-in user, a full-screen
  "Setting up your data…" block renders and no data screen is reachable. (Maps
  to plan outcome: app usage blocked until `bootstrap_completed_at` non-null.)
- The block dismisses and the normal routes render within the foreground window
  once `bootstrap_completed_at` is set.
- On a cycle error (non-`AUTH_REQUIRED`), the block shows the error message and
  a single Retry button; Retry fires one cycle via `requestSync()`.
- When the latest outcome is `AUTH_REQUIRED`, the gate routes to the sign-in
  screen (t1) and does NOT render or fire Retry.
- A boot-time scheduler wiring failure is not swallowed into a gate error state
  (it surfaces as the crash it is).
- Jest covers: gate renders block when flag null; dismisses when flag set;
  shows error + Retry on cycle error; Retry calls `requestSync()` exactly once;
  `AUTH_REQUIRED` routes to sign-in and suppresses Retry.
- UI gate: `npm run test:e2e:ios:gates` passes; a Maestro flow asserts: fresh
  install → sign-in → block visible → block dismisses after the first cycle
  completes within 1 minute of foreground.

**Output artifact:**
- A sync-gate component/screen (e.g. `apps/mobile/src/sync/SyncGate.tsx` or an
  `app/` route) wired into the route layer below the auth guard, with stable
  testIDs for the block, the error message, and the Retry button.
- A minimal read path for "bootstrap_completed_at + last-cycle-error-code"
  (exported symbol the gate consumes).
- Jest specs under `apps/mobile/app/__tests__/`.
- Updated `docs/specs/ui/navigation-contract.md` + `screen-map.md`; a Maestro
  flow under `apps/mobile/.maestro/flows/`.

**Out of scope:**
- The bootstrapper itself that SETS `bootstrap_completed_at` (t3) — t2 reads the
  flag; t3 populates it.
- The full Settings sync-status surface (t9) — t2 reads only the minimal error
  signal it needs.
- Scheduler internals / cadence changes.

**UX Contract:**
- Flow — First-sync block:
  - Trigger: signed-in user whose `bootstrap_completed_at` is null.
  - Steps: render full-screen "Setting up your data…" → first cycle drains →
    flag set → block dismisses → normal routes render.
  - Success outcome: data restored and reachable within 1 min of foreground.
  - Failure/edge: cycle error → error message + single Retry; `AUTH_REQUIRED`
    → route to sign-in (no Retry).
- Notes: single full-screen state, single primary action (Retry); reuse
  `@/components/ui` primitives; no new interaction pattern (if one is
  introduced, update `docs/specs/ui/ux-rules.md` in the same PR).
