# t8: sign-out / account-switch local wipe

**Type:** build

**Problem:** When a user signs out or a different account signs in, the device's
local entity tables still hold the previous account's data. Without a wipe, the
new account either sees stale data or — worse — the stale `bootstrap_completed_at`
suppresses the bootstrapper so the new account's catalog never restores. The
launch contract wires the auth layer's sign-out / account-switch to a LOCAL wipe
that clears the 8 entity tables and resets the runtime-state row, while
preserving the device-global monotonic counter and the client-only muscle-group
taxonomy. This wipe is LOCAL ONLY — it must not touch server data.

**Inputs:**
- t1 merged: the auth guard + sign-in route. After the wipe, an account switch
  re-enters through the auth gate and the bootstrapper (t3) runs fresh for the
  new account. (See t1 `## Output artifact`.)
- Binding design (authoritative — cite, do not restate):
  `docs/plans/sync-v2/designs/t3.md` §6.2 #1–6 — the exact wipe checklist:
  (1) clear all rows from the 8 syncable entity tables; (2) reset
  `bootstrap_completed_at` → null; (3) reset `pull_cursor` → `{}`; (4) reset
  `applied_seed_migration_app_version` → 0; (5) PRESERVE `last_emitted_ms`;
  (6) PRESERVE `muscle_groups`. All on the `'primary'` runtime-state row.
- Auth layer hook points: `apps/mobile/src/auth/service.ts` (`signOut`, and
  `handleAuthStateChange` which fires on `auth.uid()` change) +
  `apps/mobile/src/auth/provider.tsx`.
- Runtime-state shape + singleton id: `apps/mobile/src/data/schema/sync-runtime-state.ts`,
  `PRIMARY_RUNTIME_STATE_ID = 'primary'` (`apps/mobile/src/data/clock.ts`).
- The 8 entity tables (`apps/mobile/src/data/schema/*.ts`) and the existing
  full-local-wipe pattern in `apps/mobile/src/data/dev-reset.ts` (a reference
  for clearing entity tables WITHOUT clearing `muscle_groups` or
  `last_emitted_ms` — note dev-reset clears more than t8 may).
- CONSTRAINT (carry-over): t8 MUST NOT reuse `dev_wipe_my_data` /
  `wipeRemoteForCurrentUser` — that RPC is a SERVER delete and is only env-gated
  to non-prod. t8's production wipe is LOCAL: it deletes local rows and resets
  the local runtime-state row, leaving the server's data intact so the next
  sign-in restores it.
- If t8 adds any server-touching call (it should not need one), it MUST use
  `client.schema('app_public').rpc(...)`.

**Outcomes:**
- On sign-out and on an `auth.uid()` change to a different account, the local
  wipe runs and: clears the 8 entity tables; sets `bootstrap_completed_at` →
  null; sets `pull_cursor` → `{}`; sets `applied_seed_migration_app_version` →
  0; preserves `last_emitted_ms`; preserves `muscle_groups`. (Maps to plan
  outcome: sign-out / account-switch wipe per the §6.2 checklist.)
- No server delete is issued by the sign-out / account-switch path (server data
  survives; the next sign-in restores it via the bootstrapper).
- After the wipe, a subsequent sign-in re-enters the bootstrapper (t3) cleanly
  for the (possibly new) account.
- Jest covers: the wipe clears entity tables; resets the four runtime fields per
  the checklist; preserves `last_emitted_ms` and `muscle_groups`; is triggered
  by `signOut`; is triggered on an account-switch `auth.uid()` change; issues no
  server delete RPC.
- Quality gate `./scripts/quality-fast.sh frontend` passes; if any UI surface
  changes (sign-out feedback), `npm run test:e2e:ios:gates` passes.

**Output artifact:**
- A local-wipe function (e.g. `apps/mobile/src/sync/account-wipe.ts` exporting
  `wipeLocalForAccountSwitch()` or similar) implementing the §6.2 checklist,
  invoked from the auth sign-out / account-change path in
  `apps/mobile/src/auth/service.ts`.
- Jest specs under `apps/mobile/app/__tests__/`.

**Out of scope:**
- The dev-only server + local wipe affordances (t10) — different mechanism,
  different gate.
- The soft-delete repo-path conversions (t7).
- Any change to the bootstrapper logic (t3) beyond ensuring the reset fields it
  reads are cleared here.
