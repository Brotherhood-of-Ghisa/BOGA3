# t5: bundle-migration runtime loop

**Type:** build

**Problem:** Once the bootstrapper has fired on a device it never fires again,
so later bundle changes (a new field on existing seeds, a new seed slug) cannot
reach existing users through the bootstrapper. The launch contract ships the
runtime mechanism that closes that gap: a `BUNDLE_MIGRATIONS` array plus a loop
that, after the bootstrapper, applies every migration whose app-version is
greater than the persisted `applied_seed_migration_app_version` and advances
the marker. The first ship has an EMPTY migrations array â€” this task lands the
mechanism, not any migration.

**Inputs:**
- t3 merged: the bootstrapper. The bundle-migration loop runs after the
  bootstrapper on every successful sign-in cycle, whether the bootstrapper
  seeded or no-op'd. (See t3 `## Output artifact`.)
- Binding design (authoritative â€” cite, do not restate):
  `docs/plans/sync-v2/designs/t3.md` Â§7.2 (the `BUNDLE_MIGRATIONS` shape, the
  `runBundleMigrations` pseudocode, the per-migration single-transaction rule,
  the marker-advance-in-same-transaction rule, and the idempotent-predicate
  discipline), Â§8.2 #5 / #7 (transaction + idempotency constraints).
- As-built marker (already exists): `sync_runtime_state`'s
  `appliedSeedMigrationAppVersion` column (default 0) plus the read/write marker
  helpers in `apps/mobile/src/data/exercise-catalog-seeds.ts`. t5 adds ONLY the
  `BUNDLE_MIGRATIONS` array + the runtime loop â€” the column and marker helpers
  already exist.
- `CURRENT_APP_VERSION` source (resolve from the app config / a constant the
  repo already exposes; if none exists, introduce a single source-of-truth
  constant and document it).
- Writes go through the normal repo path (so migrated rows become dirty and
  reach the server next cycle); the loop must NOT write `local_dirty` directly.

**Outcomes:**
- A `BUNDLE_MIGRATIONS` array exists in the bundle module, EMPTY for the first
  ship. (Maps to plan outcome: bundle-migration mechanism ships;
  `BUNDLE_MIGRATIONS` exists, empty for first ship.)
- A `runBundleMigrations()`-style loop runs after the bootstrapper, applies
  every migration with `appVersion > applied_seed_migration_app_version` (and
  `<= CURRENT_APP_VERSION`) in ascending order, each in its own transaction, and
  advances `applied_seed_migration_app_version` in that same transaction.
- With an empty array, the loop is a no-op except it advances the marker to
  `CURRENT_APP_VERSION` (so a later non-empty migration with a lower version is
  not skipped or re-run incorrectly â€” the loop's version arithmetic is correct).
- A partial failure leaves the marker at the last successful migration's
  version; the next launch resumes from there (per-migration transaction).
- Jest covers: empty-array no-op path advances the marker; a fixture migration
  list applies in order, advances the marker per step, is idempotent on re-run,
  and resumes after a simulated mid-loop failure; `applied >= current` short-
  circuits.
- Quality gate `./scripts/quality-fast.sh frontend` passes.

**Output artifact:**
- `BUNDLE_MIGRATIONS` (empty) + the `BundleMigration` type and the
  `runBundleMigrations()` loop, in/near
  `apps/mobile/src/data/exercise-catalog-seeds.ts` (or a sibling module), wired
  to run after the bootstrapper (t3).
- A `CURRENT_APP_VERSION` constant source if one does not already exist.
- Jest specs under `apps/mobile/app/__tests__/`.
