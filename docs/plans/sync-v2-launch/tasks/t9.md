# t9: Settings sync-status surface (+ scheduler state accessor)

**Type:** build

**Problem:** Settings has no surface telling the user whether sync is healthy.
v1's `apps/mobile/src/sync/profile-status.ts` is deleted, so this builds the
surface from scratch. The scheduler also exposes no PRODUCTION state read API —
only `__getSchedulerStateForTests` (test-only). The launch contract: add a real
scheduler state accessor and a Settings panel showing last successful sync
time, dirty count, error state, and network state.

**Inputs:**
- t3 merged: the bootstrapper / runtime-state shape is finalised (the surface
  reads `sync_runtime_state` and the entity dirty bits). (See t3
  `## Output artifact`.)
- Binding design (authoritative — cite, do not restate):
  `docs/plans/sync-v2/designs/t4.md` §5.2 (the deferred user-facing sync-status
  surface: last successful sync time, dirty count, error state, network state;
  the scheduler "already exposes enough internal signal — current state, timer
  deadline, last cycle duration, last cycle error"). Note: t4 §5.2 also lists a
  "Sync now" affordance as a candidate; it is OPTIONAL here — include it only if
  it falls out cheaply from the same accessor (`requestSync()`); it is not a
  launch-contract outcome.
- Scheduler as-built: `apps/mobile/src/sync/scheduler.ts` exposes only
  `startSyncScheduler` / `stopSyncScheduler` / `requestSync` (no `reason` param)
  / the state types / `__getSchedulerStateForTests`. t9 ADDS a production read
  API: current state, last-cycle error, and the online projection. Network
  state = the scheduler's NetInfo `isInternetReachable` projection (the
  `onlineProjection` boolean the machine already tracks).
- Status sources: derive from `sync_runtime_state` + the structured scheduler
  log events (`sync_scheduler_transition`, `sync_scheduler_cycle_error`,
  `sync_scheduler_ignored_event`, `sync_scheduler_foreground_sync_requested`,
  `sync_scheduler_start_failed`) via `apps/mobile/src/logging/logEvent.ts`.
  Dirty count = `SELECT count(*) WHERE local_dirty = 1` across the 8 entity
  tables.
- If t9 reads any server-side state via RPC, it MUST use
  `client.schema('app_public').rpc(...)` (match
  `apps/mobile/src/auth/profile.ts`). Prefer deriving status locally (runtime
  state + scheduler) to avoid a server round-trip.
- Settings host: `apps/mobile/app/(tabs)/settings.tsx` (the `UiSurface` card
  layout; the dev-tools card is `isDevMode()`-gated and separate).
- UX inputs (binding): `docs/specs/08-ux-delivery-standard.md`,
  `docs/specs/ui/ux-rules.md`, `docs/specs/ui/screen-map.md`,
  `docs/specs/ui/components-catalog.md`.

**Outcomes:**
- The scheduler exposes a production state accessor (NOT the test-only one)
  returning at least: current state, last-cycle error (code/message or null),
  and the current online projection. (Maps to plan outcome: scheduler gains a
  real state accessor.)
- The Settings panel shows, for the signed-in user: last successful sync time,
  dirty count (count of `local_dirty = 1` rows across the 8 tables), error state
  (last cycle's error if any), and network state (online/offline from the
  NetInfo projection). (Maps to plan outcome: Settings sync-status surface per
  t4 §5.2.)
- The surface updates as state changes (the accessor + dirty count refresh on a
  predictable trigger — focus, interval, or subscription — chosen and
  documented in the PR).
- No reference to the deleted `profile-status.ts` remains; the surface is
  net-new.
- Jest covers: the scheduler accessor returns current state / last error /
  online projection; the dirty-count query sums across the 8 tables; the panel
  renders each field from a stubbed status source incl. the error and offline
  states.
- UI gate: `npm run test:e2e:ios:gates` passes; a Maestro flow asserts the
  sync-status fields render on the Settings screen.

**Output artifact:**
- A production scheduler state accessor exported from
  `apps/mobile/src/sync/scheduler.ts` (e.g. `getSchedulerStatus()`).
- A sync-status module (e.g. `apps/mobile/src/sync/sync-status.ts`) that
  composes runtime state + scheduler status + dirty count into the surfaced
  shape.
- A Settings sync-status panel/component wired into
  `apps/mobile/app/(tabs)/settings.tsx`, with stable testIDs for each field.
- Jest specs under `apps/mobile/app/__tests__/`; a Maestro flow under
  `apps/mobile/.maestro/flows/`; updated `docs/specs/ui/screen-map.md` for the
  new Settings section.

**Out of scope:**
- The dev-only wipe affordances (t10) already in the `isDevMode()` card.
- Battery / data quantification (deferred per t4 §5.2).
- Background-task-status display (deferred per t4 §5.2) unless it falls out of
  the accessor cheaply.
- Changing scheduler cadence or the four-state machine behaviour (read-only
  accessor; no behavioural change).
