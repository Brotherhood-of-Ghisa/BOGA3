# t1: login-on-start enforcement — redirect to sign-in when unauthenticated

**Type:** build

**Problem:** v2 is an account-linked product with no anonymous-use journey, but
today the app renders its tab screens regardless of auth state — sign-in is
buried inside the Profile detail screen (`apps/mobile/app/profile.tsx`,
signed-out card), reachable only by navigating Settings → Profile. A user who
launches with no session lands on data screens that cannot sync. Launch
requires a route-layer guard that sends an unauthenticated user to a dedicated
sign-in entry point before any data screen renders, and routes the structured
`AUTH_REQUIRED` sync outcome to that same entry point rather than surfacing it
as a generic error. This is the root gate the rest of the launch surface sits
behind.

**Inputs:**
- No task dependencies (DAG root).
- Existing auth state machine: `apps/mobile/src/auth/service.ts` +
  `apps/mobile/src/auth/provider.tsx` (`useAuth()` exposes `status`,
  `session`, `user`, `signInWithPassword`, `signOut`, `isConfigured`,
  `disabledReason`, `lastError`).
- Existing sign-in form pattern in `apps/mobile/app/profile.tsx` (signed-out
  card: `profile-email-input`, `profile-password-input`,
  `profile-sign-in-button`) — reuse the credential-entry pattern; do not invent
  a new one.
- Root layout `apps/mobile/app/_layout.tsx` and the `(tabs)` group
  (`apps/mobile/app/(tabs)/_layout.tsx`).
- The sync cycle surfaces a structured `AUTH_REQUIRED` `SyncCycleError` (see
  `apps/mobile/src/sync/cycle.ts` `classifyRpcResult`); plan 1 granted
  `sync_push`/`sync_pull` execute to `anon`, so a pre-login cycle returns the
  `AUTH_REQUIRED` envelope rather than a raw 401 — treat it as a route-to-sign-in
  signal, not an exception.
- `apps/mobile/src/auth/supabase.ts` may report auth as unconfigured
  (`isConfigured === false`); the guard must render gracefully (its own
  disabled-state message) rather than trapping the user on a dead sign-in
  screen.
- UX inputs (binding): `docs/specs/08-ux-delivery-standard.md` (required UX
  Contract), `docs/specs/ui/navigation-contract.md`,
  `docs/specs/ui/screen-map.md`, `docs/specs/ui/ux-rules.md`,
  `docs/specs/ui/components-catalog.md` (use `@/components/ui` primitives). Add
  the new route + transition to `navigation-contract.md` and `screen-map.md` in
  the same PR per the UI docs maintenance rules.

**Outcomes:**
- When the auth snapshot reports no session (`user`/`session` null) and auth is
  configured, the route layer redirects to a sign-in screen before any
  `(tabs)` data screen renders. No anonymous path reaches a data screen. (Maps
  to plan outcome: redirect-to-sign-in; no anonymous-use path.)
- The sign-in screen accepts email + password and, on success, lets the app
  proceed to the normal route. On failure it shows the inline auth error and
  stays on sign-in.
- While the auth snapshot is `restoring` (session restore in flight), the route
  layer shows a neutral loading state — it does NOT flash the sign-in screen and
  does NOT flash a data screen.
- A sync cycle that ends in `AUTH_REQUIRED` is treated as "not signed in" and
  drives the user to the sign-in route, not a generic error toast/screen.
- A scheduler/listener wiring failure at boot (the re-throw from
  `startSyncScheduler`) propagates as a crash, NOT a recoverable auth/UI error
  state — the guard does not attempt to catch it into a sign-in redirect.
- Jest covers: guard redirects when unauthenticated; renders children when
  authenticated; shows loading during `restoring`; maps an `AUTH_REQUIRED`
  outcome to the sign-in route.
- UI gate: `npm run test:e2e:ios:gates` passes; a Maestro flow asserts a launch
  with no session lands on sign-in and renders no data screen.

**Output artifact:**
- A sign-in route/screen (e.g. `apps/mobile/app/sign-in.tsx`) with stable
  testIDs for the email/password inputs and submit button.
- A route-layer auth guard wired in `apps/mobile/app/_layout.tsx` (and/or a
  guard component the layout renders) that redirects on null session and
  exposes a way for the sync layer to drive an `AUTH_REQUIRED` redirect.
- A reusable "treat AUTH_REQUIRED as route-to-sign-in" hook/selector the sync
  gate (t2) consumes.
- Jest specs under `apps/mobile/app/__tests__/`.
- Updated `docs/specs/ui/navigation-contract.md` + `screen-map.md`; a Maestro
  flow under `apps/mobile/.maestro/flows/`.

**Out of scope:**
- The bootstrap/first-sync full-screen block and its retry CTA (t2).
- The sign-out / account-switch local wipe (t8).
- Changing the in-Profile email/password edit affordances (leave the existing
  Profile screen behavior intact).
- Any change to scheduler cadence or the `requestSync()` signature.

**UX Contract:**
- Flow — Launch unauthenticated:
  - Trigger: cold launch with no restored session.
  - Steps: root layout resolves auth (`restoring` → neutral loader) → no session
    → redirect to sign-in.
  - Success outcome: sign-in screen visible; no data screen rendered.
  - Failure/edge: auth unconfigured → sign-in screen shows the disabled-reason
    message instead of an actionable form.
- Flow — Sign in from gate:
  - Trigger: user submits email + password on the sign-in screen.
  - Steps: submit → on success the app proceeds to the normal route; on error
    the inline auth error renders and the screen stays.
  - Success outcome: data screens become reachable.
  - Failure/edge: invalid credentials → inline error, stay on sign-in.
- Notes: reuse the existing signed-out credential pattern + `@/components/ui`
  primitives; primary action obvious within 3s; no new interaction pattern
  introduced (if one is, update `docs/specs/ui/ux-rules.md` in the same PR).
