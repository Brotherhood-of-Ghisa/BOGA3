# t1: design the muscle_groups synced-entity contract

**Type:** design

**Problem:** `muscle_groups` is being promoted from a client-only taxonomy to a per-user synced
entity. Every downstream build task (server DDL, client schema, sync registry, seeding, drift,
docs, final test) depends on one shared, exact contract: the client column set, the server table
DDL, the mappings FK promotion, the wire field map, and the topological placement. Two reasonable
builders would otherwise ship divergent column sets / index names / FK shapes. This design pins ONE
decision — the canonical `muscle_groups` synced-entity contract — so the fanout is unambiguous.

**The question (one):** What is the exact synced-entity contract for `muscle_groups` — its final
client columns, its server table DDL, the `exercise_muscle_mappings.muscle_group_id` FK promotion,
its wire field map, and its topo-layer placement — modeled on `exercise_definitions`?

**Inputs:**
- The locked decisions and grounded change set in the plan PR body and this card's constraints
  (do not re-litigate them — they are CEO-locked).
- Template entity `exercise_definitions`:
  - Client: `apps/mobile/src/data/schema/exercise-definitions.ts`.
  - Server: `app_public.exercise_definitions` block in
    `supabase/migrations/20260525120000_sync_v2_clean_room.sql` (table at §3.2 ~line 153; indexes;
    `touch_server_received_at` + `owner_user_id_immutable` triggers; the four owner RLS policies;
    grants to `authenticated` + `service_role`).
- Current client `muscle_groups`: `apps/mobile/src/data/schema/muscle-groups.ts`.
- Current mapping FK: `apps/mobile/src/data/schema/exercise-muscle-mappings.ts`
  (`muscleGroupId → muscleGroups.id`, NOT NULL).
- Sync contract reference: `docs/specs/tech/sync-v2-server-contract.md` (A.1 ground rules, A.2.5
  `exercise_definitions` mapping, A.2.6 `exercise_muscle_mappings`, A.2.6.1 the to-be-removed
  client-only rationale, A.5.2 deferrable FKs, A.6 RLS, A.7.7 topo layering).
- Sync engine field/table maps: `apps/mobile/src/sync/cycle.ts` (`ENTITY_FIELDS`, `ENTITY_TABLES`,
  `TS`/`SC` field-spec helpers) and `apps/mobile/src/sync/topo-order.ts` (`TOPO_LAYERS`).

**Decision authority:** The designer recommends; the human decides in PR review if there is
disagreement. The locked decisions below are NOT up for grabs.

**Constraints (locked — bake in, do not re-decide):**
- No bespoke muscle-group logic. It flows through the generic synced-entity machinery exactly like
  `exercise_definitions`.
- DROP the `muscle_groups_non_editable_guard` CHECK. Keep only generic data guards.
- No UI / no editing path this iteration; rows are system-seeded then synced. Schema must not block
  future editability.
- Server stays a typed mirror with zero CHECK constraints (A.1 no-server-validation rule).
- Topo placement is **Layer 0** (no FK dependencies, parent of `exercise_muscle_mappings` in Layer 1).
- Client keeps the FK `muscleGroupId → muscleGroups.id`; server promotes it to the composite FK.

**The contract this design must pin (the `## Decision`):**
1. **Final CLIENT column set** for `apps/mobile/src/data/schema/muscle-groups.ts`: add sync columns
   `deletedAt` (timestamp_ms, nullable), `localDirty` (boolean, notNull, default false),
   `localUpdatedAtMs` (integer, notNull, default 0); add an `id` default
   `lower(hex(randomblob(16)))` (system seeds still pass explicit ids such as `chest`); keep
   `displayName` / `familyName` / `sortOrder` / `isEditable` / `createdAt` / `updatedAt`; for
   `isEditable` — keep as a plain synced scalar, DROP the non-editable guard, KEEP the boolean guard
   and the `sortOrder >= 0` guard. Confirm and record which indexes remain.
2. **Final SERVER table DDL** mirroring `exercise_definitions`: composite PK `(owner_user_id, id)`;
   entity columns `display_name`, `family_name`, `sort_order`, `is_editable`, `created_at`,
   `updated_at`, `deleted_at`; the universal `client_updated_at_ms` + `server_received_at`;
   `muscle_groups_owner_received_idx` plus the `family_name` / `sort_order` / `display_name` /
   `deleted_at` indexes; the two structural triggers; the four owner RLS policies; grants to
   `authenticated` and `service_role`. Pin the exact index names and column types.
3. **Promote `exercise_muscle_mappings.muscle_group_id` to a composite FK** server-side:
   `(owner_user_id, muscle_group_id) → app_public.muscle_groups(owner_user_id, id)`,
   `deferrable initially deferred`. Pin the constraint name and on-delete action. The client FK
   `muscleGroupId → muscleGroups.id` stays.
4. **The wire field map for `muscle_groups`** (typed columns crossing the wire):
   `display_name, family_name, sort_order, is_editable, created_at, updated_at, deleted_at`.
   Pin which are `TS` (timestamp) vs `SC` (scalar) field specs.
5. **Topo placement:** Layer 0.

- The `## Decision` fully pins items 1–5 above so downstream builders implement without ambiguity.
- Downstream cards t2, t3, t4, t5, t6, t8, tFINAL are updated with a pointer marker

**Output artifact:** `docs/plans/muscle-groups-synced-entity/designs/t1.md` plus pointer-only edits

**Out of scope:** Any implementation (DDL files, schema files, registry edits, seeds) — those are

either side. Migrating existing installed devices.
