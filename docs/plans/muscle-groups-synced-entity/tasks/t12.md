# t12: push FK-preflight edge for muscle_groups + client-only straggler sweep

**Type:** build

> Updated from t1: see designs/t1.md ## Decision (muscle_groups is now a pushed synced parent of
> exercise_muscle_mappings).

**Problem:** The client-side push FK-closure preflight (`apps/mobile/src/sync/fk-graph.ts`) models,
for each syncable child, the FK edges whose parent is itself syncable — so a dirty child whose
required parent is missing locally (and not in-batch / not server-clean) is caught BEFORE the push
RPC instead of being rejected by the server. That graph was built when `muscle_groups` was a
client-only catalog, so it **deliberately omits** the
`exercise_muscle_mappings.muscle_group_id → muscle_groups` edge (FK_EDGES, ~lines 63-65) and its
`FK_PARENT_TABLES` entry, and carries a now-false comment (~lines 21-24) stating `muscle_groups`
"is never pushed". Now that `muscle_groups` is a pushed Layer-0 synced parent, an orphan dirty
`exercise_muscle_mappings` row (whose `muscle_group_id` parent isn't present/clean locally) escapes
the preflight and gets server-rejected — the exact wedge the preflight exists to prevent. Separately,
because `muscle_groups` being missed here shows the entity-enumeration consumers weren't exhaustively
swept, this task also sweeps `apps/mobile/src` for any OTHER consumer still treating `muscle_groups`
as client-only or hardcoding the old 8-entity set, and fixes any found.

**Inputs:** t4 merged (`muscle_groups` in `EntityTableName` / registry — `parentType` must be a valid
entity), t5 merged (`muscle_groups` pushes dirty), t7 merged (in-memory harness is FK-on by default,
so the preflight test runs under production-fidelity FK enforcement). Binding contract:
`designs/t1.md ## Decision` (item 3 — the composite FK; the client FK `muscleGroupId → muscleGroups.id`).
Key files:
- `apps/mobile/src/sync/fk-graph.ts` — add the `muscle_group_id → muscle_groups` edge to
  `FK_EDGES.exercise_muscle_mappings` (`required: true` — `muscle_group_id` is NOT NULL, mirroring the
  sibling `exercise_definition_id` edge); add `muscle_groups` to `FK_PARENT_TABLES`; rewrite the
  now-false "muscle_groups is never pushed / locally-bundled, server-seeded ... omitted" comment to
  the synced-parent reality (self-contained).
- A preflight test (extend the existing fk-graph / push-preflight suite) asserting that a dirty
  `exercise_muscle_mappings` whose `muscle_group_id` parent is absent/unclean is flagged by the
  preflight locally (not sent to the RPC), and that a valid parent+child batch still passes.

**Outcomes:**
- `FK_EDGES.exercise_muscle_mappings` includes the `muscle_group_id → muscle_groups` edge
  (`required: true`); `FK_PARENT_TABLES` includes `muscle_groups`; the false "never pushed" comment is
  gone (replaced by the synced-parent reality, self-contained).

  ⛔ N/A unless an added test needs it — this is client push-preflight logic.)

**Output artifact:** `apps/mobile/src/sync/fk-graph.ts` (edge + parent table + comment) and the
