# t8: docs — data-model + sync-v2 server contract

**Type:** build

> Updated from t1: see designs/t1.md ## Decision

**Problem:** Two always-/on-demand-loaded specs still describe `muscle_groups` as a client-only
taxonomy that never syncs, `muscle_group_id` as opaque text with no FK, and the system as having
eight entity tables. These are now false. The specs are the project source of truth that every
future agent loads, so they must describe the synced-entity reality, the boot FK enforcement, and
the invariant that client FKs only reference synced parents.

**Inputs:** t5, t6, t7 merged (the behavior the docs describe is now real). Binding contract in
`designs/t1.md ## Decision` (items 1-5). Key files:
- `docs/specs/05-data-model.md` — move `muscle_groups` from "Static/system data (not user backup
  scope)" to the user-owned synced entities; update the "Client schema drift rule" section that
  currently exempts `muscle_groups` (~lines 186-190); state FK enforcement is enabled at boot via
  `PRAGMA foreign_keys = ON`; record the invariant "client FKs only reference synced parents"; update
  any "eight entities" counts to nine.
- `docs/specs/tech/sync-v2-server-contract.md` — update the "eight client entity tables" framing to
  nine (A.1, A.2 header, A.7 derive-don't-hardcode note); add an `A.2.x muscle_groups` per-entity
  mapping (mirror A.2.5 `exercise_definitions`); change A.2.6's `muscle_group_id` row from "No FK —
  client-only" to the composite FK; replace A.2.6.1's "intentionally not synced" rationale with the
  synced-entity reality; add the `muscle_group_id → muscle_groups` FK to the A.5.2 FK table; add
  `muscle_groups` to the A.7.7 / B.3.4.1 Layer 0 listings; remove the `untyped_text_references`
  example for `muscleGroupId` from the A.3 `sync-extras.json` shape.

**Outcomes (maps to plan outcome 9):**
- `docs/specs/05-data-model.md` lists `muscle_groups` as a per-user synced entity, states boot FK
  enforcement, records the "client FKs only reference synced parents" invariant, and no longer
  exempts `muscle_groups` from the drift rule; counts read nine.
- `docs/specs/tech/sync-v2-server-contract.md` describes nine entity tables, a real
  `muscle_group_id` FK in A.2.6 and A.5.2, a `muscle_groups` per-entity mapping section, Layer 0
  membership, and no surviving opaque-text / client-only / "intentionally not synced" rationale.
- The docs match the as-built code (the migration, RPCs, schema, registry, drift checker) merged in
  t2-t7. No durable doc text references plan/card/design ids or `docs/plans/...` paths.
- `./scripts/quality-fast.sh` and `npm run test:handles` are green (some repo tests assert spec
  invariants, e.g. drift-rule / manual-wipe-doc-exists). Evidence in the PR.

**Output artifact:** `docs/specs/05-data-model.md` and `docs/specs/tech/sync-v2-server-contract.md`,
both describing `muscle_groups` as the 9th per-user synced entity with a real
`muscle_group_id → muscle_groups` FK and boot FK enforcement.
