# t9: dirty-count — include muscle_groups in the Settings pending-push count

**Type:** build

> Updated from t1: see designs/t1.md ## Decision (muscle_groups is a synced entity carrying `localDirty`).

**Problem:** Making `muscle_groups` a synced entity left one consumer behind. The Settings
sync-status panel computes the user-facing pending-push ("dirty") count by summing `local_dirty`
rows across a hardcoded `DIRTY_COUNTED_TABLES` list in `apps/mobile/src/sync/sync-status.ts` — and
that list still enumerates only the original eight entity tables, excluding `muscleGroups`. Once the
catalog seed marks `muscle_groups` rows `local_dirty = 1` (t5), those pending rows will NOT be
reflected in the count until the first successful sync clears them — a cosmetic but real undercount
that contradicts the plan Goal of treating `muscle_groups` identically to every other synced table.
A test currently encodes the old behavior: `apps/mobile/app/__tests__/sync-status-composer.test.ts`
asserts (in a comment and by construction) that `muscle_groups` "is NOT one of the eight
dirty-counted tables, so it never affects the count." The production list and that test are coupled —
fixing one without the other breaks the suite — so they move together here.

**Inputs:** t4 merged (muscle_groups registered as the 9th synced entity). `schema.muscleGroups`
(with its `localDirty` column) landed in t3. Binding contract: `designs/t1.md ## Decision`. Key files:
- `apps/mobile/src/sync/sync-status.ts` — `DIRTY_COUNTED_TABLES` (~line 28) is the hardcoded
  list summed in the dirty-count loop (~line 70). Add `schema.muscleGroups` to it, in the same
  Layer-0-first ordering used for the other entities.
- `apps/mobile/app/__tests__/sync-status-composer.test.ts` — the obsolete "muscle_groups is NOT one
  of the eight dirty-counted tables" comment (~line 88) and the surrounding construction that relies
  on `muscle_groups` never affecting the count. Reconcile to the synced-entity reality and add
  coverage that a dirty `muscle_groups` row increments the count.
- Grep the test tree for any other assertion that `muscle_groups` is excluded from / does not affect
  the dirty count (e.g. `dirty-bit-*` suites) and reconcile each hit.

**Outcomes:**
- `DIRTY_COUNTED_TABLES` in `apps/mobile/src/sync/sync-status.ts` includes `schema.muscleGroups`, so
  a `local_dirty = 1` muscle-group row contributes to the Settings pending-push count exactly like
  every other synced entity. (The list now enumerates all nine entity tables.)
- `apps/mobile/app/__tests__/sync-status-composer.test.ts` no longer asserts the old exclusion; its
  comment + construction reflect that `muscle_groups` is a dirty-counted synced table, and it asserts
  a dirty `muscle_groups` row is counted. Any other dirty-count test that assumed exclusion is
  reconciled.
- `./scripts/quality-fast.sh` and `npm run test:handles` are green. Evidence in the PR. (Slow
  backend / frontend / sim lanes ⛔ N/A — no server SQL, no UI, no migration.)

**Output artifact:** `apps/mobile/src/sync/sync-status.ts` (`muscleGroups` added to
`DIRTY_COUNTED_TABLES`) and `apps/mobile/app/__tests__/sync-status-composer.test.ts` (reconciled +
coverage), plus any other reconciled dirty-count test surfaced by the grep.

**Out of scope:** The seeding that marks the rows dirty (t5). Sync registry (t4). Drift checker (t6).
The pre-existing flaky `sync-status-panel.test.tsx` race (tracked separately — do NOT touch it).
Docs (t8). Any new dirty-count UI/feature beyond including the entity in the existing count.
