# t7: tests — FK-on harness default, coverage, comment fixes

**Type:** build

**Problem:** Several test guarantees and comments encode the old "client-only muscle_groups" world
and a factually-wrong belief that expo-sqlite (and better-sqlite3) enforce FKs by default. The
in-memory test harness defaults FK enforcement OFF, so insertion-order bugs that production (now
FK-on at boot) would catch stay hidden. The `local-data-bootstrap-fk-integrity` guarantee
("boot seeds muscle_groups first") no longer holds — the new guarantee is "topo order pulls Layer 0
before Layer 1". And `muscle_groups` needs coverage in the per-entity sync suites.

**Inputs:** t4 and t5 merged (registry + seeding reflect the synced entity). Binding contract in
`designs/t1.md ## Decision`. Key files:
- `apps/mobile/app/__tests__/helpers/in-memory-db.ts` — flip the harness default to FK-on (production
  fidelity; the `foreignKeys` option / `pragma('foreign_keys = ON')` is at ~line 70-85). Surfacing
  any insertion-order issue is the point.
- `apps/mobile/app/__tests__/local-data-bootstrap-fk-integrity.test.ts` — rewrite the guarantee from
  "boot seeds muscle_groups first" to "topo order pulls Layer 0 (`muscle_groups`,
  `exercise_definitions`) before Layer 1 (`exercise_muscle_mappings`)" under FK enforcement.
- Per-entity sync suites — add `muscle_groups` coverage to round-trip, per-entity dirty-bit, wire,
  and soft-delete: `apps/mobile/app/__tests__/sync/cycle-round-trip.test.ts`,
  `apps/mobile/app/__tests__/sync/dirty-bit-per-entity.test.ts`, and the wire/soft-delete suites that
  enumerate entities. Mirror the `exercise_definitions` cases.
- Factually-false FK comments to correct (expo-sqlite / better-sqlite3 default `foreign_keys` OFF;
  production never set the pragma before this change). In the current tree the false
  "FKs enforced by default / ships enforcement" claim lives in
  `apps/mobile/app/__tests__/sync/cycle-round-trip.test.ts` (~lines 247-248,
  "enforces foreign keys by default") and `apps/mobile/app/__tests__/helpers/in-memory-db.ts`
  (~line 73, "off by default, matching SQLite's per-connection default") — rewrite both to
  "enforcement we enable at boot via `PRAGMA foreign_keys = ON`" (self-contained, no plan/card
  references). Grep the test tree for any further `foreign_keys`/"by default" FK claims and the
  build-card-referenced files; correct any others found. (Do NOT touch
  `apps/mobile/app/__tests__/sync-cycle-pull.test.ts` — its FK references are a legitimate
  "rolls back on FK violation" test, not a false comment.)
- Obsolete "client-only / non-syncable / never-syncs" muscle-group assertions that now break under
  the synced-entity reality:
  `apps/mobile/app/__tests__/muscle-group-bootstrap-idempotent.test.ts` (its whole premise — a
  client-only taxonomy seeded every launch via `seedMuscleGroups`, exercising the non-editable guard
  — is gone; rewrite to the synced-seed-via-`seedSystemExerciseCatalog` reality or remove if its
  guarantee is fully subsumed by the per-entity coverage above) and
  `apps/mobile/app/__tests__/seed-catalog-slug-shape.test.ts` (the "client-only taxonomy — no lineage
  prefix" comment and any verification that assumes `muscle_groups` is non-dirty/non-syncable).
  Grep the tree for `client-only`, `non-syncable`, `never syncs`, `seedMuscleGroups`, and the
  non-editable guard and reconcile each hit with the synced-entity reality.
