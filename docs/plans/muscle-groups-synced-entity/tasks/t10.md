     `docs/specs/tech/sync-v2-server-contract.md §…` instead, or just describe the invariant).

2. **`sync_pull` contract suite — scenario 1 `has_more` failure (verify causality, then fix if real).**
   The full `./scripts/quality-slow.sh backend` wrapper was reported aborting in
   `supabase/tests/sync-pull-contract.sh` "scenario 1: snapshot pull" with
   `has_more: expected false, got true` (~line 271). Scenario 1 seeds exactly 3 Layer-0 rows and
   pulls `{layer:0, limit:10}` expecting `has_more=false`; `got true` means the pull RPC saw **>10
   Layer-0 rows for the test user**. This was observed in a worktree whose Supabase slot had an
   orphaned/leftover stack, so it MAY be a non-hermetic-DB / environment artifact rather than a code
   regression. **First reproduce on a CLEAN Supabase reset** at current `main`. Then:
   - If scenario 1 PASSES on a clean reset → it was environment pollution, not a code defect. Record
     that finding (and, if the suite is non-hermetic by design — e.g. scenario 1's `has_more` depends
     on no other Layer-0 rows existing — harden it to be hermetic so it can't false-fail under a
     dirty DB).
   - If scenario 1 FAILS on a clean reset → it is a real regression. Root-cause it and FIX: if the
     muscle_groups addition legitimately changed a Layer-0 count/expectation, correct the test; if the
     pull RPC's `has_more`/cursor logic is wrong for the expanded Layer 0, fix the RPC
     (`supabase/migrations/20260525130100_sync_v2_pull_rpc.sql`). Surface the exact root cause in the
     PR's Tests / Deviations section.
