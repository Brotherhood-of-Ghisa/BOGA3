# Design: the `muscle_groups` synced-entity contract

## Question

`muscle_groups` is being promoted from a client-only, version-bundled taxonomy to a per-user
**synced** entity, modeled on `exercise_definitions`. Seven downstream build tasks each need one
shared, exact contract or two reasonable builders ship divergent column sets / index names / FK
shapes. This design pins that single contract: the final client column set, the server table DDL,
the `exercise_muscle_mappings.muscle_group_id` FK promotion, the wire field map, and the topo-layer
placement.

## Constraints

(Locked by the card â€” baked in, not re-decided.)

- No bespoke logic: flows through the generic synced-entity machinery exactly like
  `exercise_definitions`.
- DROP the `muscle_groups_non_editable_guard` CHECK. Keep only generic data guards (boolean guard on
  `is_editable`, `sort_order >= 0` guard).
- No UI / no editing path this iteration; rows are system-seeded then synced. Schema must not block
  future editability.
- Server is a typed mirror with **zero CHECK constraints** (A.1 no-server-validation rule).
- Topo placement is **Layer 0** (no FK deps; parent of `exercise_muscle_mappings` in Layer 1).
- Client keeps the FK `muscleGroupId â†’ muscleGroups.id`; server promotes it to the composite FK.

## Options considered

The only genuinely open sub-decisions are (a) how `muscle_groups`-specific entity columns
(`display_name`, `family_name`, `sort_order`, `is_editable`) map onto the wire and the server type
system, and (b) the **on-delete action** of the new composite FK. Everything else is mechanically
fixed by the `exercise_definitions` template. Two end-to-end shapes:

### Option A â€” full structural clone of `exercise_definitions` (template-faithful)

Mirror the template exactly: client gets the three sync columns + `id` default; server gets composite
PK, `bigint` epoch-ms timestamps, the universal envelope, the standard index/trigger/RLS/grant set,
and the FK promoted with `on delete cascade` (the same action its sibling
`exercise_muscle_mappings_exercise_definition_fk` already uses on the same table). The entity columns
`display_name`/`family_name` are `text`, `sort_order` is a typed `integer`/`bigint` scalar,
`is_editable` is a typed scalar; the two CHECK guards stay client-only, the server stays CHECK-free.

- **Pros:** zero new patterns â€” drift checker, push/pull RPC branches, topo assertion, and FK tests
  all already understand this exact shape; smallest blast radius; trivially reversible (it is the
  same diff every other entity already carries).
- **Cons:** none material â€” `sort_order`/`is_editable` are plain scalars with no special handling.

### Option B â€” treat the taxonomy columns as server metadata, not wire fields

Sync only `display_name`/`family_name`, and keep `sort_order`/`is_editable` as client-local
presentation state that never crosses the wire (analogous to the local-only bookkeeping columns).

- **Pros:** slightly narrower wire payload.
- **Cons:** breaks the "generic machinery, no bespoke logic" constraint â€” it invents a per-entity
  wire-exclusion rule the engine does not have; `sort_order`/`is_editable` would then drift between
  devices (the exact cross-version skew this plan exists to kill); contradicts plan outcome 5, which
  fixes the wire set at all seven columns. Rejected.

### On-delete sub-options (within Option A)

- `set null` â€” **structurally impossible**: `muscle_group_id` is `NOT NULL`.
- `no action` â€” deleting a referenced muscle group would error mid-drain; worse ergonomics, no
  upside for a system-seeded parent.
- `cascade` â€” matches the sibling FK on the same table and the soft-delete model (a tombstoned
  parent cascades to its mappings cleanly). **Chosen.**

## Decision

**Option A â€” full structural clone of `exercise_definitions`, FK promoted with `on delete cascade`.**
It is the only option satisfying "generic machinery, no bespoke logic," carries the smallest blast
radius, and reuses every existing test/assertion. Option B loses by inventing a non-existent
wire-exclusion rule and reintroducing the very skew the plan eliminates.

The binding contract, by card item:

**1. Final CLIENT column set** (`apps/mobile/src/data/schema/muscle-groups.ts`):
- Keep: `id` (text, PK, notNull) **plus a new default** `sql\`(lower(hex(randomblob(16))))\``
  (system seeds still pass explicit ids such as `chest`); `displayName`, `familyName`,
  `sortOrder` (integer, notNull, default 0), `isEditable` (integer, notNull, default 0),
  `createdAt`, `updatedAt` (both `timestamp_ms`, notNull, default `unixepoch() * 1000`).
- Add the three sync columns, exactly as on `exercise_definitions`: `deletedAt`
  (`timestamp_ms`, nullable), `localDirty` (boolean, notNull, default false), `localUpdatedAtMs`
  (integer, notNull, default 0).
- Guards: DROP `muscle_groups_non_editable_guard`. KEEP `muscle_groups_sort_order_non_negative`
  (`sort_order >= 0`) and `muscle_groups_is_editable_boolean_guard` (`is_editable in (0,1)`).
- Indexes (all retained, none added): `muscle_groups_family_name_idx`,
  `muscle_groups_sort_order_idx`, `muscle_groups_display_name_idx`.

**2. Final SERVER table DDL** (`app_public.muscle_groups`, mirroring `exercise_definitions`):
- Add `drop table if exists app_public.muscle_groups cascade;` in the reset block, ordered as a
  Layer 0 parent (dropped after its Layer 1 child `exercise_muscle_mappings`).
- Columns: `owner_user_id uuid not null default auth.uid() references auth.users(id) on delete
  cascade`; `id text not null`; `display_name text not null`; `family_name text not null`;
  `sort_order integer not null`; `is_editable integer not null`; `created_at bigint not null`;
  `updated_at bigint not null`; `deleted_at bigint`; `client_updated_at_ms bigint not null`;
  `server_received_at timestamptz not null default now()`.
- PK: `constraint muscle_groups_pkey primary key (owner_user_id, id)`. **Zero CHECK constraints.**
- Indexes (exact names): `muscle_groups_owner_received_idx` on `(owner_user_id, server_received_at)`;
  `muscle_groups_family_name_idx` on `(family_name)`; `muscle_groups_sort_order_idx` on
  `(sort_order)`; `muscle_groups_display_name_idx` on `(display_name)`; `muscle_groups_deleted_at_idx`
  on `(deleted_at)`.
- Triggers: `muscle_groups_touch_server_received_at` (before update, for each row, when `new is
  distinct from old`, `execute function app_public.touch_server_received_at()`) and
  `muscle_groups_owner_user_id_immutable` (before update, for each row, `execute function
  app_public.enforce_owner_user_id_immutable()`).
- RLS: `enable row level security` + four owner policies `muscle_groups_owner_{select,insert,update,
  delete}` gated on `owner_user_id = auth.uid()` (same shape as `exercise_definitions`).
- Grants: `grant select, insert, update, delete on table app_public.muscle_groups` to both
  `authenticated` and `service_role`.

**3. `exercise_muscle_mappings.muscle_group_id` composite FK promotion** (server-side):
- Constraint name: `exercise_muscle_mappings_muscle_group_fk` (matches the sibling
  `..._exercise_definition_fk` naming).
- Shape: `foreign key (owner_user_id, muscle_group_id) â†’ app_public.muscle_groups(owner_user_id, id)`,
  `on delete cascade`, `deferrable initially deferred`.
- Drop the in-file comment claiming `muscle_groups` is client-only / opaque text with no FK.
- Client FK `muscleGroupId â†’ muscleGroups.id` is unchanged.

**4. Wire field map** for `muscle_groups` (`ENTITY_FIELDS`, in this order):
`SC('display_name','displayName')`, `SC('family_name','familyName')`,
`SC('sort_order','sortOrder')`, `SC('is_editable','isEditable')`, `TS('created_at','createdAt')`,
`TS('updated_at','updatedAt')`, `TS('deleted_at','deletedAt')`. The three epoch-ms columns are `TS`;
the four taxonomy columns are `SC`. The two local-only bookkeeping columns never cross the wire.

**5. Topo placement:** **Layer 0**, alongside `gyms` and `exercise_definitions` (no FK
dependencies). It becomes the 9th member of `EntityTableName`; `exercise_muscle_mappings`
(Layer 1) now has a legitimate Layer 1 â†’ Layer 0 edge.

## Open questions / follow-ups

- The card defers `is_editable`'s eventual editability path; this schema does not block it
  (`is_editable` remains a plain synced scalar). No follow-up needed this iteration.
- Existing installed devices are out of scope (fresh-installs-only, per plan goal). Any future
  migration of in-place taxonomies is a separate task.
